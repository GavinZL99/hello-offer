# Agent 开发知识总结

> 主线：LLM 不够 → 为什么要 Agent → Agent 怎么转（推理范式）→ 核心三件套（工具/记忆/上下文）→ 工程化（Prompt、评测、踩坑）→ 多 Agent → 选型。

## 一、为什么需要 Agent：LLM 本身不够用

**LLM 的局限**：一个纯 LLM 调用，是一个"无状态、无动作"的函数——你问、它答，仅此而已。它有三大短板，正好对应 Agent 要补的三件事：
1. **它不知道现在的事**：训练有截止日期，问"今天股价""最新文档"答不了或瞎编 → 需要**工具**（搜索、查库）。
2. **它干不了事**：只能输出文字，不能真的下单、改文件、调接口 → 需要工具去**执行动作**。
3. **它记不住、做不完长任务**：单次调用上下文有限、一问一答就结束，没法自己规划"先查 A 再查 B 然后综合"的长链路 → 需要**循环 + 记忆 + 上下文管理**。

**Agent 的定义**：**LLM + 循环 + 工具**——让模型在一个循环里反复"思考→调工具→看结果→再思考"，直到完成任务。核心是**模型自己决定下一步干什么**（自主性），而不是人写死的流程。

**类比**：纯 LLM 是个**只能口头给建议的顾问**（你说症状它说"可能是感冒"，但它不能给你开药、不能去查病历）；Agent 是个**能动手的全科医生**——它能自己去查你的化验单（工具）、记下上次的诊断（记忆）、按"检查→诊断→开药→复诊"的步骤推进（循环），自己决定每一步。

**关键判断：什么时候才该上 Agent**（别滥用）：四条都满足才值得——
1. **复杂度**：任务多步、没法在一条 prompt 里完全说清（"把这需求文档变成 PR" vs "从 PDF 提个标题"——后者一次调用就够）。
2. **价值**：结果值得承担更高的成本和延迟。
3. **可行性**：模型在这个任务类型上足够强。
4. **错误可承受**：错了能被发现和恢复（测试、评审、回滚）。
四条有一条不满足，就停留在"单次调用"或"固定工作流"，别上 Agent——Agent 是复杂度和成本的放大器。

## 二、Agent 怎么转：推理范式（ReAct 等）

Agent 的"怎么转"本质是**推理范式**——模型在循环里按什么套路思考。

**ReAct（Reasoning + Acting）——最经典**：每一步先**推理**（Thought：我现在该干嘛、为什么）再**行动**（Action：调哪个工具），看到结果（Observation）再进入下一轮 Thought。Thought→Action→Observation 反复，直到给出最终答案。
- 价值在于把"想"和"做"交织：不是闷头调一堆工具，而是每调一个就重新判断方向，能纠错。
- 现代 Claude 等模型的 **extended thinking（扩展思考）** 本质就是 ReAct 里"Thought"的强化版——模型先内部推理再输出动作。

**Plan-and-Execute（先规划后执行）**：先一口气把任务拆成计划（Plan），再逐条执行（Execute）。适合步骤可预知的长任务；缺点是计划容易脱离实际——执行中发现计划错了要回头改。

**Reflection（自我反思）**：让模型对自己的输出再审一遍、挑毛病、改一版。典型用于"生成→自评→修订"的迭代（如代码生成了再自审）。

**实践现状**：别教条地选范式。**最强做法是把任务一次性说清、开扩展思考、给足工具，让模型自己决定怎么转**（现代强模型如 Opus 4.8 在长程任务上自主性很强）。ReAct/Plan/Reflection 更像是你在 system prompt 里"建议"它怎么推进，而不是你手写的状态机。

## 三、核心概念速览（每节一句话 + 深挖链接）

Agent 开发有五大核心概念,下面每节只给一句话概要,深挖在 `Agent专题/` 各篇里。

### 工具（Tool Use）
Agent 能干活靠工具——一个**函数 + 给模型的 schema**;模型决定调哪个/传什么参数,你的代码执行。工具设计的关键:描述写"何时调用"、bash vs 专用工具的取舍、参数校验三层、权限管控(分级审批/沙箱/vault)。→ **深挖见 `Agent专题/工具调用.md`**

### 记忆（Memory）
Agent 跨会话延续靠记忆——分三层(短期=对话上下文、中期=摘要、长期=持久化文件/向量库);核心机制是怎么总结(把对话变记忆)、怎么压缩(把长上下文压短)、怎么加载(把记忆取回来用)、怎么淘汰(别只存不删)。→ **深挖见 `Agent专题/Agent记忆.md`**

### 上下文管理
Agent 循环里 context 不断累积,会爆窗口也会漂移(早期指令逐渐不被遵守)。管理由轻到重:前置重申、context editing 裁剪、compaction 压缩、子 Agent 分流、状态外移到记忆。→ **深挖见 `Agent专题/上下文管理与长对话稳定.md`**

### 执行与容错
ReAct 循环按 stop_reason 驱动(组请求→看返回→tool_use 就执行→回填→下一轮)。容错三层:工具失败 is_error 回填让模型纠错、模型 API 失败自动重试、任务级超时降级。死循环靠 7 层防护(从 max_iterations 到重复检测到 token 上限)。→ **深挖见 `Agent专题/ReAct执行链路.md`**

### Prompt 工程
Agent 的 prompt 在循环里被反复读,差一句跑偏很多步。要点:给角色+目标+边界、规定何时用工具、给自主性边界;新模型别过度 prescribing、给完整规格让它自主推进。结构化输出用 `output_config.format`/`strict`,别用 prefill(已 400)。

### 评测（Eval）
Agent 输出非确定,要建 golden set + 程序断言/LLM-as-judge + 指标(成功率/步数/token/延迟)+ 回归。改 prompt/换模型/加工具都要跑 eval,别凭感觉。

### 安全
Agent 有工具能执行真实动作,prompt 注入是最核心威胁(模型分不清指令和数据)。防御靠纵深:指令数据分离(软)+ **最小权限+审批(硬,最可靠——被骗也干不了坏事)** + 凭据 vault + 输出校验 + 全量审计。→ **深挖见 `Agent专题/Agent安全.md`**

### 运行优化
Agent 是 token 放大器,线上贵慢乱调。降本靠 caching+子 Agent 便宜模型+effort+task budget;提速靠流式+并行工具+异步子 Agent+PTC;减无效调用靠描述写何时调用+tool search+先思考+去重。→ **深挖见 `Agent专题/Agent运行优化.md`**

## 五、多 Agent 协作（概要）

单个 Agent 上下文/能力有限,复杂任务拆成多个 Agent 协作。常见模式:协调者-工人(最常用)、并行扇出、流水线、多视角。核心价值是**子 Agent 独立上下文**——跑完只交结论,主 Agent 不被中间过程撑爆。协作要解决:结果冲突(仲裁/投票)、资源竞争(锁/隔离)、状态不一致(消息协议+共享文件为准)、死循环(超时+一层委派)。→ **深挖见 `Agent专题/多智能体协作.md`**

## 六、框架与产品全景：从范式到选型

这一章把前面散落的概念串成一条逻辑链：**Agent 靠什么范式转（ReAct）→ 用什么框架表达（链式 vs 循环，LangChain/LangGraph）→ 图能否做通用 Agent → 个人 Agent 和企业级 Agent 的本质分歧（单机 vs 分布式等）→ 各自的代表产品/框架**。理解了这条链，选型就是顺理成章的。

### 1. 先分清：范式、框架、产品三个层次

面试别把它们混着比：
- **范式**：Agent 怎么转的方法论，如 ReAct（Thought→Action→Observation 循环）。**不是框架**，是思路。
- **框架**：你写代码用的库，把范式+工具+循环+状态封装好。如 LangChain/LangGraph、AgentScope、Managed Agents。
- **产品**：造好的、装上就能用的成品 Agent。如 Claude Code、OpenClaw、Hermes Agent。

同一个 ReAct 范式，可以由不同框架实现、跑在不同产品里。下面按"先范式后框架，先框架后产品，先个人后企业"展开。

### 2. 范式层：ReAct 与"链式"的根本区别

ReAct 是 Agent 最经典的转法：每步先**推理**（Thought：现在该干嘛）再**行动**（Action：调工具），看结果（Observation）再进下一轮，直到完成。它和"链式"的根本区别在**控制流谁定**：

- **链式（LangChain 的起点）**：你**预先把流程焊死**（取文档→塞 prompt→调模型→解析…），像设计流水线，每个工位提前接好。控制流在你手里，适合步骤固定、能事先想清的流程（典型 RAG）。
- **ReAct（循环/自主）**：你不预焊流程，只给模型**目标 + 工具**，模型运行时**自己决定每步干啥**，流程是**运行时涌现**的。控制流在模型手里，适合步骤不可预知、要按结果纠偏的开放任务。

类比：链式 = 你画好的**流水线**，原料按固定工位走；ReAct = 你给个**带工具箱的工人**一个目标，他自己决定先拧哪个螺丝、看完结果再决定下一步。一个"我设计流程、模型填空"，一个"我只给目标和工具、模型自己规划"。

### 3. 框架层：LangChain → LangGraph 的演进

**LangChain** 是通用 LLM 框架，重心是"链/可组合"（Runnable 串联），快速搭、生态广、能接各种数据源/模型；但抽象重、老 Agent 抽象（AgentExecutor）把 ReAct 硬塞进链里、分支循环状态管不好、调试难。

**LangGraph** 是同公司（LangChain Inc.）后来出的、**图式 Agent 编排**：用显式**状态图**（节点=一个函数/一次模型调用，边=节点间连线，State=贯穿流程的共享状态）正儿八经表达"动态循环 + 条件分支 + 中断恢复"。本质是把 ReAct 式自主循环从"隐式 while 黑盒"变成"可控、可观测的图"——**LangGraph 是 LangChain 团队对 ReAct 模式的正式回答**，老 AgentExecutor 已被它取代。两者可搭配（LangGraph 里能用 LangChain 的工具/Runnable）但 LangGraph 可独立用。

### 4. 关键澄清：LangGraph 能做"通用/发散" Agent 吗

常见误解：一听"状态图"就以为只能写 A→B→C→END 的固定脚本。**错。关键在条件边可以交给 LLM 在运行时决定走哪**——所以"图"≠"固定路径"。

LangGraph 的图定义的是**"有哪些节点 + 到每个岔路口怎么选下一步"**，而"怎么选"这个函数里可以写"读当前 State，让模型决定走哪"。一个完全通用的 ReAct Agent 用它写，其实就是 **2 节点 + 1 条件边**：
```
节点 call_model：调模型（带工具）
      │
 条件边（运行时判）：模型这次返回 tool_use 吗？
      ├── 是 → 节点 exec_tools → 回 call_model
      └── 否（end_turn）→ END
```
这是个 self-loop：模型每步自己决定调不调工具、调哪个、何时停——**完全发散的通用 Agent**，你没焊任何业务路径。**通用 Agent 本身就是个带自循环的小图**（call→tools→call + 模型决定的出口），所以"通用 Agent"和"图"不对立。

- 固定的是：你**枚举出有哪些节点/能力**、以及**岔路口的判断逻辑**——这些有限、你设计。
- 发散的是：**走到哪个节点、走几次、何时停，是模型运行时按 State 决定的**。

**诚实的取舍**：LangGraph 要你显式命名节点和路由，对**真正无结构探索**（你连"大概有哪几类动作"都不愿预先想）的场景，这点显式性是额外开销——那种场景直接一个裸 `while`（调模型→要工具就执行→再调）更省事。但这是**简单 vs 显式的取舍，不是能力上限**：LangGraph 能做通用 Agent，只对极简发散场景偏重。而且生产里几乎没有真正"无约束 Agent"——工具集本就有限，"自由循环"也只是"call→tools→call"这一个固定形状在转，LangGraph 把它显式化、还白送检查点/中断/可观测/审批插入，对复杂通用 Agent 反而加分。

一句话：LangGraph 既能做固定流程（固定边）也能做发散通用 Agent（条件边交模型）——不是"只能固定场景"，而是"让你把控制流显式化、可观测、可控制"。

### 5. 核心分歧：个人 Agent vs 企业级 Agent

讲完范式和框架，要落到选型必须先认清一个本质分歧：**给个人用的 Agent 和给企业用的 Agent，面对的问题根本不一样**。很多框架/产品的差异，根源就在它服务的是哪一边。

| 维度 | 个人 Agent（单机/单用户） | 企业级 Agent（分布式/多用户） |
|---|---|---|
| 运行形态 | **单机**跑一台机器上，进程内状态就够 | **分布式**部署、多副本水平扩展，状态要跨机器 |
| 状态/记忆 | 本地目录/文件即可，重启还在就行 | 状态必须**外置**到 Redis/MySQL，任意副本能接续任意用户的上下文 |
| 用户隔离 | 单用户，不用隔离 | **多租户**：每个用户的工作区/记忆/沙箱要隔离，数据不能互窜 |
| 部署/环境 | 本地路径写死也无所谓 | 本地/容器/分布式环境路径要**统一抽象**，一套代码跑全环境 |
| 稳定性 | 偶尔挂了重跑可接受 | 模型接口超时/限流/报错要有**容错重试+备用模型**，不能任务中断 |
| 安全 | 工具调用风险低、个人自负 | 工具调用要**权限风控**、高危操作审批、沙箱隔离、审计 |
| 上下文 | 个人长对话撑爆了顶多慢 | 长对话+多用户并发，**上下文压缩/记忆治理**是成本和稳定性双重要求 |

**这个分歧决定了产品/框架的定位**：
- OpenClaw、Hermes、Claude Code 这类**成品 Agent**主要服务个人/单机——它们假设"一个本地工作区、一个用户"，简单直接。但一旦要"服务对外、多用户隔离、多副本共享工作区"，这套本地目录假设就崩了。
- **企业级框架**（如 AgentScope）正是为后者而生：把分布式部署、多租户隔离、安全权限、容错机制做成**框架原生能力**，而非让开发者自己拼。

### 6. 企业级代表：AgentScope（2.0 / HarnessAgent）

**AgentScope**（阿里巴巴开源、Apache-2.0、Python/Java/TS 多语言）补的就是上面表格里"企业级"那一列——**企业级工程化框架**：给你造"能在生产环境长期稳定跑"的 Agent 的工程底座。

**1.0 → 2.0 的转向**：1.0 主打"透明开发"（消息流转/工具调用/协作过程全可见，便于调试和原型）；2.0（2026 年发布）是破坏性大重构，转向"**稳定落地**"——把企业落地时真正头疼的问题做成框架**原生能力**。它解决的是"跑通一次 Demo"和"长期稳定运行"之间的鸿沟。

**核心：HarnessAgent —— ReActAgent 之上的薄封装**。2.0 在原有 `ReActAgent`（Thought→Action→Observation 推理循环，原样保留）之上，抽出 `HarnessAgent` 作为**推荐入口**，用一个 Builder 把长期运行 Agent 必备的工程能力串起来：
- **工作区（Workspace）**：Agent 的人格、知识、技能、记忆、子 Agent 规格统一沉淀在一个结构化工作区里，每次运行自动加载上下文、结束后自动回写记忆——能力随时间演化。
- **双层记忆 + 上下文压缩**：对话压缩、记忆沉淀、全文检索，解决长对话上下文膨胀和跨会话记忆丢失。
- **子 Agent 编排**：声明式定义子 Agent、同步/异步委派，支持隔离沙箱执行、多轮间沙箱状态可恢复。
- **沙箱与权限**：工具执行可配 Docker/E2B 沙箱；三层权限引擎（规则+模式+运行时检查），高危操作人工确认。
- **事件流**：一次回复流式产出"模型调用开始/文本增量/工具调用/工具结果/用户确认/外部执行"等类型化事件——执行过程可展示、可交互、可干预。
- **Agent Service**：多租户多会话开箱即用、**分布式会话恢复**（状态外置到 Redis/MySQL，任意副本接续）、无状态水平扩展、多租户隔离贯穿全链路（userId/sessionId 一路传到工作区路径、KV 命名空间、沙箱状态槽）。

**设计哲学（"Harness"= 马具/约束/跑道）**：裸 Agent 是匹"野马"（有能力但无约束、无续航），HarnessAgent 给它配全套装备和标准化跑道。关键设计是**薄封装 + Hook 驱动**：不重写推理循环（"给手机加壳，而非重造手机"），工程能力通过 ReActAgent 的 Hook/扩展点注入、能力正交（像流水线工位各干各的）。理念是**让模型的原生推理主导 Agent 行为、框架只提供生产基础设施**，而不是用流水线/提示词模板约束模型（随着 LLM 推理变强，那种约束反成瓶颈）。

**它直面的就是上面"企业级"那列的痛点**：① 长对话上下文无限膨胀、成本失控；② 多轮会话记忆混乱、关键信息丢失；③ 本地/容器/分布式环境路径不统一、部署适配成本高；④ 多 Agent 任务无序、无法分层编排、不可追溯；⑤ Agent 自主工具调用无风控、有安全隐患。还多了分布式部署的硬要求：会话状态外置、容器漂移时沙箱快照重建、多租户隔离贯穿执行链路、滚动发布零停机。

**Java 版特别点**：Java 2.0 用 JDK 17（Records/Sealed Classes 做不可变消息模型）、Project Reactor 全链路异步、GraalVM 原生镜像冷启动 <200ms、对接 Spring Boot/Nacos/K8s——对 Java 团队"不换技术栈就能上生产级 Agent"是重要卖点，也是 Java 生态少数专注"企业级智能体工程化"的框架（vs Spring AI 偏 Spring 生态标准化调用）。

> 注：以上为 2026 年 5–6 月发布信息梳理，具体 API/字段以官方仓库（`agentscope-ai/agentscope`、`agentscope-ai/agentscope-java`）为准；面试讲清"HarnessAgent = ReActAgent + 工程化薄封装"这层定位和"让模型主导、框架提供生产基础设施"的理念即可。

### 7. 个人/成品代表：OpenClaw 与 Hermes Agent

回到个人/单机那列，两个对标 Claude Code 的开源成品 Agent：

- **OpenClaw**：奥地利工程师 Peter Steinberger 2025 年底发的开源自主 Agent（图标龙虾、俗称"小龙虾"）。**网关优先（gateway-first）**——常驻 Gateway 进程跑你机器上，接 WhatsApp/Telegram/Slack/飞书等，把消息路由给模型、让模型调本机工具（Shell/文件/浏览器）、结果送回聊天平台。卖点是**多渠道接入 + 本地数据可控 + 后台自主执行**。不是模型，接什么模型都行。
- **Hermes Agent**：**Nous Research** 2026 年 2 月发的开源自主 Agent（MIT）。同类但哲学相反——**学习优先（learning-first）**：核心是**闭环学习回路**，完成任务后自动反思、把成功流程提炼成可复用"技能（Skill）"存下来，下次类似任务直接调技能而非从零推理，号称"越用越聪明"。关键设计：**system prompt 全程不变、不中途换工具集**，保证 prompt caching 持续命中、省成本。注意别和奢侈品牌"爱马仕（Hermès）"混，也别和 Nous 之前发的 Hermes 系列**模型**（Hermes-2/3，基于 Llama 微调、擅长函数调用）混——Agent 是产品，模型是它可选的推理引擎之一。

两者都是"自托管 + 多渠道 + 调工具"的开源 Agent，差别在进化方向：OpenClaw 偏**网关/接入/本地可控**，Hermes 偏**自我进化**（技能沉淀、跨会话记忆）。它们和 AgentScope 的关系是**产品 vs 框架**：那俩是拿来即用的成品（面向个人/单机），AgentScope 是给你造企业级 Agent 的底座（面向多用户、分布式、可商用）——但都借了同一种"Harness Engineering"理念（用结构化工作区+上下文管理+工具约定，替代"每次对话各自为战"）。

### 8. 选型总表与表态

| 方案 | 层次 | 是什么 | 适合 | 取舍 |
|---|---|---|---|---|
| **Claude API + 手写循环** | 框架（最薄） | 自己写 Thought-Action 循环 | 要最大控制（审批/日志/条件执行）、学原理 | 要自己处理循环/上下文/重试 |
| **Claude tool runner** | 框架 | SDK 封装的自动循环（`@beta_tool`） | 自定义 Agent、自己托管计算 | 比手写省事，仍要管上下文 |
| **Managed Agents（CMA）** | 框架（托管） | Anthropic 托管 Agent + 沙箱 + 事件流 | 不想自己跑循环/托管工具、要持久化版本化 Agent | beta、生态新；Bedrock/Vertex 不可用 |
| **LangChain** | 框架 | 通用 LLM 框架，链式编排 | 快速搭、接各种数据源/模型、生态广 | 抽象重、Agent 抽象老、调试难 |
| **LangGraph** | 框架 | 图式状态机编排 | 复杂多步/多 Agent、要显式控制状态流转 | 学习曲线、绑 LangChain 生态 |
| **AgentScope** | 框架（企业级） | 企业级工程化框架（HarnessAgent） | 多用户/分布式/多租户/要安全合规的生产 Agent | 偏重、对极简个人场景过载 |
| **OpenClaw / Hermes** | 产品 | 自托管成品 Agent（对标 Claude Code） | 个人/单机、多渠道接入、本地可控 | 是产品不是框架；企业级场景能力不足 |
| **AutoGen / CrewAI** | 框架 | 多 Agent 对话/角色化协作 | 多 Agent 协作、模拟团队 | 偏特定范式、抽象较重 |

**给面试的表态**：我会按"控制力需求 vs 想省的事 + 个人 vs 企业"两个维度选——要完全掌控用 Claude API + 手写循环/tool runner；想省掉循环和工具托管用 Managed Agents；流程是复杂状态机且要多 Agent 用 LangGraph；**面向企业生产、要分布式多租户安全合规用 AgentScope**（Java 团队尤其合适、不换栈）；个人单机提效直接用 OpenClaw/Hermes 这类成品。**但首选最简的能 cover 需求的方案**，Agent 框架的抽象成本不低——别为了用框架而用框架。

## 七、面试快问快答清单

1. Agent 和 LLM 区别？→ LLM 是无状态一问一答；Agent = LLM + 循环 + 工具，模型自主决定下一步。
2. 什么时候上 Agent？→ 任务多步难预指定、价值够、模型可行、错误可恢复——四条都满足才值得。
3. ReAct？→ Thought→Action→Observation 循环，思考和行动交织、能纠错；现代模型靠扩展思考强化 Thought。
4. 工具调用的本质？→ 函数 + 给模型的 schema；模型只决定调哪个/传什么参，你的代码执行。
5. Agent 循环怎么写？→ 调模型→看 stop_reason 是否 tool_use→执行工具→把 tool_result 塞回 messages→再调，直到 end_turn。
6. bash 工具 vs 专用工具？→ bash 广但不透明难 gating；需要审批/审计/并发/渲染时提升为专用工具。
7. 工具描述怎么写？→ 不只写"做什么"，要写"何时调用"，触发条件写进 description 提升调用率。
8. 记忆分几层？→ 短期(对话上下文)、长期(文件/memory store)、外部(向量库 RAG)。
9. 上下文怎么管？→ caching 省钱 + editing 裁剪 + compaction 压缩 + 子agent 分流 + memory 外移。
10. Prompt caching 怎么才有效？→ 前缀字节稳定，别塞 datetime/UUID、别中途换工具/模型；查 cache_read_input_tokens。
11. 自调用为什么失效？→ this.method() 绕过 Agent 循环/代理对象，工具不触发。
12. Agent 评测怎么做？→ golden set + 程序断言/LLM-judge(rubric) + 指标(成功率/步数/token) + 回归。
13. 多 Agent 为什么要分而不是分函数？→ 子 Agent 有独立上下文，跑完只交结论，主 Agent 不被中间过程撑爆。
14. 多 Agent 协作要点？→ 异步不阻塞、共享文件不共享对话、只一层委派。
15. 框架怎么选？→ 最简能 cover 需求的；要控制用 API+手写循环/tool runner、要省循环托管用 Managed Agents、复杂状态机多 Agent 用 LangGraph。
16. 怎么控成本？→ caching、子 agent 用便宜模型、effort 按任务调、task budget、工具内过滤大结果。
17. 安全要点？→ 破坏性工具加审批、凭据绝不进 prompt(会持久化)、用 vault/自定义工具把密钥留在己方。
