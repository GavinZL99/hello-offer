# RAG：检索增强生成

## 一、解决什么问题

LLM 有两大知识短板：① **知识有截止日期**，不知道最新事件；② **不知道你的私有数据**（公司文档、内部知识库）。把所有知识塞进模型参数（微调）成本高、更新慢、还装不下、改一条要重训。

**RAG（Retrieval-Augmented Generation，检索增强生成）** 的思路：不动模型本身，而是**在生成前先检索相关知识、拼进 prompt、让模型基于检索内容作答**。等于给模型一本"开卷参考书"，每次答题前先翻书。比微调便宜、知识可随时更新（改文档重新入库即可）、还能带引用。

## 二、流程是什么

分**离线建库**和**在线问答**两段：

```
离线（建知识库，做一次）：
  原始文档 → ① chunking(切分) → ② embedding(每块向量化) → ③ 存入向量库

在线（每次问答）：
  用户 query → ④ embedding(同款模型向量化)
            → ⑤ 检索(在向量库找最相似的 top-k) + ⑥ 重排(精排取 top-N)
            → ⑦ 拼进 prompt("基于下列内容回答..." + 检索块 + 问题)
            → ⑧ 模型生成答案
```

下面把每个环节展开讲清"怎么做、有哪些选择、怎么取舍"。

### ① chunking：怎么切文档

切的好坏直接决定召回质量——切得不好，要么关键信息被切两半（检索到一半没用），要么一块里塞太多无关内容（相似度被稀释）。

**怎么切（按粒度从粗到细）**：
- **固定长度切**：按字符/ token 数（如每 500 字一段）最简单。问题：可能把一句话/一个语义单元切两半。加 **overlap**（相邻块重叠 50~100 字）缓解边界丢失。
- **按结构/语义切**：按段落、标题、Markdown 结构、代码块边界切，保证语义完整。比定长好，是主流。
- **父子分块（递归切）**：把文档切成大块（父），每个大块再切成小块（子）。**检索用子块**（小、相似度精），**返回用父块**（大、上下文全）。兼顾检索精度和上下文完整性，是进阶做法。

**chunk 大小取舍**：太小→丢上下文、检索准但喂给模型的信息残缺；太大→相似度被稀释、召回不准、还费 token。经验值几百 token，按内容调。文档结构化的（有标题/章节）优先按结构切。

### ② embedding：把文本变成向量

embedding 模型把一段文本映射成一个高维向量（如 768/1024 维），**语义相近的文本向量也相近**（向量距离小）。这是"语义检索"的基础——不靠字面匹配，靠意思相近。

**关键点**：
- **query 和文档必须用同一款 embedding 模型**（否则向量空间不对齐，没法比距离）。
- **选型**：中文常用 bge-m3 等；要选和你的语种/领域匹配的。embedding 质量直接决定召回上限。
- **双塔 vs cross-encoder**：embedding 是"双塔"——query 和文档各自独立编码再比距离，快但粗（适合从百万文档里粗筛）；cross-encoder 是把 query 和文档拼一起喂给模型打分，准但慢（适合从几十个里精排，即下面的 rerank）。

### ③ 向量库：存向量、做相似度检索

存的是"向量 + 原文 + 元数据"，检索时给一个 query 向量，找出最相似的 top-k。相似度一般用**余弦相似度**或点积。

**为什么不用关系库直接搜**：百万级文档里精确比距离太慢，向量库用 **ANN（近似最近邻）索引**（如 HNSW）做近似检索，牺牲一点精度换巨大速度。选型：Milvus、PGVector（Postgres 插件，已有 PG 的话省事）、Qdrant 等，按规模和现有栈选。

### ⑤ 检索 + ⑥ 重排

**检索（粗排）**：向量检索召回 top-50（保 recall，宁多勿漏）。也可同时跑 **BM25 关键词检索**（字面匹配强，对专有名词/型号/代码这种"必须字面对上"的更准），两路结果用 **RRF（Reciprocal Rank Fusion）融合**排序——向量管语义、BM25 管字面，互补。

**重排（精排）**：粗排的 50 个用 **cross-encoder rerank 模型**（如 bge-reranker）逐个和 query 精细打分，取 top-5 喂模型。重排比双塔 embedding 准得多，是召回质量**最立竿见影**的提升点——粗排保召回、重排提精度。

还可加**元数据过滤**：检索前先按时间/来源/类别过滤（"只查 2024 年后的政策"），大幅提准、减噪。

### ⑦ 拼进 prompt + ⑧ 生成

把重排后的 top-N 块拼进 prompt，格式大致：
```
基于以下检索内容回答用户问题。检索内容没有的信息就说不知道，不要编造。
[1] <块1内容>
[2] <块2内容>
...
用户问题：<query>
```
要点：**强约束**（"只基于检索内容、没有就说不知道"）压幻觉；**带引用编号**让模型对号入座、可追溯；检索块太长可先用 LLM 压成要点再拼（省 token、降噪）。

## 三、使用中会遇到哪些问题

三类顽疾，根因不同，要分别定位：

1. **召回为空**：该有的文档没进 top-k，模型"无米下锅"。根因多是 **query 和文档表达对不上**（用户问"退货流程"，文档写"退款政策"，向量相似度低）。
2. **召回不准**：top-k 里一堆不相关的，相关的反而不在前。根因是 chunking 不好、embedding 模型弱、没重排、没过滤。
3. **幻觉严重**：检索到了正确内容，模型却不照着答、瞎编。根因在生成层——prompt 没约束、检索质量差模型不敢信转去靠记忆编。

## 四、解决方案

**召回为空**：
- **Query 改写/扩展（最有效）**：先用 LLM 把口语化 query 改写成接近文档表述的检索 query，或扩展多个同义 query（"退货流程"→"退款政策/退货步骤/如何退货"）分别检索再合并。这一步对"表达对不上"收益最大。
- **混合检索**：向量 + BM25 融合，专有名词/字面匹配能补上。
- 放大 top-k、别卡死相似度阈值；真为空时明确告诉模型"未检索到，请说明无法回答"，别硬编。

**召回不准**：
- **加重排（收益最高）**：粗排保 recall、重排提准。
- **chunking 策略**：按语义/结构切、父子分块，别无脑定长。
- **元数据过滤**：先按时间/来源/类别过滤再检索。
- 换/调 embedding 模型；多轮对话先做指代消解把 query 改写完整再检索。

**幻觉严重**：
- **Prompt 强约束**：明确"只基于检索内容、没有就说不知道、禁止用自己知识编造"。
- **带引用（Citation）**：让模型每句标注来自哪个检索片段，强制对号入座。
- **后置校验**：生成后用独立 LLM 按"是否忠于检索内容"打分（faithfulness），低分重生成或转人工。
- 先把召回治好（召回不准/为空，模型自然倾向编）；事实性强场景用强模型 + 降随机性。

**评测闭环**：建 query→期望文档的 golden set，召回层看 recall@k / MRR，端到端看 faithfulness；改任何一环都跑回归，别凭感觉调参。

一句话：RAG = chunking（切好块）+ embedding（选好模型）+ 向量库（ANN 检索）+ 混合检索 + 重排（精排）+ 带约束的 prompt 拼装。召回问题靠 query 改写 + 重排 + chunking 治，幻觉靠 prompt 约束 + 引用 + 后置校验治。
