# Agent 运行优化：降本、提速、减少无效工具调用

## 一、解决什么问题

Agent 是 **token 放大器**——循环 × 多轮，每轮都要调模型、可能调工具，成本和延迟远高于单次 LLM 调用。线上 Agent 普遍三个痛点：**贵**（token 烧得多）、**慢**（往返多/串行）、**无效工具调用**（乱调、重复调，既费 token 又费延迟还可能误操作）。

## 二、流程是什么（成本/延迟构成）

先看清 Agent 运行的成本和延迟从哪来：

```
一轮循环 = 模型调用(输入token + 输出token) + 工具执行(IO耗时)
总成本  ≈ Σ(每轮输入token + 输出token) × 单价
总延迟  ≈ Σ(模型往返RTT + 工具执行时间)，串行则相加、并行则取最长
```

**两个关键洞察**（决定优化往哪使劲）：
1. **输入 token 里大半是反复重读的"稳定前缀"**（system + 工具定义 + 早期历史），每轮都重算钱——降本盯这里。
2. **延迟大半是往返 RTT × 轮次**，串行工具调用尤其慢——提速盯这里。

所以降本盯"重读的 token"，提速盯"往返和串行"，减无效调用盯"该不该调"。

## 三、使用中会遇到哪些问题

1. **贵**：上下文越滚越长、每轮重读大量重复 token；子 Agent 用强模型干简单活；effort 拉满。
2. **慢**：串行工具调用一个个等；子 Agent spawn-and-block 干等；非流式长输出超时；cache 冷启动首屏慢。
3. **无效工具调用**：工具描述没写"何时调用"导致乱调/漏调；工具太多模型选错；不思考就调；重复调同一工具。

## 四、解决方案

### 降本（盯重读 token + 模型分级）

**1. Prompt Caching（收益最大、最该先做）**

原理：稳定前缀（system、工具定义）缓存后，重复读只 ~0.1× 成本。Agent 每轮都重读这些前缀，不缓存等于每轮全价重算。

怎么生效：缓存是**前缀字节精确匹配**——前缀任何一个字节变了，后面全部失效。所以：
- 别在 system prompt 塞 `datetime.now()`/UUID/时间戳（每轮变 → 全失效）。
- 别中途换工具集/换模型（前缀变 → 全失效，cache 还是模型隔离的）。
- 稳定内容放前面、易变内容（每轮不同的用户消息）放最后。

怎么验证：查 `usage.cache_read_input_tokens`——反复请求它是 0 就说明有 silent invalidator（前缀在悄悄变）。

**2. 子 Agent 用便宜模型**

主 Agent（编排/决策）用强模型（Opus），子 Agent（检索/读文件/简单提取）用 Haiku——重活分流到便宜模型，主 Agent 只看结论。这是人为可控降本最大点之一：子 Agent 的中间过程不用最强模型也能干好。

**3. effort 按任务调**

effort 控制模型"想多深、花多少 token"。简单任务 `low`、常规 `medium`、难任务 `high`/`xhigh`，别无脑 `max`。子 Agent/简单判定用 `low` 尤其省。Opus 4.8 等模型 effort 影响大，值得按路由调。

**4. Task Budget**

给 Agent 一个总 token 预算（`task_budget`），模型看倒计时自我节制（"我还有多少额度，该收尾了"）。比硬上限 `max_tokens` 智能——后者是模型不知道的强制截断，task_budget 是模型能感知、会主动规划的软目标。

**5. 工具内过滤大结果**

工具返回几万行 DB 结果别原样塞 context——在工具函数里就过滤/截断/落文件只回路径。省的是 context 里被每轮反复重读的 token。

### 提速（盯往返和串行）

**1. 流式（streaming）**

长输出务必流式：避免 SDK HTTP 超时（>16K token 非流式会超时）；用户早看到第一个字，体感快很多。用 `.stream()` + `.get_final_message()` 既有流式体验又能拿到完整结果。

**2. 并行工具调用**

一轮里模型要调多个工具时，无依赖的并行执行（IO 密集工具尤其明显），别串行一个个等。注意有依赖的要串行。

**3. 子 Agent 异步**

派子 Agent 后主 Agent 继续干别的，别 spawn-and-block（干等慢子 Agent 是大头延迟）。长程任务里 spawn-and-block 会把主 Agent 卡在慢子 Agent 上。

**4. Cache 预热**

启动时发个 `max_tokens:0` 请求把前缀写进 cache，第一个真实请求就能命中（否则首请求冷启动慢）。适合首屏延迟敏感的交互场景。注意：cache_control 要放在和真实请求共享的前缀上，不是占位 user 消息上。

**5. 程序化工具调用（PTC）**

标准工具调用每次一往返：调工具→结果进 context→模型推理→调下一个，三次串行动作=三次往返。PTC 让模型把这些调用**合成一个脚本**在 code exec 容器里跑，工具结果回脚本（不进模型 context），只有脚本最终输出回模型——减往返、减 token、减延迟。适合"很多串行工具调用"或"中间结果很大要先过滤"的场景。

### 减少无效工具调用（盯"该不该调"）

**1. 描述写"何时调用"（最被低估）**

模型靠 description 决定调不调。只写"这工具做什么"不够，要写"**什么情况该调它**"（如"当用户问当前价格/近期事件时调用"）。Opus 4.8 等用工具偏保守，触发条件写进 description 能明显提升"该调时调、不该调时不乱调"。

**2. 工具数量控制 / tool search**

几十个直接放；上百个用 **tool search**——模型按需检索加载相关工具的 schema，不全塞 context。工具多不仅费 token，模型还容易选错（在几十上百个里挑一个，准度下降）。

**3. 先思考再调**

开扩展思考（adaptive thinking），模型先推理"这步到底需不需要调工具"，比闷头调更准。ReAct 的 Thought 本来就是干这个的。

**4. 历史去重**

模型重复调同一工具查同样东西——在工具返回时带"你刚查过 X，结果是 Y"提示，或循环层检测重复调用注入制止。

**5. 粒度合并**

太细的工具（要调好几次才能完成一件事）促使多次调用，适当合并成"高阶工具"一次完成一个完整动作；太粗不灵活。按业务动作边界定粒度。

### 三者的关系

减一次无效工具调用，**同时**省 token（少一轮模型调用）又省延迟（少一次往返）又降误操作风险——三者常交集，所以"减无效调用"是性价比最高的优化点之一。

一句话：降本盯"重读的稳定前缀"（caching + 子 Agent 便宜模型 + effort + task budget + 工具内过滤）；提速盯"往返和串行"（流式 + 并行 + 异步 + cache 预热 + PTC）；减无效调用盯"该不该调"（描述写清何时调用 + 先思考 + 去重 + 粒度）。
