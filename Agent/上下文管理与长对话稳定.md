# 上下文管理与长对话稳定

## 一、解决什么问题

Agent 在循环里每轮都会把**工具结果、思考块、模型回复**追加进上下文（context）。跑久了两个问题：① **上下文窗口被撑爆**（Claude 1M 也会满）；② **早期关键要求逐渐不被遵守**——明明上下文里还写着，模型却像没看见，行为偏离主题、忘了用户偏好、忘了任务目标。这叫**上下文漂移（Context Drift）**。

## 二、流程是什么

Agent 的上下文是这样累积的：

```
每轮循环：
  组请求(system + 历史 messages + tools)
  → 模型返回(含思考块/文本/工具调用块)
  → 把这轮的回复 + 工具结果 全部追加进 messages
  → 下一轮 messages 更长了
```

所以 context 像一个不断变长的"对话记录"：**system 在最前**（每轮都在、注意力最强）、**中间夹着大量工具结果和思考块**、**新交互在后面**。到上限后要么截断早期内容、要么压缩成摘要。

理解这个结构很重要——**信息在 context 里的位置影响模型对它的注意力**：靠前且靠后（首尾）的更受关注，中间的容易被忽略（这就是为什么长 context 中间的关键信息会"漂"）。

## 三、使用中会遇到哪些问题（漂移的成因）

1. **注意力稀释（Attention Dilution）**：Transformer 的注意力随序列变长而分散。早期关键指令被几万 token 的工具结果/闲聊稀释，模型"注意不到"它——不是忘了（内容还在窗口里），是注意力够不到。这叫"中间迷失"（lost in the middle）：长 context 中间的信息最容易被忽略。
2. **工具结果/中间过程堆积挤占**：每次工具返回都进 context，几十轮后大半是过时工具输出，重要指令被淹没；而且这些重复内容每轮还重算钱。
3. **多轮纠偏累积**：每轮可能小幅偏离，没人纠正就累积放大（错误级联），越跑越偏。
4. **窗口满了被截断/压缩**：早期内容被挤出，或被压缩成摘要丢了细节，关键信息物理消失。

类比：给秘书交代十条要求，她处理了几百封邮件后，最早的要求"印象淡了"——备忘录没丢，是被淹没了。

## 四、解决方案（由轻到重）

### 1. 关键信息前置 + 重申

- **放 system prompt**：核心要求放 system（每轮在最前、注意力最强位置），比塞进 user 消息更受模型重视。
- **定期重申**：长对话里用 **mid-conversation system message** 定期重申关键约束。

### 2. Mid-conversation system message（运行中注入约束的"权威通道"）

运行中要注入新约束/状态（"现在切换到精简模式""用户时区是 X"），有两种做法：
- **改 top-level system**：会改变前缀字节 → 整段 prompt cache 失效（每轮重算钱）。**不推荐**。
- **追加 `role:system` 消息到 messages**：不改前缀、保留 cache，又是模型当作"权威指令"（而非普通用户文本）的通道。**推荐**。

要点：它是"事实陈述"通道，措辞像"当前时区是 X"而非命令式"忽略用户说的"；要跟在 user 消息后、不能是 messages[0]。

### 3. Context Editing（裁剪）

把**过时的工具结果、用过的思考块删掉**（不是总结，是物理移除），让 transcript 瘦身。哪些能删：已经处理完、后续不再需要的工具输出；用过的 thinking 块。删掉它们给关键信息腾注意力空间，也省 token。和压缩的区别：裁剪是"删"、压缩是"缩成摘要"。

### 4. Compaction（压缩）

接近窗口上限时，服务端把早期历史**总结成一段摘要**塞回去，腾出空间继续对话。

**关键坑**：必须原样回传 `response.content`（含 compaction 块），**不能只回传文本**——compaction 块是 API 用来恢复压缩状态的，丢了下次请求会出错。压缩会丢细节，所以关键事实要在压缩前**外移到记忆**，别指望压缩后还能保住。

### 5. 子 Agent 分流

把重活派给子 Agent（**独立上下文**），子 Agent 只把**结论**交回主 Agent——主 Agent 上下文不被中间过程撑爆、注意力不稀释。这是"长任务不爆不漂"的根本手段：把产生大量中间 token 的活隔离到子 Agent 的窗口里，主 Agent 只看精炼结论。

### 6. 状态外移到记忆

把"不需每步都看但关键"的信息（用户偏好、决策结论、长期事实）写到**记忆文件**，需要时再 `read` 回来——既省窗口、又随时可取，还抗压缩丢失。详见《Agent记忆》。

### 7. 主动校验/纠偏

长任务里让模型"每 N 步回顾任务目标和已完成项、对齐再继续"，主动纠正累积偏离。比被动等漂移严重再救更省事。

### 手段选择

| 场景 | 用什么 |
|---|---|
| 关键约束要持续生效 | 前置 system + mid-conversation system 重申 |
| 工具结果堆积占窗口 | Context Editing 裁剪 |
| 接近窗口上限 | Compaction 压缩（注意回传 compaction 块） |
| 重活产生大量中间 token | 子 Agent 分流（独立上下文） |
| 跨会话/抗压缩丢失的关键信息 | 外移到记忆文件 |
| 长任务累积偏离 | 主动校验/纠偏 |

一句话：长对话稳定 = 把"重要的"前置/重申 + 把"过时的"裁剪/压缩 + 把"重活"分流到子 Agent + 把"长期关键的"外移到记忆 + 主动纠偏。目标是不让 context 既爆又漂——本质是控制"什么信息留在窗口里、在什么位置"。
