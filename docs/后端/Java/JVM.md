# JVM 知识总结

> 主线：一段 Java 代码怎么跑起来 → 运行时内存怎么分区 → 垃圾怎么回收（重头）→ 类怎么加载 → 怎么调优排查。
> 风格沿用中间件/并发系列：先类比建直觉，术语给中英文 + 说明。
> 注：JMM（Java 内存模型）、volatile、happens-before 属于"并发可见性"，在《Java并发.md》里讲，本篇不重复；本篇讲的"内存"是 **JVM 运行时数据区**（堆/栈等物理划分），两者别混。

---

## 一、JVM 是什么：一段 Java 代码怎么跑起来

Java 号称"**一次编写、到处运行**"，靠的就是 JVM。流程是：
```
你写的 User.java
   │ javac 编译
   ▼
User.class（字节码 bytecode，一种"中间语言"，和平台无关）
   │ JVM 加载、执行
   ▼
解释执行（一行行翻译）+ JIT 即时编译（热点代码编译成机器码缓存）
   ▼
对应操作系统/CPU 的机器码
```

**关键**：`.class` 字节码不是机器码，它是一种"通用中间语言"。**不同平台装不同的 JVM**（Windows 版、Linux 版、ARM 版…），同一份字节码丢给各平台的 JVM，由 JVM 翻译成各自的机器码——所以你编译一次，到处能跑。

**类比**：字节码像"**世界语**"，JVM 像各国的**翻译官**。你用世界语写一份文件（编译成字节码），到了哪个国家，当地翻译官（那个平台的 JVM）就把它翻成当地话（机器码）——你不用为每个国家重写一遍。

JVM 还顺带干了另一件大事：**自动内存管理（垃圾回收）**——你 `new` 对象不用手动 `free`，JVM 帮你回收，这是第四章的重点。

**JDK / JRE / JVM 的关系**（常被问）：JVM 是执行字节码的虚拟机；JRE = JVM + 核心类库（运行 Java 程序的最小环境）；JDK = JRE + 编译工具（javac 等开发工具）。包含关系：JDK ⊃ JRE ⊃ JVM。

## 二、运行时数据区（JVM 内存结构）

JVM 运行时把内存划成几块，**有的线程私有、有的线程共享**。先记这张图：

```
┌─────────────────── 线程私有（每个线程一份）───────────────────┐
│  程序计数器        虚拟机栈            本地方法栈                │
│ （记执行到哪行）   （Java 方法调用）   （native 方法调用）        │
└──────────────────────────────────────────────────────────┘
┌─────────────────── 线程共享（全局一份）──────────────────────┐
│  堆（Heap）：放所有对象实例，GC 的主战场                        │
│  方法区（元空间 Metaspace）：放类的元信息、常量、静态变量         │
└──────────────────────────────────────────────────────────┘
```

**线程私有的三块**（随线程生灭）：
- **程序计数器（PC Register）**：记录当前线程执行到字节码的哪一行。线程切换后能恢复到正确位置。**唯一一块不会 OOM 的区域。** 类比：每个人手里的**书签**，记着自己读到哪页。
- **虚拟机栈（JVM Stack）**：每个线程一个栈，**每调用一个方法就压入一个"栈帧（Stack Frame）"**，方法返回就弹出。栈帧里装**局部变量表、操作数栈、方法返回地址**等。类比：一叠盘子，调方法放一个盘子上去、方法结束拿走。
  - 栈太深（如无限递归）→ `StackOverflowError`；栈无法再扩展申请内存 → `OutOfMemoryError`。
- **本地方法栈（Native Method Stack）**：和虚拟机栈类似，但服务于 `native` 方法（用 C/C++ 写的底层方法）。

**线程共享的两块**：
- **堆（Heap）**：**几乎所有对象实例都分配在这**——`new` 出来的东西都在堆里。它是**垃圾回收的主战场**（第四章全在讲它）。堆放不下新对象且无法扩展 → `OutOfMemoryError: Java heap space`。类比：一个大**仓库**，存所有货物（对象）。
- **方法区（Method Area）/ 元空间（Metaspace）**：放**类的元信息**（类结构、方法字节码、字段）、运行时常量池、静态变量。
  - **重要变化**：JDK7 及以前叫"永久代（PermGen）"、在堆里、大小固定容易 OOM；**JDK8 起改成"元空间（Metaspace）"、移到了本地内存（堆外）**，默认随物理内存增长，缓解了永久代易 OOM 的问题。这是高频考点。

（另：**字符串常量池**在 JDK7 起从方法区移到了**堆**里——所以 `String.intern()` 行为和早期版本不同。）

## 三、对象在堆里：创建与布局

**对象的内存布局**分三部分（呼应《Java并发.md》synchronized 讲的对象头）：
- **对象头（Header）**：① Mark Word（存 hashCode、**GC 分代年龄**、锁状态标志——锁升级和 GC 年龄都在这）；② 类型指针（指向它属于哪个类）。
- **实例数据**：你定义的字段的值。
- **对齐填充**：补齐到 8 字节的整数倍（JVM 要求对象大小是 8 的倍数）。

**对象怎么访问**（两种方式，了解）：句柄方式（栈里存的引用指向句柄池，句柄再指向对象——对象移动时只改句柄，引用不变）vs **直接指针方式**（引用直接指向对象，少一次跳转、快——HotSpot 用的是这种）。

## 四、垃圾回收（GC）—— 重头戏

自动内存管理的核心：JVM 自动找出"没用的对象"并回收它们的内存。要回答三个问题：**怎么判断是垃圾？用什么算法回收？谁来执行（收集器）？**

### 1. 怎么判断对象是垃圾

**① 引用计数法（Java 没用）**：每个对象记"有多少个引用指向我"，归 0 就是垃圾。简单，但有**致命缺陷——循环引用**：A 引用 B、B 引用 A，但外界已经没人用它俩了，它们的计数都还是 1、永远回收不掉。类比：两个人互相抱着、谁都不肯先松手，其实旁边没人需要他们了。

**② 可达性分析（Java 用的）**：从一组**根对象（GC Roots）**出发，顺着引用链往下走，**能"够到"的对象就是活的，够不到的就是垃圾**。这样循环引用的 A、B 因为从 GC Roots 够不到，照样被判定为垃圾。类比：**顺藤摸瓜**——从藤（GC Roots）开始摸，摸得到的瓜留下，摸不到的（哪怕瓜之间还连着）全扔。
- **GC Roots 有哪些**（能作为"根"的）：虚拟机栈里正在用的局部变量、静态变量、常量、native 方法引用的对象等——本质是"**当前确实在被使用、不该回收**"的起点。

### 2. 四种引用强度（强软弱虚）

决定对象"多容易被回收"：
- **强引用（Strong）**：`Object o = new Object()` 这种普通引用。**只要强引用还在，对象绝不回收**（哪怕 OOM）。
- **软引用（SoftReference）**：**内存够时留着、内存不够（要 OOM 前）才回收**。适合做缓存（内存紧张时自动让出）。
- **弱引用（WeakReference）**：**只要发生 GC 就回收**（不管内存够不够）。`ThreadLocal` 的 key 就是弱引用。
- **虚引用（PhantomReference）**：最弱，几乎等于没有，唯一用途是**对象被回收时收到一个通知**（管理堆外内存等）。

### 3. 垃圾回收算法（怎么回收）

- **标记-清除（Mark-Sweep）**：先标记所有垃圾，再清除。简单，但**产生大量内存碎片**（清完东一块空、西一块空，大对象放不进）。
- **标记-复制（Mark-Copy）**：把内存分两半，只用一半；回收时把活对象**复制到另一半**、原来那半整块清空。**没碎片、快**，但**浪费一半空间**。适合存活对象少的场景（新生代）。
- **标记-整理（Mark-Compact）**：标记后，把活对象**都往一端推、挪到一起**，再清掉边界外的。**没碎片、不浪费空间**，但移动对象有开销。适合存活对象多的场景（老年代）。
- 类比：标记-清除像扫地只捡垃圾（地上坑坑洼洼）；标记-复制像在两块地轮流种、把苗移到新地旧地铲平；标记-整理像扫完把家具都推到墙边、腾出中间整片空地。

### 4. 分代收集（GC 的核心思想）

**关键洞察**：大部分对象**朝生夕死**（方法里的临时对象，用完就没用了），少部分活很久。于是把堆**分代**，不同代用不同策略：

```
堆
├── 新生代（Young）：朝生夕死的对象，占比小，频繁回收
│     ├── Eden 区（伊甸园，新对象在这出生）
│     ├── Survivor 0（幸存区 S0）
│     └── Survivor 1（幸存区 S1）          默认比例 Eden:S0:S1 = 8:1:1
└── 老年代（Old）：活得久的对象，占比大，很少回收
```

**对象的一生 + Minor GC（新生代回收，用标记-复制）**：
1. 新对象在 **Eden** 分配；
2. Eden 满了触发 **Minor GC**：把 Eden 和"正在用的那个 Survivor"里的**存活对象，复制到另一个空的 Survivor**，年龄 +1，然后把 Eden 和原 Survivor 整块清空（这就是新生代用复制算法、且只浪费一个 Survivor 的空间）；
3. 对象每熬过一次 Minor GC 年龄 +1，**年龄到阈值（默认 15）就"晋升"到老年代**；
4. **大对象**（如大数组）直接进老年代（避免在 Survivor 之间来回复制）；
5. **动态年龄判定**：如果 Survivor 中同年龄对象总和超过一半，超过该年龄的也直接晋升。

**Full GC（整堆回收，含老年代，用标记-整理/清除）**：老年代满了等情况触发，**比 Minor GC 慢得多、STW 时间长**。调优很大程度就是在**减少 Full GC 的频率和停顿**。

**STW（Stop-The-World）**：GC 工作时（尤其某些阶段）要**暂停所有业务线程**，否则对象引用关系边扫边变会乱。STW 时间长 = 业务卡顿。所以现代收集器都在拼命缩短 STW。类比：盘点仓库时让所有人停下别动，盘点越久、业务停越久。

### 5. 垃圾收集器（谁来执行）

收集器是 GC 算法的具体落地。演进主线就一句话：**停顿（STW）越来越短**。先把两组容易混的概念和搭配关系理清，再逐个讲。

**先分清"并行"和"并发"**（GC 语境，高频混淆）：
- **并行（Parallel）**：**多个 GC 线程**同时回收，但此时**业务线程仍然全部 STW 暂停**——只是"多个清洁工一起扫，但商场还是关着门"。
- **并发（Concurrent）**：**GC 线程和业务线程同时运行**，业务大部分时间不暂停——"边营业边打扫"。
低停顿收集器（CMS/G1/ZGC）拼的就是把更多工作做成"并发"。

**经典收集器的搭配**（JDK8 及以前是"新生代收集器 + 老年代收集器"配对使用）：
```
新生代（都用复制算法）        老年代
Serial ───────────────── Serial Old   （单线程，客户端/小内存）
ParNew ───────────────── CMS           （低延迟组合）
Parallel Scavenge ────── Parallel Old  （吞吐量优先组合，JDK8 默认）
```

#### 经典收集器逐个看

- **Serial / Serial Old**：单线程回收、全程 STW。简单、没有线程切换开销，适合单核、几十 MB 的小堆（客户端）。Serial Old 也常作为 CMS 失败时的"兜底"。
- **ParNew**：就是 Serial 的**多线程并行版**（新生代），唯一能和 CMS 搭配的新生代收集器。
- **Parallel Scavenge + Parallel Old（吞吐量优先）**：多线程并行，目标是**高吞吐**（`吞吐量 = 业务时间 / (业务时间 + GC时间)`，GC 占比越低越好）。适合**后台批处理、计算型**任务（不在乎单次停顿、在乎总效率）。还能"自适应调节"（`-XX:+UseAdaptiveSizePolicy` 自动调各代大小）。**JDK8 默认组合**。

#### CMS（Concurrent Mark Sweep）—— 第一个主打低停顿

CMS 用**标记-清除**算法回收老年代，最大特点是**大部分阶段和业务线程并发**。它分**四个阶段**（必考）：
1. **初始标记（Initial Mark，STW）**：只标记 GC Roots **直接关联**的对象。范围小，**停顿很短**。
2. **并发标记（Concurrent Mark，不 STW）**：从上一步的对象出发，**遍历整个对象图**找出所有存活对象。最耗时，但**和业务并发、不停顿**。
3. **重新标记（Remark，STW）**：并发标记期间业务还在改引用，这一步**修正那段时间内变动的对象**。停顿比初始标记长、比并发标记短。
4. **并发清除（Concurrent Sweep，不 STW）**：清除垃圾，和业务并发。

CMS 把最耗时的"标记"和"清除"都做成并发，所以停顿短。但**四个缺点**（也是它被淘汰的原因）：
- **内存碎片**：标记-清除算法天生有碎片，碎片多了大对象放不下，会触发一次带整理的 Full GC（长 STW）。
- **浮动垃圾**：并发清除时业务还在产生新垃圾，这次清不掉、只能等下次。
- **CPU 敏感**：并发阶段 GC 线程要占用 CPU，业务可用 CPU 变少、吞吐下降。
- **Concurrent Mode Failure**：并发回收**还没结束、老年代就被业务填满了**，只能临时**退化成 Serial Old 单线程做一次 Full GC**——长时间 STW，是 CMS 最怕的情况。
**JDK9 起废弃、JDK14 移除**，被 G1 取代。

#### G1（Garbage First，JDK9+ 默认）—— 分区 + 可预测停顿

G1 是为了取代 CMS 而生，解决了 CMS 的碎片和停顿不可控问题。核心设计：
- **Region 分区**：把堆切成约 2048 个**大小相等的小块（Region）**。不再有物理上连续的新生代/老年代——每个 Region **动态扮演** Eden / Survivor / Old / Humongous（专放大对象）角色，逻辑上仍分代、物理上打散。
- **可预测停顿（最大卖点）**：G1 给每个 Region 估算"回收它能腾多少空间、要花多久"。你设一个目标停顿（`-XX:MaxGCPauseMillis=200`），G1 就**挑性价比最高的若干 Region** 在这个时间预算内回收——这就是 "Garbage First"（垃圾最多的优先回收）。类比：仓库分成很多小隔间，每次清理只挑"垃圾最多、最划算"的几个隔间，并控制每次清理时长。
- **无碎片**：Region 之间用**复制**算法（把存活对象复制到空 Region），天然**不产生碎片**——比 CMS 的标记清除强。
- **RSet（Remembered Set）**：解决"跨 Region 引用"——每个 Region 记一份"**谁引用了我**"。这样回收单个 Region 时，不用扫描整个堆去找谁引用了它的对象，看自己的 RSet 即可。
- **G1 的 GC 类型**：Young GC（只回收新生代 Region）→ Mixed GC（回收全部新生代 + 部分收益高的老年代 Region，G1 的常态）→ Full GC（前两者跟不上时的兜底，单线程、慢，**要尽量避免**）。

#### ZGC / Shenandoah（超低延迟）

目标更狠：**停顿 < 1ms，且几乎不随堆增大而变长**——几百 GB 甚至 TB 级的堆，停顿也能稳定在亚毫秒。
- 关键技术：**着色指针（Colored Pointers）**——把 GC 标记信息直接存在对象**指针的几个空闲位**里；**读屏障（Load Barrier）**——程序读取对象引用时插入一小段逻辑，使得**对象的标记、转移（移动整理）都能和业务线程并发进行**，不需要长 STW。
- 代价：吞吐略有损失、内存占用略高。适合**超大堆 + 对延迟极敏感**的服务（如大型在线交易）。
- Shenandoah（Red Hat 主导）和 ZGC（Oracle 主导）思路类似，都是并发整理、超低延迟。ZGC 在 JDK15 转正、JDK21 支持分代。

#### 深入：三色标记与"漏标"（并发标记为什么需要 Remark/SATB）

并发标记（CMS/G1 都有）面临一个难题：**标记的同时业务在改引用，可能把本该存活的对象错标成垃圾（漏标）**。原理用"三色"描述对象：**白**（还没被扫到）、**灰**（自己被扫到了、但它引用的对象还没扫完）、**黑**（自己和它引用的都扫完了）。漏标发生在两件事同时满足时：一个黑对象**新增了**指向某白对象的引用，且这个白对象**唯一的其他引用被删掉了**——白对象没人再扫到它，被当垃圾。
解决靠"写屏障"记录这种变动：
- **CMS 用增量更新（Incremental Update）**：记录"黑对象新增的引用"，Remark 时重新扫描这些黑对象。
- **G1 用 SATB（Snapshot At The Beginning）**：记录"被删除的旧引用"，相当于保住标记开始那一刻的快照。
（知道"并发标记会漏标、所以要 Remark/写屏障来补"，这道题就到位了。）

**怎么选**：JDK8 默认 Parallel（吞吐优先），但**一般 Web 服务更在乎停顿、推荐 G1**（JDK9+ 默认）；**延迟极敏感 + 大堆 → ZGC**；批处理/计算型不在乎停顿 → Parallel；老系统还在 CMS 的建议升级到 G1。

## 五、类加载机制

### 1. 类加载过程（一个 class 怎么进 JVM）

一个类从 `.class` 文件到能用，经过五步（**加载 → 验证 → 准备 → 解析 → 初始化**）：
1. **加载（Loading）**：读取 `.class` 字节流，在方法区生成类的元信息，在堆里生成代表这个类的 `Class` 对象。
2. **验证（Verification）**：校验字节码合法、安全（防止恶意/损坏的 class 危害 JVM）。
3. **准备（Preparation）**：给**静态变量分配内存、赋"零值"**（`static int x = 10` 这里 x 先是 0，不是 10）。
4. **解析（Resolution）**：把常量池里的**符号引用**（"某个类的某个方法"这种名字）替换成**直接引用**（真实内存地址）。
5. **初始化（Initialization）**：真正执行静态变量赋值和静态代码块（`static int x = 10` 这里才变 10）。**类的初始化是懒的**——第一次用到（new、访问静态成员等）才触发。

### 2. 类加载器与双亲委派

**类加载器（ClassLoader）** 负责把 class 加载进来。有几层（父子关系）：
- **启动类加载器（Bootstrap）**：加载核心类库（`java.lang.*` 等 `JRE/lib` 下的），C++ 实现。
- **扩展/平台类加载器（Extension/Platform）**：加载扩展类库。
- **应用类加载器（Application）**：加载我们自己写的类（classpath 下的）。
- **自定义类加载器**：可继承 ClassLoader 自己实现（如热部署、加密类加载）。

**双亲委派模型（Parent Delegation）**——核心机制：一个类加载器收到加载请求时，**不自己先加载，而是先委托给父加载器**，父加载器再往上委托……一直到顶层 Bootstrap；**父加载器能加载就用父加载的，加载不了才轮到自己**。
- 类比：公司里你接到任务先**上报给上级**，上级能处理就上级处理，上级处理不了才退回给你做。
- **为什么要这样**：① **安全**——防止核心类被篡改。你写一个 `java.lang.String` 想冒充，请求会一路上交给 Bootstrap，它加载的是正版的，你的假冒类根本没机会被加载。② **避免重复加载**——同一个类不会被不同加载器各加载一份。
- **怎么打破双亲委派**：重写 `loadClass` 方法。典型场景：**Tomcat**（每个 web 应用一个类加载器、相互隔离，且要优先加载自己应用的类而非父加载器的）、**JDBC 的 SPI**（核心接口由 Bootstrap 加载，但实现类在 classpath、要由应用加载器加载，靠线程上下文类加载器"逆向"打破委派）。

## 六、JVM 调优与排查

### 1. 常见 OOM（OutOfMemoryError）类型

- `Java heap space`：堆放不下对象。最常见，多因内存泄漏（对象该回收却被一直引用）或堆设太小。
- `Metaspace`：元空间满，多因动态生成大量类（如反射、CGLIB 代理、热部署）。
- `StackOverflowError`：栈深度溢出（通常是无限递归）。
- `GC overhead limit exceeded`：GC 花了大量时间却回收不了多少——内存泄漏的征兆。
- `unable to create new native thread`：线程开太多，超了系统限制。

### 2. 常用 JVM 参数

- 堆大小：`-Xms`（初始堆）、`-Xmx`（最大堆，**生产建议 Xms=Xmx**，避免动态扩容抖动）。
- 新生代：`-Xmn`；元空间：`-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize`。
- 选收集器：`-XX:+UseG1GC` / `-XX:+UseZGC`。
- OOM 时自动 dump：`-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...`（排查内存泄漏的关键，出事时留下"现场快照"）。

### 3. 排查工具与思路

- **命令行**：`jps`（看 Java 进程）、`jstat`（看 GC 频率/各代占用，判断 GC 是否异常）、`jstack`（打线程栈，排查死锁/线程卡住/CPU 高）、`jmap`（导出堆快照 heap dump）。
- **分析工具**：**MAT**（Eclipse Memory Analyzer，分析 heap dump 找内存泄漏的"罪魁"对象）、**Arthas**（阿里开源，线上实时诊断神器——看方法耗时、热点、类加载等，不用重启）。
- **典型排查思路**：
  - **CPU 飙高** → `top` 找进程 → `top -Hp 进程` 找占 CPU 的线程 → 线程 ID 转 16 进制 → `jstack` 里找这个线程在干嘛。
  - **内存泄漏/频繁 Full GC** → `jstat` 看老年代是否一直涨、Full GC 是否频繁 → `jmap` dump 堆 → MAT 分析谁占了大头、谁拿着不放。

## 七、面试快问快答清单

1. 字节码为什么能跨平台？→ .class 是平台无关的中间语言，各平台装各自的 JVM 翻译成本地机器码（一次编译到处运行）。
2. JDK/JRE/JVM？→ JDK ⊃ JRE ⊃ JVM；JVM 执行字节码、JRE 加核心库能运行、JDK 加编译工具能开发。
3. 运行时数据区有哪些？→ 线程私有：程序计数器、虚拟机栈、本地方法栈；线程共享：堆、方法区(元空间)。
4. 哪块不会 OOM？→ 程序计数器。
5. 永久代 vs 元空间？→ JDK8 起永久代(在堆、定长易OOM)改成元空间(在本地内存、随物理内存增长)。
6. 怎么判断对象是垃圾？→ 可达性分析(从 GC Roots 够不到就是垃圾)，不用引用计数(解决不了循环引用)。
7. GC Roots 有哪些？→ 栈中局部变量、静态变量、常量、native 引用的对象。
8. 四种引用？→ 强(绝不回收)、软(内存不足才回收，做缓存)、弱(GC 就回收，ThreadLocal的key)、虚(回收时通知)。
9. 三种 GC 算法？→ 标记清除(有碎片)、标记复制(无碎片浪费一半，用于新生代)、标记整理(无碎片可移动，用于老年代)。
10. 为什么分代？→ 大部分对象朝生夕死：新生代小而频繁回收(复制)、老年代大而少回收(整理)。
11. 对象怎么晋升老年代？→ 熬过 N 次 Minor GC(默认15)、大对象直接进、动态年龄判定。
12. Minor GC vs Full GC？→ Minor 只回收新生代、快；Full 含老年代、慢、STW 长，调优重点是减少 Full GC。
13. 什么是 STW？→ GC 时暂停所有业务线程，时间长则业务卡顿，现代收集器拼命缩短它。
14. CMS vs G1 vs ZGC？→ CMS 低停顿但有碎片(已废弃)；G1 分 Region、可预测停顿(默认)；ZGC 亚毫秒停顿、适合超大堆。
15. 类加载过程？→ 加载→验证→准备(静态变量赋零值)→解析(符号引用变直接引用)→初始化(才真正赋值)。
16. 双亲委派？→ 加载类先委托父加载器，父加载不了才自己加载；为了安全(防核心类被篡改)和避免重复加载。
17. 怎么打破双亲委派？→ 重写 loadClass；典型 Tomcat 应用隔离、JDBC 的 SPI(线程上下文类加载器)。
18. 怎么排查 CPU 飙高？→ top 找进程→top -Hp 找线程→线程ID转16进制→jstack 看它在干嘛。
19. 怎么排查内存泄漏？→ jstat 看老年代/Full GC→jmap dump 堆→MAT 找占大头、拿着不放的对象。
