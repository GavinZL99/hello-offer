# 后端场景题总结（设计题/手撕方案）

> 场景题考的不是"X 是什么"，而是"你会不会用 X 解决真实问题、踩过哪些坑、知不知道边界"。
> 答题套路统一为：**先问清需求边界 → 给最朴素的方案 → 指出它的问题 → 一步步演进到生产级 → 主动说出剩下的坑**。能演进、能证伪自己的方案，比直接背最终答案更打动面试官。
> 风格沿用 Redis 文档：每题先用类比建立直觉，再进技术细节。
>
> **重要原则：不要张口就 Redis。** 场景题真正考的是"会按约束选工具"。同一个问题，数据量小的时候 MySQL 一个索引就解决了，没必要上 Redis；要全文检索可能是 ES；要削峰是 MQ；单机限流是本地内存。**先讲清"什么规模/什么约束下该用什么"，再给方案**——能说出"为什么不用 Redis"和"什么时候才需要 Redis"，比只会用 Redis 高一个层次。本文很多题主推 Redis，但每题都会补"其他方案/选型边界"。

## 〇、方法论总纲（三维度 + 由浅入深 + 点状速查）

> 任何场景题（秒杀、支付、排行榜…）本质都是在 **高性能 / 高并发 / 高可用** 三件事里做取舍。下面这套总纲是所有正文的"骨"——遇到新题先套这五步，再对号入座查点状方案。

### 一、三维度框架：场景题永远是这三件事

| 维度 | 解决什么 | 本质手段 |
|---|---|---|
| **高性能** | 响应快、吞吐高 | 减少单次耗时 + 提升并发能力 |
| **高并发** | 扛住海量请求 | 分流 + 削峰 + 异步化 |
| **高可用** | 不宕机、不丢单、能容灾 | 冗余 + 故障转移 + 最终一致兜底 |

**三者常互相制约**（这是面试要主动讲取舍的地方）：强一致 ↔ 高可用（CAP）、同步强一致 ↔ 高性能（支付"实时"靠受理快+异步，不是同步等扣款）、单点强一致 ↔ 水平扩展（DB 行锁正确但扛不住并发）。**没有全都要的方案，只有按业务约束选的权衡**。

### 二、由浅入深五步法（任何场景题都套这五步）

1. **问清边界**：数据量级（1万 vs 1亿）、QPS、一致性要求（强一致还是最终一致够）、延迟容忍、预算。**不问边界直接给方案都是耍流氓**——10 QPS 单机 MySQL 就够，10万 QPS 才需要 Redis+MQ+分库分表。
2. **朴素方案**：先给单机、单 DB、同步的能跑方案。这一步证明你懂基本原理，也作为演进起点。
3. **找瓶颈**：朴素方案在哪扛不住——是 DB（连接/锁）、单点（一台机器）、还是同步阻塞？瓶颈点决定下一步手段。
4. **逐层演进**：按瓶颈对症下药——读瓶颈上缓存、写瓶颈上 MQ 异步、单机瓶颈上分片、可用性瓶颈上冗余+故障转移。**每一步只解决一个瓶颈，演进路径本身就是答案**。
5. **主动留坑 + 兜底**：讲清演进后剩余的取舍（一致性窗口、缓存击穿、消息丢失）+ 监控/对账/补偿兜底。**能证伪自己的方案、说清哪里可能出问题怎么兜，比背最终方案加分**。

### 三、三维度各自手段（查表）

**高性能**：索引优化 / 避免大事务减少 IO / 缓存（读多写少）/ 异步化（MQ、线程池）/ 并行（多线程、ForkJoin）/ 静态化+CDN / 连接池对象池复用 / 批量代替循环单条。

**高并发**：缓存挡读 / MQ 异步+削峰挡写 / 限流保护下游 / 分库分表水平扩展 / 读写分离 / 热点分桶 / 连接池复用。

**高可用**：冗余（主从/集群/多机房）/ 故障转移（哨兵/Cluster/健康检查）/ 熔断降级（依赖挂了不拖垮自己）/ 限流（防雪崩）/ 最终一致+对账兜底 / 幂等+重试 / 异地多活容灾。

### 四、常见点状问题速查（一句话方案 → 正文）

| 点状问题 | 一句话方案 | 正文 |
|---|---|---|
| 缓存穿透（查不存在的）| 布隆过滤器 / 缓存空值 | 一-2 |
| 缓存击穿（热点 key 失效）| 互斥锁重建 / 永不过期+异步刷新 | 一-2 |
| 缓存雪崩（大面积失效）| 过期时间加随机 / 多级缓存 / 限流降级 | 一-2 |
| 缓存与 DB 一致性 | 先更 DB 再删缓存 + 延时双删 | 一-3 |
| 限流 | 固定窗口/滑动窗口/漏桶/令牌桶，分布式用 Redis+Lua | 一-4 |
| 热点 Key | 本地缓存 + 多副本 key 分散 | 一-5 |
| 分布式锁 | SET NX PX + Lua 解锁 + 看门狗 / Redlock / Redisson | 二-1 |
| 分布式 ID | 雪花算法（时钟回拨注意）/ 美团 Leaf | 二-2 |
| 分布式事务 | 最终一致：本地消息表/RocketMQ 事务消息（首选）；强一致：TCC | 二-3 |
| 幂等 | 唯一索引 + 状态机 + token + 对账；五阶段防重看 request_id | 二-4 |
| 余额并发扣减 | Redis Lua 原子+分桶+异步落库+对账，不用 DB 行锁 | 二-5 |
| 支付系统设计 | 模型+Auth-Capture 状态机+防重+余额扣减+对账 | 二-6 |
| 消息丢失 | 生产端确认+MQ 持久化+消费端手动 ack | 三-1 |
| 消息重复 | 消费端幂等（唯一 msgId + 去重表） | 三-2 |
| 消息顺序 | 同 key 进同一分区，分区单线程消费 | 三-3 |
| 消息积压 | 扩消费者 / 临时降级跳过堆积 | 三-4 |
| 延迟队列 | Redis ZSet/键过期+事件 / RocketMQ 延迟消息 / 时间轮 | 三-5 |
| 削峰填谷 | MQ 异步缓冲 / 限流保护 | 三-6 |
| 深分页 | 延迟关联 / 游标翻页 / 覆盖索引 | 四-7 |

### 五、一句话

场景题 = 先按**三维度**（高性能/高并发/高可用）认清要解决什么 + 按**五步法**（问边界→朴素方案→找瓶颈→逐层演进→留坑兜底）由浅入深推进 + 对**点状问题**用对应手段（查表）+ 主动讲**取舍和兜底**。能演进、能证伪、能说清"为什么不用某方案"，比直接背答案高一个层次。

**万能金句**："这个方案的核心权衡是 X 换 Y，在当前业务（读多写少/能容忍最终一致）下这么换是划算的；如果需求变成 Z，我会改用……"——展示你不是背方案，而是会按场景做决策。

## 📖 Redis 速查表（不熟 Redis 先看这个，正文随时翻回来查）

**先理解一件事**：Redis 是个"内存里的键值仓库"，每个数据都有一个 **key（键，字符串）**，key 对应的 **value（值）** 可以是下面几种不同"形状"的数据结构。选哪种结构，取决于你要存的数据长什么样。

**5+2 种数据结构：**

| 结构 | 是什么（类比 Java） | 典型用途 |
|------|--------------------|---------|
| **String** 字符串 | 一个 key 配一个值，值可以是文本/数字/二进制 | 缓存、计数器、锁 |
| **Hash** 哈希 | 一个 key 下挂多个 `field→value`，像 `Map<String,String>` | 存一个对象（如用户的多个字段） |
| **List** 列表 | 有序、可重复，两头都能插入弹出，像 `LinkedList` | 队列、最新消息列表 |
| **Set** 集合 | 无序、不重复，像 `HashSet` | 去重、共同好友（求交集） |
| **ZSet** 有序集合 | 每个成员额外带一个**分数 score**，自动按分数排序 | 排行榜、延迟队列 |
| **Bitmap** 位图 | 本质是 String，但按"位"操作，每位只存 0/1 | 签到、布隆过滤器 |
| **GEO** 地理 | 底层就是 ZSet，存经纬度坐标 | 附近的人/店 |

**本文用到的命令（按结构分组，记不住就查这）：**

| 命令 | 作用 | 属于 |
|------|------|------|
| `SET k v` / `GET k` | 存值 / 取值 | String |
| `SET k v NX` | **N**ot e**X**ists：key 不存在时才设置成功（用于加锁、去重） | String |
| `SET k v PX 30000` | 设置的同时给 30000 毫秒（PX=毫秒，EX=秒）过期时间 | String |
| `INCR k` | 把值 +1 并返回新值（原子操作，计数器靠它） | String |
| `DECRBY k n` | 把值减 n（扣库存靠它） | String |
| `EXPIRE k 60` | 给 key 设 60 秒后自动删除；`PEXPIRE` 是毫秒版 | 通用 |
| `DEL k` | 删除 key | 通用 |
| `HMGET k f1 f2` / `HMSET k f1 v1` | 取/设 Hash 里的多个字段 | Hash |
| `SADD k v` | 往 Set 加元素（重复加无效，天然去重） | Set |
| `SINTER k1 k2` | 求两个 Set 的交集（共同关注） | Set |
| `ZADD k score member` | 往 ZSet 加成员，并指定它的分数 | ZSet |
| `ZINCRBY k n member` | 给某成员的分数 +n（加分） | ZSet |
| `ZCARD k` | 数 ZSet 里有多少个成员 | ZSet |
| `ZREVRANGE k 0 9` | 按分数**从高到低**取第 0~9 名（排行榜前10） | ZSet |
| `ZREVRANK k member` | 查某成员按分数从高到低排第几名 | ZSet |
| `ZRANGEBYSCORE k min max` | 取分数在 [min,max] 区间的成员（延迟队列取到期任务） | ZSet |
| `ZREMRANGEBYSCORE k min max` | 删除分数在 [min,max] 区间的成员（清理过期记录） | ZSet |
| `SETBIT k offset 1` / `GETBIT k offset` | 把第 offset 位设为 1 / 读第 offset 位 | Bitmap |
| `BITCOUNT k` | 数一共有多少位是 1（签到了多少天） | Bitmap |
| `GEOADD` / `GEOSEARCH` | 存坐标 / 查某范围内的坐标 | GEO |

> **Lua 脚本**：一段发给 Redis 执行的小程序。因为 Redis 单线程执行命令，**一段 Lua 在执行期间不会被其他命令插队**，所以能把"读—判断—写"多步操作变成一个不可分割的原子操作（防并发出错）。下文很多地方用它就是图这个原子性。

# 一、高并发与性能

## 1. 秒杀系统（场景题之王，一题串起全篇）

> 为什么是王：它同时考限流、缓存、库存防超卖、Lua 原子性、异步削峰、幂等——几乎所有高并发知识点的集大成。

**类比**：春运抢票。1 万张票，1000 万人同一秒点"购买"。如果让 1000 万人全冲进售票窗口（数据库），窗口当场挤爆。秒杀系统的全部设计，就是**在人流冲到窗口之前，一层层把人挡掉**——能在家门口劝返的，绝不放进小区；能在小区门口拦的，绝不放到楼下；最后真正能摸到窗口的，只剩一两万人。

**核心矛盾**：瞬时请求量（百万 QPS）远超数据库承载（几千 QPS），且**绝对不能超卖**（卖出去的不能多于库存）。

**分层拦截（从外到内，每层把请求量砍一个数量级）**：

```
1000万请求
   │
① 前端层：按钮置灰 + 答题/验证码 + 静态资源 CDN     ──挡掉一大半机器人和重复点击
   │ 100万
② 网关层：限流(令牌桶) + 黑名单 + 同一用户去重      ──只放有限流量进后端
   │ 10万
③ 应用层：Redis 预扣库存(Lua 原子)                ──库存为0直接拒绝，不碰DB
   │ 1万（约等于库存量）
④ 异步层：扣减成功的请求丢进MQ                      ──削峰，DB按自己节奏消费
   │
⑤ DB层：消费MQ，真实下单扣库存(乐观锁兜底)
```

**关键设计点逐个拆：**

**① 库存预扣为什么放 Redis + 必须用 Lua？**
DB 扛不住百万并发的 `update`，所以库存先放 Redis，用内存的高吞吐挡住。但"读库存→判断>0→扣减"是三步，并发下会超卖（A、B 同时读到库存=1，都判断通过，都扣，变成 -1）。Redis 单线程 + Lua 把三步合成一个原子操作：

```lua
-- KEYS[1]=库存key, ARGV[1]=购买数量
local stock = tonumber(redis.call('get', KEYS[1]))
if stock == nil or stock < tonumber(ARGV[1]) then
    return 0   -- 库存不足
end
redis.call('decrby', KEYS[1], ARGV[1])
return 1       -- 扣减成功
```

为什么 Lua 能保证原子：Redis 命令执行是单线程的，一段 Lua 脚本执行期间不会插入任何其他命令（呼应 Redis 文档的单线程模型）。

**② 为什么扣完 Redis 还要丢 MQ，不直接写 DB？**
就算 Redis 把请求砍到 1 万，这 1 万个"真实下单"如果同时打 DB（建订单、扣真实库存、记流水，是个事务），DB 依然可能被瞬时打满。MQ 在这里做**削峰填谷**：1 万个下单请求瞬间进队列，DB 以自己舒服的速度（比如 2000/s）慢慢消费。用户端体验是"排队中，请稍候"。类比：餐厅爆满时先发号码牌，后厨按节奏出餐，而不是让所有人挤到后厨。

**③ 怎么防止"少卖"（Redis 扣了，DB 没下成单）？**
Redis 预扣成功但后续 MQ 消费失败/超时，会出现"库存扣了但订单没建"，即少卖。兜底：① MQ 消费失败重试；② 设置订单超时（用户 Redis 扣减后 15 分钟不付款），超时则**回补 Redis 库存**（延迟队列实现，见第三章）；③ 最终以 DB 库存为准，活动结束对账。

**④ DB 层为什么还要乐观锁？**
Redis 是缓存，可能与 DB 不一致（Redis 重启丢数据、预扣和真实扣减有时间差）。DB 这层是最后防线，用乐观锁兜底防止真正超卖：

```sql
update stock set count = count - 1
where id = #{id} and count >= 1;   -- count>=1 是关键，扣不动就影响0行
```

`where count >= 1` 让"判断"和"扣减"在一条 SQL 里原子完成，返回影响行数为 0 就是没扣成功。这是数据库层面防超卖的标准写法，比 `select ... for update` 悲观锁吞吐高。

**高频追问：**
- **Redis 挂了怎么办？** 库存数据丢失。方案：开启持久化 + 主从；活动开始前预热库存到 Redis；Redis 不可用时降级为 DB 乐观锁直接扛（限流要更狠）。
- **超卖和少卖你更能接受哪个？** 业务上**少卖可接受（无非剩货），超卖是事故（要赔付/砍单，体验极差）**。所以所有设计向"宁少卖不超卖"倾斜——预扣 + DB 乐观锁双保险。
- **同一个用户狂刷怎么办？** 网关层按 userId 去重/限流；一人一单的活动用 `SET userId 1 NX` 标记已购买（NX 保证只有第一次能设置成功，第二次失败即"已买过"）。

## 2. 缓存三大问题（穿透/击穿/雪崩）

> Redis 文档讲过底层，这里从"场景设计"角度给一句话辨析 + 方案，面试能快速区分三者即可。

**一句话区分**（类比"图书馆找书"）：
- **穿透**：找一本**根本不存在**的书。每次都查不到，每次都跑去仓库（DB）翻一遍。恶意攻击常用。
- **击穿**：找一本**超级热门**的书，刚好被人借走（缓存过期）的那一瞬间，一万个人同时涌向仓库。
- **雪崩**：图书馆**整层书架同时塌了**（大量 key 同时过期，或 Redis 宕机），所有人全涌向仓库。

| 问题 | 方案 | 关键细节 |
|------|------|---------|
| 穿透 | 缓存空值 + 布隆过滤器 | 空值给短 TTL（如 60s）；布隆过滤器"说不存在一定不存在"，挡在查询前 |
| 击穿 | 互斥锁 / 逻辑过期 | 互斥锁：只放一个请求去重建，其他人等；逻辑过期：不设物理TTL，异步重建期间返回旧值 |
| 雪崩 | TTL加随机 + 高可用 + 熔断 | 基础TTL ± 随机10%，把过期时间打散；Redis集群保证不整体宕机；DB前加熔断限流 |

**追问：互斥锁重建会不会把请求堆死？** 拿不到锁的请求不要无限等，可以短暂自旋后返回旧值或兜底默认值，保证不雪崩到 DB。

## 3. 缓存与数据库一致性

> Redis 文档已深入讲过时序，这里浓缩成"面试可直接背"的决策链。

**类比**：一份文件，办公室抽屉里有副本（缓存），档案室有原件（DB）。改文件时，是先改原件还是先改副本？副本删了还是改了？

**结论链（按推荐顺序）：**
1. **删缓存，不是更新缓存**——并发更新会乱序导致缓存停留旧值；且缓存值常是计算出来的，更新浪费。
2. **先更新 DB，再删缓存**（Cache Aside）——两种顺序都有不一致窗口，但"先删后更"的窗口是毫秒级必现，"先更后删"要求"读比写慢"才触发，概率极低。
3. **删缓存失败的兜底**：延迟双删（更新DB→删缓存→延迟几百ms再删一次，覆盖读回填和主从延迟），或订阅 binlog（Canal+MQ）异步删除并重试。
4. **能做到强一致吗？不能。** 缓存和 DB 是两个系统，没有跨系统事务，一切方案都是**缩小不一致窗口，不是消除**。要绝对强一致只能读写都加分布式锁，等于放弃缓存性能——这么表态就是满分。

## 4. 限流（4 种算法，必考）

**先想清楚限流到底在干嘛**：系统的处理能力有上限（比如每秒只能处理 1000 个请求），超过就崩。崩了的后果是**所有人都用不了**。限流就是主动把超出的请求挡掉（拒绝或排队），**牺牲一部分请求，保住整个系统活着**——宁可让 100 个人里的 10 个排队，也不能让 100 个人一起把餐厅挤垮、一个都吃不上。

下面四种算法，按"从笨到巧"的顺序，每种都解决前一种的一个毛病。

---

### 先说清"限流粒度"：限的是谁？

限流算法和"限谁"是两件正交的事——**算法决定怎么数/怎么放行，粒度决定这个计数器是给谁建的，区别只在 Redis 的 key 怎么拼**：

| 粒度 | 限什么 | key 示例 | 典型场景 |
|------|--------|---------|---------|
| 接口/全局级 | 一个接口被**所有用户加起来**的总调用量 | `ratelimit:order_create` | **保护系统不被整体流量打垮（最常见）** |
| 用户级 | **单个用户**的访问频率 | `ratelimit:order_create:user123` | 防刷、防单个用户恶意占用资源 |
| IP 级 | 单个 IP | `ratelimit:login:1.2.3.4` | 防爬虫、防暴力破解登录 |

所以**同一套算法，把 key 拼上不同维度就切换了粒度**。本节开头讲的"保护系统不被打垮"对应的是**接口级**（很多人一起访问，加起来别超过系统能扛的量），下面的示例就以接口 `order_create` 为例；如果要按用户限流，把 `userId` 拼进 key 即可。

---

### ① 固定窗口计数器——最简单，但有突刺

**怎么做**：把时间切成固定的一段段窗口（比如每 1 分钟一段），每段里放一个计数器。来一个请求计数 +1，超过阈值就拒绝；到了下一个整分钟，计数器清零重来。

**类比**：景区"每小时整点开闸，放 100 人，放完关闸；下一个整点重新放 100"。

**实现**：Redis `INCR` 计数 + `EXPIRE` 让 key 到期自动清零，一行 Lua 搞定。

**致命毛病：临界突刺**。看一个具体例子（限流 100 次/分钟）：

```
时间轴 ──────────────────────────────────────────▶
       10:00:00 ............ 10:00:59 │ 10:01:00 ............ 10:01:59
       └──────── 窗口A ────────┘ │ └──────── 窗口B ────────┘
                          ↑整点清零↑
       窗口A 的最后 1 秒来了 100 个 ──┤├── 窗口B 的第 1 秒又来了 100 个
                                   │
       在 10:00:59 ~ 10:01:01 这 2 秒内，实际放进了 200 个请求！
```

问题出在哪：每个窗口单独算账，恰好卡在两个窗口交界处的瞬间，前一个窗口的尾巴 + 后一个窗口的头，加起来是阈值的 **2 倍**。你按"100/分钟"配置的系统，瞬间被 200 个请求打了脸。窗口越大、这个突刺越危险。

**Redis 命令实战**：核心技巧是**把时间窗口编码进 key**——按分钟限流就用 `限流:接口:分钟` 当 key，到了下一分钟自然就是另一个 key，旧 key 让它自己过期。下面限制 `order_create` 接口每分钟最多 100 次（不分用户，所有人加起来）：

```bash
# order_create 接口在 10:30 这一分钟的限流 key
> INCR  ratelimit:order_create:1030     # 全局第1个请求，返回 1
> EXPIRE ratelimit:order_create:1030 60  # 给它设60秒过期，到点自动清零
> INCR  ratelimit:order_create:1030     # 又来一个请求（可能是另一个用户），返回 2
...
> INCR  ratelimit:order_create:1030     # 第101个请求，返回 101 > 100，拒绝
# 到了 10:31，key 变成 ratelimit:order_create:1031，从头计数
# 若要改成"每个用户每分钟100次"，key 拼上 userId 即可：ratelimit:order_create:user123:1030
```

> **key 不是 Redis 自动往上加的**，是**应用代码每次请求都按当前时间现拼出来的**：
> ```java
> long minute = System.currentTimeMillis() / 60000;       // 当前"第几分钟"，一个递增整数
> String key = "ratelimit:order_create:" + minute;        // 时间进入下一分钟，拼出的字符串自然不同
> Long count = redis.incr(key);                            // 新 key 第一次 INCR 自动从0开始(返回1)
> if (count == 1) redis.expire(key, 60);
> if (count > 100) return 拒绝;
> ```
> 所以"到点清零"不是真去清零，而是**时间走到下一分钟、代码拼出一个新 key**，旧 key 摆着等它自己过期。示例里的 `1030/1031` 是为好读代表 10:30/10:31，真实代码里就是上面这种 epoch 分钟数。

裸命令有个坑：`INCR` 和 `EXPIRE` 是两条命令，如果第一次 INCR 后、EXPIRE 前进程崩了，这个 key 就**永不过期**、计数永远卡在上限。所以生产用 Lua 把它俩变原子：

```lua
-- KEYS[1]=限流key  ARGV[1]=窗口秒数  ARGV[2]=阈值
local current = redis.call('INCR', KEYS[1])
if current == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[1])  -- 只有第1个请求才设过期
end
if current > tonumber(ARGV[2]) then
    return 0   -- 超限，拒绝
end
return 1       -- 放行
```

### ② 滑动窗口——把突刺磨平

**毛病的根源**是"固定窗口整点清零"——一清零，过去的请求就不算数了。滑动窗口的思路：**不要整点清零，永远统计"从现在往前数 60 秒"内的请求数**，窗口随时间平滑滑动，没有突变的边界。

**类比**：不再是"看本自然小时进了几人"，而是"看**刚刚过去的 60 分钟**进了几人"——这个 60 分钟的尺子是跟着当前时刻一起往前挪的。

接着上面的例子：到了 10:01:00，固定窗口会清零、再放 100；但滑动窗口往前看 60 秒（10:00:00 ~ 10:01:00），发现 10:00:59 那 100 个还在这 60 秒内，已经满了，**后面的直接拒绝**。突刺没了。

**实现（精确版）**：Redis ZSet，每个请求存一条（member 是请求唯一标识，score 是时间戳）。每次来请求：① 用 `ZREMRANGEBYSCORE` 删掉 60 秒之前的旧记录；② `ZCARD` 数一下还剩几个；③ 没超阈值就 `ZADD` 把这次请求加进去。
**代价**：每个请求都要存一条记录，QPS 极高时 ZSet 很占内存。

**Redis 命令实战**：用 ZSet，每个请求是一个成员，`score` 和 `member` 都用"时间戳"（member 要唯一，可以用 `时间戳-随机数`）。下面限制 `order_create` 接口 3 次/60 秒，当前时间戳是 `10075`（秒）：

```bash
# 来一个请求时，按顺序做三件事：
# ① 删掉 60 秒之前的旧记录（score < 10075-60 = 10015 的全删）
> ZREMRANGEBYSCORE ratelimit:order_create 0 10015
# ② 数一下窗口内还剩几个
> ZCARD ratelimit:order_create        # 返回 2（说明这60秒内已经有2个）
# ③ 2 < 3，没超，把本次请求加进去
> ZADD ratelimit:order_create 10075 "10075-8842"
> EXPIRE ratelimit:order_create 60    # 顺便续期，避免不活跃的key堆积
# 下一个请求来时重复①②，如果 ZCARD 返回 3，就拒绝、不再 ZADD
```

三步之间有竞态（两个请求同时 ZCARD 都读到 2，都通过），所以生产同样用 Lua 打包：

```lua
-- KEYS[1]=限流key  ARGV[1]=当前时间戳(ms)  ARGV[2]=窗口大小(ms)  ARGV[3]=阈值  ARGV[4]=本次请求唯一id
local now = tonumber(ARGV[1])
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, now - tonumber(ARGV[2]))  -- 清窗口外
local count = redis.call('ZCARD', KEYS[1])                            -- 数窗口内
if count < tonumber(ARGV[3]) then
    redis.call('ZADD', KEYS[1], now, ARGV[4])
    redis.call('PEXPIRE', KEYS[1], ARGV[2])
    return 1   -- 放行
end
return 0       -- 超限拒绝
```

**实现（生产折中版，最常用）**：上面 ZSet 存每个请求太耗内存。折中：把 1 分钟切成 6 个 10 秒的小格，每格只存一个计数（用 Hash，field 是格子编号）。当前窗口 = 最近 6 个格子的计数之和。时间每过 10 秒，丢掉最老的格子、加一个新格子。既近似了滑动效果、又不用存每个请求。**格子越多越精确、也越耗资源**——固定窗口其实就是"只有 1 个格子"的滑动窗口。

---

> 前两种本质都是**"数数"**：在某段时间内数请求个数，超了就拒。它们不管请求进来的节奏，只管总数。
> 后两种换了个思路——**控制"节奏"**，让请求以某种速率被处理。这是漏桶和令牌桶与前两者的根本区别。

---

### ③ 漏桶——强制匀速，保护下游

**类比**：一个**底部有个小孔的水桶**。水（请求）从顶上倒进来，速度时快时慢；但水只能从底部小孔**匀速漏出去**（被处理）。如果倒水太快、桶装满了，多出来的水就溢出（请求被拒绝）。

```
          请求忽快忽慢地倒进来
              ↓ ↓↓↓ ↓  ↓↓
          ┌─────────────┐
          │░░░░░░░░░░░░░│ ← 桶（缓冲），满了就溢出=拒绝
          │░░░░░░░░░░░░░│
          └──────┬──────┘
                 ↓ 小孔：永远匀速漏出（比如固定每秒处理10个）
            稳定的、恒定速率的请求流 → 下游
```

**关键特性：不管进水多猛，出水永远是恒定速率。** 下游收到的永远是平滑的流量。

**毛病**：太死板，**不给突发留余地**。哪怕桶是空的、系统其实闲着，请求也只能按固定速率一个个漏出去，没法利用"系统当前有空"这件事快速处理一波。该快的时候快不起来。

**适合**：必须严格保护下游的场景——比如下游是个脆弱的老系统、或调用第三方有严格的 QPS 上限，你需要无论如何都给它平滑、恒定的流量。

**Redis 命令实战**：漏桶不需要真的有个定时器去"漏水"。技巧是**记两个值——当前水量 `water` 和上次更新时间 `last`，每次请求来的时候，根据"距上次过了多久"现算出这期间漏掉了多少水**（懒计算）。用一个 Hash 存这两个值：

```lua
-- KEYS[1]=桶key  ARGV[1]=漏出速率(个/秒)  ARGV[2]=桶容量  ARGV[3]=当前时间戳(秒)
local rate     = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local info  = redis.call('HMGET', KEYS[1], 'water', 'last')
local water = tonumber(info[1]) or 0      -- 当前水量，初始空桶
local last  = tonumber(info[2]) or now    -- 上次更新时间
-- 关键：根据流逝的时间，算出这期间漏掉了多少水
local leaked = (now - last) * rate
water = math.max(0, water - leaked)       -- 漏完后的剩余水量（不会小于0）
if water < capacity then
    redis.call('HMSET', KEYS[1], 'water', water + 1, 'last', now)  -- 桶没满，加入这滴水
    redis.call('EXPIRE', KEYS[1], math.ceil(capacity / rate) + 1)  -- 闲置一阵自动清理
    return 1   -- 放行
else
    redis.call('HMSET', KEYS[1], 'water', water, 'last', now)      -- 桶满，溢出
    return 0   -- 拒绝
end
```

**带数字走一遍**（速率 10/秒、容量 10）：10:00:00 桶空，连来 10 个请求，水量涨到 10（满）；第 11 个来，桶满拒绝。等到 10:00:01（过了 1 秒），漏掉 `1秒 × 10/秒 = 10` 滴，水量归 0，又能再放 10 个。所以无论怎么挤，出口稳定在 10/秒。

---

### ④ 令牌桶——允许突发，最常用

**类比**：一个桶，有人**按固定速率往里丢令牌**（比如每秒丢 10 个，桶最多装 100 个）。请求来了，**先从桶里拿一个令牌，拿到才能通过，拿不到就被拒**。

```
     按固定速率发令牌（每秒10个），桶满了就不再发
              ↓
          ┌─────────────┐
          │ 🪙🪙🪙🪙🪙🪙 │ ← 桶里平时攒着令牌（上限=桶容量）
          └──────┬──────┘
                 │ 请求来 → 拿走一个令牌 → 放行
       请求 ─────┤ 没令牌了 → 拒绝
                 │ 一波突发来 → 一次拿走一批（只要桶里攒够了）
```

**关键特性：令牌会攒着。** 系统空闲时令牌一直在桶里累积（攒到桶容量上限）。这时来一波突发流量，可以**一次性取走一批令牌、瞬间全部放行**（最多放行"桶容量"那么多）。这就是它和漏桶最大的不同——**允许突发**。

平时放令牌的速率，限制了**长期平均速率**；桶容量，决定了**能容忍多大的瞬时突发**。

**最常用**：Guava 的 `RateLimiter`、Sentinel、各种网关限流默认都是令牌桶。因为它既能控住平均速率、又能容忍真实流量里正常的小突发，体验好。

**Redis 命令实战**：和漏桶几乎对称——同样懒计算，记 `tokens`（当前令牌数）和 `last`（上次更新时间），每次请求来时根据流逝时间**现算这期间补充了多少令牌**，再尝试拿一个：

```lua
-- KEYS[1]=桶key  ARGV[1]=生成速率(个/秒)  ARGV[2]=桶容量  ARGV[3]=当前时间戳(秒)
local rate     = tonumber(ARGV[1])
local capacity = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local info   = redis.call('HMGET', KEYS[1], 'tokens', 'last')
local tokens = tonumber(info[1]) or capacity   -- 当前令牌数，初始装满
local last   = tonumber(info[2]) or now
-- 关键：根据流逝的时间补充令牌，但不超过桶容量
tokens = math.min(capacity, tokens + (now - last) * rate)
if tokens >= 1 then
    redis.call('HMSET', KEYS[1], 'tokens', tokens - 1, 'last', now)  -- 拿走一个令牌
    redis.call('EXPIRE', KEYS[1], math.ceil(capacity / rate) + 1)
    return 1   -- 放行
else
    redis.call('HMSET', KEYS[1], 'tokens', tokens, 'last', now)
    return 0   -- 没令牌，拒绝
end
```

**带数字走一遍**（速率 10/秒、容量 100）：系统空闲了 10 秒，令牌补满到 100（封顶，不会超过容量）。这时来一波 100 个请求的突发——令牌够，**100 个全部瞬间放行**（这就是"允许突发"）；第 101 个请求来，桶空了，拒绝；之后每秒匀速补 10 个，就只能按 10/秒放行了。对比漏桶的代码会发现：两者结构几乎一样，唯一区别是**漏桶在"加水"、令牌桶在"减令牌"，方向相反**——这也正是"漏桶限出口、令牌桶可突发"的本质。

> **注意**：Lua 脚本里**不能用 `os.time()` 或 `redis.call('TIME')` 拿时间**（Redis 为保证主从复制和重放的确定性，禁止脚本依赖这类非确定值），所以时间戳统一**由应用层传进来**（上面都是 `ARGV[3]`）。这是写 Redis 限流脚本最容易踩的坑。

---

### 漏桶 vs 令牌桶——用同一个例子彻底讲清（最爱混的点）

两个都配成"**平均 100 个/秒**"。现在来一波突发：**100 个请求在 10 毫秒内同时到达**，然后这一秒剩下的时间没有请求了。

| | 漏桶 | 令牌桶（桶容量=100） |
|---|---|---|
| 这波突发怎么处理 | 按 100/秒匀速漏，即每 10ms 放 1 个，**这 100 个要花将近 1 秒才陆续处理完** | 桶里早攒满了 100 个令牌，**这 100 个请求瞬间拿光令牌、立刻全部放行** |
| 下游看到的流量 | 平滑的 100/秒，**没有尖峰** | 一个 100 的**尖峰**，然后归零 |
| 一句话 | 管的是**出水匀速**（保护下游不被冲击） | 管的是**令牌匀速生成**，但请求可以攒着突发消费（提升体验） |

**怎么选**：
- 下游脆弱、必须平滑（保护第三方接口、老系统）→ **漏桶**。
- 只想限制平均速率、能接受合理突发（绝大多数 Web API、用户请求）→ **令牌桶**。

记忆钩子：**漏桶盯着"出口"恒定，令牌桶盯着"令牌生成"恒定而出口可突发。** 同样是"平均 100/秒"，漏桶把突发摊平，令牌桶让攒下来的额度一次性爆发。

---

### 四种算法总结表

| 算法 | 一句话 | 优点 | 缺点 | 典型实现 |
|------|--------|------|------|---------|
| 固定窗口 | 整段时间数个数，到点清零 | 最简单 | 临界突刺（可达 2 倍） | Redis INCR+EXPIRE |
| 滑动窗口 | 永远数"最近 N 秒"的个数 | 消除突刺 | 精确版耗内存 | Redis ZSet / Sentinel |
| 漏桶 | 请求匀速漏出 | 输出绝对平滑，护下游 | 不容忍突发，浪费瞬时算力 | Nginx limit_req |
| 令牌桶 | 令牌匀速生成，可攒着突发用 | 限均值又容突发，体验好 | 实现略复杂 | Guava RateLimiter、Sentinel |

### 限流用什么实现？不一定是 Redis

**算法**（怎么数、怎么放行）和**实现位置**（状态存哪）是两件正交的事。同一个令牌桶，可以跑在 JVM 内存、Redis、或 Nginx 里。本节示例全用 Redis，是因为开篇框架定的是"保护整个集群不被打垮"——这种**跨多台机器的全局限流**才需要 Redis 当共享计数器。

**类比**：安检设在哪一层——城门口（网关）、大楼入口（应用入口）、每间办公室门口（应用内部）都行。而"所有门加起来不超过 100 人"需要多个门**共用一个登记本**，那个共享登记本才是 Redis 的活；各门各管各的，门后拿张纸记就行，不用 Redis。

| 实现 | 状态存哪 | 能限的范围 | 代价 |
|------|---------|-----------|------|
| **单机限流** Guava `RateLimiter`、Resilience4j | JVM 本地内存 | 只能限**本台机器** | 零依赖、最快（无网络） |
| **分布式限流** Redis+Lua、Sentinel 集群流控 | Redis（共享） | 限**整个集群**总量 | 每次判断多一次网络往返，Redis 成了依赖 |
| **网关/接入层** Nginx `limit_req`、Spring Cloud Gateway、APISIX | Nginx 共享内存 / Redis | 请求**进应用前**就拦掉 | 拦得早、保护最彻底 |

**为什么单机不够、要 Redis**：集群 10 台机器，每台用 Guava 各限 100/秒，全局其实放进了 1000——各台内存计数器互不知情。要精确控住"全集群合计 100/秒"，计数器必须放在大家共享的地方，Redis 最自然（够快、Lua 保原子、栈里通常已有）。

单机限流长这样，**完全不碰 Redis**：
```java
RateLimiter limiter = RateLimiter.create(100);   // Guava，纯JVM内存的令牌桶，每秒补100个
if (limiter.tryAcquire()) { /* 放行 */ } else { /* 拒绝 */ }
```

**怎么选**：能接受按机器限 → 本地 Guava，最快；要全局精确 → Redis / Sentinel 集群模式；想在流量进应用前挡掉 → 网关层。实践常**组合**：网关粗粒度兜底 + 应用内 Sentinel 精细控制；Sentinel 集群模式还会"本地缓一批令牌、定期找 token-server 批发"，兼顾性能与全局精度。

### 开源项目的关键实现代码（面试加分：我看过源码）

#### 1. Guava RateLimiter（单机令牌桶，SmoothBursty）

**核心思想（最精妙的一点）**：不记"上次请求什么时候"，而是记**"下次请求可以从什么时候开始"**（`nextFreeTicketMicros`）。这样不用后台线程不停发令牌，**请求来了才根据时间差懒计算该补多少令牌**——零线程开销。

**关键属性**（`SmoothRateLimiter`）：
```java
double storedPermits;       // 桶里当前存量令牌
double maxPermits;          // 桶容量上限（= permitsPerSecond × maxBurstSeconds，默认存1秒）
double stableIntervalMicros;// 两个令牌的间隔（微秒）= 1秒/permitsPerSecond
long   nextFreeTicketMicros;// 下次请求能被处理的时间戳（核心！）
```

**懒计算补令牌（resync）**——请求来时先按时间差补令牌：
```java
void resync(long nowMicros) {
    if (nowMicros > nextFreeTicketMicros) {
        // 过去了多久 → 该补多少令牌（有上限 maxPermits）
        double newPermits = (nowMicros - nextFreeTicketMicros) / coolDownIntervalMicros();
        storedPermits = min(maxPermits, storedPermits + newPermits);
        nextFreeTicketMicros = nowMicros;  // 补完，对齐到现在
    }
}
```

**预支令牌（reserveEarliestAvailable）**——令牌不够也先放行，把"欠的"推给下次：
```java
long reserveEarliestAvailable(int permits, long nowMicros) {
    resync(nowMicros);                              // 先补令牌
    long stored = min(permits, storedPermits);      // 能用存量的部分
    long fresh = permits - stored;                  // 不够的，要新生成
    long waitMicros = fresh * stableIntervalMicros; // 新生成要等多久
    nextFreeTicketMicros += waitMicros;             // ★ 把"欠的时间"推给下次
    storedPermits -= stored;
    return nextFreeTicketMicros;                    // 返回本次该等到何时
}
```
**关键设计**：`nextFreeTicketMicros += waitMicros`——本次令牌不够不阻塞、先放行，把欠的时间累加到 `nextFreeTicketMicros`，**下次请求要为本次的透支买单**（等更久）。这就是 Guava 令牌桶"允许突发、但后续偿还"的实现精髓。面试能说出这句就到位了。

#### 2. Sentinel 滑动窗口（LeapArray，环形数组）

**核心思想**：用**固定大小的环形数组**做滑动窗口——时间推进、旧格子自然过期被覆盖，不创建新对象（减 GC）、用 CAS 更新（免锁高并发）。比"每个请求存一条"省内存、比"固定窗口"精确。

**数据结构**：
```java
abstract class LeapArray<T> {
    int windowLengthInMs;   // 每格时长（默认500ms）
    int sampleCount;        // 格子数（默认2）→ 总窗口=1秒
    // ★ 环形数组，固定大小，格子复用
    AtomicReferenceArray<WindowWrap<T>> array;
}
class WindowWrap<T> {
    long windowStart;  // 格子起始时间
    T value;           // 统计桶（MetricBucket：通过数/阻塞数/异常数/RT）
}
```

**时间→格子索引映射**（这就是"环形"的关键）：
```java
int calculateTimeIdx(long timeMillis) {
    return (int)(timeMillis / windowLengthInMs) % sampleCount;  // 取模环回
}
long calculateWindowStart(long timeMillis) {
    return timeMillis - timeMillis % windowLengthInMs;  // 对齐到格子起点
}
```
例：`windowLength=500, sampleCount=2`，`time=2200ms` → idx=`2200/500%2=0`、start=`2000`——**复用 array[0]**，旧数据被覆盖。

**currentWindow：拿当前格子、过期就重置（CAS 免锁）**：
```java
WindowWrap<T> currentWindow(long timeMillis) {
    int idx = calculateTimeIdx(timeMillis);
    long windowStart = calculateWindowStart(timeMillis);
    while (true) {
        WindowWrap<T> old = array.get(idx);
        if (old == null) {
            // 格子还没建 → CAS 建一个
            WindowWrap<T> wrap = new WindowWrap<>(windowLengthInMs, windowStart, newEmptyBucket());
            if (array.compareAndSet(idx, null, wrap)) return wrap;
        } else if (windowStart == old.windowStart()) {
            return old;                              // 格子还是当前窗口，直接用
        } else if (windowStart > old.windowStart()) {
            // 格子过期了 → 加锁重置起始时间+清零统计（复用对象，不 new）
            if (updateLock.tryLock()) {
                try { return resetWindowTo(old, windowStart); }
                finally { updateLock.unlock(); }
            }
        }
        // else: 时间倒退（不可能），忽略
    }
}
```
**关键设计**：① **CAS 建格子**（无锁快速路径）、只有"过期重置"才加锁（低争用）；② **复用 WindowWrap 对象**只重置不 new（减 GC）；③ **取模环回**让固定数组表达无限滑动。统计时把所有非过期格子的值加起来就是当前窗口的 QPS。

一句话：Guava 用"下次可用时间 + 懒计算 + 预支"实现零线程令牌桶；Sentinel 用"环形数组 + CAS + 格子复用"实现高并发滑动窗口——都是"用时间换空间、用无锁换性能"的工程典范。

#### 3. Redis-Cell（分布式令牌桶，GCRA 算法）

单机令牌桶有 Guava，**分布式**令牌桶最经典的开源实现是 **redis-cell**（Rust 写的 Redis 模块）。它只暴露一个命令 `CL.THROTTLE`，底层用 **GCRA（Generic Cell Rate Algorithm，通用信元速率算法）**——这是令牌桶的数学等价变体，但**不用真的维护令牌数**，而是维护一个"理论到达时间"（TAT, Theoretical Arrival Time），更省、更精确。

**GCRA vs 令牌桶的关系**：令牌桶记"桶里有多少令牌"；GCRA 记"下一个请求的理论最早到达时间"。两者数学等价，但 GCRA 只存**一个时间戳**（一个 Redis key），不存令牌数——状态更小、计算更简单。

**核心参数**：
- **发射间隔（Emission Interval, e）**：两个请求的最小间隔 = 窗口时长 / 令牌数。
- **延迟变化容差（Delay Variation Tolerance, τ）**：允许的最大突发 = 桶容量 × e。

**算法逻辑**（每次请求来）：
```
tat = Redis 里存的"理论到达时间"（上次算出来的）
now = 当前时间
允许的最早到达 = tat - τ          // 往前推一个容差窗口（突发余量）
if now < 允许的最早到达:
    拒绝，返回还需等待多久
else:
    新tat = max(now, tat) + e     // 更新理论到达时间：推后一个间隔
    写回 Redis
    允许
```
**关键**：`tat - τ` 就是"加上突发余量后的最早可放行时间"——`τ` 越大（桶越大）允许的突发越多。请求来了如果 `now` 在这之后就放行，并把 `tat` 推后一个间隔。**整个状态就一个时间戳**，一个 Redis key 搞定，不存令牌数、不需要后台线程补令牌。

**CL.THROTTLE 命令**：
```
CL.THROTTLE user123 15 30 60 1
          │       │   │  │  └─ 本次申请令牌数（默认1）
          │       │   │  └─── 60秒窗口
          │       │   └──── 30个令牌/窗口
          │       └──── 桶容量(max_burst)=15
          └──── key
返回: [是否拒绝(0/1), 桶容量, 剩余, 需等待秒数, 桶满秒数]
```
**为什么适合分布式**：① Redis 单线程 + 模块内执行 = 天然原子，多机器并发请求无竞态；② 单 key 存一个时间戳，状态极小、0.1ms 级；③ 语言无关，任何语言的客户端调一条命令就行，不用自己写 Lua。

**和自己写 Redis+Lua 令牌桶对比**：自己写要手动维护令牌数 + 时间戳 + 补令牌逻辑（前面④令牌桶那节的 Lua 脚本），容易写错；redis-cell 把 GCRA 封装成一个原子命令，更精确（滚动窗口而非固定窗口）、更省（单 key 单时间戳）、更快（0.1ms）。但代价是要装 Redis 模块（Redis 4.0+ 支持），不是原生命令。

一句话：分布式令牌桶的工业级实现 = **GCRA（只存一个"理论到达时间"时间戳）+ Redis 单线程原子性**——redis-cell 把它封装成 `CL.THROTTLE` 一条命令，比手写 Lua 精确、省、快。

## 5. 热点 Key

**类比**：明星塌房，全网都来刷同一条微博。这条微博（热点 key）所在的服务器被打爆，其他服务器却闲着——因为一个 key 只在一个分片上，**集群能解决容量，解决不了单 key 热点**。

**发现**：`redis-cli --hotkeys`、客户端埋点统计、云产品热 key 分析。
**治理**：
1. **本地缓存（多级缓存）**：热点数据在应用本地 JVM 缓存（Caffeine）扛第一层，哪怕只缓存 1 秒，10 万 QPS 也只剩个位数到 Redis。
2. **key 打散**：`hotkey` 拆成 `hotkey#1`...`hotkey#N` 存 N 个副本到不同分片，读时随机选一个，写时全部更新。用写放大换读分散。

# 二、分布式基础设施

## 1. 分布式锁

> Redis 文档已深挖 RedLock 论战，这里给"选型决策"视角：Redis 锁 vs ZooKeeper 锁。

**类比**：会议室门口的钥匙。单进程时 `synchronized` 是房间内部的暗号，出了这栋楼（跨进程）就没人认了，需要一个**所有人都认的、放在公共位置的物理钥匙**。

**Redis 实现要点**（一句话带过，详见 Redis 文档）：`SET key uuid NX PX 30000` 原子加锁——`NX` 保证"只有 key 不存在时才设置成功"（即同一时刻只有一个人能加锁成功，这就是互斥），`PX 30000` 给锁设 30 秒过期（防止持锁的人崩了导致锁永远不释放），`uuid` 是加锁人的唯一标识（解锁时校验"这锁是不是我的"防误删）。解锁用 Lua 原子执行、看门狗续期、主从切换可能丢锁。

**ZooKeeper 实现**：利用**临时顺序节点**。每个客户端来加锁就创建一个临时顺序节点（zk 保证编号递增），**编号最小的获得锁**；其他节点监听前一个节点（而不是都监听最小节点，避免惊群）。客户端断连，临时节点自动删除，锁自动释放。

**Redis 锁 vs ZK 锁（核心选型题）**：

| 维度 | Redis | ZooKeeper |
|------|-------|-----------|
| 性能 | 高（内存） | 较低（每次加锁要写磁盘、过半节点确认） |
| 可靠性 | AP，主从异步复制可能丢锁 | CP，基于 ZAB 共识，不丢锁 |
| 锁释放 | 靠过期时间兜底（可能误放） | 临时节点，客户端断连自动释放（更及时可靠） |
| 实现复杂度 | 简单，Redisson 开箱即用 | 较重，需引入 ZK 集群 |

**结论**：追求性能、能容忍极小概率锁失效（配业务幂等兜底）→ Redis；追求强一致正确性（金融、不能出一点错）→ ZooKeeper/etcd。**大多数互联网业务用 Redis（Redisson）。**

## 2. 分布式 ID 生成

**类比**：全国连锁店开发票，每张发票号必须全局唯一、最好还能大致看出开票先后。单店时用本地流水号（数据库自增）就行，连锁后各店各自增会撞号，需要一个全局发号机制。

**需求**：全局唯一、趋势递增（对 DB 索引友好，避免页分裂）、高性能、高可用。

**方案演进：**

**① UUID**——简单，本地生成无网络开销。但**无序**（导致 B+ 树索引频繁页分裂、写入慢）、太长（36 字符，占空间、不可读）。**不适合做数据库主键**。

**② 数据库自增**——单点瓶颈，且暴露业务量（竞对看你订单号就知道你日销）。

**③ 号段模式（美团 Leaf-segment）**——"批发代替零售"。不是每次找 DB 要一个 ID，而是一次要一段（如 1000 个），用完再要下一段。DB 压力降低 1000 倍。类比：不是每写一张发票就去税务局领一个号，而是一次领一本（1000 张）回来慢慢用。双 buffer 优化：当前号段用到一定比例就异步预加载下一段，避免用完时卡顿。

**④ 雪花算法（Snowflake）**——最经典。64 位 long 拆成：

```
0 | 41位时间戳(毫秒) | 10位机器ID | 12位序列号
符号位  约69年            1024台机器   每毫秒每机器4096个ID
```

本地生成无网络开销、趋势递增、每台机器每秒可生成 **400 万+**（每毫秒 4096 个）。**致命缺陷：时钟回拨**——机器时间被 NTP 校正往回跳，会和之前生成重复的 ID（同一毫秒+机器+序列号撞了）。处理：
- **小幅回拨**（几毫秒）：等待，时间追上再生成。
- **大幅回拨**（几秒以上）：拒绝服务告警，或切备用机器号。
- 兜底：记录上次最大时间戳，回拨时用"最大时间戳 +1"生成，保证单调不重复。

**和其它发号方案对比**：

| 方案 | 问题 | 适用 |
|---|---|---|
| UUID | 乱序 → B+ 树索引页分裂慢；太长 | 不做主键时 |
| DB 自增 | 单点瓶颈、暴露业务量 | 单库小规模 |
| Redis 自增 | 依赖外部、网络开销、Redis 挂发不出号 | 计数器类 |
| **雪花** | **本地生成、无网络、趋势递增、每秒百万级** ✓ | 分布式主键主流 |

**生产别自己搓**，用成熟方案：
- **美团 Leaf**：号段模式（一次领一段号，DB 压力降千倍）+ 雪花模式，双 buffer 预加载防卡顿，时钟回拨内部处理。
- **百度 UidGenerator**：雪花变体，秒级时间戳 + 更优化的位分配。

这两个把时钟回拨、机器 ID 分配、高可用都封装好了。

**追问：雪花算法的机器 ID 怎么分配？** 小规模手动配；大规模用 ZK/Redis 动态分配，或用美团 Leaf、百度 UidGenerator 等成熟方案。

## 3. 分布式事务

> 单机事务靠数据库 ACID，跨服务/跨库后一个操作要改多个数据源，没有统一的事务管理器，这是分布式系统最难的问题之一。

**类比**：异地恋情侣约定"同时分手"。没有一个上帝视角的协调者能保证两人**真正同一瞬间**说出口——总有先后、总有人反悔、消息总会延迟。分布式事务的所有方案，都是在"无法做到真正同时"的现实下，尽量逼近"要么都成、要么都不成"。

**方案按一致性强度排列：**

**① 2PC（两阶段提交）**——强一致，靠一个协调者。
- 阶段一（准备）：协调者问所有参与者"能提交吗？"，各自执行但不提交，锁住资源，回复 yes/no。
- 阶段二（提交）：全 yes 则通知所有人提交，有一个 no 则全部回滚。
- **问题**：① 同步阻塞，整个过程资源被锁，性能差；② 协调者单点；③ 阶段二如果协调者挂了，参与者一直锁着资源（脑裂）。实际用得少（XA 协议），主要存在于面试和概念里。

**② TCC（Try-Confirm-Cancel）**——把 2PC 上升到业务层，最终一致。
- **Try**：预留资源（如冻结余额，而不是直接扣）。
- **Confirm**：确认执行（真正扣掉冻结的钱）。
- **Cancel**：取消（解冻）。
- 类比：网购下单先**冻结**库存（Try），付款成功后**确认**扣减（Confirm），超时未付则**解冻**（Cancel）。
- 特点：性能比 2PC 好（不锁数据库资源，锁的是业务层的"冻结"状态），但**侵入业务**——每个操作都要写 Try/Confirm/Cancel 三个方法，且要处理空回滚、幂等、悬挂三大坑。

**③ 本地消息表**——最常用、最接地气，最终一致。
- 核心思想：**把"分布式问题"转化成"本地事务 + 重试"**。
- 做法：业务操作和"写一条消息到本地消息表"放在**同一个本地事务**里（要么都成功）；然后一个后台任务轮询消息表，把消息发给 MQ/下游；下游处理成功后回调标记消息已完成；失败就重试。
- 类比：你给朋友转账后，在自己的记事本上记一笔"已转，待对方确认"，然后不断打电话确认，直到对方说收到才划掉。记账和转账是你一个人能保证的原子操作，剩下的靠重试。
- 优点：实现简单、可靠、不强依赖中间件。缺点：业务和消息表耦合，要建表。

**④ MQ 事务消息（RocketMQ）**——本地消息表的中间件升级版。
- RocketMQ 把"消息表"的能力内置了：先发**半消息**（对消费者不可见）→ 执行本地事务 → 根据结果 commit/rollback 半消息 → 如果中间状态丢失，MQ 会**回查**本地事务状态。
- 比本地消息表少了自己建表和写轮询任务。
- 完整机制（半消息+二次确认+回查的每一步、为什么保证一致、它不保证什么）见《消息队列.md》事务消息一节。

**⑤ Saga**——长事务方案，最终一致。
- 把大事务拆成一串本地事务，每个都有对应的**补偿操作**。正向执行，某步失败就**反向依次补偿**（撤销前面已成功的）。
- 类比：旅行预订（订机票→订酒店→租车），租车失败就反向退酒店、退机票。
- 适合**步骤多、链路长、不要求强隔离**的业务（如旅游、电商履约）。

**怎么选（决策树）：**
- 强一致、能接受性能差、跨库同一应用 → 2PC/XA（或 Seata AT 模式，对业务无侵入）。
- 资金类、要求较高一致性、能接受业务改造 → TCC。
- 普通业务、最终一致够用、想简单可靠 → **本地消息表 / RocketMQ 事务消息（首选）**。
- 长流程、多步骤 → Saga。

**一句话表态**：互联网业务 95% 的场景，**最终一致性 + 本地消息表/事务消息**就够了，强一致的 2PC/TCC 成本高，能不用就不用。

## 4. 幂等设计

> 分布式系统中"重试"无处不在（超时重发、MQ 重投、用户重复点击），所以接口必须幂等——**同一个请求执行一次和执行多次，结果一样**。

**类比**：电梯按钮。你按一次和连按十次，电梯只来一次。幂等就是给接口装上这种"按多少次都一样"的特性。

**哪些操作天然幂等**：查询（select）、删除（delete 同一条）、`update set status=1`（赋值而非 `+1`）。**不幂等**：插入（重复下单）、`update set count=count+1`（重复累加）。

**方案：**

**① 唯一索引/唯一键**——最简单可靠。给订单号、流水号建唯一索引，重复插入直接被 DB 拒绝（抛唯一键冲突，捕获后当成功处理）。

**② Token 机制（防表单重复提交）**——
- 进入页面时服务端发一个一次性 token；提交时带上 token，服务端校验并**原子地删除** token（用 Redis `DEL` 返回值判断，或 Lua）；删除成功才放行，重复提交时 token 已不存在，拒绝。
- 类比：演唱会门票，检票时撕掉一角，同一张票进不了第二次。

**③ 去重表 / Redis 标记**——
- 用业务唯一标识（如 `订单ID + 操作类型`）作为 key，处理前先 `SET key 1 NX`（NX = 不存在才设置成功）：设置成功说明是第一次、放行处理；设置失败说明 key 已存在、之前处理过，直接返回。
- MQ 消费幂等常用这个：消息带全局唯一 msgId，消费前查 Redis/DB 是否处理过。

**④ 状态机**——
- 订单状态 `待支付→已支付→已发货`，每个操作只允许从特定前置状态流转。`update ... where status='待支付'`，重复的支付请求因为状态已变，影响 0 行，自然幂等。

**追问：幂等的 key 怎么设计？** 必须是**业务唯一**且**请求方能传递**的——比如客户端生成的请求 ID、订单号，而不是服务端生成的（服务端生成的话重试时变了就失效了）。

**⑤ 分阶段防重（以转账为例，连环问终极版）**

很多人答幂等只会"用唯一索引挡"——但**唯一索引建在 order_no 上，挡的是"同一单重复处理"，挡不了"创建单接口被调两次生成两个 order_no"**。用户点两次 / 网络重试，会生成两个单。防重必须分阶段。

**类比**：你去银行转账，柜员不会因为你来两次就开两个单——你拿着同一个"业务受理号"（request_id）来，柜员查"这号开过单了"，直接给你**原来的单**（order_no），不会开第二张。

**关键：区分两个号**（容易混的就在这）：

| | order_no（业务单号） | request_id（幂等键） |
|---|---|---|
| 谁生成 | **服务端**（发号器/雪花） | **前端可生成**（UUID） |
| 是什么 | 转账单的终身身份证 | "这次转账意图"的受理号 |
| 暴露业务量吗 | 是（连续递增会暴露日销） | 否（随机 UUID 看不出量） |
| 防哪个阶段 | 阶段 ③ 同一单重复扣款 | 阶段 ② 创建单接口被重复调 |

之前说"前端不能生成 order_no"——对；但**前端可以生成 request_id**（它只是请求身份证，不是业务单号，不暴露量）。

**五阶段防重表（任一层漏了下一层兜）**：

| 阶段 | 防什么 | 手段 |
|---|---|---|
| ① 提交前 | 用户点两次/手抖 | 前端禁用 + 进页 token（Redis SETNX，用完作废） |
| ② 创建单 | **创建单接口被调两次生成两个单** | **request_id 幂等键**（同 request_id 只创建一个单、返回同一 order_no） |
| ③ 处理单 | 同一单被重复扣款 | order_no 唯一约束 + 状态机（只有初态能扣款，重复请求返回上次结果） |
| ④ 回调 | 渠道异步回调重发 | 回调标识/状态机去重（已 AUTHORIZED 的再收 auth 回调状态不变跳过） |
| ⑤ 兜底 | 极小概率全漏 | 每日对账 + 冲正 |

**阶段 ② 代码骨架**（这是整个连环问的核心）：

```java
// 前端为"一次转账意图"生成 UUID 并复用，重试也带同一个 request_id
String orderId = redis.get("req:" + requestId);
if (orderId != null) return orderId;                      // 这意图已创建过，返回原 order_no
if (!redis.setnx("req:lock:" + requestId, "1", 10))       // 并发抢锁，只一个进来
    return queryByRequestId(requestId);
orderId = snowflake.next();                                // 生成 order_no（服务端发号器）
insert(orderId, ..., requestId);                           // 落库，记录 request_id→order_no
redis.set("req:" + requestId, orderId);
return orderId;
```

**核心**：order_no 是阶段 ② 成功之后才有的东西，它防的是阶段 ③，**防不了阶段 ② 本身**（点两次时还没有 order_no）。阶段 ② 必须靠 request_id（前端为一次意图生成 UUID、期间复用，服务端用它做幂等键）。叠加 token（防手抖）和对账（兜底），五阶段才闭环。

## 5. 高并发余额扣减（支付场景）

> 追问"支付前余额校验通过、但发起时不够怎么办"——本质是 **TOCTOU（检查-使用时间差）**。有人答"用 DB 行锁 / `UPDATE ... WHERE balance>=amount` 原子更新"——正确但**扛不住高并发**：热点账户所有扣减打同一行，行锁排队，DB 直接被打爆。真实高并发支付（秒杀、红包）**不靠 DB 扛扣减**，而是把扣减前移到 Redis/内存层，DB 只最终落库。

**核心思路：分两层**

- **性能层（Redis/内存）**：扛并发、原子扣减、快速判断够不够、不够立刻拒。
- **持久层（DB）**：异步落库、最终记账、靠对账保一致。

DB 不参与实时扣减，压力从"每笔都打"降到"异步批量落库"。

**类比**：超市结账。所有顾客挤一个收银台（DB 行锁）会排死；开 N 个临时收银台（Redis 分桶）快速扣减，闭店后统一和总账（DB）对账。Redis 是"快速扣减层"，DB 是"最终账本"。

**① Redis Lua 原子扣减（防透支，不碰 DB）**

余额/额度放 Redis，扣减用 Lua 脚本——Redis 单线程执行 Lua 不被打断，原子完成"读当前值→判断够→扣减"，等价于 DB 原子条件更新但跑内存，QPS 几十万：

```lua
-- KEYS[1]=余额key, ARGV[1]=扣减金额
local cur = tonumber(redis.call('get', KEYS[1]) or 0)
if cur >= tonumber(ARGV[1]) then
    redis.call('decrby', KEYS[1], ARGV[1])
    return 1          -- 扣成功
else
    return 0          -- 余额不足，快速拒绝，不碰 DB
end
```

判断和扣减在 Lua 里一次完成，没有 TOCTOU 缝隙，不会透支。**这就是"校验通过但发起时不够"的真正解——校验和扣减在 Redis 一次 Lua 里完成，根本不分离**。

**② 热点账户分桶（单账户超高并发）**

单个热点账户（平台营销户）即使 Redis 也成热点（一个 key）。拆：一个账户额度分到 N 个桶（bucket_0..N-1），扣减随机/轮询打到不同桶，各桶独立扣、无竞争，再异步汇总回主账户。和 ConcurrentHashMap 分段锁一个思路，并发能力 ×N。

**③ 异步落库 + 对账兜底**

Redis 扣减成功 → 发 MQ → 消费者批量/异步落 DB（攒一批一起写降压力）。DB 是最终账本，**不扛实时并发**。Redis 与 DB 短暂不一致，靠对账兜底：Redis 余额 vs DB 余额 vs 流水三方对平，不平则补偿/冲正。资金系统最终一致靠对账，不靠每笔强一致同步。

**④ Redis 挂了怎么办**

- Redis 集群 + AOF 持久化：高可用 + 不丢。
- 降级：Redis 整体挂 → 降级到 DB 慢速模式（限流排队），或直接熔断拒服务（宁可不可用也不错账）。
- 冷启动：重启后从 DB 重建余额缓存。

**和缓冲支付/AE 的关系**：AE"缓冲支付"hold 住按速率处理的是**调渠道（ipay 慢、渠道限流）**，不是余额校验；余额预扣（Redis Lua）是**同步前置**的，不进 MQ。Auth-Capture 的 Auth 预冻结阶段也是先在 Redis 占额度，capture 才落 DB。

**一句话**：高并发不能用 DB 行锁扛扣减。真实做法 = 余额放 Redis 用 **Lua 原子判断+扣**（防透支、无 TOCTOU）+ 热点**分桶**降单 key 并发 + 扣成功发 MQ **异步批量落 DB** + **对账**保最终一致；Redis 挂了降级/熔断，宁可不可用不错账。

## 6. 支付系统设计（综合应用，串起分布式事务/幂等/余额扣减）

> 场景题："设计一个转账/支付系统"。这题是上面几节（分布式事务、幂等、分布式 ID、余额扣减）的综合应用，把模型、状态机、并发、对账串成一条完整链路。答题别一上来写代码，按"本质→模型→状态机→并发→对账"层层推进。以下用**通用模型**表述（参考真实跨境支付架构，但不搬内部主体代号）。

### ① 本质：转账 = 改余额 + 记流水 + 凭证可追溯

两个人异地转账不可能物理搬钞票，本质是**记账**：改两个账户余额 + 留一条谁转给谁多少的流水凭证。所以核心是**账本**。资金系统遵循**复式记账**：每笔记两条流水（付款方减、收款方加），**有借必有贷、借贷必相等**——资金守恒可验证、可对账。

### ② 模型：账户表 + 流水表 + 转账单，业务单与支付执行单分层

- **账户表**（当前状态）：`account(id, balance, frozen, version, status)`——balance 当前余额、frozen 冻结额度、version 乐观锁。
- **流水表**（历史凭证，append-only 不可改）：`transaction(id, account_id, order_no, amount, direction, balance_after)`——direction 出/入，balance_after 变更后余额便于追溯。
- **转账单**（业务单，状态机）：`transfer_order(order_no, from, to, amount, status, ...)`。
- **真实支付更进一步——业务单层和支付执行层分离**：业务单（订单/退款单/打款单）记录"用户要干嘛"，支付执行单（支付计划/支付指令）记录"钱怎么走"；一笔订单可对应多笔支付计划（拆单/组合支付），分开才灵活。业务流驱动资金流，资金流和渠道交互。

**为什么账户表和流水表分开**：账户是"当前快照"（可变、高频更新），流水是"历史凭证"（不可变、只 append）。分开后余额更新走账户表（高频），审计/对账走流水表（只读），不互相锁死。

### ③ 状态机：Auth-Capture 二段式（对应 TCC）

真实跨境支付普遍是 **Auth-Capture 二段式**，不是一步扣款：

```
INIT(下单) → AUTHORIZING(申请授权) → AUTHORIZED(已授权,额度冻结未扣) → PAID(已请款,真正扣款)
                                       ↘ FAILED(授权失败)
AUTHORIZED → CANCELED(未扣款撤销) / REVERSED(已扣款冲退)
```

- **Auth = 预授权冻结**（对应 TCC 的 Try：冻结额度，不真扣）。
- **Capture = 请款扣款**（对应 Confirm：把冻结转成实扣）。
- **取消（CANCELED） vs 冲退（REVERSED）要区分**：取消是钱还没真扣（还在 AUTHORIZED），撤销授权即可、不产生反向资金流；冲退是钱已扣（PAID），要发反向流水退回、产生真实资金回流。两个动作两套逻辑，别混。
- 渠道交互靠**异步回调推进状态机**（auth/capture 消息回调推进）+ **回调幂等**（已 AUTHORIZED 的再收 auth 回调状态不变、跳过）。这要求状态机能"中断-恢复-幂等重试"。

### ④ 防重复转账 → 见《4. 幂等设计》⑤五阶段防重

五阶段闭环：① 进页 token 防手抖 → ② **request_id 幂等键**防"创建单被调两次生成两个单"（同 request_id 返回同一 order_no）→ ③ order_no 唯一约束 + 状态机（只初态能扣）→ ④ 渠道回调幂等 → ⑤ 对账兜底。order_no 由服务端发号器（雪花/Leaf）生成，前端生成的是 request_id（幂等键，不暴露量），别混。

### ⑤ 余额并发扣减 → 见《5. 高并发余额扣减》

不用 DB 行锁扛扣减：余额放 Redis 用 **Lua 原子判断+扣**（防透支、无 TOCTOU）+ 热点**分桶** + 扣成功 **MQ 异步落 DB** + 对账保最终一致。

### ⑥ 高并发下的支付实效

**核心认知**：支付实时性 ≠ 同步强一致。用户感觉"实时"靠**受理快 + 结果推送**，不是真同步等到扣款完成。
- **同步受理极短**：接单→校验→预扣→落库→返回"处理中"，耗时压到毫秒级。
- **真正处理异步**：调渠道、Auth-Capture 靠状态机 + 回调推进。
- **MQ 削峰缓冲支付**：大促时支付请求先入 MQ，按渠道承受速率消费，防打爆下游。
- **渠道侧限流 + 熔断 + 重试**：外部渠道是瓶颈和不可控点。

### ⑦ 缓冲支付下"第二笔怎么立刻失败"

**关键**：MQ 缓冲的是**调渠道**（慢、限流），**不是余额校验**。余额预扣冻结必须在**同步受理阶段**用 Redis Lua 完成——第一笔同步预扣后余额 10→0，第二笔受理时 Redis Lua 一看不够**立刻拒**，根本不进 MQ。入 MQ 的是"已预扣、待渠道处理"的请求。预扣后若渠道异步失败要**回滚释放额度**（TCC Cancel）。资金系统保守优先——**宁可误拒不超扣**（误拒可恢复、超扣难恢复）。

### ⑧ 拒付链路（跨境信用卡特有，加分项）

拒付（用户向发卡行拒付）→ 卡组先退用户钱、扣商户、启动调查 → 商户**拒付举证**（提供证据）→ 卡组判有效 → **拒付反转**（钱退回商户、撤销用户退款）。这是逆向资金流的复杂场景，讲出来显懂真实跨境。

### ⑨ 对账兜底（最终一致的最后保险）

资金流三阶段：**清算**（算各方应收应付，虚拟账务指令）→ **结算**（央行支付系统实际划转）→ **入账**（到最终收款账户，可支配）。每日**对账**：账户余额 vs 流水累加 vs 外部渠道账单三方对平，不平则补偿/冲正。即使代码有 bug，对账能发现——资金系统的兜底。

### 完整答题框架（一句话）

支付系统设计 = **模型**（账户表/流水表/转账单，业务单与支付执行单分层，复式记账）+ **状态机**（Auth-Capture 二段式 INIT→AUTHORIZING→AUTHORIZED→PAID，对应 TCC，区分取消/冲退）+ **防重**（五阶段：token/request_id/order_no+状态机/回调/对账）+ **余额扣减**（Redis Lua 原子+分桶+异步落库，不用 DB 行锁）+ **高并发实效**（同步受理极短+异步推进+MQ削峰+渠道限流）+ **缓冲支付下预扣同步前置**（第二笔立刻失败，宁可误拒不超扣）+ **拒付链路** + **对账兜底**（清算→结算→入账，三方对账）。本质是"最终一致 + 对账兜底"，不是每笔强一致同步。

# 三、消息队列与异步

> MQ 三大作用：**解耦**（上下游不直接调用）、**异步**（非核心流程异步化提升响应）、**削峰**（缓冲瞬时流量）。代价是引入了新问题：消息会丢、会重复、会乱序、会积压。面试就考这四个。

## 1. 消息丢失（分三个阶段排查）

**类比**：寄快递，包裹可能在三个环节丢——你交给快递员前（生产端）、快递公司中转仓（MQ 自己）、派送到收件人（消费端）。三段都要保。

**① 生产端丢**：消息没成功发到 MQ。
- 方案：用**同步发送 + 确认机制**（RocketMQ 的 SYNC、Kafka 的 acks=all、RabbitMQ 的 confirm 机制）。发送失败就重试。

**② MQ 自身丢**：消息到了 MQ 但还没落盘，MQ 宕机。
- 方案：**持久化 + 集群副本**。Kafka 设 `acks=all` + `replication.factor>=2` + `min.insync.replicas>=2`（消息要写入多个副本才算成功）；RabbitMQ 开持久化 + 镜像队列。

**③ 消费端丢**：消费者拿到消息还没处理完就宕机，但 MQ 以为已消费。
- 方案：**手动 ACK**。处理完业务逻辑后再手动确认，而不是拿到消息就自动 ACK。处理失败/超时不 ACK，消息会被重新投递。

**追问：三段都保了，能 100% 不丢吗？** 理论上接近，但代价是性能（同步、多副本、手动 ACK 都慢）。**要不要全保看业务**——日志类可以丢一点换吞吐，交易类必须全保。

## 2. 消息重复消费 → 用幂等解决

为了不丢，重试是必然的，重试就会重复（生产重发、消费端 ACK 前宕机导致重投）。**所以 MQ 几乎无法保证"恰好一次"，业界共识是"至少一次 + 消费幂等"**。方案见第二章幂等设计：消息带全局 msgId，消费前查去重表/Redis。

## 3. 消息顺序

**类比**：流水线上"组装→喷漆→质检"必须按序，不能质检完才喷漆。但不同产品之间互不影响。

- **全局有序**：所有消息进**单分区/单队列**，单消费者串行消费。代价是完全没有并发，吞吐极低，很少用。
- **局部有序（实际方案）**：只保证**同一业务实体**有序即可。把同一个 key（如同一订单 ID）的消息路由到**同一个分区/队列**（按 key 取模），分区内有序、分区间并行。类比：同一辆车的工序在同一条流水线上按序，不同车在不同流水线并行。

**追问：消费端多线程会打乱顺序吗？** 会。即使同分区消息按序到达消费者，多线程处理也会乱序。解决：消费者内部再按 key 哈希到不同的内存队列，每个队列单线程处理。

## 4. 消息积压

**类比**：双十一快递爆仓，包裹堆积如山。要么加派送员（扩消费者），要么先把包裹堆到临时仓库回头再处理（转存）。

**先定位原因**：是消费者挂了、消费太慢，还是生产突增？

**应急方案**：
1. **扩容消费者**——但受限于分区数（Kafka 一个分区只能被一个消费者消费，消费者数 > 分区数则多出来的闲置）。所以要同时**临时扩分区**。
2. **消费者降级**——临时关掉非核心的耗时逻辑，先快速消费。
3. **紧急转存**——写个临时消费者，光速把积压消息原样转存到另一个 topic / DB，先把队列腾空避免消息过期被删，回头再慢慢处理转存的数据。

**根治**：提升消费者性能（批量消费、异步处理、优化慢 SQL），合理设置分区数。

## 5. 延迟队列（订单超时取消）

**类比**：寄存柜定时——存进去时设"2 小时后没人取就自动清空"。订单"30 分钟未支付自动取消"就是这个模型。

**方案演进：**

**① 定时任务轮询 DB**——每分钟扫一遍订单表找超时的。简单，但**有延迟**（最多差一个轮询周期）、**扫全表浪费**（绝大多数订单没超时）、量大时 DB 压力大。小业务能用。

**② Redis ZSet**——利用 ZSet 按分数排序的特性：member 存订单 ID，score 存"应取消的时间戳"。轮询时用 `ZRANGEBYSCORE key 0 当前时间戳`（取出分数在 0~现在 之间的成员，也就是**已经到期的订单**）来处理。比扫 DB 高效（只取到期的），但仍需轮询、且要处理多个 worker 取到同一条（取+删用 Lua 保证原子）。

**③ RocketMQ 延迟消息**——下单时发一条"30 分钟后投递"的延迟消息，到点了消费者收到再检查订单状态。**最优雅**，无需轮询。RocketMQ 原生支持延迟级别；RabbitMQ 用死信队列（TTL + DLX）实现。

**④ 时间轮（Netty/Kafka 内部用）**——类比钟表，指针每秒走一格，每格挂着该时刻要执行的任务。高效处理海量定时任务，是中间件内部的实现，业务层一般直接用现成 MQ。

**关键：到点了只是"提醒检查"，不能直接取消。** 收到超时消息后必须**再查一次订单状态**——用户可能在最后一秒付款了。状态是"已支付"就忽略，还是"待支付"才取消（并回补库存）。这一步本质是幂等 + 状态机。

## 6. 削峰填谷

就是秒杀场景里 MQ 的作用：瞬时洪峰请求先进队列缓冲，下游按自己的处理能力匀速消费。类比水库——上游山洪暴发（峰），水库蓄起来，下游农田按需放水（填谷），避免冲毁。代价是牺牲实时性（用户要排队等），所以适合**能异步、用户能接受延迟**的场景。

# 四、典型业务建模

## 1. 点赞 / 计数系统

**类比**：演唱会实时统计举起的荧光棒数量。如果每举一次就记一笔到档案室（DB），档案室会被写爆。所以先在现场用计数器累计（Redis），定期汇总归档。

**朴素方案的问题**：每次点赞直接 `update count=count+1` 打 DB，高频写（热门内容每秒上万赞）DB 扛不住，且行锁竞争激烈。

**方案**：
1. **Redis 承接写**：点赞数用 `INCR`（值+1，原子操作）计数；用 Set 记录"谁赞了这条"——`SADD like:postId userId`（往集合加元素，同一个 userId 重复加无效，所以**天然防重复点赞**，还能用 `SISMEMBER` 查某人是否赞过）。
2. **异步落库**：定时任务/MQ 把 Redis 的增量**批量**刷回 DB（合并写，1 万次赞变 1 次 `update count=count+10000`）。
3. **读**：直接读 Redis，DB 只是持久化备份。

**追问：Redis 挂了计数会丢吗？** 两次落库之间的增量会丢。容忍度高的场景（点赞数差几个无所谓）可接受；要求准的用 AOF + 主从降低风险，或关键计数双写。

**什么时候不用 Redis**：点赞量不大（一篇内容几十几百赞）时，直接 `update count=count+1` 打 MySQL 完全够，加个唯一索引（userId+postId）防重复赞即可——别为了用 Redis 而用。Redis 的价值只在"高频写顶不住 DB"时才显现。再大规模（统计类、要分析）可以走"埋点日志 → Kafka → 数仓/ClickHouse 聚合"，连实时计数都不需要。

## 2. 实时排行榜

**类比**：游戏天梯榜。表面就是"按分数从高到低排个序"，但藏着几个 DB 很难做的硬需求，正是它们让排行榜成为好题：
1. 取榜单前 N 名（这个 DB 也能做）；
2. **查"我排第几"**（DB 要 `COUNT(*) WHERE score > 我的分`，全表/索引扫描，慢）；
3. **查我前后几名**（看看追我的人和我追的人）；
4. 分数**高频变动**还要求排名**实时**更新；
5. 同分怎么排、日榜周榜怎么分、亿级用户怎么扛。

---

### 朴素方案：MySQL `ORDER BY`，以及为什么撑不住

榜单不大、更新不频繁（如"公司内部积分榜"几千人）时，直接 `SELECT * FROM score ORDER BY score DESC LIMIT 10` + 给 score 建索引就够了，**别一上来就 Redis**。

但规模/频率一上去就崩在两点：① 分数每次变动，`ORDER BY` 都要重新排序，高频更新下 DB 扛不住；② 查"我排第几"要 `SELECT COUNT(*) WHERE score > 我的分`，本质是一次扫描，榜越大越慢，做不到实时。**排行榜真正难的不是"取前 10",而是"实时维护顺序 + 随时查任意人的排名"**——这正是 Redis ZSet 的主场。

---

### 主流方案：Redis ZSet（有序集合）

**为什么天造地设**：ZSet 里每个**成员（member）**都绑定一个**分数（score）**，并且**永远按分数自动排好序**。这正好对应"用户 + 积分 + 名次"，且插入、改分、查排名都是 **O(logN)**（底层是跳表，详见 Redis 文档），不需要像 DB 那样重排。

**数据怎么放**：一个榜就是一个 ZSet，key 是榜名，member 存 userId，score 存积分。比如总榜 `rank:total`：

```bash
# ZADD k score member：把 member 以 score 分加入/更新到 ZSet k
> ZADD rank:total 1000 userA     # userA 1000分
> ZADD rank:total 1500 userB     # userB 1500分
> ZADD rank:total 800  userC     # userC 800分
# 此刻 ZSet 内部已自动排好：userB(1500) > userA(1000) > userC(800)
```
> **`ZADD k score member` 三个参数**：`k` = 榜的 key（如 `rank:total`）；`score` = 分数（数字，ZSet 靠它排序）；`member` = 成员（排名主体，如 userId）。同一个 member 再次 ZADD 会**覆盖**它的分数（用于"设为某分"）。

**四个核心操作**（对应上面的硬需求，括号里是参数含义）：

```bash
# ① 加分：ZINCRBY k 增量 member —— 给 userA 加 50 分（在原分数上累加，返回新分数）
> ZINCRBY rank:total 50 userA            # userA 变成 1050

# ② 取榜单前10：ZREVRANGE k 起 止 [WITHSCORES] —— REV=从高到低，0~9 即前10名
> ZREVRANGE rank:total 0 9 WITHSCORES    # 返回前10名的 member + 分数

# ③ 查我排第几：ZREVRANK k member —— 返回从高到低的名次(从0开始，记得+1展示)
> ZREVRANK rank:total userA              # 返回 1，表示第2名

# ④ 查我前后3名：先用③拿到我的名次 r，再取 [r-3, r+3]
> ZREVRANGE rank:total 7 13 WITHSCORES   # 假设我排第10(r=9)，看第8~14名
```
（`ZREV...` 系列都是"从高到低"；不带 REV 的 `ZRANGE`/`ZRANK` 是"从低到高"。`ZSCORE rank:total userA` 单独查某人分数。）

**关键**：第 ③ 步"查任意人排名"是 O(logN)、瞬间返回——这是 DB 做不到的杀手锏（ZSet 跳表的每个节点记了 span 跨度，能直接算出排名）。

---

### 方案选型（不只是 Redis）

- **数据量小、更新不频繁** → MySQL `ORDER BY` + 索引，省一套同步。
- **高频更新 + 实时排名（主流）** → Redis ZSet。
- **离线榜 / 复杂加权**（"本周最热"按播放+点赞+评论加权算）→ Flink/Spark 离线算好，结果写 Redis 或 DB 供查询。
- **多条件筛选的榜**（按地区+品类+时间段排）→ ES，支持多维聚合排序。

---

### 实战要点与高频追问

**追问 1：同分怎么排？（先到先得）** ZSet 里分数相同的成员，默认按 **member 的字典序**排（即按 userId 字符串比大小），这通常不符合业务的"先达到者排前面"。技巧：**把时间编进 score**，让一个数字同时表达"分数高优先、同分则早到优先"：
```
组合分 = 真实分数 × 10^10 + (10^10 - 时间戳)
```
高位是真实分数（主排序）；同分时，越早达到时间戳越小、`(10^10 - 时间戳)` 越大、排越前（次排序）。
**但有个精度坑（必须说）**：ZSet 的 score 是 64 位浮点 double，只能**精确表示约 15~16 位十进制整数**（超过 2^53 就丢精度）。所以两段位数加起来不能超 ~15 位——比如真实分数留 6 位（≤999999）、时间戳用"相对某个起始点的秒数"压到 9 位，6+9=15 刚好。位数不规划好，同分排序会因精度丢失而错乱。

**追问 2：日榜/周榜/月榜/总榜怎么做？** 一个时间维度一个 ZSet，把时间编进 key：`rank:daily:20260615`、`rank:weekly:2026W24`、`rank:monthly:202606`、`rank:total`。
- 用户加分时，对**所有相关榜同时 `ZINCRBY`**（日+周+月+总，几条命令可以用 pipeline 一次发，减少网络往返）。
- 日榜/周榜加上 `EXPIRE`（如日榜留 2 天）自动清理过期榜，不用手动删。
- 查"本周榜"就读 `rank:weekly:2026W24` 这个 key。

**追问 3：榜单要显示头像、昵称，ZSet 里只有 userId 怎么办？** member **只存 userId**（小、唯一、改名也不影响榜）。取出前 10 的 id 后，**再批量捞详情**——从另一个存用户信息的 Redis Hash（`HMGET`）、或 DB、或本地缓存一次性取回昵称头像。**不要把昵称头像塞进 member**：会让 ZSet 变臃肿，且用户改名时整个榜都要改。

**追问 4：分数从哪来？Redis 挂了榜就没了吗？** ZSet 通常是**服务/展示层**，真实分数的源头一般在 DB 或由事件算出。两种配合：① 写穿——每次加分同时写 DB 和 `ZINCRBY`；② 重建——Redis 重启或新建榜时，从 DB 拉数据 `ZADD` 重灌。纯游戏分等场景也可让 Redis 配 AOF 做主存，但要承担丢数据风险。

**追问 5：亿级用户的榜怎么扛？**

先说清**为什么单个 ZSet 装 1 亿人会崩**：
- **大 key**：1 亿成员，每个成员在 ZSet 里（跳表节点 + 字典项 + member 字符串 + score）大约占 60~100 字节，整个 key **轻松上 8GB**。
- **集群也救不了**：Redis Cluster 是按 key 分片的，**一个 key 只能落在一个分片**——你没法把一个 ZSet 拆到多台机器上。于是这 8GB + 所有读写流量全压在一个节点，集群"分摊负载"的意义在这个 key 上彻底失效（呼应热点 key 那节）。
- **附带伤害**：这么大的 key 会让 RDB fork、AOF 重写、主从同步都变重；某些范围操作还可能阻塞单线程。

**思路一：只维护 Top N（最常用）**

绝大多数产品**只展示头部**（前 100/1000 名）。既然只展示头部，那就**只把头部这 N 个人存进一个小 ZSet**，让它永远只有 N 个成员——key 就小了，1 亿人也不怕。

**最朴素的做法（先理解这个）**：每当一个用户分数变化，就两步走——

```bash
# 第1步：把这个用户以新分数放进榜（已在榜就更新，不在就加入，此刻榜里可能变成 N+1 个人）
> ZADD rank:top  1050  userA
# 第2步：把排在 N 名之后的人裁掉，让榜恢复成只有 N 个
> ZREMRANGEBYRANK rank:top  0  -(N+1)
```

**`ZREMRANGEBYRANK` 怎么裁?先弄懂"排名下标"**：ZSet 默认**从低到高**给成员编号(rank)，最低分是第 0 名；负数从尾部(最高分)倒数，-1 是最高分、-2 是第二高……`ZREMRANGEBYRANK key 起 止` 就是**删掉排名在 [起, 止] 区间的成员**。

**用 N=3 走一遍**就懂了。假设要留 Top 3，但某次 ZADD 后榜里有了 4 个人，分数从低到高是：
```
分数:   100    200    300    400
从低到高rank: 0      1      2      3
从尾部倒数:  -4     -3     -2     -1
```
我们要**留下最高的 3 个(200/300/400)、删掉最低的(100)**。执行 `ZREMRANGEBYRANK rank:top 0 -(3+1)` 即 `0 到 -4`：
- 起=0 → 最低那个(100)
- 止=-4 → 从尾部倒数第 4 个，正好也是 100
- 于是只删掉 100，剩下 200/300/400。✓

再看裁掉两个的情况：榜里 5 个人(100/200/300/400/500)，要留 Top 3：`ZREMRANGEBYRANK rank:top 0 -4`，删的是 rank 0~(-4) = 100 和 200，剩 300/400/500。✓

所以 **`-(N+1)` 的含义是"从最高分倒数第 N+1 个"**——从最低分一路删到"第 N 名的下一个"，剩下的正好是 Top N。不管榜里临时多了几个，这一句都能精确削回 N 个。

**优化（避免每次都写这个热点 key）**：1 亿人频繁变分，如果每次都 ZADD+裁剪，这个 key 会被写爆。所以先判断"够得着榜吗"——读出当前**第 N 名的分数(入榜门槛)**：`ZRANGE rank:top 0 0 WITHSCORES`（取从低到高第 0 名，即榜里最低分=门槛）。新分数**超过门槛**(或榜还不满 N 个、或这人本来就在榜上)才执行上面两步；够不着就直接忽略，根本不碰这个 key。

**前提**：这套"只存 Top N"假设**分数只增不减**(积分榜、累计榜都是)。如果分数会降，榜上某人掉下去后，该由谁递补进 Top N？——那个"第 N+1 名"根本不在这个小榜里，无从得知，就得用思路三(分桶)或离线全量重算。

代价：榜外的 99.99% 用户查不到精确名次——这就引出思路二（直方图算近似排名）。

**追问：每次点赞都 ZADD + 裁剪（还要 Lua 保原子），不会把榜写爆吗？**（以"点赞最多的 10 个视频"为例）会，但**生产不会每个赞都更新榜**——要把"计数"和"维护榜单"拆开：
- **先澄清这俩命令不慢**：榜只有 10 个成员，ZADD/ZREMRANGEBYRANK 都是 O(logN)≈常数，Lua 也极小，单次开销可忽略。真正的问题是 ① 爆款视频每秒上万赞，每赞都写榜 = **每秒上万次写挤到同一个 key**（单分片热点）；② 它已经第 1，再多赞顺序没变，**绝大多数更新是无用功**。
- **点赞计数走 `INCR`**（这才是赞数源头），它本身是热点 key，用本地合并/key 打散抗，**和榜单无关**。
- **榜单是"派生视图"，异步维护**：
  - **定时刷新（最简单）**：后台每隔几秒把候选视频的当前赞数同步进 Top10 ZSet、裁剪一次。"最热10视频"晚几秒更新无人在意，把"每秒上万次写"压成"几秒一次"。候选集还可只取近 24h 活跃视频。
  - **门槛过滤 + 去抖（要更实时）**：维护第 10 名赞数当门槛，视频 INCR 后只有超过门槛（或已在榜）才更新榜，99% 的视频够不着、根本不碰这个 key；已在榜的爆款加去抖（同一视频最多每秒更新榜一次）。
- **原子性其实要求不高**：榜是派生视图，ZADD 写绝对计数，并发下顶多写个稍旧值、下次刷新自纠，不影响最终正确——所以未必需要 Lua，定时刷新用 pipeline 批量做即可。
- **一句话**：点赞用 INCR 扛高频，榜单当成"每隔几秒重算的派生视图"异步维护，别把榜单更新挂在点赞主路径上。

**思路二：用"分数直方图"算排名/百分位（亿级查排名的关键技巧）**
大多数业务其实不需要给第 8000 万名算出"精确第几名"，只要告诉他"**你击败了全国 95% 的玩家**"——这是**百分位**，不是精确排名。做法：维护一个**直方图**（每个分数有多少人），而不是给每个人排序：
- 用一个 ZSet 或 Hash 存 `分数 → 该分数的人数`。它的条目数 = 不同分数的个数（比如分数范围 0~10万，最多 10 万条），**和用户数无关**，从"1 亿成员"变成"几万条"，小了几个数量级。
- 用户从 s1 分变到 s2 分：`直方图[s1]--`、`直方图[s2]++`，O(1) 更新。
- 我的近似排名 = 比我分高的总人数 + 1 = 直方图里所有 > 我分数的人数之和；我的百分位 = 比我分低的人数 / 总人数。
这就是"亿级排名"的真实解法：**头部用精确 ZSet，全员排名/百分位用直方图近似算**。

**思路三：分桶（需要全员相对精确时）**
按分数段拆成多个 ZSet：`rank:bucket:0`（0-999 分）、`rank:bucket:1`（1000-1999）……每个桶是独立 key、可落在不同分片，单个 key 也变小。
- 取 TopN：从最高分的桶开始取，不够再往下一个桶取。
- 查个人排名 = 桶内 `ZREVRANK` + 所有更高分桶的人数之和（各桶 `ZCARD` 相加）。
- 坑：分数分布通常**极不均匀**（低分人巨多、高分人很少），等宽分桶会让低分桶又成大 key，需要在低分段切得更细或动态分桶；跨桶的同分排序也更麻烦。所以分桶一般也只对头部要求精确、长尾容忍近似。

**一句话总结**：亿级榜的核心认知是"**别真给 1 亿人排序**"——头部精确（Top N 小 ZSet）、全员近似（直方图百分位），需要全员相对精确才上分桶。

### 离线 / 准实时方案展开（复杂榜单的正解）

前面 ZSet 的方案有个隐含前提：**排名分数能简单累加**（`ZINCRBY` 加一下就行）。但很多榜的"热度分"根本不是简单累加的，这时 Redis 增量维护就力不从心，要靠离线/流式计算。

**什么时候必须走离线/准实时：**
1. **算分逻辑复杂**：热门榜的分数 = 播放量×a + 点赞×b + 评论×c + 完播率×d + **时间衰减**，多因子加权，没法用一个 `ZINCRBY` 维护。
2. **要时间衰减**：热门榜不能让老内容靠总量永远霸榜，得让热度随时间冷却。常用类似 Hacker News 的公式 `分数 = 热度 / (发布距今小时数 + 2)^重力因子`——这种每时每刻都在变、依赖"当前时间"的分数，**只能定时重算**，不可能增量累加。
3. **数据量大且不要求秒级**：日榜、周榜、热搜榜通常分钟级/小时级更新就够，"准实时"完全可接受。

**典型链路（计算和存储分离）：**
```
用户行为(播放/点赞/评论)
   │ 埋点上报
   ▼
日志 → Kafka ──┬─→ 离线批：Spark/Hive 定时(小时级)拉全量，按复杂公式算分、排序、取TopN
               └─→ 准实时流：Flink 消费流，用时间窗口(如滑动1小时)持续算"近1小时热度"
   │
   ▼ 算好的榜单(TopN + 分数)回写
Redis ZSet / DB  ←── 服务层从这里读，又快又是算好的结果
```
关键认知：**离线 ≠ 不用 Redis**。重的计算放离线/Flink 做，**算完的榜单结果照样写回 Redis ZSet 供前台快速查询**。两者是"后厨算账、前台出餐"的分工，不是二选一。

**离线批 vs 准实时流，怎么选：**
| | 离线批（Spark/Hive） | 准实时流（Flink） |
|---|---|---|
| 延迟 | 小时级 / T+1 | 秒级~分钟级 |
| 适合 | 昨日榜、历史总榜、月榜 | 实时热搜、近1小时热门 |
| 算分能力 | 最强，能跑很重的全量逻辑 | 窗口聚合，增量算 |
| 代价 | 延迟大 | 资源常驻、运维复杂 |

**进阶：Lambda 架构**。要"实时 + 准确"兼得时，常用"**离线批算准确基准 + 实时流补最近增量**"，前台把两者合并展示——既有离线的全量准确，又有流式的最新热度。这是大厂热榜的常见形态。

**一句话**：能简单累加的榜用 Redis ZSet 实时维护；带复杂加权/时间衰减的榜用 Flink/Spark 算、结果回写 Redis 服务——别硬用 ZSet 去表达算不动的分数。

## 3. Feed 流 / 关注时间线（推拉模型，高频）

**类比**：你关注了 100 个博主，怎么生成你的首页时间线？两种极端思路——**报纸投递（推）** vs **自己去报刊亭买（拉）**。

**① 推模型（写扩散 / push）**——博主发一条，**主动推**到所有粉丝的"收件箱"。
- 类比：报社印好报纸，主动投递到每个订阅者家门口。
- 读快（用户直接读自己的收件箱），但**写爆炸**——大 V 发一条要写 1000 万个粉丝的收件箱（写扩散）。且存储冗余大。
- 适合：粉丝少的普通用户、关系链稳定的场景。

**② 拉模型（读扩散 / pull）**——博主发的存自己"发件箱"，用户刷首页时**实时去拉**所关注的所有人的最新内容再聚合排序。
- 类比：你自己跑去报刊亭，把关注的 100 份报纸各买一份回来拼着看。
- 写轻（只写自己），但**读爆炸**——每次刷首页要拉 100 个人的内容实时聚合，慢。
- 适合：关注数少、发文频繁的场景。

**③ 推拉结合（实际方案）**——
- **普通用户用推**（粉丝少，写扩散成本低，读体验好）。
- **大 V 用拉**（避免一条微博写千万收件箱）——粉丝刷首页时，把"推模型收件箱里的普通博主内容" + "实时拉取的大 V 最新内容"合并。
- 类比：小报社挨家投递（推），大报社你自己去买（拉），首页是两者的拼合。
- 微博、抖音等都是这种混合模式。

**追问：怎么界定大 V？** 按粉丝数阈值（如 > 10 万）动态划分,且可结合活跃度。

**收件箱用什么存？（这才是 Redis 容易被滥用的点）** 推模型的"收件箱"是一个个有序列表，Redis List/ZSet 读快，但**全量存 Redis 内存成本极高**（亿级用户 × 每人几百条）。实际分层：**热数据（最近几天、活跃用户的收件箱）放 Redis**，**全量历史落 HBase/Cassandra 这类海量列存**（按 userId 做 rowkey，天然适合"读某人的时间线"）。所以推拉模型是**架构思路**，存储不绑定 Redis——Redis 只做热数据加速层。

## 4. 签到系统（连续签到）

**类比**：日历上打勾。一个月 31 天，每天一格，签了就涂黑。用什么存最省？

**方案：Bitmap（位图）**。Bitmap 本质是一长串二进制位，每位只存 0 或 1，用"第几位"来表示"第几天"。`key = sign:userId:202606`（某用户某月一个 key），第 N 天签到就 `SETBIT key N 1`（把第 N 位设成 1）。
- 一个用户一个月只占 31 bit ≈ 4 字节，1 亿用户一个月也才几百 MB。比每天存一条记录省几个数量级。
- 查某天是否签到：`GETBIT key N`（读第 N 位是 0 还是 1）；查当月签到总数：`BITCOUNT key`（数一共有多少位是 1）。

**追问：怎么算"连续签到天数"？** 用 `BITFIELD`（一次性读出一段连续的位）把整月的位取出来，或从今天往前逐位 `GETBIT` 检查直到遇到 0。连续签到送奖励的活动就靠这个。

**追问：为什么不用 Set 存签到的日期？** Set 存 31 个整数，每个 int 至少几十字节（含编码开销），比 Bitmap 的 31 bit 浪费几十倍。Bitmap 的前提是"用稠密的整数偏移"，签到天数（1~31）正好是稠密的。

**什么时候不用 Redis Bitmap**：如果签到记录要**长期留存、还要按日期查明细/做报表**（比如"查某用户全年每天的签到时间"），那 Bitmap 只存"签没签"不够，该用 **MySQL 签到流水表**（userId, date, sign_time），一天一行，查询和统计都灵活。Bitmap 的优势场景很专一：**只关心"签没签/连续几天"、用户海量、不需要明细**。两者也可组合：MySQL 存流水做持久化和报表，Redis Bitmap 做"是否已签/连续天数"的实时判断。

## 5. 附近的人 / 附近的店

**类比**：找你周围 1 公里内的奶茶店。地球是二维的（经纬度），怎么把二维的"远近"变成数据库能快速查的一维问题？

**核心思路（与具体工具无关）**：GeoHash——把二维经纬度通过"不断二分、交错取位"编码成一维字符串，**前缀相同的编码在空间上相邻**。于是"找附近"就变成"找编码前缀相同的"，把二维最近邻降维成一维范围查询。类比：把地图切成棋盘格并给每格编号，编号相邻的格子地理上也相邻，找附近就是找编号相近的格子。**这套思路是地理检索的通用底层，下面几种实现都基于它或类似原理：**

**方案选型（这题尤其不该只答 Redis）**：
- **Redis GEO**：`GEOADD key 经度 纬度 名称`（存一个坐标点）、`GEOSEARCH key ... BYRADIUS 1 km`（查指定中心 1 公里内的所有点），底层就是 ZSet + GeoHash（把坐标编码成一个分数存进 ZSet）。适合**动态位置、高频更新**（如"附近的人"，每个人实时移动）。
- **MySQL 空间索引（Spatial Index / R-Tree）**：用 `POINT` 字段 + `ST_Distance` 函数。适合数据量不大、本来就在 MySQL、不想引入额外组件的场景（如"附近的门店"，店铺位置基本不变）。
- **PostGIS（PostgreSQL 地理扩展）**：专业 GIS 能力最强，支持复杂空间运算（多边形、路径、面积），地图类业务首选。
- **Elasticsearch geo_point**：已经在用 ES 做搜索时，顺带做"附近 + 关键词 + 筛选"的复合查询最方便（如"附近 1 公里、评分 4 分以上的火锅店"）。
- 一句话选型：附近的**人**（高频动）→ Redis GEO；附近的**店**（静态、少量、要复合筛选）→ MySQL 空间索引 / ES。

## 6. 短链系统（短网址）

**类比**：把一长串身份证号映射成一个简短的会员卡号，扫卡号就能查到完整信息。`长URL → 短码`，访问短码时 302 跳转回长 URL。

**核心问题：怎么生成短码？**

**① 发号器 + 进制转换（主流）**——
- 用分布式 ID 发号器生成一个全局唯一的自增数字 ID，再把这个十进制 ID 转成 **62 进制**（0-9 + a-z + A-Z）。
- 为什么 62 进制：6 位 62 进制能表示 62^6 ≈ 568 亿个短码，足够用，且短。
- 类比：把"第 100000000 号"这个长数字，换成更紧凑的"进制表示"，像把 1000000 秒说成"11 天 13 小时"一样压缩位数。

**② 哈希（如 MurmurHash）**——对长 URL 哈希取短码。问题：**哈希冲突**（不同 URL 得到相同短码），要处理冲突（加盐重试）。

**关键流程**：
- **写**：长 URL → 发号 → 转 62 进制 → 存映射（`短码 → 长URL`）到 DB + 缓存。
- **读（高频）**：短码 → 查 Redis 缓存 → 命中直接 302 跳转，未命中查 DB 回填。读多写少，缓存命中率极高。

**追问：为什么用 302 而不是 301？** 301 永久重定向会被浏览器缓存，导致后续请求不再经过你的服务器，**统计不到点击量**；302 临时重定向每次都经过服务器，能统计 PV/UV。但 302 服务器压力大些——看是否需要统计来权衡。

## 7. 分库分表 + 深分页

**类比**：一个图书馆书太多（单表上亿行），一栋楼装不下也找不快，于是**按类别分到不同楼（分库），每栋楼再按字母分到不同楼层（分表）**。

**什么时候分**：单表数据量过大（经验值千万级开始考虑，但更要看实际查询性能）导致查询慢、索引臃肿。**先穷尽其他手段**（加索引、读写分离、归档冷数据、上缓存），分库分表是最后选择——因为它带来一堆新麻烦。

**怎么分（分片键 sharding key 的选择是关键）**：
- **范围分片**：按时间/ID 区间。简单、易扩容，但有热点（最新的库最忙）。
- **哈希分片**：按 `userId % N`。数据均匀，但扩容要迁数据（N 变了取模结果全变，用一致性哈希缓解）。
- 分片键要选**最高频的查询维度**——电商按 userId 分（用户查自己订单最频繁），而不是按 orderId。

**分库分表带来的新问题（面试重点）**：
1. **跨库 join 没了**——只能改成多次查询在应用层拼，或做数据冗余/宽表。
2. **分布式 ID**——不能再用单表自增（见第二章）。
3. **跨库分页/排序/聚合**——查"全局第 100 页"要从每个库都取出来归并，很重。
4. **分布式事务**——跨库写需要分布式事务（见第二章）。

### 深分页问题（`LIMIT 1000000, 10` 为什么慢，怎么优化）

**先看懂这条 SQL 在干嘛**：
```sql
SELECT * FROM orders ORDER BY create_time LIMIT 1000000, 10
```
`LIMIT 1000000, 10` 的意思是：**跳过前 100 万行，取接下来的 10 行**（第一个数字是"跳过多少"offset，第二个是"取多少"）。如果每页 10 条，这就是在翻"第 10 万零 1 页"。翻头几页很快，翻到这么深就会慢到几秒——这就是深分页问题。

**核心原因：MySQL 没法"直接跳"到第 100 万行，只能从头一行行数过去。**

类比：一本书只有正文、没印页码，你跟管理员说"从第 100 万行开始给我读 10 行"。他没法直接翻到那里，只能**从第 1 行开始一行一行数**：1、2、3……数到 100 万，前面这 100 万行数完就扔，然后才读你要的 10 行。慢就慢在"数过并丢弃前 100 万行"这个纯浪费的过程。`LIMIT 1000000, 10` 实际让 MySQL 处理了 1000010 行，只为返回 10 行。**翻得越深、白扔的越多。**

**为什么有索引也救不了、甚至更糟？这要先懂两个概念：**
- **聚簇索引**：InnoDB 里，主键索引的叶子节点**直接存着整行数据**。按主键找，找到就拿到全部字段。
- **二级索引**（比如建在 `create_time` 上的索引）：叶子节点**只存 `create_time + 主键 id`**，没有整行数据。
- **回表**：用二级索引找到主键后，**再拿主键回到聚簇索引里捞整行**，这个二次查找叫"回表"。

现在 `ORDER BY create_time ... SELECT *` 走 `create_time` 二级索引时发生了什么：MySQL 沿着 create_time 索引顺序往下走，对走过的**每一行**都拿主键去**回表**捞整行——回表是随机 IO，很慢。结果：回表 1000010 次，再把前 100 万次的结果统统扔掉。**深分页的真正杀手是这上百万次无用的回表。**

**为什么"会被扔掉的行"也要回表？根子在 MySQL 的两层架构和它们的对话方式。**

MySQL 内部分成上下两层，各管各的：

- **Server 层**：解析 SQL、优化、执行。`LIMIT`、`ORDER BY`、JOIN、函数计算都在这层。**`LIMIT` 的"跳过/丢弃"逻辑在这里。**
- **存储引擎层（InnoDB）**：只负责按要求把数据从磁盘/内存取出来。**回表（拿主键去聚簇索引捞整行）在这里。**

两层之间通过一个固定的接口对话，协议是"**一次要一行**"——Server 层喊一声"给我下一行"，InnoDB 取一行还回去，如此循环。**关键就在这个协议的粒度上**：Server 层要么要"完整的一行"、要么要"只含索引列的一行"，这取决于查询要哪些列，而且**在扫描开始前就定死了**（Server 层会先把"我需要哪些列"告诉 InnoDB）。

`SELECT *` 要全部列 → InnoDB 每还一行都必须**回表**把整行凑齐才能交差。于是完整时序是：

```
┌─ Server 层 ───────────────┐         ┌─ InnoDB 存储引擎层 ──────────┐
│ 执行器：喊"给我下一行" ──────┼────────▶│ ① 沿 create_time 二级索引读下一条│
│                            │         │   (只有 create_time + 主键id)  │
│                            │         │ ② 要的是SELECT *→拿主键【回表】 │
│                            │◀────────┼─── 把凑齐的"整行"还回去          │
│ ③ LIMIT 计数器这时才判断：   │         └──────────────────────────────┘
│   还在前100万 → 丢弃         │     ↑
│   到第100万零1~10 → 留下     │     回表已经发生在②，是在③丢弃【之前】
└────────────────────────────┘
```

**注意①②（InnoDB 回表）发生在③（Server 层丢弃）之前。** InnoDB 根本不知道这行马上会被 Server 层扔掉，它只是按协议"老实凑齐整行交差"；Server 层拿到整行**之后**才数 `LIMIT`、决定丢还是留。所以前 100 万行全都是"InnoDB 先花力气回表凑齐 → 递上来 → Server 层一看在 offset 范围内、扔掉"。两层没法商量"这些反正要丢、就别回表了"——因为接口粒度是"一整行",而且"要不要回表"在开扫前就按"要哪些列"定死了,整趟扫描要么全回表(`SELECT *`)、要么全不回表(覆盖索引),没有逐行的小聪明。

**类比（后厨 vs 前厅，正好对应两层）**：InnoDB 是**后厨**（负责做菜=取数据、回表），Server 层是**前厅经理**（按 `LIMIT` 数数、决定上不上桌），传菜口的规矩是"**一次只能传一道做好的整菜**"。经理要第 100万零1~10 道，但后厨只能按顺序一道道把菜**完整做好**(回表)从传菜口递出来，经理收到才数数、把前 100 万道退掉。经理没法隔着传菜口喊"前 100 万道只报个菜名别真做"——因为传菜口的协议就是"整菜",而且开餐前已经告诉后厨"这桌要全套"(`SELECT *`)。

**延迟关联为什么破局**：它把里层查询改成只要 id —— `SELECT id ... ORDER BY create_time LIMIT 1000000,10`。id 和 create_time 二级索引里全有（**覆盖索引**），于是 Server 层开扫前告诉 InnoDB"我只要这两列"，InnoDB **整趟都不用回表**，相当于"只报菜名不下厨"，前 100 万行只在索引上轻轻扫过、便宜；挑出最终 10 个 id 后，外层再只回表这 10 行。**等于把回表挪到了 `LIMIT` 丢弃之后。**

**一句话记牢**：`LIMIT` 在 Server 层、回表在 InnoDB 层，两层一次一行地对话；InnoDB 先回表凑整行、Server 层后用 LIMIT 丢——所以丢弃救不回回表。而回表是 `SELECT *` 逼的，只查索引里有的列（覆盖索引）就整趟不回表。

**优化方案 1：游标 / 书签翻页（最推荐）**

不要用"跳过 N 行"，改成"**从上次的位置接着往后取**"。记住上一页最后一条的 id，下一页这样查：
```sql
SELECT * FROM orders WHERE id > 上次那一页的最大id ORDER BY id LIMIT 10
```

**为什么这个就快？关键对比：**
- `LIMIT 1000000, 10`——"跳过 100 万行"，没有任何值可供查找，MySQL 只能老老实实从头数 → 慢。
- `WHERE id > 1000000`——这是个**带值的查找条件**，索引可以像查字典一样**二分直接定位**到 `id=1000000` 的位置，从那里读 10 行就走 → 几乎 O(1)，不管翻多深都一样快。

类比：与其每次从第一行数起，不如上次读完夹一张**书签**，下次直接翻到书签处往后读 10 行。

缺点：只能"上一页/下一页"顺序翻，**不能直接跳到第 100 页**（因为你没存第 99 页末尾的 id）。但绝大多数 App 的信息流、下拉加载本来就是顺序翻，所以这是首选。

**优化方案 2：延迟关联（必须支持跳页时用）**

如果产品就是要"跳到第 N 页"，没法用书签，就尽量**砍掉那上百万次回表**：
```sql
SELECT * FROM orders o
JOIN ( SELECT id FROM orders ORDER BY create_time LIMIT 1000000, 10 ) t
ON o.id = t.id
```
拆成两步看：
- 里层 `SELECT id ... LIMIT 1000000,10`：只查 `id`。而 id 和 create_time **都在二级索引里**（这叫"覆盖索引"，要的字段索引里全有），所以**全程不用回表**，扫 100 万个索引项虽然还是要扫，但只在索引上扫、不碰整行数据，快得多，最后只拿出 10 个 id。
- 外层再用这 10 个 id 回表捞整行——**回表从 100 万次降到 10 次**。

它没消除"扫描 100 万个索引项"（offset 跳过避免不了），但消除了"回表 100 万次"这个大头。

---

**分库分表下更糟**：要查"全局第 10 万页",得**从每个库都取出前 1000010 行**再归并排序(因为不知道全局第 100 万名落在哪个库)，量翻几倍。所以分库分表后产品上更要避免"跳到第 N 页"，强制用游标式"下一页"翻页。
