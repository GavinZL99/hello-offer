# 消息队列（MQ）面试知识总结

> 主线：消息队列要解决什么问题 → 它靠什么机制解决 → 主流产品差在哪 → 不同场景怎么选。
> 每个点按"问题 → 方案 → 为什么不用别的 → 还有什么坑"往下挖，目标是被追问三层也接得住。风格沿用 Redis 文档：先类比建立直觉，再进技术细节。
> 注：消息"不丢/不重/顺序/积压"从**面试场景设计**角度的答法在《后端场景题.md》里有；本篇从**底层机制**角度讲，两者互补。

## 一、消息队列要解决什么问题

**一句话类比**：消息队列就是**快递驿站**。寄件人（生产者）不用亲自把包裹送到收件人手上、也不用管收件人在不在家，把包裹往驿站（Broker）一放就走；收件人（消费者）有空了自己来取。寄收双方被驿站隔开了——这一隔，就同时带来下面三个核心价值。

### 1. 解耦——寄件人不需要认识收件人

**问题**：系统直接调用会绑死。下单成功后要做：扣库存、加积分、发短信、推数据给风控……如果订单服务直接挨个调用这些下游，每加一个下游就要改订单服务的代码，任何一个下游挂了/变慢，订单服务跟着挂/慢。**上下游被硬绑在一起。**

**方案**：订单服务只往 MQ 发一条"订单已创建"的消息就返回；库存、积分、短信、风控各自去订阅这条消息、各干各的。订单服务**根本不知道有谁在消费**，新增/下线一个下游，订单服务一行代码都不用改。

**类比**：寄件人只管把包裹放驿站，驿站后面有几个收件人、收件人换了地址，跟寄件人毫无关系。

### 2. 异步——寄完就走，不等签收

**问题**：下单主流程里如果同步等"扣库存(50ms)+加积分(30ms)+发短信(200ms)"全做完才返回，用户要等 280ms，而其中发短信这种**非核心**操作不该拖慢主流程。

**方案**：核心操作（扣库存）同步做，非核心的（积分、短信）发个消息就返回，下游异步慢慢处理。用户响应从 280ms 降到几十毫秒。

**类比**：你把包裹交给驿站就能去忙别的，不用站在那儿等它送到、等对方签收。

### 3. 削峰填谷——驿站扛住爆仓

**问题**：秒杀、双十一瞬时流量是平时的几十倍，直接打到数据库会被打垮，但为这种偶发峰值去扩容一堆数据库又太浪费。

**方案**：洪峰请求先进 MQ 排队，下游按自己**舒服的速度**匀速消费。峰值被"蓄"在队列里，慢慢消化。

**类比**：双十一包裹爆仓，驿站先把包裹堆在仓库里，快递员按自己的节奏一车一车发，而不是让所有包裹同一秒挤上路。代价是收件人要多等一会儿（牺牲实时性换系统不崩）。

### 4. 引入 MQ 的代价（要主动讲，体现思辨）

MQ 不是免费的，它把"一次同步调用"换成了"一套异步链路"，引入三类新问题——**这恰好是第二章要逐个解决的**：
- **可靠性下降**：多了 MQ 这个环节，消息可能在"发→存→收"任一段丢；还可能重复、乱序。
- **一致性变弱**：异步意味着下游处理有延迟，系统从"强一致"变"最终一致"（用户下单成功但积分过几秒才到）。
- **复杂度和运维成本上升**：多一个要部署、要高可用、要监控的中间件；排查问题链路变长。

**所以选型第一问永远是**：这个场景**真的需要 MQ 吗**？能同步调用解决的简单场景，别为了用 MQ 而用——它解决的是"解耦/异步/削峰"这三个明确痛点，没有这些痛点就是徒增复杂度。

## 二、消息队列是怎么解决的（核心机制）

### 1. 基本模型：两种消息传递方式

**① 点对点（队列模型）**：一条消息**只能被一个**消费者消费。类比驿站的包裹——一个包裹只能被一个人取走，取走就没了。适合"任务分发"（一个任务只能被一个 worker 处理）。

**② 发布-订阅（Pub/Sub）**：一条消息可以被**多个**订阅者各消费一份。类比报纸——同一份内容投递给所有订阅者。适合"一个事件多方关心"（订单创建 → 库存、积分、短信都要收到）。

**现代 MQ（Kafka/RocketMQ）用"消费者组"把两者统一了**（见下文），既能广播又能分摊。

### 2. 关键概念（看懂这几个，MQ 就通了）

用一张图把概念串起来：

```
生产者们                     Broker（驿站集群）                      消费者们
Producer ──发送──▶  ┌─ Topic: order（订单这个主题/货架区）──┐
                    │  Partition 0: [msg][msg][msg]...    │◀─ 消费者组A（库存服务，3个实例）
                    │  Partition 1: [msg][msg][msg]...    │     组内3个实例分摊3个分区
                    │  Partition 2: [msg][msg][msg]...    │
                    └──────────────────────────────────┘◀─ 消费者组B（积分服务）
                                                              另一个组，再独立消费一整份
```

- **Broker**：MQ 的服务节点（驿站），负责存消息、收发消息。多个 Broker 组成集群。
- **Topic（主题）**：消息的逻辑分类（驿站里的一个货架区），如 `order`、`payment`。生产者发到某个 Topic，消费者订阅某个 Topic。
- **Partition（分区）**：把一个 Topic 的消息**拆成多条并行的队列**（货架区里的多个货架）。这是 Kafka/RocketMQ **高吞吐和并行消费的关键**——一个分区只能被组内一个消费者消费，所以分区数决定了并行度上限。
- **消费者组（Consumer Group）**：一组消费者协作消费一个 Topic。**组内分摊**（每个分区只发给组内一个实例，实现负载均衡/点对点），**组间广播**（每个组都能独立收到全量消息，实现 Pub/Sub）。这一个设计就把点对点和发布订阅统一了。
- **Offset（位点）**：消费者在分区里"读到第几条了"的进度记录（取件进度）。**消费者自己维护 offset**，所以可以重置 offset 重新消费（消息回溯），这也是 Kafka 的一大特点。

这几个概念怎么协作，分"写入侧"和"消费侧"两头看，就把 MQ 的核心机制串通了。

#### 写入侧：一条消息只落到一个分区

生产者发消息时，按规则**选定一个分区**就只写那一个：指定了 key（如订单 ID）就 `hash(key) % 分区数`——同一个 key 永远落同一分区，这正是"同一订单的消息有序"的实现；没指定 key 就轮询/粘性，均匀分散。

为什么不能写多个分区？因为顺序只在分区内保证，而且同一条消息落进两个分区，会被组内两个消费者各消费一次（重复）——**"一消息一分区"是 offset、顺序、不重复这一整套机制的前提**。

这里要分清三个都带"多"字、但只有一个真把数据复制成多份的概念：
- **分区**：一条消息只进一个，不复制。
- **副本（Replica）**：那一个分区会复制到多台 Broker（1 Leader + N Follower 热备），消息物理上在多机——但那是**同一个分区的备份**，不是多个分区，读写只走 Leader。（货架被拍照备份到别的仓库，还是"那一个货架"。）
- **消费者组**：多个组能各消费一份——但**消息物理上只有一份、在一个分区**，靠的是每个组各记自己的 offset、各读一遍同一分区。（同一个货架，库存组和积分组各派人走一遍各抄一份，包裹没被复制、只被看了两次。）

#### 消费侧：分区怎么分给组内的消费者

分区和消费者是 **N:1** 的关系——**一个分区只能被组内一个消费者消费，但一个消费者可以同时消费多个分区**（注意方向，别反）。所以**整个组合起来一定覆盖全部分区、消费到全量**。以 4 个分区为例：

| 组内消费者数 | 分配 | 结果 |
|---|---|---|
| 1 | 1 个消费者吃下全部 4 个分区 | 全量 ✅ |
| 2 | 各 2 个 | 全量 ✅ |
| 4 | 各 1 个 | 全量 ✅，并行度拉满 |
| 5 | 4 个各 1 个，第 5 个闲置 | 全量 ✅，但多的浪费 |

可见**分区数 = 组内并行度的上限**：消费者比分区少，就有人身兼多分区；比分区多，就有人闲着。（类比：一个快递团队必须包完所有货架，人少了一人多管几个，人比货架多就有人没活干。）

#### 消费组、应用、消费者实例怎么对应

- **一个应用 = 一个消费组**：同一应用的多个实例配**同一个 group id**，组成一个消费组、分摊该 Topic 的所有分区。这正是消费端**横向扩容**的方式——加实例 = 加消费者 = 多分摊分区、消费更快，直到实例数 = 分区数到顶。
- **多个应用 = 多个消费组**：不同应用用不同 group id，各自独立消费**全量**。例：库存服务 3 实例（同一 group id）合起来消费全量，积分服务是另一个组再独立收一遍全量。
- **消费者 ≈ 消费实例**，通常一个应用进程一个消费者，但别和"进程"画死等号：一个进程能起多个消费实例，一个消费者拿到分区后内部还能多线程加速处理（代价是打乱顺序、offset 提交变复杂——所以**分区才是并行分配的最小单位**，线程只是分区内部的加速）。

把上面串成一条因果链就记牢了：**因为"分区是组内分配的最小单位、且一个分区只归一个消费者"，才同时得到三个结果——组内不重复消费（一分区一消费者）、能横向扩容（加实例多分摊分区）、有并行上限（实例多于分区就闲置）。** 这三件事是同一个设计的三个推论。

#### 分区数怎么定（不是越多越好）

分区多 → 并行度高、吞吐大，但也有代价：① 每个分区是一组文件，太多会拖慢 Broker（文件句柄多、顺序写退化成随机写）；② Rebalance（重平衡）更慢；③ 端到端延迟可能上升。按目标吞吐和消费者数量规划，别盲目设几千个。

### 3. 为什么 Kafka 能那么快（经典深挖题，类比 Redis 的"为什么快"）

Kafka 单机吞吐可达百万级。核心思路一句话：**把"写磁盘"和"传数据"这两件最慢的事，开销榨到极致。** 下面五点，①② 让"写入"快，③ 让"读取/发送"快，④⑤ 让整体能批量和水平扩展。

#### ① 顺序写磁盘 —— 磁盘没你想的慢，慢的是"随机"

很多人以为"磁盘一定比内存慢"。不准确——**磁盘慢的是"随机访问"，顺序访问其实很快**。

为什么随机慢？机械硬盘读写要把磁头**物理移动**到目标磁道（寻道，毫秒级），随机写每条数据都要重新移一次磁头，时间全耗在"找位置"上；SSD 没有磁头，但随机写也有寻址和写放大开销。而顺序写**不挪位置、一直往后写**，省掉了所有寻找。两者差距有多大？随机写大约每秒几百次（IOPS），顺序写可达每秒几百 MB——**差几个数量级**。

Kafka 正是吃这个红利：每个分区是一个 **append-only log（只追加日志）**，消息**只往文件末尾追加，永不修改、不插入中间**，所以天然是纯顺序写。
类比：图书管理员上架——随机写 = 每本书都要按编号找到对应位置插进去（来回跑、找空位）；顺序写 = 所有书直接摞到最后一堆上（走到队尾放下就走）。

#### ② Page Cache —— "写磁盘"其实大多是写内存

**Page Cache 是操作系统在内存里给文件留的一块缓存。** Kafka 没有自己在 JVM 堆里搞缓存，而是直接用它：

- **写**：Kafka 写消息，其实是写进 page cache（内存，极快）就返回了，**真正刷到磁盘由 OS 攒一批择机异步做**。所以"写磁盘"绝大多数时候是写内存。
- **读**：消费者要读的，往往是"刚写进来不久"的新消息，这些还在 page cache 里，**直接内存命中、根本不碰磁盘**。

**为什么不用 JVM 堆自己缓存？** ① 堆里放几十 GB 缓存会被 GC 扫得卡死；② Java 对象有额外内存开销，同样数据占得更多；③ 进程重启堆缓存就没了，而 page cache 在**操作系统层**，Kafka 重启它还在。
和 ① 的关系：Kafka 顺序写的对象就是 page cache，OS 之后也顺序刷盘——两个优化叠加。
代价：page cache 里还没刷盘的数据，机器断电会丢——所以才需要多副本兜底（见可靠性一节）。
类比：page cache = 前台的"快取台"，东西先放台上（秒回），后台再批量搬进仓库；要拿的东西大多还在台上，不用进仓库。

#### ③ 零拷贝 —— 数据从磁盘到网卡，不在应用内存里绕一圈

消费者来拉消息时，Kafka 要把文件里的数据发到网络。**传统做法要拷贝 4 次、在用户态/内核态之间来回切换 4 次**：

```
磁盘 ─DMA→ ①内核读缓冲(page cache) ─CPU→ ②用户态应用缓冲
                                              │ ─CPU→ ③内核socket缓冲 ─DMA→ ④网卡
（数据进了应用内存又原样出去，②③两次 CPU 拷贝纯属白搬；read()和write()两个系统调用来回切内核态）
```

**零拷贝（`sendfile` 系统调用）**：让数据**直接从内核的 page cache 送到网卡**，根本不进用户态应用内存，CPU 也不参与搬运（现代网卡用 DMA 直接取）。拷贝从 4 次降到 2 次（且都是 DMA、不耗 CPU），上下文切换从 4 次降到 2 次。

**为什么 Kafka 能用零拷贝？** 因为它发消息时只是把文件字节**原样转发**到 socket，不需要在应用里逐条解析/加工消息——正好符合 sendfile"只搬运、不加工"的前提。
类比：传统做法 = 包裹从仓库先搬进办公室、你看一眼、再搬出去装车（白搬两趟）；零拷贝 = 仓库直接把包裹装上门口的快递车。

#### ④ 批量 + 压缩

生产者攒一批消息一次性发、Broker 一批落盘、消费者一批拉取。把"每条消息一次系统调用 / 一次网络往返 / 一次磁盘 IO"这些**固定开销摊薄到一整批上**——批越大，单条分摊的开销越低。再对整批做压缩，网络和磁盘占用进一步下降。

#### ⑤ 分区并行

一个 Topic 拆成多个分区、分布在多台 Broker 上，多个生产者/消费者并行读写，**吞吐随分区数水平扩展**。前四点榨干的是"单条链路"的效率，这一点提供"加机器就能更快"的扩展性。

#### 串起来记

- **写入快** = 顺序写（不寻道）+ page cache（写内存）+ 批量。
- **读取快** = page cache 命中（新消息在内存）+ 零拷贝（不绕用户态）+ 批量。
- **规模** = 分区并行水平扩展。
- 一句话：**Kafka 没有黑魔法，就是把磁盘和网络这两个最慢环节的开销，用"顺序、内存、零拷贝、批量"四招榨到极致。**

### 4. 推（Push）还是拉（Pull）？

消息从 Broker 到消费者，有两种方式：

- **Push（RabbitMQ）**：Broker 主动把消息推给消费者。**延迟低**（来了就推），但**消费者可能被压垮**——Broker 不知道消费者处理得过来不过来，推太快就把慢消费者打挂（需要靠 QoS/prefetch 限流）。
- **Pull（Kafka/RocketMQ）**：消费者按自己的节奏主动来拉。**消费者不会被压垮**（自己控制速度），天然适合削峰和大数据吞吐；缺点是没消息时要轮询（用"长轮询"——拉不到就 hold 住一会儿再返回，缓解空轮询）。

**类比**：Push 是驿站主动送货上门（快，但你家放不下也得收）；Pull 是你自己按节奏去取（不会被塞爆，但要自己跑一趟）。**大吞吐场景几乎都用 Pull。**

### 5. 可靠性：消息怎么不丢（从机制角度，分三段）

消息从生产到消费经过三个环节，**每段都可能丢，每段都有对应机制**（类比寄快递：你交寄前、驿站中转、派送给收件人）：

- **① 生产端**：消息没成功到 Broker。机制——**发送确认（ack）**：Kafka `acks=all`（要求消息写入所有同步副本才算成功）、RocketMQ 同步发送、RabbitMQ 的 confirm 机制；发送失败就重试。
- **② Broker 端**：消息到了但还没落盘/还没复制就宕机。机制——**持久化刷盘 + 多副本**：Kafka 靠 ISR（同步副本集合）+ `replication.factor≥2` + `min.insync.replicas≥2`；刷盘有同步刷盘（可靠但慢）和异步刷盘（快但宕机可能丢一点）之分。
- **③ 消费端**：消费者拿到消息还没处理完就崩了，但 offset 已经提交（Broker 以为消费成功）。机制——**处理完再手动提交 offset/ACK**，而不是拿到就自动提交。

**追问：三段都保了能 100% 不丢吗？** 接近，但代价是性能（同步发送、多副本、同步刷盘、手动提交都更慢）。**要不要全保看业务**：日志可丢一点换吞吐；交易消息必须全保。这又回到"可靠性 vs 性能"的取舍。

### 6. 可靠性：消息重复与顺序（机制要点）

- **重复**：为了不丢就要重试，重试必然带来重复。**业界共识是"至少一次（at-least-once）投递 + 消费端幂等"**——MQ 很难做到真正的"恰好一次"。消费端用业务唯一 ID（msgId/订单号）查去重表或 Redis 来幂等。（Kafka 有"幂等 Producer + 事务"能在 Kafka 内部做到 exactly-once，但跨到外部系统时仍需业务幂等。）
- **顺序**：**分区内天然有序**（顺序写、顺序读）。要让相关消息有序，就把它们用同一个 key（如同一订单 ID）路由到**同一个分区**（分区间并行、分区内有序）。**全局有序**要所有消息进一个分区、单消费者串行——吞吐极低，几乎不用。RocketMQ 还提供专门的"顺序消息"支持。

### 7. 高级特性：延迟消息、死信、事务消息

这几个是**业务型 MQ（RocketMQ）相对 Kafka 的差异化能力**。延迟消息和死信先一句话带过，事务消息最绕、最常考，单独展开。

- **延迟消息**：发一条"N 分钟后才可被消费"的消息。用于订单超时取消等。RocketMQ 原生支持（4.x 固定延迟级别、5.x 任意延迟）；RabbitMQ 用 TTL+死信队列变相实现；Kafka 无原生支持（需外部时间轮/额外 topic）。
- **死信队列（DLQ）**：消息反复消费失败（超过重试次数）后，挪到一个专门的队列，避免它堵住正常消费、也便于人工排查。RocketMQ/RabbitMQ 原生支持，Kafka 需自己实现。

#### 事务消息：让"本地事务"和"发消息"一起成功或一起失败

**它解决什么问题**：下单时，订单服务要做两件事——① 在自己的库里**写订单（本地数据库事务）**，② 发一条"订单已创建"消息让**下游扣库存**。这两件事分属数据库和 MQ 两个系统，没有统一事务，naive 地做一定出错：

- **先写库、再发消息**：库写成功了，发消息时网络抖动/MQ 挂了——**订单建了但消息没发出去**，下游永远不扣库存（少发）。
- **先发消息、再写库**：消息发出去了（下游已扣库存），但本地写订单失败回滚了——**库存扣了但订单不存在**（多发）。

根因：两个独立系统的操作无法原子。事务消息就是来解决"本地事务 + 发消息"这一步原子性的。

**机制：半消息 + 二次确认 + 回查**。核心妙招是把"发消息"拆成两步，并加一个"兜底问询"：

```
① 生产者发"半消息"(half) ──▶ Broker 收下并持久化，但【对消费者不可见】(下游拉不到)
② Broker 回复"半消息已收到"
③ 生产者执行本地事务（写订单）
④ 生产者按结果发二次确认：
     成功 → COMMIT  ──▶ Broker 把半消息【转为可见】，下游开始消费（扣库存）
     失败 → ROLLBACK ──▶ Broker【丢弃】半消息，这条消息永不投递（下游根本不知道有过它）
⑤ 回查（兜底）：若 ④ 因生产者宕机/网络丢失，Broker 迟迟等不到二次确认，
     就【主动回调生产者】问"你那个本地事务到底成没成？"
     生产者查自己的 DB（订单建了没）→ 再答 COMMIT 或 ROLLBACK（可重试多次）
```

**为什么这样就一致了**：关键在"半消息"——它是一条**已被 Broker 存下、但暂不对外生效**的占位消息。正因为 Broker 知道"有一个待定的事务挂在这儿"，即使生产者在第④步前宕机，Broker 也能靠**回查**主动问出本地事务的真实结果，最终决定这条消息到底"转可见"还是"丢弃"。于是"消息发不发"和"本地事务成不成"**最终一定一致**。

**它不保证什么（重要，别答漏）**：事务消息只保证**上游**"本地事务 + 消息发出"的原子性，**不保证下游一定消费成功**。下游消费失败 MQ 会重投，所以下游仍要靠**消费幂等 + 重试**兜底。完整链路 = 事务消息（消息一定发出且和本地事务一致）+ 消费端幂等重试（下游最终处理成功）= **最终一致性**，不是强一致。

**和本地消息表的关系**：事务消息本质就是把"本地消息表"方案**内置进了 MQ**——用"半消息"代替你自己建的消息表，用"回查"代替你自己写的轮询补发任务，省了建表和写定时任务。（本地消息表方案见《后端场景题.md》分布式事务一节。）

**类比**：登记结婚——先去民政局把申请**存档但盖"暂不生效"章**（半消息，存了但对外不算数）→ 回去办婚礼（本地事务）→ 办成了回民政局确认生效（COMMIT）/ 黄了就撤销申请（ROLLBACK）。万一你办完婚礼后失联、没回去确认，民政局过段时间会**主动打电话问你"到底结没结"**（回查），按你的回答决定那份申请生效还是作废。

**横向对比**：RocketMQ 原生强支持（上面这套）；Kafka 的"事务"是为**多条消息 + offset 一起原子提交**（流处理 exactly-once）设计的，**不带"回查本地 DB 事务状态"机制**，做不了这个跨系统场景，要做还得自己上本地消息表；RabbitMQ 无原生事务消息。

## 三、主流产品对比

先一句话给每个产品定性，再上对比表。

- **Kafka**：为**高吞吐和流处理**而生。把消息当"日志流"，消费者自己管 offset，设计简单粗暴、吞吐极致。大数据生态的事实标准。
- **RocketMQ**：阿里从 Kafka 思想出发、为**电商/金融业务**补齐能力。在高吞吐基础上加了事务消息、延迟消息、顺序消息、消息轨迹、海量 Topic 支持——业务消息的首选。
- **RabbitMQ**：基于 AMQP 协议，**灵活路由 + 低延迟**。Exchange-Binding-Queue 的路由模型很灵活，push 模型延迟低，适合企业系统集成和复杂路由；但吞吐和堆积能力弱。
- **Pulsar**：新一代，**存算分离**（计算层 Broker 无状态 + 存储层 BookKeeper）。弹性扩展和多租户是强项，云原生友好。

**核心对比表：**

| 维度 | Kafka | RocketMQ | RabbitMQ | Pulsar |
|------|-------|----------|----------|--------|
| 定位 | 高吞吐日志/流 | 业务消息(电商/金融) | 灵活路由/企业集成 | 云原生/存算分离 |
| 语言 | Scala/Java | Java | Erlang | Java |
| 吞吐 | 极高(百万级/s) | 高(十万级/s) | 中(万级/s) | 高 |
| 延迟 | 毫秒级(批量略高) | 毫秒级 | **最低(微秒~毫秒)** | 毫秒级 |
| 消息模型 | 分区 log，pull | 类 Kafka，pull | Exchange 路由，push | 分区，存算分离 |
| 顺序 | 分区有序 | 分区有序 + 顺序消息 | 队列有序 | 分区有序 |
| 事务消息 | Kafka 内部 exactly-once | **强(半消息+回查)** | 无原生 | 支持 |
| 延迟消息 | 无原生 | **支持** | TTL+DLX 变相 | 支持(任意延迟) |
| 死信队列 | 无原生 | 支持 | 支持 | 支持 |
| 堆积能力 | **极强(磁盘)** | 强 | 弱(堆积性能差) | 极强 |
| 多 Topic | 几千分区后退化 | **支持海量 Topic** | 中 | 强 |
| 协议 | 自有 | 自有 | **AMQP/MQTT 多协议** | 自有 |
| 运维复杂度 | 中(依赖 ZK，新版去 ZK) | 中 | 低 | 高(组件多) |

**几个关键差异展开：**

- **Kafka vs RocketMQ**：思想同源（都是分区 + pull + 顺序写），RocketMQ 是"Kafka + 业务能力"。**Kafka 强在超高吞吐和流生态（Kafka Streams、Connect）**；**RocketMQ 强在业务特性（事务/延迟/顺序消息）和海量 Topic**。Kafka 一个 Broker 上 Topic/分区太多会因"顺序写退化成随机写"而性能下降，RocketMQ 用"所有消息写一个 CommitLog、再建消费队列索引"的设计扛住了海量 Topic。
- **RabbitMQ 的独特点**：Exchange 路由（direct/topic/fanout/headers 多种路由规则）极其灵活，能表达复杂的消息分发逻辑；push + 低延迟适合 RPC 式、小流量、要求快的场景。但它消息堆积在内存里，**大量堆积时性能急剧下降**——不适合削峰扛海量积压。
- **Pulsar 的存算分离**：Broker 不存数据、只做计算转发，数据存在 BookKeeper。**扩容 Broker 不用迁移数据**（无状态），扩存储独立加 BookKeeper 节点——弹性比 Kafka（分区和数据绑在 Broker 上、扩容要迁移）好。代价是架构组件多、运维更重。

## 四、技术选型（不同场景选什么）

**按场景对号入座：**

- **日志收集、监控数据、大数据管道、流处理** → **Kafka**。吞吐第一、生态最全（和 Flink/Spark/数仓无缝），消息能回溯重放。这是 Kafka 的绝对主场。
- **电商交易、金融、订单履约——要事务消息/延迟消息/顺序消息/金融级可靠** → **RocketMQ**。业务特性最齐全，国内互联网公司业务线的主流选择。
- **企业系统集成、复杂路由、RPC 式低延迟、小到中等流量、需要多协议（AMQP/MQTT）** → **RabbitMQ**。路由灵活、延迟低、上手快；但别拿它扛海量削峰。
- **云原生、多租户 SaaS、需要存算分离弹性伸缩、超大规模** → **Pulsar**。架构先进，但要扛得住运维复杂度。
- **轻量场景、已经有 Redis、能容忍极端情况丢消息、不想多引入组件** → **Redis Stream**（详见《Redis.md》消息队列一节）。适合小型异步任务，可靠性要求高就别用。

**选型的通用思考框架（比记结论更重要）：**
1. **先问要不要 MQ**：能同步调用解决就别上（回到第一章的代价）。
2. **吞吐量级**：百万级日志 → Kafka；十万级业务消息 → RocketMQ；万级企业集成 → RabbitMQ。
3. **要什么特性**：事务消息/延迟消息 → RocketMQ；复杂路由 → RabbitMQ；流处理生态 → Kafka。
4. **可靠性要求**：金融交易 → RocketMQ/Kafka（多副本同步刷盘）；日志可丢一点 → 怎么快怎么来。
5. **团队和生态**：已有大数据栈 → Kafka；阿里系/国内业务 → RocketMQ；运维力量弱 → 别选 Pulsar。

**万能表态**："没有最好的 MQ，只有最合适的。我会先确认是不是真需要 MQ，再按‘吞吐量级 + 必需特性 + 可靠性 + 团队生态’四个维度选——比如纯日志管道我选 Kafka，电商交易要事务和延迟消息我选 RocketMQ。"

## 五、面试快问快答清单

1. MQ 解决什么问题？→ 解耦、异步、削峰；代价是可靠性/一致性下降、复杂度上升。
2. 引入 MQ 带来什么新问题？→ 消息丢失、重复、乱序、积压，以及最终一致性。
3. Topic / Partition / 消费者组 / Offset？→ 主题分类 / 并行队列 / 组内分摊组间广播 / 消费进度。
4. 消费者组怎么同时实现点对点和发布订阅？→ 组内一个分区只给一个实例(点对点)，组间各收全量(发布订阅)。
5. Kafka 为什么快？→ 顺序写磁盘 + Page Cache + 零拷贝(sendfile) + 批量压缩 + 分区并行。
6. Push 还是 Pull？→ Kafka/RocketMQ 用 Pull(消费者不被压垮、适合大吞吐)，RabbitMQ 用 Push(延迟低)。
7. 消息怎么不丢？→ 生产端 ack 重试 + Broker 多副本同步刷盘 + 消费端处理完再提交 offset。
8. 为什么 MQ 做不到 exactly-once？→ 为不丢必须重试、重试必带重复；业界用"至少一次 + 消费幂等"。
9. 顺序怎么保证？→ 分区内有序；同一业务键路由到同一分区；全局有序要单分区(吞吐极低)。
10. 消息积压怎么办？→ 扩消费者(受分区数限制要同时扩分区)、消费者降级、紧急转存腾队列。
11. Kafka vs RocketMQ？→ 同源；Kafka 胜在超高吞吐和流生态，RocketMQ 胜在事务/延迟/顺序消息和海量 Topic。
12. RabbitMQ 适合什么？→ 复杂路由、低延迟、企业集成；不适合海量堆积削峰。
13. Pulsar 的特点？→ 存算分离，Broker 无状态、扩容不迁数据，云原生弹性好，但运维重。
14. 事务消息原理(RocketMQ)？→ 半消息(消费者不可见)→执行本地事务→提交/回滚→状态丢失则 Broker 回查。
15. 怎么选 MQ？→ 先问要不要；再按吞吐量级 + 必需特性 + 可靠性 + 团队生态四维度选。
