# Redis 面试知识总结（深挖版）

> 主线：Redis 要解决什么问题 → 它采用了什么方案 → 近些年怎么演进的 → 实际工作中怎么用。
> 每个知识点都按"问题 → 方案 → 为什么不用别的方案 → 还有什么坑"的链条往下挖，目标是被追问三层也接得住。

---

## 一、Redis 要解决什么问题

### 1. 根本矛盾：磁盘数据库扛不住高并发读

先把数量级说清楚，这是所有后续设计的根源：

| 操作 | 耗时量级 |
|------|---------|
| CPU 访问 L1 缓存 | ~1ns |
| 访问内存 | ~100ns |
| SSD 随机读 | ~100μs（0.1ms） |
| 机械盘寻道 | ~10ms |
| 同机房网络一次 RTT | ~0.5ms |

内存和磁盘之间差 **3~5 个数量级**。MySQL 即使有 buffer pool，也受限于 B+ 树的多层查找、锁、事务日志刷盘、连接模型（一个连接一个线程），单机一般几千 QPS。而互联网业务读写比普遍 10:1 以上，热点数据集中（二八定律），让 MySQL 硬扛读流量既浪费又危险。

**结论：需要把热数据放进内存，挡在数据库前面。** 这是 Redis 的第一身位——缓存。

### 2. 第二个问题：应用集群化之后，"单机内存"失效了

应用从 1 台变 N 台后，这些原来放本机内存的东西全都没了着落：

- **登录态**：单机 `HashMap<sessionId, User>` 就够；N 台机器时请求会被负载均衡到任意一台，必须有集中存储。
- **计数/限流**：单机 `AtomicLong` 就够；N 台机器各数各的，加起来就超限了。
- **互斥**：单机 `synchronized` 就够；跨进程的互斥需要一个所有节点都认的"裁判"。

**结论：需要一个高性能、所有实例可达、操作具备原子性的共享内存。** 这是 Redis 的第二身位——分布式协调的基础设施。注意第二身位对**原子性**的要求，直接决定了 Redis 选单线程模型（后面会呼应）。

### 3. 为什么是 Redis，不是别的

- **本地缓存（Caffeine）**：纳秒级，比 Redis 还快（Redis 还要一次网络 RTT ~0.5ms），但各节点数据不一致、容量受限、无法做跨节点原子操作。所以实践中它是**一级缓存**，和 Redis 组成多级缓存，不是替代关系。
- **Memcached**：多线程，纯 KV，value 只能是字符串。没有持久化（重启即空，缓存雪崩）、没有原生高可用、没有丰富数据结构（排行榜、去重、延迟队列都做不了）、多线程导致单 key 操作需要客户端自己保证原子。它只能做第一身位，做不了第二身位。
- **Redis**：数据结构服务器 + 可持久化 + 主从/哨兵/集群 + 单线程原子性。两个身位都能干。

一句话定位：**Redis = 高性能共享内存 + 数据结构服务器**，"缓存"只是它最常见的用法。

## 二、Redis 采用了什么方案

### 1. 为什么快 —— 先一口气答完，再逐层深挖

被问"为什么快"，先给这套全景答案，再挑一点往下钻（每点后文都有专门小节）：
1. **纯内存**：内存比磁盘快 3~5 个数量级，这是根本。代价是贵、断电易失、容量有限——Redis 后面一半的复杂设计（持久化、淘汰策略）都是在为"数据在内存里"擦屁股。
2. **单线程免锁 + IO 多路复用**：命令执行单线程，没有锁竞争和上下文切换、天然原子；再用 epoll 一个线程管住海量连接 → 详见下文《2. 单线程 + IO 多路复用》。
3. **高效数据结构**：每种类型按数据规模在"紧凑编码 / 专业结构"间自动切换 → 详见《4. 数据结构》。
4. **RESP 协议**：长度前缀的极简协议，解析快、二进制安全 → 详见《3. RESP 协议》。

**贯穿全节的前提**：Redis 的瓶颈在**网络和内存带宽、不在 CPU**——正因如此单线程才敢成立，省下锁和上下文切换的开销反而更划算。下面把后三个原因逐个展开（纯内存上面已说尽）。

### 2. 单线程 + IO 多路复用 —— 重点，往死里问的地方

#### 先把 socket 本身讲清楚：它到底是个什么东西

socket 不是网线、不是端口，它是**内核里的一个数据结构**——一条网络连接在操作系统里的"化身"，由四元组（源 IP、源端口、目的 IP、目的端口）唯一标识，**一条 TCP 连接 = 一个 socket 对象**。对用户进程来说，它表现为一个 **fd（文件描述符，就是个整数）**：每个进程有一张 fd 表（Linux"一切皆文件"），read/write 一个 fd，背后操作的就是 fd 指向的内核对象。

所以一个进程有几个 socket？**每条连接一个，要几个有几个**，上限是 fd 数限制（`ulimit -n`，默认 1024，生产服务器一般调到几十万）。客户端进程（比如 Java 应用）：连 Redis 的连接池 50 个连接就是 50 个 socket，加上 MySQL 池、下游 HTTP 连接，通常几十到几百个；服务端进程则是 **1 个监听 socket + N 个连接 socket**。以 1 万客户端连接的 Redis 为例：

```
Redis Server 进程的 fd 表：
fd=0,1,2    → stdin/stdout/stderr（fd 不全是 socket）
fd=3        → 监听 socket（0.0.0.0:6379）  ← 永远只有 1 个，不传数据
fd=4        → epoll 实例                   ← eventpoll 本身也占一个 fd
fd=5        → 连接 socket（客户端A）
fd=6        → 连接 socket（客户端B）
...
fd=10004    → 连接 socket（第 1 万个客户端）
```

监听 socket 很特殊：它不传输数据，只维护"已完成三次握手的连接队列"；`accept()` 从中取出一条连接、**创建一个新的 socket 对象、返回新 fd**。所以后面 epoll 监视这些 fd 时，"监听 socket 可读"= 有新连接该 accept 了，"连接 socket 可读"= 有命令来了，处理逻辑完全不同。用 `ls /proc/<pid>/fd`（Linux）或 `lsof -p <pid>` 可以亲眼看到一个进程的全部 fd。

每个连接 socket 的内部有三样核心的东西：**接收缓冲区、发送缓冲区、等待队列**。

```
用户进程（Redis Server）                                  用户态
┌────────────────────────────────────────────┐
│ fd 表:  fd=3        fd=5        fd=6  ...   │
└─────────┬───────────┬───────────┬───────────┘
══════════╪═══════════╪═══════════╪══════════════════════ 系统调用边界
          ▼           ▼           ▼                       内核态
   ┌────────────┐ ┌────────────────────────────┐
   │ 监听 socket │ │ socket（客户端A 的连接）      │
   │ 等新连接,    │ │                            │
   │ accept 从这 │ │ 四元组: 10.0.0.2:53301      │
   │ 里取        │ │     ↔  10.0.0.9:6379       │
   └────────────┘ │ ┌────────────────────────┐ │
                  │ │ 接收缓冲区 ◀── 网卡来的数据│ │
                  │ │ 发送缓冲区 ──▶ 待发出的数据│ │
                  │ │ 等待队列: [谁在等它就挂谁] │ │
                  │ └────────────────────────┘ │
                  └────────────────────────────┘
```

**`read(fd)` 不是去网卡读数据**，而是从这个 socket 的**接收缓冲区**往用户内存拷；`write(fd)` 也不是把数据发出去，只是拷进**发送缓冲区**，什么时候真正发走是内核协议栈的事。进程从头到尾不碰网卡。

**"阻塞"的本质**：`read()` 时接收缓冲区是空的，内核就把这个进程挂到该 socket 的**等待队列**上、让出 CPU 睡过去。"挂到等待队列"具体是三个动作（背景：进程在内核里也是一个数据结构 task_struct，有状态字段；CPU 调度器**只从运行队列里挑进程执行**，不在运行队列里的进程一纳秒 CPU 都不耗）：

```
read() 发现接收缓冲区为空，内核做三件事：
  ① socketA 的等待队列（一个链表）里记一个指向本进程的指针
  ② 进程状态 RUNNING → SLEEPING，并从 CPU 运行队列里摘掉
  ③ schedule() 让出 CPU

睡眠期间： CPU 运行队列: [进程X, 进程Y, ...]  ← 没有 Redis 进程，0 消耗
          Redis 进程冻结在 read() 内部的某一行内核代码上

数据到达： ④ 协议栈写入 socketA 接收缓冲区
          ⑤ 顺着 socketA 等待队列找到在等的进程 → 改回 RUNNING、放回运行队列
          ⑥ 下次被调度到，从冻结处继续：缓冲区有数据了 → 拷贝 → read() 返回
```

类比奶茶店叫号：餐没好（缓冲区空），店员把你的号记在本子上（等待队列记指针），你去睡觉（让出 CPU）；餐好了翻本子叫号（遍历等待队列唤醒），你醒来取餐（继续执行 read 后半段）。**等待队列的本质作用：让"数据"能反向找到"在等它的人"。** 所谓阻塞/非阻塞、select/epoll，全部是围绕"缓冲区空了怎么等、数据到了怎么通知"做文章——select 把自己挂到所有 socket 的等待队列上（哪个来数据都叫得醒它），epoll 的关键一招是把队列里的"进程指针"换成"回调函数"，后面马上展开。

注意等待队列是个链表，但这**不代表多个线程同时读一条连接是常态**——恰恰相反。链表设计是为了真实存在的多等待者场景：① **多个 worker 等同一个监听 socket**（Nginx 等多进程服务器都阻塞在同一个监听 socket 上 accept；新连接到达时早期内核会全部唤醒、只有一个抢到、其余白醒——著名的**惊群问题**，后来用 wake-one（等待队列项加 EXCLUSIVE 标记只唤醒一个）和 `SO_REUSEPORT`（每个 worker 各自一个监听 socket，内核按四元组哈希分流）治理）；② fd 表是进程级的，同进程的线程天然共享，fork 后父子进程也共享 socket，内核不禁止多线程读同一个 fd。但对**连接 socket** 这么干是反模式：内核只保证单次 read 原子，不保证消息完整——100 字节的命令可能被线程 A 读走前 60 字节、线程 B 读走后 40 字节（缓冲区是消费式的，每个字节只能被读走一次），一条消息撕成两半谁也解析不了。所以成熟框架全是"**一条连接固定归一个线程**"：Netty 的 Channel 终生绑定一个 EventLoop；**Redis 6.0 IO 多线程同理**——主线程把就绪 socket 分配给 IO 线程，每个 socket 同一轮只属于一个线程，这正是它不需要加锁的原因之一。

#### 数据到达的完整链路：内核怎么知道该唤醒谁

```
客户端A 发来 "GET name"
   │
   ▼
① 网卡收到数据帧
   │  DMA（不经过CPU，直接写内存）
   ▼
② 内存中的环形缓冲区 ──③ 硬中断──▶ ④ 内核协议栈解析 IP 头、TCP 头
                                      │
                                      │ 拿四元组查表：这是哪条连接的数据？
                                      ▼
                         ⑤ 数据放进 socket A 的接收缓冲区
                                      │
                                      ▼
                         ⑥ 看 socket A 的等待队列上挂着谁：
                            · 挂着睡眠的进程 → 唤醒它（BIO/select 的情况）
                            · 挂着一个回调   → 执行回调（epoll 的情况）
```

第 ⑤⑥ 步是理解一切的钥匙：**内核按四元组把数据"分拣"到对应 socket，然后处理这个 socket 等待队列上挂的东西**。BIO、select、epoll 的全部区别，就是"等待队列上挂的是什么、唤醒之后干什么"。

#### BIO 为什么不行

一个线程调用 `read()` 后只能睡在**一个** socket 的等待队列上，于是只能一个连接配一个线程：

```
线程1 ──read()阻塞──▶ [socketA 等待队列: 线程1]
线程2 ──read()阻塞──▶ [socketB 等待队列: 线程2]      1万连接 = 1万线程
线程3 ──read()阻塞──▶ [socketC 等待队列: 线程3]      每线程栈 ~1MB → 光栈 10GB
 ...                                                上下文切换 1~10μs/次 → CPU 全耗在切换
```

这就是经典的 **C10K 问题**。

#### 多路复用：一个线程监视所有 socket

核心问题只有一个：**内核怎么把"谁就绪了"告诉用户进程？** select 和 epoll 的全部区别就在这件事的做法上。

**select 的做法（挨个问）**：进程把自己**同时挂到所有被监视 socket 的等待队列**上睡觉，任何一个来数据都会把它唤醒——但醒来时**不知道是谁唤醒的**，只能全部扫一遍。

```
              ┌──▶ [socketA 等待队列: 本进程]
单线程 ─select─┼──▶ [socketB 等待队列: 本进程]   ← 挂在所有 socket 上
              └──▶ [socketC 等待队列: 本进程]
                          │
              任何一个来数据 → 唤醒进程
                          │
                          ▼
              醒来后: for fd in 全部N个fd: 你就绪了吗？   ← O(n) 扫描
              返回用户态后: 用户再扫一遍位图找就绪的 fd    ← 又一次 O(n)
```

完整的四宗罪：① 每次调用都要把 fd 集合（fd_set 位图，上限 FD_SETSIZE=1024——这就是 1024 限制的来源）**全量拷贝进内核**；② 内核 O(n) 遍历；③ 返回后用户态再 O(n) 遍历位图；④ 位图被内核改写，下次调用还得重新构造。poll 只是把位图换成数组解除了 1024 上限，**拷贝和两次遍历原封不动**。几万连接里往往只有几十个真有数据，绝大部分扫描是纯浪费。

**epoll 的做法（就绪的自己来登记）**：进程不再挂到每个 socket 上，而是只睡在 **eventpoll 这一个对象**上；每个 socket 的等待队列上挂的不是进程，而是**一个回调函数**——数据到达时，socket 通过回调把自己登记进就绪链表。

```
                        eventpoll 对象（epoll_create 创建）
                     ┌──────────────────────────┐
 epoll_ctl(ADD,fd) ──▶│ 红黑树: 所有被监视的 fd     │ fd 只在注册时拷贝这一次
 注册时顺便在 socket   ├──────────────────────────┤
 等待队列放一个回调     │ 就绪链表 rdllist           │◀─┐
                     ├──────────────────────────┤  │
                     │ 等待队列: [睡着的Redis进程]  │  │ ③ 回调把 socketA
                     └──────────┬───────────────┘  │   登记进就绪链表
                                │ ④ 唤醒            │
                                ▼                  │
                     epoll_wait 返回                │
                     只拿链表里的 fd，零扫描          │
                                                   │
[socketA 等待队列: 回调函数] ──① 数据到达──② 执行回调──┘
[socketB 等待队列: 回调函数]      （没数据的 socket 全程安静，无人理会）
[socketC 等待队列: 回调函数]
```

三个系统调用对应图上三步：`epoll_create` 建 eventpoll 对象；`epoll_ctl(ADD)` 把 fd 挂上红黑树（O(logN)，顺便去重）并在 socket 等待队列上放回调（ep_poll_callback）；`epoll_wait` 只收割就绪链表——**返回的全是就绪的，一个都不用扫**。

一句话对比：**select 是内核每次替你把所有人问一遍；epoll 是每个 socket 就绪时自己来登记。** 把 O(n) 轮询变成事件驱动的 O(1)。优势在"连接多、活跃少"的场景最明显——Redis 正是这种：几万个长连接，每轮真正有命令到达的只有几百个。

#### 三种模型放在一起看：差别到底在哪

**类比：老师收作业。** 60 个学生（连接）写作业，写完时间不定，老师（线程）要收（读数据）：

- **BIO** = 每个学生配一个专属老师盯着，没写完老师就在旁边睡。能用，但 60 个学生要 60 个老师——贵在人头。
- **select** = 一个老师管全班，每隔一会儿从第 1 排走到第 60 排挨个问"写完没？"。一轮问下来可能只有 2 个写完。班级越大每轮越慢，可白问的占绝大多数。
- **epoll** = 一个老师管全班，改规则："谁写完，谁自己把作业放讲台的筐里。"老师只看筐，筐空就睡。**写没写完，老师不用问任何人。** 筐 = 就绪链表，"自己送上来" = 回调。

**算笔账**：1 万个连接，某时刻只有 10 个有数据——

| | BIO | select | epoll |
|---|---|---|---|
| 需要几个线程 | 1 万 | 1 | 1 |
| 怎么发现那 10 个 | 各自线程被各自唤醒 | 1 万个全问一遍筛出 10 个 | 那 10 个自己登记了，直接拿名单 |
| 每轮成本 | 万级线程的内存+切换 | 拷 1 万 fd + 内核扫 1 万 + 用户再扫 1 万 | O(就绪数)=取 10 个 |

**回到内核视角，差别只在一处：socket 等待队列上挂的是什么、唤醒时带不带"是谁"的信息。**

```
BIO：    每个 socket 等待队列上睡着专属线程
         数据到 → 唤醒它 → 天然知道是自己那条连接（1对1，不存在"找"）
         代价：线程数 = 连接数

select： 一个进程挂在全部 1 万个 socket 的等待队列上
         任何一个来数据 → 进程被唤醒
         ★ 唤醒只是"推一把"，不附带信息——只知道"有事"，不知道"谁的事"
         → 只能 1 万个全扫一遍

epoll：  进程只睡在 eventpoll 一个对象上；socket 等待队列上挂的是回调
         数据到 → 回调执行 → ★ 把"我是 fd=5"写进就绪链表（信息被记下了）
         → 醒来直接看链表，零扫描
```

点破：**select 的唤醒丢失了"是谁"的信息，醒来只能全员排查；epoll 用回调在数据到达那一刻就把"是谁"登记进就绪链表，醒来直接拿名单。** 三者是递进关系：BIO 的问题是人太多 → select 解决人数（一人看全部）但留下"醒来不知道是谁" → epoll 让就绪者自己报到，补上最后一块。

**两个加分细节**：
- **LT vs ET**：类比闹钟——LT（水平触发）是"不起床就每隔一分钟再响一次"：只要缓冲区还有没读完的数据，每次 epoll_wait 都会再报告这个 fd；ET（边沿触发）是"只响一声，没听见拉倒"：只在"从无到有"的瞬间报告一次，必须一次性循环读到 EAGAIN，否则剩下的数据再也不会被通知。**Redis 用 LT**（实现简单不易丢事件），Nginx 用 ET（减少重复唤醒）。
- **辟谣**：网上流传"epoll 用 mmap 共享内存避免拷贝"是**错的**，epoll_wait 返回时就是正常的 copy_to_user，只是只拷就绪的那一小部分，本来就不大。

**Redis 的事件循环（ae 库）**：封装 epoll（Linux）/kqueue（macOS、BSD）/evport（Solaris）/select（兜底），按平台选最优。循环处理两类事件——**文件事件**（socket 可读/可写）和**时间事件**（serverCron：过期扫描、rehash 推进、持久化触发等）。每轮循环：算出最近一个时间事件还差多久到期 → **以这个差值作为 epoll_wait 的超时时间**（既不空转烧 CPU，也不会睡过头耽误定时任务）→ 处理就绪的文件事件（读命令 → 执行 → 写回复）→ 处理到期的时间事件。这就是单线程 Reactor 的完整形态。

#### 回到"单线程"：为什么敢、代价、到底指什么

**为什么命令执行敢用单线程？**
1. 瓶颈不在 CPU。纯内存操作微秒级就完成，单线程能跑 10w+ QPS，瓶颈在网络 IO 和内存带宽。
2. 单线程**免锁**。如果多线程执行命令，每个 key 甚至每个数据结构内部都要加锁，锁竞争 + 上下文切换的开销可能比多核收益还大（Memcached 就为此付出了复杂度）。
3. 单线程给了**天然的原子性**：`INCR`、`SETNX`、Lua 脚本不需要任何额外机制就是原子的——这正是第二身位（分布式锁/计数器）的根基。**如果 Redis 是多线程执行命令的，分布式锁的实现会复杂一个量级。**

**单线程的代价（要主动讲，体现思辨）**：任何一个慢命令都会阻塞所有请求。`KEYS *`、`HGETALL` 一个百万级 Hash、`DEL` 一个大 key、Lua 死循环——全队请求一起等。所以 Redis 的演进史里很大一块就是在拆这个雷（lazyfree、IO 多线程，见演进部分）。

**追问：那"单线程"到底指什么？** 严格说从来不是整个进程一个线程。命令执行始终单线程；但 2.6 起就有后台线程做 AOF fsync、关闭大文件；4.0 加了 lazyfree 线程；6.0 加了 IO 线程。准确表述是"**命令执行是单线程的事件循环**"。

### 3. RESP 协议 —— 看懂真实的字节流

RESP（REdis Serialization Protocol）是 Redis 在 TCP 之上的应用层协议。设计目标三条：实现简单、解析快、人可读。

**RESP2 的五种类型，由首字节区分**：

| 首字节 | 类型 | 线上字节流示例 | 含义 |
|--------|------|---------------|------|
| `+` | 简单字符串 | `+OK\r\n` | 状态回复 |
| `-` | 错误 | `-ERR unknown command\r\n` | 错误回复 |
| `:` | 整数 | `:100\r\n` | INCR、LLEN 等的返回 |
| `$` | 批量字符串 | `$5\r\nhello\r\n` | **带长度前缀**的二进制安全字符串；`$-1\r\n` 表示 nil |
| `*` | 数组 | `*2\r\n:1\r\n:2\r\n` | 元素可以是任意类型、可嵌套；命令本身就是数组 |

**一条真实命令的完整往返。** 客户端执行 `SET name redis`，TCP 上实际传输的是：

```
*3\r\n$3\r\nSET\r\n$4\r\nname\r\n$5\r\nredis\r\n
```

逐段拆：`*3` = 一个 3 元素的数组（命令就是"字符串数组"）；`$3\r\nSET` = 第一个元素是 3 字节的字符串 SET；`$4\r\nname`、`$5\r\nredis` 同理。服务端回 `+OK\r\n`。再执行 `GET name` 会收到 `$5\r\nredis\r\n`；key 不存在则收到 `$-1\r\n`（客户端把它翻译成 null）。

**关键设计：长度前缀，而不是分隔符。为什么这是性能点？** 对比 HTTP/JSON 这类靠分隔符和括号界定内容的协议：
1. **解析是跳跃式的**：读到 `$5` 就知道"接下来 5 个字节是内容，再后面 2 字节是 \r\n"，**指针直接跳过去**，不需要逐字节扫描找结尾。解析复杂度只和元素个数有关，和内容长度无关。
2. **天然二进制安全，不需要转义**：内容里就算出现 `\r\n` 也无所谓——长度说了算，扫都不扫，自然不会被骗。所以 value 可以是图片、序列化对象等任意字节，这和 SDS 用 len 而不是 `\0` 判断结尾是同一个思想在协议层的体现。
3. **解析器是个极简状态机**：几十行代码就能写一个能用的客户端——这是 Redis 客户端生态在各语言全面开花的原因之一。

**追问：为什么用文本协议，不用更省字节的二进制协议（Protobuf/Thrift 那种）？** antirez 的取舍：协议开销在整个请求耗时里占比极小（瓶颈在网络 RTT 和内存拷贝，不在解析这点 CPU），而文本协议带来的**可调试性**收益巨大——`telnet`/`nc` 连上去直接敲命令就能调试。Redis 还兼容 **inline command**：首字节不是 `*` 就按裸文本解析，所以 telnet 里直接敲 `PING` 也能得到 `+PONG`。又是一次"性能够用的前提下选简单"的典型取舍。

**追问：那为什么不直接用 HTTP？** 类比公函 vs 对讲机：HTTP 是正式公函，每封信都要信封、抬头、落款（请求行 + 一堆 header），哪怕正文只有一个"嗯"；RESP 是架好的对讲机——专线长连接、彼此认识，按下按钮直接说内容。Redis 的通信特征是**每句话极短、说话极频繁、永远是同一个对端**，用公函聊天，成本全在信封上。展开成三笔账：
1. **字节账**：`GET name` 用 RESP 一来一回约 36 字节；用 HTTP，请求行 + Host/Accept/Connection 等头 + 响应状态行 + Date/Content-Type/Content-Length 等头，轻松 300+ 字节，差近 10 倍。10 万 QPS 下白吃 20~30MB/s 带宽。
2. **解析账（大头）**：Redis 执行一条命令本身只要 1~10μs。RESP 解析是亚微秒（读长度、跳指针）；HTTP 要逐行扫 `\r\n`、解析大小写不敏感的 header、靠 Content-Length/chunked 定 body 边界，几微秒起步还带内存分配——**协议开销和命令执行同量级，换 HTTP 等于 QPS 直接砍半**。
3. **模型账（最本质）**：① HTTP/1.1 是严格一问一答，pipelining 现实中不可用；RESP 天然支持 pipeline，攒 100 条一次发，RTT 从 100 次变 1 次。② HTTP 无状态，而 Redis 连接有状态——SUBSCRIBE 的订阅模式、MULTI 的事务上下文、RESP3 的服务端主动推送，请求-响应模型表达不了。③ HTTP 的信封是给中间人（代理、缓存、网关）看的，Redis 客户端和服务端之间没有中间人——付了通用性的成本，用不到通用性的收益。

**反例佐证**：Elasticsearch 就用 HTTP/JSON，而且用得很好——单次查询几十毫秒、payload 本来就是大块 JSON，HTTP 的几百字节和几微秒摊进去可忽略，还白送整个 HTTP 生态（curl 可调、网关/认证/LB 现成）。**判断标准一句话：看"协议开销 ÷ 单次请求真实工作量"的比值**——ES 趋近 0 所以选生态，Redis 接近 1 所以选极简。

**RESP3（Redis 6.0，发 `HELLO 3` 切换）解决了 RESP2 的两个真问题**：
1. **类型太少，语义靠客户端猜**：RESP2 里 HGETALL 返回的是一个扁平数组 `[k1, v1, k2, v2]`，"这是个 map"这件事协议表达不了，要客户端按约定自己拼。RESP3 新增 Map（`%`）、Set（`~`）、Double（`,`）、Boolean（`#`）、大数、verbatim string 等类型，**回复自描述**，客户端不用再为每个命令硬编码"返回值该怎么理解"。
2. **服务端不能主动说话**：RESP2 是严格的请求-响应模型，服务端推不了消息（Pub/Sub 实际是把整条连接整体切换进订阅模式的 hack，这条连接就不能干别的了）。RESP3 新增 **Push 类型（`>`）**，允许服务端在普通连接上插入与请求无关的推送——**这是 6.0 客户端缓存（tracking）的协议基础**：key 变更时服务端在原连接上推 invalidate 消息，客户端据此失效本地缓存。没有 RESP3，客户端缓存这个特性就做不出来。

### 4. 数据结构 —— 记三条设计哲学 + 一个典型例子

> **这一章在论证什么**：前面"为什么快"列的"高效数据结构"那条——每种类型都按场景做了定制。不用背每种结构的底层细节，记住**三条设计哲学**，再用一个典型例子把它们串起来就够了：
>
> **① 同一个类型，两套编码，按规模自动切换**（整章最重要的模式）：
>
> | 类型 | 数据小时（省内存、缓存友好） | 数据大时（保操作复杂度） |
> |------|---------------------------|------------------------|
> | String | embstr（一次分配、连续内存） | raw |
> | Hash | listpack（连续内存，O(n) 但 n 小） | hashtable（O(1)） |
> | ZSet | listpack | skiplist + dict |
> | Set | intset / listpack | hashtable |
> | List | ——（quicklist 本身就是两者杂交） | quicklist |
>
> 背后的洞察：**指针结构的复杂度优势要 n 大才能兑现**——n 只有几十时，连续内存的线性扫描比指针跳跃还快（CPU 缓存命中），而指针本身比数据还占地方。切换阈值就是 `hash-max-listpack-entries 128` 这类配置。
>
> **② 一切为单线程让路，大停顿必须拆碎**：渐进式 rehash（一次搬完→每次搬一桶）、listpack 换 ziplist（消灭级联 realloc）、SDS 预分配（N 次扩容→O(logN) 次）。这条线还延伸到后文的 lazyfree、Multi-part AOF。
>
> **③ 每个字节都算过账**：44 字节对齐 jemalloc 档位、跳表 P=0.25 换平均 1.33 个指针、prevlen 的 1/5 字节两档——"奇怪的数字"全是内存与性能的精确折中。
>
> 面试用法：答"为什么快"时说出主线①那句话——"每种类型都有紧凑编码和专业编码两套，按数据规模自动切换"——然后用下面 Hash 的例子展开即可。

#### 典型例子：一个 Hash 的一生（把主线①②走一遍）

**出生（数据少）→ listpack 编码**：所有 field/value 顺序铺在一块连续内存上，像写在同一张纸上的购物清单。查找就是从头扫，O(n)——但 n 不到 128 时这反而比 hashtable 快：连续内存 CPU 缓存全命中，还省掉了所有指针（指针经常比数据本身还大）。

**长大（元素超 128 个，或单个值超 64 字节）→ 转 hashtable**：清单太长扫不动了，换成真正的哈希表，O(1) 查找。阈值就是配置项 `hash-max-listpack-entries / value`——所有类型的编码切换都是这一个套路。

**继续长大（hashtable 装满要扩容）→ 渐进式 rehash**：扩容要把几百万个 key 搬进新表，一次搬完会卡住主线程几百毫秒，单线程的 Redis 不能接受。做法类比图书馆搬新馆但**不闭馆**：每个来借书的读者顺手帮忙搬一格书架（每次增删改查顺手迁一个桶，后台定时任务也帮着搬）；找书先翻旧馆、再翻新馆（查两张表）；新书直接上新馆（新增只写新表——旧馆只出不进，总有搬空的一天）；搬完新馆转正。代价是期间两张表并存、内存多占一点——**用平摊换不卡顿**。

这一个例子就把两条主线都走完了：编码随规模切换（主线①）、大停顿拆成小碎步（主线②）。被追问到更深（负载因子、rehashidx 游标）可以坦然说"具体参数记不全，但设计思路就是图书馆不闭馆搬家"。

#### 其余类型一句话备查（答出这句就够，再追问就回到设计思路上）

- **String（SDS）**：比 C 字符串多存了一个长度字段 → 取长度 O(1)、二进制安全（能存图片/序列化对象）、修改前先检查空间并预分配（减少扩容次数）。
- **ZSet（跳表 + dict 双结构）**：跳表像公交快线——底层链表站站停，上层索引只停大站，先快线再换慢车，O(logN)。**为什么不用红黑树**：范围查询找到起点后沿底层链表顺着走就行，而且实现简单得多。旁边的 dict 负责按 member 查 score 的 O(1)。

  **顺带把红黑树本身讲清（它在 TreeMap/HashMap/epoll 里到处都是）**：地基是二叉搜索树——猜数字游戏，每次"大了/小了"砍一半，O(logN)；但按顺序插入会歪成一根"糖葫芦"（链表，O(n)），所以需要自平衡。AVL 是强迫症（左右高度差 ≤1，查询最快但每次写都大整队），红黑树是"差不多先生"（只保证**最长路径 ≤ 2×最短**，写操作调整少，工业界默认选它）。靠两条涂色规则实现：黑=正式工、红=实习生，①实习生手下不能再带实习生（不许红红相连）②根到任何出口路过的正式工数量相同——最短路径全正式工、最长顶多红黑交替，自动推出"最长 ≤ 2×最短"。规则被破坏就变色（O(1)）或旋转（三代人斜成一条线时，把中间的"爸爸"提上来当家长，一条线变小三角）。落点：TreeMap/TreeSet 就是它，HashMap 桶超 8 转它，epoll 管理 fd、Linux CFS 调度也是它——**有序 + 增删查都快的默认答案，ZSet 选跳表是少数例外**。
- **List（quicklist）**：双向链表套 listpack 的杂交——宏观链表（两端插入 O(1)），微观连续内存（省指针、缓存友好）。
- **7.0 用 listpack 换掉 ziplist**：旧结构每个元素要记"前一个元素的长度"，一个元素变长会连锁引发后面全部扩容（**级联更新**，像多米诺）；listpack 每个元素只记自己的长度，彻底解决。

#### 跳表深入（选读）：排名 `ZREVRANK` 和改分 `ZINCRBY` 凭什么也是 O(logN)

> 上面"备查"层够应付大多数面试。这一节是想搞清"为什么排行榜查名次、改分都这么快"时再看——它解释了 ZSet 作为排行榜底座的两个关键操作。

**先理解 ZSet 为什么"天生有序"**：它不是查询时才排序，而是**数据从插入起就一直按 score 排好**——读榜单就是顺着底层有序链表走，不需要任何排序动作。跳表（公交快线：底层慢车站站停、上层快线只停大站）保证的是"插入/查找该放哪"够快，O(logN)。

**关键字段 span（跨度）**：每根指针上额外记一个数——**这一跳跨过了底层的几个节点 = 终点位置 − 起点位置**（HEAD 算位置 0）。它就是排名能 O(logN) 的秘密。

**① `ZREVRANK`（查排第几）= 沿途 span 累加**。
没有 span 的话，查"F 排第几"得从头一个个数（O(N)）；跳表能跳，但一跳就跨过好几个，**span 就是来记"刚跨了几个"**。看例子（A~H 共 8 个，分数 1~8）：
```
L2: HEAD ════(span4)════> D(4) ════(span4)════> nil
L1: HEAD ═(span2)═> B(2) ═(span2)═> D(4) ═(span2)═> F(6) ═(span2)═> nil
L0: HEAD → A → B → C → D → E → F → G → H        (每跳 span=1)
```
查 F：① L2 上 HEAD→D（分4<6），跳，累加 span 4 → 到 D；② D 在 L2 再往右越界，下沉到 L1；③ L1 上 D→F，跳，累加 span 2 → 到 F。累计 4+2=**6**，即 F 是第 6 个。全程 2 跳（O(logN)），没逐个数。
（`ZRANK` 是 0 基返回 5；`ZREVRANK` 从高到低 = 总数 8 − 6 = 2。）
类比：上楼梯能迈大跳，每个跳跃口挂牌写"此跳=4级"，想知道站在第几级，把沿途的牌子加起来即可，不用一级级数。

**② `ZINCRBY`（给成员加分）= dict 查老分 + 跳表删 + 跳表插**。
分数变了，它在"按分排好的队"里的位置可能变，所以不是改个数字，而是"摘下来、按新分插回去"：
1. **查当前分**：用旁边的 dict（member→score）O(1) 拿到老分（如 1000）；
2. **算新分**：1001；
3. **跳表里挪位**：按"老分 1000+member"找到并摘除（O(logN)）→ 按"新分 1001"找位置插入（O(logN)）；
4. **更新 dict**：member→1001。
合计 **O(logN)**。类比：按身高排好的队里有人垫高了 1cm，不重排全队，只把他抽出来插到新位置。

**③ 摘/插时 span 怎么 ±1 修正**（这一步保证排名一直准）。以插入新节点 X 为例：
- **X 参与的层**：把"原来横跨插入点的那根指针"**拆成两段**（前驱→X、X→原目标），两段 span 之和 = 老 span + 1（多出的 1 是 X 自己）。拆出的具体数值，靠查找时**顺手记下的每层前驱位置 `rank[i]`** 一减就得到。
- **X 够不到的更高层**：横跨插入点的指针 span 直接 **+1**（底下多了一个节点，这一跳就多跨一个）。
- **删除反过来**：参与的层把两段合并回一根（span = 两段和 − 1），更高层 span −1。

**最关键的收益**：无论插入还是删除，**只有"搜索路径上每层那一根指针"要改 span**——每层一根、共 logN 层，所以是 O(logN) 根指针，不用动全表。这就是改分/查排名都能 O(logN) 的根本：span 让你既不必全局重数排名，维护时也只碰路径上那几根指针。

### 5. 持久化 —— 把每个"为什么"挖出来

#### RDB：快照 + 写时复制

`bgsave` 流程：主进程 **fork** 出子进程，子进程遍历内存写出 RDB 文件，主进程继续服务。

**核心机制 COW（写时复制）**：类比父子共用一本厚相册——fork 只是给儿子复印了一份**目录**（页表），相册本体（物理内存页）共用一套；父亲要改哪一页，才把**那一页**单独复印出来改，儿子手里看到的永远是当初那一页。技术表述：fork 不复制内存数据，只复制页表（虚拟地址 → 物理页的映射），所有页标记只读；父进程写某页时触发缺页异常，内核复制**这一个 4KB 页**给父进程改，子进程看到的还是 fork 那一刻的旧页。所以子进程拿到的是**时间点一致**的快照，且大部分内存不用复制——但改得越多、复印得越多，这就是下面第 2 个坑的来源。

**三个深坑（生产必知）**：
1. **fork 本身是阻塞的**：复制页表的耗时和内存量成正比，经验值约 10GB 内存 20ms 量级。实例越大，fork 停顿越明显——这是"单实例不建议超过 10GB"说法的来源之一。
2. **写入越猛，内存膨胀越凶**：快照期间父进程改到的每个页都要复制，极端情况（快照期间全量写一遍）内存翻倍。所以 maxmemory 一般只配机器内存的 50%~65%，给 COW 留余量。
3. **要关透明大页（THP）**：THP 把页从 4KB 变 2MB，COW 时改 1 字节也要复制 2MB，写放大 512 倍，可能导致快照期间内存暴涨和延迟尖刺。`echo never > /sys/kernel/mm/transparent_hugepage/enabled` 是 Redis 部署的标准动作。

**RDB 的根本缺陷**：两次快照之间的数据丢失。快照不能太频（fork + 全量写盘开销大），所以丢失窗口是分钟级。

#### AOF：命令日志

**为什么是写后日志（先执行命令，再写日志）？** 和 MySQL 的 WAL 相反。类比记账：Redis 是"先花钱、后记账"——账本上绝不会出现一笔花不出去的错账（不用校验），但最后一笔可能没来得及记就断电；MySQL WAL 是"先记账、后花钱"——账先落纸，钱怎么花都能照账恢复。具体两个原因：
1. Redis 不做命令语法检查，先执行就等于天然校验——执行失败的命令不会进 AOF，**避免恢复时重放错误命令**；
2. 不阻塞当前写操作。
代价：命令执行完、日志没落盘时宕机，这条命令就丢了。**Redis 的取舍一贯如此：性能优先，持久化是尽力而为。**

**写盘的三个层次**要分清楚：命令先写进 `aof_buf`（用户态缓冲）→ 每次事件循环结束 `write()` 进内核 page cache → `fsync()` 才真正落盘。三种策略的区别只在 fsync 的时机：
- `always`：每个事件循环 fsync。最多丢一个事件循环的命令，但每条写命令都付出一次磁盘 IO（即使是 SSD 也把 QPS 拉到几千的量级）。
- `everysec`（默认）：**由后台线程**每秒 fsync 一次。号称最多丢 1 秒，**实际最多丢 2 秒**：主线程写 aof_buf 时会检查上一次 fsync 的完成时间，如果超过 2 秒还没完成（磁盘抖动），**主线程会阻塞等待 fsync 完成**——这是"磁盘慢会卡死 Redis"的根源，也是面试的好细节：everysec 既可能丢 2 秒数据，也可能因为慢盘阻塞主线程。
- `no`：只 write 不主动 fsync，交给 OS（通常 30 秒级）。

**AOF 重写**：日志会无限增长（对同一个 key 的 1 万次 INCR 记了 1 万条，其实一条 SET 就等价）。重写 = fork 子进程，**根据当前内存数据**生成等价的最小命令集（不是分析旧 AOF 文件！）。
- **7.0 之前的双写问题**：重写期间新的写命令要同时写进旧 AOF 的 `aof_buf` **和** `aof_rewrite_buf`（重写缓冲区），重写完成后主线程把 rewrite_buf 追加到新文件——大缓冲区追加时还会阻塞。同一份数据写两遍，CPU 和磁盘都浪费。
- **7.0 Multi-part AOF**：AOF 变成一组文件——一个 base 文件（重写时由子进程生成，RDB 或 AOF 格式）+ 多个 incr 文件（增量命令）+ 一个 manifest 清单。重写期间新命令**只写新的 incr 文件**，不再需要 rewrite_buf，双写问题和追加阻塞都消失了。

**混合持久化（4.0 引入）**：`aof-use-rdb-preamble yes`，重写时 base 部分直接用 RDB 格式（小、加载快），之后的增量是 AOF 命令。恢复速度接近 RDB，丢失窗口等于 AOF。7.0 的 multi-part 结构里这就是常态。

### 6. 高可用 —— 主从 → 哨兵 → 集群，每层解决上一层的遗留问题

#### 主从复制

**全量同步流程**：slave 发送 `PSYNC <replid> <offset>` → master 判断无法增量（首次连接/偏移量已被覆盖）→ 回复 `+FULLRESYNC`，执行 bgsave 生成 RDB → 发送给 slave → slave **清空自己的旧数据**、加载 RDB → master 把 bgsave 期间积累的写命令（存在该 slave 的 **replication buffer**，即客户端输出缓冲区）补发过去。

**增量同步**：类比上课板书——老师（master）边讲边写黑板，黑板写满就从头擦了重写（环形覆盖）；缺课的学生（断线的 slave）回来，要补的内容**还在黑板上**就自己抄一遍（增量同步），**已经被擦掉了**就只能找老师要全套讲义（退化为全量同步）。黑板太小等于刚写就擦，谁缺课都得重领讲义。技术对应：master 维护一个 **repl_backlog_buffer**（环形缓冲区，所有 slave 共享一个，默认仅 1MB），所有写命令都会写进去；slave 断线重连后带着自己的 offset 来，这段数据还在环里就只补差量，被覆盖了就全量。

**三个生产级细节**：
1. **repl_backlog_size 怎么配**：`写入速率 × 平均重连耗时 × 2（安全系数）`。例如写入 1MB/s、故障重连平均 10s，就至少配 20MB。默认 1MB 在生产几乎必然导致"一断线就全量"。
2. **复制风暴**：master 重启或网络分区恢复时，N 个 slave 同时请求全量同步，master 要生成 RDB 并同时向 N 个 slave 推送，带宽和内存(输出缓冲区)被打爆。解法：树形复制拓扑（slave 挂 slave）、错峰重连。
3. **输出缓冲区导致的死循环**：replication buffer 有上限（`client-output-buffer-limit slave 256mb 64mb 60`：超 256MB 或连续 60 秒超 64MB 就断开连接）。如果 RDB 太大、加载太慢，buffer 积压超限 → master 主动断开 slave → slave 重连又触发全量 → 又超限……**无限全量循环**。大实例必须调大这个限制。

**复制是异步的** —— master 写完本地就返回客户端，不等 slave。这是后面"主从切换丢数据/锁失效"所有问题的根源，先在这里钉死。

#### 哨兵（Sentinel）：解决"故障了没人切"

哨兵自身是一个集群（至少 3 个、奇数个），对 master 做三件事：监控、自动故障转移、通知客户端新地址。

**判定下线的两步**：
1. **主观下线（SDOWN）**：某个哨兵连续 `down-after-milliseconds`（默认 30s）ping 不通 master。单个哨兵的网络抖动不能作数，所以要第二步。
2. **客观下线（ODOWN）**：该哨兵向其他哨兵发 `is-master-down-by-addr` 询问，**达到 quorum 个**哨兵都认为下线，才定性。

**quorum 和 majority 是两个数，别混**（高频深挖点）：
- **quorum**：判定客观下线需要的票数，配置文件里配的那个数。
- **majority（哨兵总数的过半）**：执行故障转移前，哨兵之间要先**选举一个 leader** 来主刀，当选需要**过半数哨兵**投票（Raft 风格：每个 epoch 每个哨兵只投一票，先到先得，过半当选，超时随机延迟后进入下一个 epoch 重选）。
- 类比小区物业：quorum 是"几户邻居联名就可以认定这家出事了"（**报案门槛**）；majority 是"业主大会到场过半才能选出代表去处理"（**行动门槛**）。联名够了但开会人数不够，事认定了也没人能去办。
- 推论：5 个哨兵、quorum=2 时，2 票就能判定下线，但**至少 3 个哨兵活着**才能选出 leader 执行切换。quorum 决定"多敏感"，majority 决定"还能不能动手"。

**新 master 怎么选**：过滤掉断线/响应慢的 slave 后，依次比较：① `slave-priority` 配置优先级 → ② 复制偏移量最大（丢数据最少）→ ③ runid 最小（兜底的确定性规则）。

**哨兵没解决什么**：写入还是只有一个 master 扛，容量受限单机内存。规模上去了就需要分片——集群。

#### Cluster：解决"单机写不下、存不下"（数据分片）

**它解决什么**：主从、哨兵都是"**一个主节点扛全部写、存全部数据**"，受限于单机内存（容量）和单机写吞吐（性能）。Cluster 把数据**分片（sharding）**——拆开放到多个主节点，每台只存一部分、只扛一部分写，**容量和写性能都随节点数水平扩展**。
类比：一个图书馆书太多、一栋楼装不下、借书也挤爆一个柜台 → 开多个**分馆**，每个分馆只存一部分书、各接待一部分读者。

**集群长什么样**：N 个**主节点**，每个主节点负责一部分数据，并各自带**从节点**做备份。典型"3 主 3 从"：3 个主各管约 1/3 的数据，每个主配 1 个从。节点之间用 **Gossip 协议**互相发消息，所以每个节点都掌握"全集群的数据分布图"。

**一个 key 放在哪台机器？两层映射：key → 槽 → 节点**
1. 整个集群固定划 **16384 个哈希槽（slot）**。
2. 一个 key 算 `CRC16(key) % 16384` → 得到它属于**哪个槽**。
3. 这 16384 个槽**分配给各个主节点**（如 A 管 0~5460、B 管 5461~10922、C 管 10923~16383）。
所以定位一个 key：先算它落在哪个槽，再看这个槽归哪个节点。
类比：不是把书直接分到"几号分馆"，而是先分进 16384 个**标准箱子（槽）**，箱子再分给各分馆。

**为什么要"槽"这个中间层，不直接 `CRC16(key) % 节点数`？**
如果直接对**节点数**取模，一旦增/删节点，节点数变 → 几乎所有 key 的归属全变 → 要全量搬数据，灾难。加了槽这层后：**key → 槽 永远不变**（算法固定），会变的只是**槽 → 节点这张映射表**。增删节点时，只需把**一部分槽连同其中数据**搬给新节点，其他槽纹丝不动，迁移量小、且哪些槽要动完全可控、进度可见。
类比：加分馆时，只从老分馆搬几箱书过去，不用把所有书重新分箱。

> **澄清：Redis Cluster 用的是哈希槽，不是一致性哈希**（这俩常被搞混）。"决定 key 放哪台机器"有三种方案，一个比一个好：
> 1. **取模 `hash(key)%节点数`**：最朴素，但增删节点 → 几乎全量迁移，灾难。
> 2. **一致性哈希**：把节点和 key 都放到一个**哈希环**（0~2³²首尾相接）上，key 顺时针找到的第一个节点就归它。增删节点只影响环上相邻的一小段 key、不全量迁移——但缺点是**数据易倾斜**（节点少时分布不均，要靠"虚拟节点"补救）、且**哪些 key 迁移由环上位置决定、运维不可控**。
> 3. **哈希槽（Redis Cluster 选这个）**：固定 16384 个槽，key→槽 固定、槽→节点 是张可人工管理的表。相比一致性哈希：槽**均匀**不倾斜、迁移以槽为单位**人工可控可见**。
>
> 一句话：②③ 都靠"加中间层"避免全量迁移——一致性哈希的中间层是"环上位置区间"（隐式），哈希槽是"固定 16384 个槽"（显式、可调度）。Redis 选后者。

**客户端怎么读写——重定向（MOVED）**：客户端可以连**任意**一个节点。发 `GET key` 时，该节点先算 key 属于哪个槽、看是不是自己负责：是 → 直接处理；不是 → 回 `MOVED 槽号 正确节点地址`，让客户端改去对的节点。**智能客户端（Lettuce/Jedis Cluster）会缓存"槽→节点"映射表**，之后直接找对的节点，不用每次被重定向。
类比：你去任意分馆借书，管理员一算这本在几号箱、箱归哪个分馆，不归这儿就说"去 X 分馆"；你是聪明读者，记下来下次直接去对的分馆。

**为什么是 16384 个槽、不是 65536**（antirez 本人在 GitHub issue 里答过）：节点间心跳包（PING）要携带自己负责的槽位图，16384 bit = 2KB；65536 bit = 8KB，每秒大量心跳时带宽浪费明显。而集群规模官方建议不超过 1000 节点（再多 Gossip 本身就是负担），1000 个节点分 16384 个槽粒度足够。槽数少、位图填充率低时压缩效果也更好。

**扩容/缩容 = 搬槽**：加节点就把现有节点的一部分槽（连数据）迁过去、缩容反之。迁移可在线进行、不停服。迁移中"一个槽正从 A 搬到 B、部分 key 已过去"的临时状态，就用下面的 ASK 处理。

**MOVED vs ASK（必须分清）**：
- `MOVED 1234 ip:port`：这个槽**已经永久地**属于另一个节点。智能客户端（Lettuce/Jedis Cluster）收到后**更新本地槽位映射表**，以后直接去新节点。
- `ASK ip:port`：这个槽**正在迁移中**，你要的这个 key 已经搬到目标节点了，但槽还没整体交接。客户端去目标节点时要先发 `ASKING` 命令（否则目标节点会因为"这个槽不归我"而拒绝），并且**不更新本地映射表**——因为迁移还没完成，下一个 key 可能还在源节点。
- 一句话：MOVED 是"改户口了"，ASK 是"人临时在那边，户口还没迁"。

**集群的故障转移（不需要哨兵）**：节点间用 Gossip 协议互发 PING/PONG 交换状态。某 master 被一个节点标记 PFAIL（疑似下线），当**集群中过半数的 master** 都标记它 PFAIL，升级为 FAIL 并广播。它的 slave 发起选举：向所有 master 拉票（带 epoch），**获得过半数 master 投票**的 slave 升级为新 master，接管槽位。复制偏移量大的 slave 会更早发起拉票（延迟和 offset 挂钩），优先当选。

**集群的限制及绕法**：多 key 命令（MSET、SINTER）、事务、Lua 里的多个 key 必须在同一个槽。绕法是 **hash tag**：`{user:1000}.following` 和 `{user:1000}.followers`——只对 `{}` 里的内容算 CRC16，强制同槽。代价是热点风险：tag 用得太集中，数据全挤到一个槽。

### 7. 过期删除与内存淘汰

#### 过期删除：惰性 + 定期，缺一不可

- **惰性删除**：访问 key 时先查过期字典（expires dict，单独存 key → 过期时间戳），过期就删并返回 nil。零额外开销，但**不被访问的过期 key 永远占着内存**。
- **定期删除**（activeExpireCycle）：serverCron 每 100ms 执行一次（慢周期），每个 db **随机抽 20 个**带 TTL 的 key，删掉其中过期的；**如果过期比例超过 25%，立刻再抽一轮**，循环直到比例降下来或者超时（单次慢周期上限 25ms，防止吃掉主线程）。事件循环之前还有快周期（上限 1ms）。
- **为什么是抽样而不是全扫**：全扫 O(n) 阻塞主线程；抽样 + "超 25% 就继续"在数学上能把整体过期未删比例压在约 25% 以下，用常数开销换近似效果。
- **主从的过期语义**：slave **不主动删过期 key**，等 master 删了之后同步 `DEL` 命令过来（保证主从一致的删除顺序）。3.2 之前从库读过期 key 会返回旧值（bug 级行为）；3.2 之后从库读时做逻辑判断，过期返回 nil，但物理删除仍等 master 的 DEL。

#### 内存淘汰：到达 maxmemory 时的兜底

8 种策略 = （allkeys / volatile）×（lru / lfu / random）+ volatile-ttl + noeviction（默认，写入直接报 OOM 错误）。

**Redis 的 LRU 是近似 LRU，为什么以及怎么近似**：
- 真 LRU 要维护一个全局双向链表，每次访问都把节点挪到头部——**每个 key 多两个指针（16 字节）**，且每次读都有链表操作。对千万级 key，光指针就是 GB 级开销。Redis 不肯付这个钱。
- 做法：每个 redisObject 里有一个 **24bit 的 lru 字段**，存最近访问时间（秒级精度，约 194 天回绕一次）。淘汰时**随机采样 5 个 key**（`maxmemory-samples`，可调），淘汰其中最久未访问的。
- 3.0 的优化：维护一个 **16 个槽位的淘汰池（eviction pool）**，每次采样出来的候选和池里的存量比，池子始终保留"见过的最旧的 16 个"——跨多次采样积累信息，采样数调到 10 时效果已接近真 LRU。

**LFU（4.0+）解决 LRU 的什么问题**：偶发的批量扫描（比如一次全量遍历）会把所有 key 的"最近访问时间"都刷新，把真正的热数据挤出去。LFU 按**频率**淘汰，复用那 24bit：**高 16 位存最后一次衰减时间（分钟级），低 8 位存访问频率计数器**。
- 8 位最多 255，怎么表示百万次访问？**对数概率递增**：每次访问以 `1/((counter - 5) × lfu_log_factor + 1)` 的概率 +1——计数越大越难加，factor=10 时 255 大约对应百万级访问。
- 不衰减的话旧热点永远占着高计数，所以有 `lfu-decay-time`：每过 N 分钟计数减 1。

## 三、Redis 近些年的演进

> 演进主线：**每一个版本都在拆"单线程被阻塞"的雷，同时把周边能力收编进来。**

### Redis 3.0（2015）—— Cluster
官方分片方案落地。此前业界用客户端分片（无中心但扩容难）或代理方案（Codis/Twemproxy，多一跳延迟、组件多）。Cluster 把路由知识下放给智能客户端，去掉了代理这一跳。

### Redis 4.0（2017）—— 异步化，拆"删除阻塞"的雷
- **lazyfree / UNLINK**：DEL 一个百万元素的 Hash 要在主线程里逐个释放，百毫秒级阻塞。UNLINK 把 key 从键空间摘除（瞬间完成，逻辑上已删除），实际内存释放丢给后台 bio 线程。**细节**：UNLINK 会先评估释放成本，元素数 >64 才走异步，小对象直接同步删（任务投递本身也有成本）。配套的 `lazyfree-lazy-expire`/`lazyfree-lazy-eviction`/`lazyfree-lazy-server-del` 把过期删除、淘汰删除、隐式删除（如 RENAME 覆盖）也都异步化——否则就算你只用 UNLINK，一个大 key 过期时照样卡主线程。
- **混合持久化**、**LFU**、**MEMORY 命令**。
- **Module 系统**：第三方可用 C 写扩展数据类型，RedisBloom/RedisJSON/RediSearch 由此而来——为后来 Redis Stack 和 8.0 的功能合并埋下伏笔。

### Redis 5.0（2018）—— Stream
对标 Kafka 核心语义的持久化消息队列：消费者组、ACK、PEL（pending 列表）、消息回溯。弥补 List（无 ACK、无广播）和 Pub/Sub（不持久化、掉线即丢）的缺陷。底层用 Radix 树索引消息 ID，消息体用 listpack 存。

### Redis 6.0（2020）—— IO 多线程，拆"网络瓶颈"的雷
**背景**：内存和 CPU 越来越快，单线程的瓶颈实测落在了**网络 IO**上——从内核 socket 缓冲区读数据、解析协议、把回复写回 socket，这些系统调用和内存拷贝占了大头，而不是命令执行本身。

**精确的执行流程**（说清楚这个，这道题就是满分）：
1. 主线程 epoll_wait 收集所有可读的客户端，放入待处理队列；
2. 用 Round-Robin 把这些客户端**分给 IO 线程（主线程自己也算一个）**，所有 IO 线程并行地做 read + 协议解析；
3. **屏障**：主线程自旋等所有 IO 线程完成；
4. **主线程串行执行所有命令**（这一步永远单线程，原子性语义完全不变）；
5. 可写客户端再分给 IO 线程并行 write 回复；
6. 再次等待全部完成，进入下一轮循环。

**关键细节**：默认配置 `io-threads` 只用于**写回复**（写的量通常远大于读），读和解析要额外开 `io-threads-do-reads yes`。IO 线程数建议 ≤ 核数且官方说超过 8 收益递减。**为什么命令执行不并行？** 一并行就要给数据结构加锁，原子性承诺（INCR/Lua/事务）全部要重做，整个生态的假设都会被打破——收益却有限，因为执行本来就不是瓶颈。

其他：**ACL**（此前所有客户端共享一个 requirepass，运维误执行 FLUSHALL 无从限制；ACL 可以按用户限制命令和 key 模式）、SSL/TLS、**客户端缓存**（服务端 tracking key，变更时主动 invalidate 客户端本地缓存——把多级缓存的一致性问题做进了协议层）。

### Redis 7.0（2022）—— 整合与精简
- **Multi-part AOF**（机制见持久化一节）：消灭 aof_rewrite_buf 和双写。
- **Function**：替代 EVAL 脚本。EVAL 的问题——脚本以 SHA 标识存在服务端内存里，**不持久化、不复制**，重启就没，主从切换后从库上没有，客户端必须自带源码做 EVALSHA 失败重试。Function 把函数库当数据对待：持久化进 RDB/AOF、随复制传播、有名字可管理。
- **listpack 全面取代 ziplist**（级联更新问题见数据结构一节）。
- **Sharded Pub/Sub**：原 Pub/Sub 在集群里要把消息广播到**所有节点**（因为不知道订阅者连在哪），节点数越多浪费越大；sharded 版按 channel 算槽，消息只在负责该槽的分片内传播。

### 2024 至今 —— 许可证之争与生态分裂
- 2024.03：Redis 公司把许可证从 BSD 换成 **RSALv2/SSPL**（双许可），核心诉求是阻止云厂商免费拿 Redis 卖托管服务。SSPL 不被 OSI 认定为开源许可证。
- 社区反应剧烈：Linux 基金会迅速托管分叉项目 **Valkey**，AWS/Google/Oracle 等站台，多数发行版（Fedora 等）把默认包换成 Valkey。原 Redis 核心贡献者一部分流向 Valkey。
- 2025.05：Redis 8.0 增加 **AGPLv3** 作为第三个许可选项（部分回归开源），同时把原 Redis Stack 的模块能力（JSON、搜索、时序、布隆过滤器等）并入开源核心，并宣传显著的性能优化。antirez（原作者）2024 年底也回归了 Redis 公司。
- **对使用方的实际影响**：自建用户基本无感；做技术选型时若在意许可证风险或跟随云厂商，Valkey 是平替（协议、数据格式兼容）；国内云厂商的"Redis 兼容版"内核走向值得关注。

**演进总结一句话**：命令单线程执行这个核心十年没变，变的是把它周围的阻塞点一个个搬走（删除异步化、IO 并行化、重写轻量化），并把生态能力收编进核（Stream、Function、Stack 模块）。

## 四、常用场景和用法

### 1. 缓存：一致性问题要讲到时序级别

**基本模式 Cache Aside**：读——先缓存后 DB，未命中回填；写——**先更新 DB，再删缓存**。下面把"为什么"全部展开。

**为什么是删缓存，不是更新缓存？**
1. 并发写乱序：A 先更新 DB（值=1）、B 后更新 DB（值=2），但 B 先更新缓存（=2）、A 后更新缓存（=1）——**缓存停留在旧值 1，且没有任何机制会纠正它**，直到下次写。删除没有这个问题：谁删都一样，删完下次读自动回填最新值。
2. 写了不一定读（懒加载思想）：缓存值若是多表计算出来的，每次写都重算一遍是纯浪费。

**为什么先更新 DB，再删缓存？把两种顺序的失败时序摆出来对比：**

先删缓存，再更新 DB 的坏时序（**很容易发生**）：
```
线程A：删除缓存
线程B：读缓存 miss → 读 DB（旧值）→ 回填缓存（旧值）   ← 整个动作发生在 A 更新 DB 之前
线程A：更新 DB（新值)
结果：缓存里是旧值，DB 是新值，且会一直脏到下次写或过期
```
A 删缓存和更新 DB 之间隔着网络往返 + SQL 执行，窗口是**毫秒级**，并发读随时撞进来。

先更新 DB，再删缓存的坏时序（**理论存在，概率极低**）：
```
线程B：读缓存 miss → 读 DB（旧值）          ← ①
线程A：更新 DB（新值）→ 删除缓存            ← ②③
线程B：回填缓存（旧值）                     ← ④
```
要求 B 的"读 DB"发生在 A 更新之前，而 B 的"回填"发生在 A 删除**之后**——即**一次读的后半程比一次完整的写还慢**，同时还要求恰好缓存刚失效。三个低概率条件叠加，实践上可接受。**这就是 Cache Aside 选这个顺序的原因：两种顺序都不完美，选坏窗口小的那个。**

**还有第三种失败：删缓存这一步失败了怎么办？** DB 已是新值、缓存还是旧值。两个工程方案：
1. **延迟双删**：更新 DB → 删缓存 → 延迟几百毫秒**再删一次**。第二次删的延迟时间要**覆盖一次读请求"读 DB + 回填"的耗时**，再加上**DB 主从延迟**（如果读流量走从库，从库还没同步到新值时回填的还是旧值，所以延迟还要 > 主从复制延迟）。第二次删除应该异步化 + 失败重试。
2. **订阅 binlog 删缓存（Canal + MQ）**：把"删缓存"从业务代码挪到 binlog 消费者里。好处：① 不入侵业务代码；② binlog 是 DB 串行化之后的事实顺序，天然解决并发写乱序；③ MQ 的重试机制保证删除**最终一定成功**——把"尽力而为"升级成"最终一致"。代价：链路变长，秒级的不一致窗口。

**终极追问：能做到强一致吗？** 不能，除非读写都加分布式锁（性能上等于没有缓存）。缓存 + DB 是两个系统，没有跨系统事务，**一切方案都是缩小不一致窗口，不是消除**。面试这么表态就是满分。

**缓存三大问题**：

| 问题 | 本质 | 方案及其细节 |
|------|------|------------|
| 穿透 | 查**不存在**的数据，缓存层形同虚设 | ① 缓存空值，TTL 给短（如 60s），防止占内存 + 数据新增后能较快感知；② 布隆过滤器前置：误判率公式决定它"说不存在就一定不存在，说存在可能误判"，所以拦截的是不存在的查询；缺点是不支持删除（要删用布谷鸟过滤器或定期重建）；③ 恶意流量在网关层做参数校验和风控 |
| 击穿 | **单个热 key** 过期瞬间，并发全打 DB | ① 互斥重建：拿到分布式锁（或 JVM 级锁）的那一个请求去查 DB 回填，其他请求自旋短等或返回旧值；② 逻辑过期：物理上不设 TTL，value 里带一个过期时间字段，发现逻辑过期就**异步**重建、本次先返回旧值——牺牲一致性换可用性，适合极热数据 |
| 雪崩 | **大面积** key 同时失效或 Redis 宕机 | ① TTL 加随机抖动（基础 TTL ± 随机 10%）；② Redis 高可用（哨兵/集群）；③ 应用层熔断降级限流（Sentinel/Hystrix），DB 顶不住时宁可拒绝部分请求；④ 多级缓存，本地缓存兜底热点 |

### 2. 分布式锁：从一行命令挖到 RedLock 论战

**正确的加锁姿势及每个参数的存在理由**：
```
SET lock_key {uuid} NX PX 30000
```
- `NX`：不存在才能设置 = 互斥语义本身。
- `PX 30000`：**怕死锁**——持锁的进程崩了没人释放。注意 SET 命令一步完成"加锁 + 设过期"，如果拆成 `SETNX` + `EXPIRE` 两条，两条之间进程崩溃照样死锁。**原子性必须在一条命令里完成。**
- `{uuid}`（每次加锁生成的唯一值）：**怕误删**——线程 A 的锁过期被 B 拿到，A 业务做完执行 DEL，删的是 B 的锁。所以解锁前必须验证"锁还是我的"。

**解锁必须用 Lua**：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```
GET 比对和 DEL 必须原子。否则：GET 判断通过 → **就在此刻锁过期、B 加锁成功** → DEL 删掉的是 B 的锁。窗口虽小，原理上就是 check-then-act 竞态。

**锁续期（看门狗）**：业务执行时间不可预知，过期时间设短了锁提前飞、设长了故障后恢复慢。Redisson 的方案：加锁不指定 leaseTime 时默认 30s，后台线程**每 10s（即 1/3 周期）检查一次**，持锁线程还活着就把过期时间重置回 30s。进程崩溃 → 看门狗线程也死了 → 锁最多 30s 后自动释放。**注意：显式传了 leaseTime 就不启动看门狗**，这是 Redisson 一个著名的使用陷阱。

**可重入**：用 Hash 实现——key 是锁名，field 是"实例ID:线程ID"，value 是重入计数。加锁 Lua：不存在则创建并置 1；存在且 field 是自己则 +1；否则失败返回剩余 TTL。解锁对应 -1，减到 0 才真正 DEL。这就是 Redisson RLock 的结构。

**主从切换丢锁——精确时序**：
```
1. 客户端A 在 master 上 SET NX 成功，拿到锁
2. master 宕机，这条写还没异步复制到 slave        ← 根源：复制是异步的
3. 哨兵把 slave 提升为新 master，锁在新 master 上不存在
4. 客户端B 在新 master 上 SET NX 成功
5. A 和 B 同时持锁，互斥被打破
```

**RedLock 及其论战（深挖到这里就是加分项）**：
- antirez 的方案：部署 N（如 5）个**完全独立**（无主从关系）的 Redis，加锁时依次向 5 个实例 SET NX，**在锁有效期内拿到 ≥3 个**即成功，且有效时间要减去加锁过程本身的耗时。单实例宕机不影响整体互斥。
- **Martin Kleppmann 的反驳**（《How to do distributed locking》）：① 客户端拿到锁后发生长时间 **GC 停顿**（或网络延迟），停顿期间锁过期、别人拿到锁，停顿结束后旧客户端**并不知道自己已经丢锁**，继续写共享资源——任何带超时的锁都有这个问题，RedLock 也不例外；② RedLock 的安全性依赖**时钟**：某个实例时钟跳变（NTP 校时、运维改时间）导致锁提前过期，多数派就会被打破。他的结论：要正确性就用 **fencing token**（每次加锁拿一个单调递增编号，存储层拒绝旧编号的写入），Redis 给不出单调 token，所以 RedLock"既不够快也不够安全"。
- **antirez 的回应**：时钟跳变可以靠运维纪律避免（不开 NTP 步进式校时）；fencing token 要求**存储端**支持按 token 做条件写，如果存储端有这能力，它自己就能做互斥，何必要锁。
- **实践结论（表态）**：绝大多数业务用单 Redis 主从 + Redisson，接受极小概率的锁失效，**用业务幂等兜底**（锁是效率手段而非正确性的唯一防线）；真正要正确性的场景（钱、库存的最终扣减）靠数据库层面的约束（唯一键、乐观锁版本号、事务）或 Zookeeper/etcd（CP 系统，基于共识协议，没有异步复制丢数据问题，但吞吐低一个量级）。

### 3. 计数与限流

- **计数器**：`INCR` 单线程原子。库存预扣的标准 Lua：读库存 → 判断够不够 → 扣减，三步一个脚本原子完成，避免"读和写之间被插队"导致超卖。
- **固定窗口**：`INCR` + 首次 `EXPIRE`。**临界突刺问题要能算出来**：限流 100/分钟，第 59 秒来 100 个、第 61 秒（新窗口）再来 100 个——2 秒内实际通过 200 个，瞬时是限值的 2 倍。
- **滑动窗口**：ZSet，score = 时间戳。一段 Lua：`ZREMRANGEBYSCORE key 0 (now-窗口)` 清掉窗口外 → `ZCARD` 计数 → 未超限则 `ZADD`。精确，但每个请求都是一个 member，**高 QPS 下 ZSet 本身变成大 key**，要注意。
- **令牌桶**：Hash 存（上次补充时间、当前令牌数），取令牌的 Lua 里按时间差现算补充量。允许突发（桶容量），平均速率受控。

### 4. 排行榜

ZSet：`ZINCRBY` 加分，`ZREVRANGE ... WITHSCORES` 取前 N，`ZREVRANK` 查自己排名。
- **同分按时间排序**：score 是 double（53 位有效精度），组合编码 `score = 分数 × 2^32 + (2^32 - 秒级时间戳)`——高位比分数、分数相同时早到的大。注意 53 位精度上限，分数和时间戳的位数要算好。
- **亿级用户排行榜**：单个 ZSet 是大 key 且打单分片。按分数段或 ID 分桶成多个 ZSet，查全局 TopN 时各桶取 TopN 归并；查个人排名用"所在桶内排名 + 前面所有桶的基数"近似。

### 5. 分布式 Session / 登录态

String/Hash 存 `token → 用户信息`，TTL = 会话有效期，访问时续期（滑动过期）。Spring Session 开箱即用。追问点：和 JWT 的取舍——JWT 无状态省存储但**无法主动踢人下线**（只能等过期或维护黑名单——而黑名单又回到了集中存储，等于绕回来了）。

### 6. 消息队列

- **List**：`LPUSH` + `BRPOP`（阻塞读，避免轮询空转）。缺陷：无 ACK——`BRPOP` 弹出后消费者崩溃，消息就丢了；无广播；无回溯。
- **Pub/Sub**：有广播，但消息不落地、不缓存：消费者断线期间的消息**永久丢失**。只配做"在线实时通知"（如配置刷新广播）。
- **Stream（5.0+，做 MQ 的正解）**：
  - `XADD` 写入（消息 ID `时间戳-序号` 全局有序）、`XREADGROUP` 按消费者组读、`XACK` 确认。
  - **PEL（Pending Entries List）**：已投递未 ACK 的消息记录在消费者名下，消费者崩溃后，别的消费者用 `XAUTOCLAIM` 把超时未确认的消息认领过来重新消费——**这就是 List 方案缺失的可靠投递语义**。
  - 上限要心里有数：`MAXLEN` 控制长度（内存有限，不能像 Kafka 留 7 天）；没有分区概念，扩展性不如 Kafka；消费失败没有死信队列，要自己用 PEL + 重试计数模拟。
- **延迟队列**：ZSet，score = 期望执行时间戳，轮询 `ZRANGEBYSCORE key 0 now LIMIT 0 N` + `ZREM`（取和删要 Lua 原子化，否则多个 worker 会取到同一条）。
- **表态**：可靠性要求高（不可丢、要事务消息、要堆积能力）用 RocketMQ/Kafka；Redis 适合轻量、容忍极端情况丢消息、不想多引入一个组件的场景。

### 7. 去重与大数据量统计

- **Set**：精确去重。`SINTER` 共同关注、`SRANDMEMBER`/`SPOP` 抽奖。
- **HyperLogLog**：UV 统计。**原理一句话讲清**：把每个元素 hash 成二进制串，观察"低位连续 0 的最大个数"——出现过 k 个连续 0，说明大概来过 2^k 个不同元素（类比：抛硬币连续正面 10 次，说明大概抛了 1024 轮）。单个估计方差大，所以分 16384 个桶各自记录再调和平均。固定 12KB（16384 桶 × 6bit），标准误差 0.81%。`PFADD`/`PFCOUNT`/`PFMERGE`（可合并多天算月 UV）。**不能取出元素、不能问"某人来过没"**——只能计数。
- **Bitmap**：本质是 String 的位操作。签到：`SETBIT sign:{uid}:{月} {第几天} 1`；月签到天数 `BITCOUNT`；连续签到判断用 `BITFIELD` 取出位段判断。活跃统计：每天一个 bitmap（offset = 用户 ID），周活 = 7 个 bitmap `BITOP OR`。1 亿用户一天 100000000/8 ≈ **12MB**。前提：**用户 ID 必须是稠密整数**，稀疏的长 ID 浪费巨大（offset 决定分配大小）。
- **布隆过滤器**：类比一个只记特征、不记脸的保安——每个进门的人他记 k 个特征（发型、身高、衣着…）。再有人来，他说"**绝对没来过**"一定可靠（k 个特征里有一个完全没见过）；说"来过"却可能认错（k 个特征恰好都被**不同的人**凑齐了，这就是误判）。而且他没法"忘记"某个人——特征是大家共享的，抹掉会连带误伤别人（**不支持删除**）。技术对应：位数组 + k 个 hash 函数，加入时把 k 个位置置 1；查询时 k 个位置**都为 1 才"可能存在"，任一为 0 则"一定不存在"**——所以适合做"不存在"的拦截（缓存穿透）。误判率由位数组大小和元素数决定；要删除用计数布隆或布谷鸟过滤器，或定期重建。RedisBloom 模块提供，8.0 起进入核心。
- **GEO**：`GEOADD`/`GEOSEARCH` 做附近的人。底层就是 ZSet：经纬度做 GeoHash 编码（二分逼近交错取位）成 52bit 整数当 score——**GeoHash 前缀相同的点空间上相邻**，所以范围查询变成 ZSet 的 score 区间查询。知道它没有新数据结构、是 ZSet 的应用，就是加分项。

### 8. 生产实践要点

**大 key**：
- 标准（参考值）：String > 10KB；集合类元素数 > 5000 或总大小 > 10MB。
- **为什么危险，逐条对应单线程**：读写一个 1MB 的 value，网络传输和序列化都在主线程；`HGETALL` 百万级 Hash 是 O(n) 阻塞；**DEL 大 key、大 key 过期、淘汰大 key 都是同步释放内存**（不开 lazyfree 的话）；集群迁移时大 key 整体搬，阻塞两端。
- 排查：`redis-cli --bigkeys`（基于 SCAN，只报每种类型最大的，粗筛）、`redis-cli --memkeys`、RDB 离线分析（rdb-tools，最准且不影响线上）。
- 治理：拆分（大 Hash 按 `field % N` 分桶）；删除用 `UNLINK`；集合类要清空用 `HSCAN` + 批量 `HDEL` 渐进删；开 `lazyfree-lazy-expire yes` 等配置。

**热 key**：
- 危害：集群再大，一个 key 只在一个分片上，热 key 把单分片打满，其他分片闲着——**集群解决容量，解决不了单 key 热点**。
- 发现：`redis-cli --hotkeys`（要求 LFU 策略）、代理层/客户端埋点统计、云产品自带热 key 分析。
- 治理：① **本地缓存**扛第一层（哪怕只缓存 1 秒，10w QPS 也只剩 1 QPS 到 Redis）；② key 打散：`hotkey:{0~N 随机后缀}` 存 N 个副本，读随机挑一个，写要写全 N 份——用写放大换读分散。

**慢操作清单**：`KEYS *`（O(n) 全键空间扫描，线上禁用，用 `SCAN` 游标渐进遍历——注意 SCAN 基于 dict 游标的高位进位法，保证 rehash 期间不漏，可能重复，业务要幂等）；`HGETALL`/`SMEMBERS` 大集合（改 `HSCAN`/`SSCAN`）；`SINTERSTORE` 等集合运算 O(n×m)；事务里塞大量命令。排查工具：`SLOWLOG GET`（默认记录 >10ms 的命令）、`latency monitor`。

**批量与原子的选择**：
- `MGET/MSET`：一次 RTT，但集群下 key 要同槽。
- **pipeline**：把 N 条命令攒一起发，N 次 RTT 变 1 次。**不是原子的**，中间可能插入其他客户端的命令；攒太多会撑爆服务端输出缓冲，建议几百条一批。
- **Lua**：原子（执行期间不会插入其他命令）。**正因为原子，脚本必须短**——慢脚本阻塞整个实例；超过 `busy-reply-threshold`（默认 5s）后 Redis 对其他请求回 BUSY 错误，此时若脚本已执行过写命令，连 `SCRIPT KILL` 都杀不掉（杀了破坏原子性），只能 `SHUTDOWN NOSAVE`。
- **事务 MULTI/EXEC 为什么没人用**：只保证"命令连续执行不被插队"，**不支持回滚**——入队后某条命令运行时报错（比如对 String 执行 LPUSH），前面的不回滚、后面的照常执行。官方的解释是回滚机制复杂且这类错误属于编程错误。需要"要么全做要么不做"的语义，用 Lua（虽然 Lua 中途报错同样不回滚已执行的写，但至少能在脚本内先做完所有检查再写）。`WATCH` 提供乐观锁（CAS），但实践中还是 Lua 更直接。

## 五、面试快问快答清单

1. 为什么快？→ 内存 + 单线程免锁 + epoll 多路复用 + 高效编码。瓶颈在网络不在 CPU。
2. 单线程指什么？→ 命令执行单线程；2.6 起有后台 fsync 线程，4.0 lazyfree，6.0 IO 多线程（只并行读写解析，执行仍串行）。
3. 数据结构为什么快？→ 每种类型两套编码：小数据用连续内存的 listpack（省指针、缓存友好），大数据自动切 hashtable/跳表（阈值如 128 个元素）。
4. 跳表 vs 红黑树？→ 公交快线：先快线跳大站、再慢车到站；选它是因为范围查询沿底层链表顺着走就行，实现也简单。
5. ziplist 为什么被 listpack 换掉？→ 旧结构每个元素记"前一个的长度"，一个变长引发多米诺式级联更新；新结构只记自己的长度。
6. 渐进式 rehash？→ 图书馆不闭馆搬家：新旧两表并存，每次操作顺手搬一桶；查两张表、新增只写新表。
7. RDB 的坑？→ fork 阻塞与内存成正比、COW 写放大（关 THP）、maxmemory 留一半余量。
8. AOF everysec 丢多少？→ 最多 2 秒（主线程容忍上次 fsync 落后 2s，超了反而阻塞主线程）。
9. AOF 重写 7.0 改了什么？→ Multi-part：base + incr + manifest，消灭重写缓冲区双写。
10. 增量复制靠什么？→ repl_backlog 环形缓冲 + offset；默认 1MB 太小，按"写速率×重连时间×2"配。
11. quorum 和 majority？→ quorum 定客观下线票数；leader 选举要过半数哨兵，决定还能不能执行切换。
12. 为什么 16384 槽？→ 心跳带槽位图 2KB（65536 则 8KB）；集群建议 ≤1000 节点，粒度够。
13. MOVED vs ASK？→ 永久迁移（更新路由表）vs 迁移中临时跳转（带 ASKING、不更新路由表）。
14. 过期键怎么删？→ 惰性 + 定期（每 100ms 抽 20 个，过期超 25% 再抽，单轮上限 25ms）；slave 等 master 的 DEL。
15. LRU 怎么实现的？→ 近似：24bit 时间戳 + 随机采样 5 个 + 16 槽淘汰池；LFU 用 8bit 对数计数 + 衰减。
16. 缓存为什么先更新 DB 再删缓存？→ 两种顺序都有竞态窗口，反过来的窗口是毫秒级必现风险，正序要求"读比写慢"概率极低；删除失败用延迟双删或 binlog+MQ 重试兜底；强一致做不到，只能缩小窗口。
17. 分布式锁四件套？→ SET NX PX 一条命令原子加锁；uuid 防误删；Lua 原子解锁；看门狗 1/3 周期续期（Redisson 传了 leaseTime 就不续）。
18. 主从切换丢锁怎么办？→ 根源是异步复制；RedLock 多数派方案存在 GC 停顿/时钟跳变争议（Martin vs antirez）；实践用幂等兜底，强正确性用 DB 约束或 ZK/etcd。
19. Redis 6.0 IO 多线程流程？→ 主线程收集就绪客户端 → 分发 IO 线程并行读+解析 → 屏障 → 主线程串行执行 → 分发并行写 → 屏障。默认只多线程写，读要开 io-threads-do-reads。
20. Redis 和 Valkey？→ 2024 许可证改 RSALv2/SSPL 引发 Linux 基金会分叉 Valkey；2025 Redis 8.0 加回 AGPLv3 并把 Stack 模块并入核心。
