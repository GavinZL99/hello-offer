## 一、MySQL 要解决什么问题

数据为什么不直接存在内存（如 Redis）或普通文件里？因为业务要的是同时满足这四件事，而它们恰恰是关系型数据库的看家本领：

1. **持久化且不丢**：断电、宕机后数据还在、且不能只恢复一半。
2. **结构化查询**：用 SQL 灵活地按各种条件查、连表、聚合，而不是只能按 key 取。
3. **事务（ACID）**：一组操作要么全成、要么全不成（转账：扣款和入账必须同时成立）。
4. **并发安全**：成千上万的连接同时读写，不能互相踩乱。

**类比**：MySQL 像一个**有严格规章的银行**——账本永久保存（持久化）、能按任意条件查账（SQL）、每笔交易要么完整完成要么撤销（事务）、再多柜员同时办业务也不会把账搞乱（并发控制）。后面的索引、MVCC、锁、日志，全是为了把这四件事又快又稳地做到。

**和 NoSQL/Redis 的分工**：Redis 是"快但弱约束"的内存结构服务器；MySQL 是"强约束、强一致、能复杂查询"的持久化底座。**钱、订单、库存这类要强一致和事务的数据放 MySQL**，缓存/排行榜/计数放 Redis，二者配合（见《Redis.md》缓存一致性）。

## 二、架构：一条 SQL 是怎么执行的

MySQL 分**两层**（这个分层是理解很多问题的钥匙，深分页那题就靠它）：

- **Server 层**：连接器、分析器、优化器、执行器，以及 **binlog**。负责 SQL 的解析、优化、执行调度——**和具体存储引擎无关**。
- **存储引擎层**：真正存数据、取数据的地方，插件式（InnoDB、MyISAM…）。**redo log、undo log、索引、锁、MVCC 都在 InnoDB 这一层**。

**一条查询 SQL 的旅程**（`select * from user where id=1`）：
```
客户端 →① 连接器：建连接、校验权限
       →② 分析器：词法/语法分析，看懂这是条查询
       →③ 优化器：选用哪个索引、多表用什么连接顺序（基于成本估算）
       →④ 执行器：调用存储引擎接口，一行行取数据、过滤、返回
       →⑤ InnoDB：从索引/page cache/磁盘把数据捞上来
```
（MySQL 8.0 已删掉查询缓存——它一旦表有更新就整张表的缓存失效，命中率低、弊大于利。）

**一条更新 SQL 还多了写日志**（`update user set age=20 where id=1`）：在上面基础上，InnoDB 要改数据 + 写 **redo log**，Server 层要写 **binlog**，两者通过**两阶段提交**保证一致（见第六章）——这是更新流程的核心考点。

**InnoDB vs MyISAM（为什么默认 InnoDB）**：

| | InnoDB（默认） | MyISAM |
|---|---|---|
| 事务 | 支持 | 不支持 |
| 锁粒度 | 行锁（并发好） | 只有表锁（并发差） |
| 崩溃恢复 | 有 redo log，crash-safe | 无，易损坏 |
| 外键 | 支持 | 不支持 |
| 适用 | 绝大多数场景 | 只读/读多写极少的老场景 |

一句话：要事务、要高并发写、要不丢——选 InnoDB，现在基本没有用 MyISAM 的理由。

## 三、索引 —— 为什么能快

**类比**：索引就是书的**目录**。没目录找一个词要从第一页翻到最后（全表扫描）；有目录先在目录里定位、直接翻到对应页（索引查找）。代价是目录本身占地方、且正文增删时目录要跟着维护（索引占空间、拖慢写）。

### 1. 为什么用 B+ 树（核心，必考）

先看为什么不用那些"看起来也能查得快"的结构：

- **不用 Hash**：哈希按 key 取等值是 O(1)，但**不支持范围查询和排序**（`where age > 20`、`order by`），而数据库一半的查询是范围查询。所以 Hash 索引只在等值场景用，不能当主力。
- **不用二叉搜索树/红黑树**：它们是"瘦高"的——N 个节点树高约 log₂N。数据库数据在**磁盘**上，**树高 = 查一次要读几次磁盘**，而磁盘 IO 是毫秒级、最怕多。1000 万行，红黑树高约 24 层 = 最多 24 次磁盘 IO，太慢。
- **不用 B 树**：B 树已经是"矮胖"的多叉树（一个节点存多个 key、有多个孩子），但它**每个节点都存数据**，导致单个节点能放的 key 变少、树偏高；且范围查询要在树里上下来回跳。

**B+ 树怎么解决**（"矮胖 + 叶子链表"）：
1. **矮胖**：一个节点是一个**数据页（16KB）**，能放几百上千个 key，所以扇出（分叉）极大。**1000 万行数据，B+ 树通常只有 3 层**——查任意一行最多 3 次磁盘 IO（而且上面 1~2 层常驻内存，实际更少）。这是它碾压二叉树的根本。
2. **非叶子节点只存索引、不存数据**：所以每个节点能塞更多 key，树更矮。**数据全在叶子节点**。
3. **叶子节点串成有序双向链表**：范围查询（`where age between 10 and 20`）找到起点后，**顺着叶子链表横着扫**就行，不用回树里跳——范围查询和排序天然高效。

**一句话**：B+ 树是为"数据在磁盘、要少 IO、要支持范围查询"量身定做的——矮胖少 IO、叶子链表利于范围扫。

### 2. 聚簇索引 vs 二级索引、回表、覆盖索引

这是 InnoDB 索引的核心区别（深分页那题就卡在这）：

- **聚簇索引（主键索引）**：叶子节点**直接存整行数据**。所以按主键查，到叶子就拿到了全部字段。一张表只有一个聚簇索引（数据本身就按主键组织存放）。类比：新华字典正文**按拼音排序**，翻到拼音就看到字的全部解释。
- **二级索引（普通索引）**：叶子节点**只存索引列 + 主键值**，没有整行。类比：字典的部首目录，只给你"页码（主键）"，不给解释。
- **回表**：用二级索引查到主键后，**再拿主键回聚簇索引捞整行**，这次二次查找叫回表。比如 `where name='张三'` 走 name 索引拿到主键，再回表取整行。
- **覆盖索引**：如果你要的列在二级索引里**全都有**（如 `select id, name from t where name=?`，id+name 索引里全齐），就**不用回表**，直接从二级索引返回。这是索引优化的重要手段——**只查需要的列、让索引覆盖，能省掉回表**。

（深分页 `LIMIT 1000000,10` 为什么慢、怎么用覆盖索引+延迟关联优化，详见《后端场景题.md》。）

### 3. 联合索引与最左前缀

**联合索引**：把多个列建一个索引，如 `(a, b, c)`。它的排序规则是**先按 a 排，a 相同再按 b，b 相同再按 c**（像先按姓排、再按名排的电话簿）。

**最左前缀原则**：因此查询必须**从最左列开始、连续命中**才能用上索引：
- `where a=1`、`where a=1 and b=2`、`where a=1 and b=2 and c=3` → 能用。
- `where b=2`（跳过 a）、`where a=1 and c=3`（跳过 b，c 用不上索引、只能 a 部分用）→ 用不上或只用一半。
- 类比：电话簿按"姓+名"排，你知道姓能快速定位；只知道名、不知道姓，整本翻——排序帮不上忙。

**实用推论**：把**区分度高、最常用**的列放左边；范围查询（`>`、`<`、`between`）会**中断后续列的索引使用**（`a=1 and b>2 and c=3`，c 用不上），所以范围列尽量放后面。

### 4. 索引失效的常见场景（高频）

索引在这些情况会失效、退化成全表扫描：
- **对索引列做运算/函数**：`where age+1=20`、`where date(create_time)='2026-01-01'` → 失效（索引存的是原始值，算过就对不上了）。改写成 `where age=19`、`where create_time between ...`。
- **隐式类型转换**：phone 是字符串，`where phone=138...`（传了数字）→ 触发类型转换 = 相当于对列做函数 → 失效。
- **`like` 以通配符开头**：`like '%abc'` 失效（无法用前缀定位），`like 'abc%'` 可用。
- **`or` 连接的条件有一个没索引** → 可能全表扫。
- **最左前缀不满足**（见上）。
- 用 `EXPLAIN` 看执行计划（type、key、rows、Extra）是排查索引是否生效的标准手段。

### 5. 索引下推（ICP，加分项）

**场景**：联合索引 `(name, age)`，查询 `SELECT * FROM user WHERE name LIKE '张%' AND age = 20`。

**先看一个别扭的点**：按最左前缀，`name LIKE '张%'` 是**范围条件**，它一出现，后面的 `age` 就**没法用来在索引里定位（seek）了**（范围列中断后续列的索引定位，见联合索引那节）。所以索引只能帮你定位到"name 以张开头"这一段，`age=20` 不能用于缩小查找范围。
**但关键在于**：`age` 虽然不能用来 seek，它的值却**实实在在存在索引里**（它是联合索引的第二列）。扫描"张%"这段时，每条索引记录里**本来就带着 age 的值**。这就给了优化的空间。

**没有索引下推时（5.6 之前）——大量无用回表**：
```
存储引擎层：扫索引拿到所有"张%"的记录(只看了name) ──每条都回表取整行──▶ Server层：逐行判断 age=20，丢掉不符的
```
存储引擎对每一条"张%"记录**都回表**取整行、上交给 Server 层，由 Server 层再判断 age——哪怕这条 age≠20、注定要被丢掉，回表也已经白做了。

**有了索引下推时（5.6+）——回表前先过滤**：
```
存储引擎层：扫索引"张%"时，顺手用记录里已有的 age 值判断 age=20 ──只有通过的才回表──▶ Server层
```
存储引擎在**回表之前**，就用索引记录里**现成的 age 值**把 `age=20` 这个条件先过滤掉，不满足的根本不回表。

**为什么叫"下推"**：本该在 **Server 层**做的 `age=20` 过滤，被**下推到存储引擎层**、在扫索引时就顺手做了（呼应架构那节的两层划分）。

**效果（具体数字）**：假设 1000 个"张 X"，但只有 5 个 age=20——
- 无 ICP：回表 1000 次，Server 层过滤到 5 行，**995 次回表纯属浪费**。
- 有 ICP：索引扫描时用 in-index 的 age 先过滤到 5 行，**只回表 5 次**。

**前提**：下推的条件，其涉及的列必须**就在当前用的这个索引里**（age 在 `(name,age)` 里，才能在引擎层判断）。如果条件是 `AND city='北京'` 而 city 不在该索引，引擎手里没有 city、判断不了，就推不下去、只能回表后由 Server 层判断。

**和覆盖索引的区别**：覆盖索引是"要的列索引里全有 → 压根不回表"；ICP 是"还是要回表（如 `SELECT *`），但**在回表前先用索引里的列过滤、减少回表次数**"。两者都靠"索引里已有的列"做文章，但一个是免回表、一个是减回表。

**怎么确认生效**：`EXPLAIN` 的 `Extra` 列出现 **`Using index condition`** 就是用上了 ICP。

## 四、事务与 MVCC

### 1. ACID 与隔离级别

**ACID**：原子性（Atomicity，要么全做要么全不做，靠 undo log）、一致性（Consistency，最终目的）、隔离性（Isolation，并发事务互不干扰，靠锁 + MVCC）、持久性（Durability，提交后不丢，靠 redo log）。

**并发会出三个问题**（隔离性不够时）：
- **脏读**：读到了别的事务**还没提交**的数据（对方可能回滚，你读了个根本不存在的值）。
- **不可重复读**：同一事务内两次读**同一行**，值不一样（别的事务改了该行并提交）。针对 **UPDATE**。
- **幻读**：同一事务内两次按**同一条件查**，行数变了（别的事务插入了新行并提交）。针对 **INSERT**。

**四个隔离级别**，逐级更严、并发性逐级更差：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| 读未提交 RU | ✗会 | ✗会 | ✗会 |
| 读已提交 RC | ✓防 | ✗会 | ✗会 |
| 可重复读 RR（MySQL 默认） | ✓防 | ✓防 | ✓基本防* |
| 串行化 Serializable | ✓防 | ✓防 | ✓防 |

\* RR 下，InnoDB 用 **MVCC（快照读）** 防住了快照读的幻读，用 **间隙锁（当前读）** 防住了当前读的幻读，所以 InnoDB 的 RR 基本没有幻读问题——这是 MySQL 的一个特色（标准 SQL 里 RR 是允许幻读的）。

### 2. MVCC：多版本并发控制（最核心，必考）

**要解决的问题：读写并发冲突。** 注意不是"多个事务读同一数据"——读和读从不冲突，大家一起读没问题；真正的麻烦是**一个事务在读、另一个事务同时在改同一份数据**。

举个例子（转账）：事务 A 分两步执行"张三 −100、李四 +100"（**写**）；事务 B 此刻要统计"张三 + 李四"的总额（**读**）。如果 B 恰好在 A 改完第一步、还没改第二步时去读，就会读到"凭空少了 100"的**不一致中间状态**。

不用 MVCC 怎么避免？只能**加锁**：B 读时锁住这些行，A 就得等 B 读完才能改（反之亦然）。后果是**读和写互相阻塞、排队进行**——而系统大多读多写少，这样并发性极差。

**MVCC 用"多版本 + 快照"同时解决两个痛点**：
1. **读不到中间状态**：B 读的是"A 开始改之前那个一致的快照版本"，自然看不到改了一半的样子。
2. **读写不互相阻塞**：A 改数据是生成**新版本**、不动旧版本；B 读的是旧版本快照——各看各的，**读不加锁、不用等写**。（只有写和写之间仍需加锁。）

一句话：**MVCC 让"写生成新版本、读读旧快照"，从而在读写并发时既保证读到一致数据、又不像加锁那样让读写互相等待。**

**类比**：给数据**拍快照**。每个事务查询时，相当于对整个数据库"拍了张照片"，之后它看到的一直是照片里那个时刻的样子，别人后来怎么改都不影响这张照片。

**怎么实现的，三个零件**（每个事务有一个唯一且递增的**事务 id**，下面的判断都靠它）：

1. **每行的两个隐藏字段**：每行除了你写的字段，还藏着 `DB_TRX_ID`（事务 id，记"最近改过这行的是哪个事务"）和 `DB_ROLL_PTR`（回滚指针，指向这行的上一个旧版本存在哪）。
2. **版本链**：每次修改一行，旧值不删除，而是存进 `undo log`（回滚日志），再用回滚指针 `DB_ROLL_PTR` 把"最新版本 → 上一个旧版本 → 更旧版本"串成一条链。一行数据于是有了一串历史版本。
3. **`ReadView`（读视图）**：快照读时生成的一张"可见性判断依据表"，记录"生成它这一刻还有哪些事务没提交"。读某行时顺着版本链从新到旧逐个看，挑出**对当前事务可见的那个版本**返回。

**RC 和 RR 的区别，就一句话——`ReadView`（读视图）的生成时机不同：**
- **读已提交（Read Committed, RC）**：**每次查询都新建一张 `ReadView`**，每次都能看到别的事务最新提交的内容 → 会"不可重复读"。
- **可重复读（Repeatable Read, RR）**：**只在第一次查询时建一张 `ReadView`，整个事务一直复用它** → 快照始终不变 → "可重复读"。

#### 深入：`ReadView` 怎么判断"哪个版本可见"

一张 `ReadView`（读视图）里记了四个字段（左边是源码里的名字，右边是它的意思）：
- `m_ids`（活跃事务列表）：生成 `ReadView` 这一刻，**还没提交的那些事务的 id** 集合。
- `min_trx_id`（最小活跃 id）：`m_ids` 里最小的那个 id。
- `max_trx_id`（下一个待分配 id）：生成时系统**下一个准备发出去的事务 id**（= 当前用到的最大 id + 1）。
- `creator_trx_id`（本事务 id）：生成这张 `ReadView` 的事务自己的 id。

读一行时，看每个版本上记的 `DB_TRX_ID`（改过它的事务 id，下面简称 `trx_id`），按下面四条判断它可不可见；**不可见就顺 `DB_ROLL_PTR`（回滚指针）往下看更旧的版本，直到找到可见的**：
1. `trx_id == creator_trx_id`（等于本事务 id）→ 这版本是**自己改的** → 可见。
2. `trx_id < min_trx_id`（小于最小活跃 id）→ 改它的事务在 `ReadView` 生成前**早就提交了** → 可见。
3. `trx_id >= max_trx_id`（≥ 下一个待分配 id）→ 改它的事务是 `ReadView` 生成**之后才开始的** → 不可见。
4. `min_trx_id <= trx_id < max_trx_id`（落在两者之间）→ 看它**在不在 `m_ids`（活跃事务列表）里**：在（当时还没提交）→ 不可见；不在（当时已经提交）→ 可见。

一句话概括：**只有"自己改的"和"`ReadView` 生成前就已提交的"版本才可见**；当时还没提交的、以及 `ReadView` 之后才开始的事务所做的修改，一律不可见。

**一个例子走通，顺便看清 RC/RR 的差别**：一行 `余额=100`（由早已提交的事务 id=90 写入）。现在事务 200 正在跑，另有事务 300 并发。
```
事务200 第一次查询 → 生成 ReadView：
   m_ids（活跃列表）={200,300}，min_trx_id（最小活跃）=200，max_trx_id（下一个）=301，creator_trx_id（本事务）=200
事务300 把余额改成 200 并【提交】
事务200 第二次查询同一行：
```
- **可重复读（RR）**：复用第一张 `ReadView`。最新版本是 `余额=200`、它的 `trx_id`=300 → 落在 `min_trx_id`(200) 和 `max_trx_id`(301) 之间，且 **300 在 `m_ids` 里 → 不可见** → 顺回滚指针看旧版本 `余额=100`、`trx_id`=90 < `min_trx_id`(200) → 可见。**最终读到 100**（可重复读：哪怕 300 已提交，事务 200 还是看到旧值）。
- **读已提交（RC）**：第二次查询**重新建一张 `ReadView`**，此时 300 已提交、不在活跃列表里了 → `m_ids`={200}、`min_trx_id`=200、`max_trx_id`=301。最新版本 `trx_id`=300 落在区间内、但**不在 `m_ids` 里 → 可见**。**最终读到 200**（读已提交：看到了 300 的提交）。

同一条版本链、同一套规则，仅因为 `ReadView` 生成的时机不同，可重复读读到旧值、读已提交读到新值——这就是两个隔离级别差异的根本。

同一条版本链、同样的规则，只因为 ReadView 生成时机不同，RR 读到旧值、RC 读到新值——这就是两个隔离级别差异的本质。

**快照读 vs 当前读**（容易混）：
- **快照读**：普通 `SELECT`，读 MVCC 的历史快照，**不加锁**。
- **当前读**：读**最新版本**并**加锁**——`SELECT ... FOR UPDATE`（排他锁）、`SELECT ... LOCK IN SHARE MODE`（共享锁）、以及 `UPDATE`/`DELETE`（改之前必须读到最新的，否则会改错）。

### 3. RR 怎么防幻读

- **快照读**：一直读同一个 ReadView，别人插入的新行不在快照里，**天然不幻读**。
- **当前读**：读的是最新数据，可能撞见别人新插入的行 → InnoDB 用**间隙锁/临键锁**把扫描范围"锁起来"，不让别的事务往这个范围里插入，从而防住当前读的幻读（见下一章）。

## 五、锁

### 1. 按粒度分

- **全局锁**：锁整个库（`FTWRL`），用于全库逻辑备份。
- **表锁**：锁整张表，并发差。MyISAM 只有表锁，这是它并发弱的根源。
- **行锁**：InnoDB 支持，锁到具体行，并发好。**行锁是加在索引上的**——如果 SQL 没走索引，会退化成锁全表的效果（实际是锁了所有行），这是个隐藏坑。

### 2. 行锁的三种形态（InnoDB，必考）

- **记录锁（Record Lock）**：锁住**某一条具体记录**。
- **间隙锁（Gap Lock）**：锁住记录**之间的"空隙"**（一个开区间），**不让别人往这个空隙里插入**——专门用来防幻读。比如锁住 (10, 20) 这个间隙，谁都不能插入 id=15 的行。
- **临键锁（Next-Key Lock）**：= 记录锁 + 间隙锁，锁住"记录本身 + 它前面的间隙"（左开右闭区间）。**RR 隔离级别下行锁默认就是临键锁**，这正是 InnoDB 在 RR 下防住当前读幻读的机制。

**类比**：记录锁 = 锁住某个座位；间隙锁 = 锁住座位之间的过道，不让人加塞新椅子；临键锁 = 座位 + 它旁边的过道一起锁。防的就是"我数完一排座位，回头一看你又塞进来一个"（幻读）。

#### 深入：一条 SQL 到底锁了什么（RR 下加锁分析，最爱考）

面试常给一张表 + 一条 `... for update`，问锁了哪些范围。规则取决于三件事：**走的是唯一索引还是普通索引、等值还是范围、命中还是没命中**。**RR 下行锁默认是临键锁**（左开右闭 `(前一个值, 当前值]`），下面的"退化"都是在此基础上的简化。

设表 `t` 主键 `id`，已有记录 `id = 5, 10, 15, 20`，于是存在这些间隙：`(-∞,5) (5,10) (10,15) (15,20) (20,+∞)`。

- **① 唯一索引 + 等值 + 命中**（`where id=10 for update`）：临键锁**退化成记录锁**，只锁 `id=10` 这一行。因为主键唯一，等值最多一行，不存在"再插一个 id=10"的幻读风险，不需要锁间隙。
- **② 唯一索引 + 等值 + 没命中**（`where id=12 for update`）：临键锁**退化成间隙锁**，锁住 12 所在的间隙 `(10,15)`——不让别人插入 11~14。因为你查 12 现在没有，若不锁这个缝，别人插入一条 12、你再查就"幻"出来了。
- **③ 唯一索引 + 范围**（`where id>=10 for update`）：锁 `id=10` 这条记录 + 它之后的所有临键区间：`[10] 及 (10,15], (15,20], (20,+∞)`——整个范围都锁住，防止范围内任何插入。
- **④ 普通（非唯一）索引 + 等值命中**：即使命中也**不会退化成纯记录锁**。因为普通索引值可重复，要防止"往这个值附近再插一条相同值"，所以锁 = 命中记录的临键锁 + **向后再加一个间隙锁**（锁到下一个不同值之前的缝）。
- **⑤ 没有用到索引**（`where 非索引列=? for update`）：全表扫描，会给**扫到的每一条记录都加临键锁 = 几乎锁全表**。这就是第 1 节说的"行锁加在索引上，没走索引退化成锁全表"的具体表现——也是生产事故高发点（一条没走索引的更新把全表锁住）。

**怎么记**：唯一索引等值命中→只锁一行（无幻读风险）；没命中或范围或非唯一→要锁间隙/临键（防插入）；没索引→锁全表。核心永远是那句：**间隙/临键锁存在的唯一目的，就是"不让你往我扫过的范围里插新行"，从而防住幻读。**

### 3. 共享锁 / 排他锁 / 意向锁

- **共享锁（S 锁，读锁）**：多个事务可同时持有，都能读、都不能写。
- **排他锁（X 锁，写锁）**：只能一个事务持有，写时加，别人读写都挡。
- **意向锁（IS/IX）**：表级的"**意向声明**"。一个事务要给某行加行锁前，先在**表上**加一个意向锁，表示"我要在这表里加行锁了"。作用：别的事务想加**表锁**时，看一眼表上的意向锁就知道"有人在用行锁"，不用去逐行检查——**让表锁和行锁的冲突判断变成 O(1)**。

### 4. 死锁

两个事务互相持有对方想要的锁、互相等 → 死锁。InnoDB 有**死锁检测**，发现后回滚其中一个（代价小的）让另一个继续。预防：① 让事务按**固定顺序**访问资源（都先改 A 再改 B，就不会交叉等待）；② 事务尽量短小、快提交；③ 降低隔离级别/减少间隙锁范围。

## 六、日志：redo / undo / binlog（更新流程的核心）

MySQL 有三个关键日志，各管一摊，**别混**：

### 1. redo log（重做日志）—— 保证持久性、崩溃不丢

**问题**：每次更新都直接改磁盘上的数据页？太慢（随机 IO）。但只改内存（buffer pool）不落盘，宕机就丢了。
**方案：WAL（Write-Ahead Logging，预写日志）**。改数据时，先把"这一页要怎么改"**顺序写进 redo log**（顺序写极快，见《消息队列.md》Kafka 顺序写），就告诉客户端成功了；脏页之后再由后台慢慢刷盘。宕机重启后，**拿 redo log 把没来得及刷盘的改动重做一遍**，数据就不丢了（crash-safe）。
**类比**：先在小本子上记一笔"要把 A 账户改成 100"（redo log，快），实际账本（数据页）回头再慢慢誊。万一誊之前停电了，照着小本子重做一遍就行。
- redo log 是 **InnoDB 特有**的，循环写（固定大小，写满回头覆盖已刷盘的部分）。

### 2. undo log（回滚日志）—— 保证原子性 + 支撑 MVCC

记录"改之前的样子"，两个用途：① 事务要回滚时，照 undo log 把数据**改回去**（原子性：要么全成要么全不成）；② 给 MVCC 提供**历史版本链**（见第四章）。
**类比**：后悔药 / 修改前的存档。

### 3. binlog（归档日志）—— 主从复制 + 数据归档

记录所有**对数据的更改操作**（逻辑日志，记的是"做了什么 SQL/什么行变更"）。
- 在 **Server 层**，所有存储引擎都有（不像 redo 只属于 InnoDB）。
- 追加写（不覆盖），可永久保留。
- 两大用途：**主从复制**（从库拉主库 binlog 重放，见第七章）、**数据恢复/归档**（按 binlog 把库恢复到任意时间点）。

**redo vs binlog 的区别**（高频）：redo 是 InnoDB 的、物理日志（记"某页某位置改成什么"）、循环写、用于崩溃恢复；binlog 是 Server 层的、逻辑日志（记"什么变更"）、追加写、用于复制和归档。

### 4. 两阶段提交（为什么需要，必考）

更新时要写 redo（InnoDB）和 binlog（Server 层）两个日志。**如果不做两阶段、简单地先写一个再写另一个，会主从不一致**：
- 假设先写 redo、再写 binlog：redo 写完、binlog 还没写就崩溃 → 主库靠 redo 恢复后**有**这条改动，但从库靠 binlog、**没有** → 主从数据不一致。

**两阶段提交（2PC）**把它俩绑成一个原子操作：
```
① prepare：写 redo log，标记为 prepare 状态
② 写 binlog
③ commit：把 redo log 标记为 commit
```
崩溃恢复时的判断规则：redo 已 commit → 直接生效；redo 是 prepare 但 **binlog 完整** → 也提交（因为 binlog 有了、从库会同步到，必须保持一致）；redo 是 prepare 但 binlog 不完整 → 回滚。**靠这个规则保证 redo 和 binlog 的状态始终一致，主从不分叉。**

## 七、主从复制与高可用

**原理（binlog 复制）**：
```
主库：写 binlog
从库：① IO 线程拉取主库 binlog，写入本地 relay log（中继日志）
      ② SQL 线程重放 relay log，把变更应用到从库
```

**主从延迟**（常见痛点）：从库重放跟不上主库写入，导致"刚写完主库、立刻读从库读到旧数据"。原因：从库单线程重放（旧版本）、大事务、从库性能弱等。缓解：并行复制（多线程重放）、读写分离时强一致读走主库、关键读后短暂走主库。

**读写分离**：写走主库、读走从库，分摊读压力。配合延迟问题要做好"读自己刚写的内容走主库"等处理（呼应《后端场景题.md》缓存一致性里"主从延迟"的讨论）。

**高可用**：主库挂了要能切换（故障转移）。常见 MHA、MGR（MySQL Group Replication）、或云上的高可用版。核心是选出新主、让从库指向新主、保证切换时不丢已提交数据。

## 八、选型与实践

**InnoDB vs MyISAM**：要事务/行锁/崩溃恢复 → InnoDB（默认、绝大多数场景）；纯只读老系统才可能 MyISAM。

**MySQL vs PostgreSQL**（常被问）：MySQL 生态广、读多写少的互联网业务成熟、上手简单；PostgreSQL 功能更强（复杂查询、窗口函数、JSON、地理 GIS、更严格的 SQL 标准），复杂分析和数据完整性要求高时更优。互联网业务主流仍是 MySQL。

**什么时候不该用关系库**：海量非结构化/半结构化数据、超高写吞吐且弱事务（日志、埋点）→ 考虑 NoSQL（MongoDB、HBase、ClickHouse 等）；缓存/排行榜 → Redis。关系库的强项是**事务 + 强一致 + 复杂查询**，没这些需求时它不是唯一解。

**索引设计原则**：① 给高频查询条件、连接列、排序列建索引；② 联合索引按最左前缀和区分度排列；③ 尽量用覆盖索引省回表；④ 不滥建（每个索引都拖慢写、占空间）；⑤ 用 `EXPLAIN` 验证是否生效。

**分库分表、深分页优化**：见《后端场景题.md》（分片键选择、跨库问题、游标翻页、延迟关联）。

## 九、面试快问快答清单

1. 为什么索引用 B+ 树？→ 矮胖（一页存几百 key，千万行才 3 层、少磁盘 IO）+ 叶子链表（范围查询快）；不用 Hash（不支持范围）、不用二叉树（太高、IO 多）。
2. 聚簇索引 vs 二级索引？→ 聚簇叶子存整行（按主键），二级叶子存主键值，二级查整行要回表。
3. 什么是回表/覆盖索引？→ 二级索引拿主键再查聚簇取整行=回表；要的列索引里全有就不用回表=覆盖索引。
4. 最左前缀？→ 联合索引按列顺序排，查询必须从最左列连续命中才用得上；范围列会中断后续列。
5. 索引失效场景？→ 对列做运算/函数、隐式类型转换、`like '%x'`、不满足最左前缀。
6. ACID 靠什么实现？→ 原子性=undo log，持久性=redo log，隔离性=锁+MVCC，一致性=前三者共同保证的目标。
7. 三个并发问题？→ 脏读(读未提交)、不可重复读(行被改)、幻读(范围多了行)。
8. MySQL 默认隔离级别？→ 可重复读 RR；InnoDB 用 MVCC+间隙锁把幻读也基本防住了。
9. MVCC 原理？→ 行隐藏 trx_id+回滚指针 + undo log 版本链 + ReadView，读历史可见版本、不加锁。
10. RC 和 RR 的区别？→ ReadView 生成时机：RC 每次 SELECT 都新建（能读到新提交=不可重复读），RR 整个事务复用第一次的（可重复读）。
11. 快照读 vs 当前读？→ 普通 SELECT 读快照不加锁；FOR UPDATE/UPDATE/DELETE 读最新版本并加锁。
12. 间隙锁/临键锁？→ 间隙锁锁记录间空隙防插入(防幻读)；临键锁=记录锁+间隙锁，RR 下行锁默认形态。
13. redo / undo / binlog 区别？→ redo(InnoDB,物理,循环写,崩溃恢复)、undo(回滚+MVCC版本)、binlog(Server层,逻辑,追加,复制归档)。
14. 为什么要两阶段提交？→ 让 redo 和 binlog 状态一致，否则崩溃后主库(redo)和从库(binlog)数据不一致。
15. 主从延迟怎么办？→ 并行复制、关键读走主库、读自己刚写的走主库。
16. 行锁会失效吗？→ 行锁加在索引上，SQL 没走索引会退化成锁全表。
