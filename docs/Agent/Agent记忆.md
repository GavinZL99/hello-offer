# Agent 记忆：设计、存储与淘汰

## 一、解决什么问题

默认情况下，**会话一结束 Agent 就"失忆"**——不记得用户偏好、不记得上次决策、不记得之前学到的技巧。单会话内长对话也记不住全部细节（上下文有限）。Agent 要"越用越懂你"、要跨会话延续，就必须有**记忆**：把关键状态持久化下来、按需取回。

记忆和上下文的区别：**上下文**是当前会话窗口里的对话（会话结束即逝、还会爆会漂）；**记忆**是跨会话持久化、按需调取的外部状态（详见《上下文管理》）。

## 二、流程是什么

记忆的生命周期四步：

```
写入：模型主动写 / 规则自动写 → 把要点落到存储
存储：按类型存到合适的地方（文件/DB/向量库）
召回：需要时检索/读回相关记忆
注入：把召回的记忆拼进当前 context，供模型参考
```

**怎么触发写入**：可以是模型自己判断"这事值得记"主动调写记忆的工具；也可以是规则触发（每次会话结束自动摘要存档）。召回同理：模型主动调读记忆工具，或每轮自动注入相关记忆。

这四步里有三个机制最关键、也最容易被一笔带过——**总结（把对话变成记忆）、压缩（把长上下文压短）、加载（把记忆取回来用）**。单独展开讲。

### 机制一：记忆怎么总结（把对话变成可复用的记忆）

原始对话很长、废话多，不能整段存。要"总结"成精炼要点再落库。

**谁来总结**：用一个 LLM 调用，输入"这段对话/这次任务"，输出"提炼出的要点"。可以是主 Agent 自己干，也可以派一个专门的"记忆整理"子 Agent（独立上下文、不污染主对话）。

**总结什么、不总结什么**（决定记忆质量）：
- **该记**：用户偏好（"喜欢 Python"）、长期事实（"项目用 Java 17"）、关键决策及**原因**（"选 A 方案因为性能"）、已解决的坑（"某 API 要特殊 header"）、可复用的流程（→沉淀成 Skill）。
- **不该记**：代码/聊天历史已有的（别重复存）、临时状态、低价值闲聊、会过时的短期信息。

**总结的时机**：
- **实时**：模型在对话中判断"这条值得记"就立刻写——好处是新鲜不丢，坏处是可能写太多噪音。
- **会话结束批量**：每次会话结束跑一次总结，提炼本轮要点存档——更精炼，但要等会话结束。
- **混合**：实时记关键事实、结束时再整理去重压缩。主流做法。

**总结的格式**：一条要点一个文件/一条记录（别一个超大文件），带"为什么"而不只结论，方便后续检索和更新。Hermes 的做法是 MEMORY.md（工作笔记，限 2200 字符）+ USER.md（用户画像）分开存。

### 机制二：上下文怎么压缩（把长 context 压短，别和"总结记忆"混）

注意区分两个概念：**总结记忆**是"把对话变成跨会话的长期记忆"（外存）；**上下文压缩**是"把当前会话内越滚越长的 context 压短"（还在窗口里）。这里讲后者。

**为什么必须压缩**：Agent 循环里每轮追加工具结果/思考块，几十轮后 context 几万 token，既撑爆窗口、又每轮重算钱、还稀释注意力。必须把早期的"详细内容"压成"摘要"塞回去。

**怎么压缩（几种做法）**：
- **服务端 Compaction**（最省事）：接近窗口上限时，API 自动把早期历史总结成一段塞回去。关键坑：必须原样回传 `response.content`（含 compaction 块），只回传文本会丢压缩状态、下次请求出错。压缩会丢细节，关键事实要提前外移到记忆。
- **手动摘要替代**：自己跑一个 LLM 把早期 N 轮压成一段摘要，用摘要替换原文塞回 messages——比服务端 compaction 可控（你能指定压哪些、保留什么），但要自己写逻辑。
- **滑动窗口 + 摘要**：只保留最近 K 轮原文 + 更早的压成一段摘要，是长对话稳定的主流结构（近期细节全、早期有摘要保连贯）。
- ** selective 裁剪**：不总结、直接删过时的工具结果/用过的思考块（context editing），比压缩更激进、更省，但丢信息更多——适合"中间过程用完即弃"的场景。

**压缩什么、保留什么**：保留关键决策、用户偏好、未完成任务、关键工具结果；压/删已完成的中间过程、冗余思考、过时工具输出。原则是"保骨干、压细节"。

### 机制三：记忆怎么加载（把记忆取回来用）

存了记忆不等于用上——要在合适的时机把相关记忆取回来、注入 context，模型才能"想起来"。

**加载时机（几种策略）**：
- **每轮自动注入**：每轮把"和当前任务相关的记忆"拼进 system 或 context 前部——简单可靠，主流做法。缺点是每轮都占 token，要控制注入量。
- **模型主动调取**：给模型一个"读记忆"工具，它判断需要时自己调——省 token（不相关时不加载），但依赖模型主动想起来调（可能漏）。
- **会话开始全量加载**：新会话开头把用户画像/偏好全量读进来——适合"每次都要用的基础信息"（如 USER.md）。

**怎么找"相关"记忆（召回方式，按存储类型）**：
- 文件类少量记忆：直接全量读（量小，如 MEMORY.md 就几千字符，全塞进去最简单）。
- 向量库海量记忆：把当前 query/context 向量化，检索语义最相关的 top-k 条——和 RAG 一个原理。
- 结构化数据：按 key/条件查（如查"该用户的历史决策"）。

**注入到哪**：拼进 system prompt（每轮都在、注意力强、可缓存）或作为 context 前部消息。要控制注入量——别把记忆全塞进去撑爆 context，只注和当前任务相关的。

**主流组合**：会话开始全量加载基础画像（USER.md）+ 每轮向量检索相关长期记忆注入 + 模型主动调取深度记忆。三层结合，既不漏也不爆。

## 三、使用中会遇到哪些问题

1. **存哪**：不同记忆该用什么存储？少了不够、多了乱。
2. **怎么不爆**：只存不删迟早失控（撑爆存储、检索变慢、过时信息误导）。
3. **怎么不忘关键**：什么该记、什么不该记？
4. **怎么纠错**：记错了/过时了怎么办？
5. **安全**：记忆会被检索、可能泄露，敏感信息怎么办？

## 四、解决方案

### 三层记忆设计（业界共识）

分三层，各管不同时间尺度：

| 层 | 是什么 | 内容 | 生命周期 | 怎么运作 |
|---|---|---|---|---|
| **短期** | 当前会话的对话上下文 | messages 历史、当前任务状态 | 会话内、结束即逝 | 就在 context 里，靠上下文管理（裁剪/压缩）维持 |
| **中期** | 会话内/跨几天的摘要、工作笔记 | 任务进度摘要、已做决策、临时中间结论 | 跨几轮/几天 | 把早期对话**压缩成摘要**替代原文塞回 context，保连贯又省窗口 |
| **长期** | 跨会话持久化的偏好/事实/技能 | 用户偏好、长期事实、可复用 Skill | 永久、跨会话 | 落到外部存储（文件/DB/向量库），按需 read 回来 |

**设计要点**：短期靠 context 本身但要主动管理（否则爆）；中期靠摘要（压缩原文替代）；长期靠外存（按需调取）。一个文件存一条要点（别一个超大文件）；记"为什么"而不只结论；不重复存代码/历史已有的东西。

### 存储方式（按量和查询方式选，别局限 Redis）

| 记忆类型 | 存储 | 适合 |
|---|---|---|
| 短期 | 进程内 messages | 所有场景默认 |
| 中期摘要 | SQLite / KV（Redis） | 要跨进程恢复、多副本 |
| 长期-少量要点 | 文件（MEMORY.md / USER.md） | 个人 Agent、要点少，简单直接、模型用普通文件工具读写 |
| 长期-海量知识 | 向量库（Milvus/PGVector）+ 语义检索 | 量大、要"按意思找"而非按 key 找 |
| 技能 | SKILL.md 文件 + 版本 | 可复用流程、要审计/回滚 |
| 结构化用户数据 | 关系库（MySQL） | 画像/决策历史、要精确查询统计 |

**怎么选**：看"量 + 查询方式"——少量要点用文件最简单；要"按语义找"用向量库；要精确查询用关系库；要跨进程共享用 Redis/SQLite。不是上来就 Redis。

### 淘汰机制（不能只存不删）

记忆无限增长迟早失控（撑爆存储、检索变慢、过时信息误导）。手段：

1. **容量上限**：每层设上限（如 MEMORY.md 限 2200 字符、向量库每用户限 N 条），到顶触发压缩/淘汰。
2. **LRU / 时间衰减**：最久未访问的优先淘汰；时间越久置信度越低（中期记忆定期降级/删除）。
3. **摘要替代原文**：早期详细对话压成摘要后，原文可删——保信息骨干、省空间。
4. **分层衰减**：短→中→长逐级沉淀，每层独立 TTL（短期会话结束即逝、中期几天、长期永久但要维护）。
5. **主动遗忘 + 纠错更新**：发现错了/过时就**更新而非新建**；明显无用的删；别让错误记忆长期污染。让模型每次写入前先查"有没有相关的、要不要合并/更新"。
6. **重要性评分**：给记忆打分，低分优先淘汰（不是所有信息都值得留）。
7. **绝不存密钥/密码/PII**：记忆会被检索、可能泄露，密钥用 vault/凭据管理（详见《工具调用》）。

一句话：记忆 = 三层（短/中/长，各管不同时间尺度）+ 按量选存储 + 主动淘汰（容量/LRU/摘要替代/纠错更新/重要性评分）+ 安全底线（不存敏感）。核心是"该记的记牢、该忘的忘掉、别让记忆失控污染"——既不忘关键，也不被过时信息拖累。

## 五、主流 Agent 用什么记忆方案

讲完原理，看看当前主流 Agent 实际怎么落地记忆——印证上面的设计、也方便面试说"业界现状"。

| Agent / 框架 | 记忆方案 | 怎么总结 | 怎么压缩 | 怎么加载 |
|---|---|---|---|---|
| **Claude Code / Managed Agents（Anthropic）** | 文件型长期记忆 + 服务端 compaction | 模型主动写 memory 文件（一条一文件）+ 可选 memory store | 服务端 compaction 自动压缩早期历史（接近上限触发，靠回传 compaction 块续） | memory store 以 FUSE 挂载成目录，模型用普通文件工具 read/write；system 自动注入"有哪些记忆"的提示 |
| **Hermes Agent（Nous Research）** | 五层记忆架构，自我进化 | **闭环学习回路**：任务后自动反思、把成功流程提炼成 Skill 存档 | 上下文压缩器（context_compressor）+ system prompt 全程不变保 cache | 会话开始读 MEMORY.md/USER.md + 全文检索（FTS5）召回相关记忆 + Skill 按需加载 |
| **OpenClaw** | 工作区驱动 + 本地文件 | Agent 人格/知识/技能/记忆沉淀在结构化工作区，每次结束自动回写 | 靠工作区分层 + 中期摘要 | 每次运行从工作区自动加载上下文 |
| **AgentScope 2.0（HarnessAgent）** | 双层记忆 + 工作区 | 工作区统一沉淀人格/知识/技能/记忆，结束自动回写 | 内置对话压缩 + 记忆沉淀 + 全文检索 | 每次运行自动从工作区加载、按需召回 |
| **通用 RAG 式记忆** | 向量库 + 检索 | LLM 把对话总结成要点 → embedding 入库 | 滑动窗口 + 摘要替代原文 | 每轮把当前 query 向量化、检索 top-k 相关记忆注入 context |

**从这些方案能看出几个共识**（面试可这么总结）：
1. **长期记忆主流是"文件/工作区"**（MEMORY.md、SKILL.md、工作区目录），不是数据库——简单、模型可直接用文件工具读写、可审计。少量要点全量读、量大才上向量库。
2. **总结普遍用 LLM 提炼 + 会话结束批量整理 + 实时记关键事实的混合**；自我进化型（Hermes）还会把成功流程沉淀成可复用 Skill。
3. **压缩普遍靠"滑动窗口 + 摘要替代" 或服务端 compaction**；关键事实提前外移到记忆，不指望压缩后保住。
4. **加载普遍是"会话开始全量读基础画像 + 每轮检索/注入相关记忆 + 模型主动调取深度记忆"三层组合**。
5. **system prompt 保持稳定**是跨方案的工程共识（保 prompt cache、防漂移）——Hermes 甚至把它做成硬约束。

一句话：业界从"无状态对话"走向"持久化 + 自我进化的记忆型 Agent"——文件/工作区存长期记忆、LLM 总结沉淀、滑动窗口+compaction 压缩、三层加载组合，是当前主流配方。
