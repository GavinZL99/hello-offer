# 多智能体协作

## 一、解决什么问题

单个 Agent 的上下文和能力有限：复杂任务上下文装不下、一个 Agent 啥都干不够专业、串行处理太慢。**多智能体协作**就是把任务拆给多个 Agent，各司其职、并行或流水线推进，主 Agent 综合结论。

**先泼冷水**：单 Agent 能搞定就别上多 Agent——它引入协调成本、通信开销、调试复杂度。判断标准是"**单 Agent 上下文装不下、或任务能天然并行/分专业**"才值得拆。

## 二、流程是什么

### 几种协作模式（按任务形状选）

**1. 协调者-工人（Orchestrator-Worker，最常用）**：
```
主 Agent 接任务 → 拆成子任务 → 派给子 Agent(各自独立上下文/工具/模型)
              → 子 Agent 执行 → 只把结论交回主 Agent
              → 主 Agent 综合所有结论 → 产出最终结果
```
主 Agent 负责编排和最终产出，子 Agent 负责具体执行。

**2. 并行扇出**：N 个独立子任务（读 N 个文件、查 N 个候选、评审 N 个模块）同时派 N 个子 Agent 并行跑，主 Agent 归并——比串行快得多。这是最该用多 Agent 的场景（加速明显）。

**3. 流水线**：A 输出喂 B、B 喂 C，各管一段，边界清晰。

**4. 多视角/对抗**：一个建设、一个挑刺；或多个方案并行生成再选优——需要独立视角时。

### 分工逻辑（怎么拆）

- **按能力/角色分工**：每个 Agent 一个清晰职责 + 配套工具/模型/system。研究员给搜索工具、编码员给 code exec，别全给——专注 + 安全。
- **共享文件系统 + 独立对话历史**（主流做法）：多个 Agent 能读写同一份文件（间接协作），但各自对话历史独立。这样既共享中间状态、又互不撑爆对方上下文。
- **给每个 Agent 清晰边界**：职责、工具、模型、输入/输出格式都定义清楚。子 Agent 不需要知道全貌，主 Agent 在委派消息里把"要它干什么、给什么、要什么输出"说清。

**为什么分 Agent 而非分函数**：核心是**子 Agent 有独立上下文**，跑完只交结论，主 Agent 不被中间过程撑爆。函数没有自己的 context，调一个函数的中间过程全堆在主 Agent 窗口里。

## 三、使用中会遇到哪些问题

1. **结果冲突**：多个 Agent 给了不同结论/方案，听谁的？
2. **资源竞争**：多个 Agent 要改同一文件/调同一受限工具，并发写坏。
3. **状态不一致**：子 Agent 各自对话独立，A 改了共享文件 B 不知道。
4. **死循环 / 互相等待**：A 等 B、B 等 A。
5. **任务重复 / 遗漏**：两个 Agent 干了同一件事、或都没干某件事。

## 四、解决方案

### 结果冲突

- **仲裁者**：主 Agent 或专门评审 Agent 综合判断、选优。
- **投票/多数**：多个 Agent 各出一票，多数决（适合有明确对错的）。
- **优先级**：定义领域权威（编码问题以编码员为准、合规以合规员为准）。
- **人兜底**：无法自动解决时升级人工。

### 资源竞争

- **锁/队列**：共享资源加锁排队访问，防并发写坏。
- **分工时就隔离**：让各 Agent 操作不同文件/分区，从根上避免竞争——比加锁简单可靠。

### 状态不一致

- **消息协议**：子 Agent 完成后显式通知主 Agent（"我写好了 X"），主 Agent 再告知需要的其他 Agent；或主 Agent 做信息中转。
- **以共享文件为准**：Agent 间不靠"记忆对方说过什么"，而是每次读共享文件**最新状态**——最终一致性。这比让每个 Agent 都记住别人说过什么可靠得多。

### 死循环 / 互相等待

- **超时**：每个委派有 timeout，超时强制结束/降级。
- **只一层委派**：主派子，子不再派（深度>1 复杂度爆炸、通常被框架忽略）。AgentScope 等明确"深度>1 被忽略"。
- **任务依赖设计成 DAG（有向无环图）**：避免循环依赖，A 依赖 B、B 依赖 A 这种就解不开。

### 任务重复 / 遗漏

- **主 Agent 集中调度**：分工由主 Agent 统一规划，别让子 Agent 自行认领（自行认领易撞/漏）。
- **任务看板**：共享一份待办清单，领完标记，防重复/遗漏。

### 落地要点

- **只一层委派**：主派子，子不再派。
- **异步优于阻塞**：派完继续干别的，别 spawn-and-block（干等慢子 Agent 是大头延迟）。
- **子 Agent 用便宜模型**：重活/简单活分流到 Haiku，主 Agent 用强模型——降本。
- **只交结论不交过程**：子 Agent 中间过程留在自己上下文，只交最终结论给主 Agent（这是上下文隔离的核心价值）。
- **可观测**：多 Agent 调试难，每个子 Agent 的输入/输出/状态要可追溯（事件流/日志/trace）。
- **框架**：复杂状态机式编排用 LangGraph；企业级分布式多 Agent 用 AgentScope（声明式子 Agent、同步/异步委派、沙箱隔离）；多 Agent 对话式用 AutoGen。

一句话：多 Agent = 协调者拆派 + 子 Agent 独立上下文只交结论；冲突靠仲裁/锁/消息协议/超时/集中调度解决；核心价值是上下文隔离和并行，代价是协调复杂度——所以"该用才用"。
