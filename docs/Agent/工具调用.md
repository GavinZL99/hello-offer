# 工具调用（Tool Use）

## 一、解决什么问题

LLM 本身只能输出文字，不能执行实际动作、也不知道实时信息。**工具**让 Agent 能调用外部函数——查天气、查库存、发邮件、改文件、跑代码。模型决定调哪个工具、传什么参数，你的代码真正执行、把结果回填给模型。工具是 Agent 从"能说"到"能干"的桥梁。

## 二、流程是什么

一个工具 = **一个你能执行的函数 + 一份给模型看的说明书（schema：name/description/input_schema）**。完整流程：

```
① 写函数（业务逻辑）
② 写 schema（给模型的说明书：名字、参数、何时用）
③ 注册进 Agent 的 tools 列表
④ 模型在循环里返回 tool_use 块（含 name + input）
⑤ 你的代码 dispatch 到对应函数、传 input、执行
⑥ 把结果包成 tool_result（tool_use_id 对上）回填给模型
⑦ 模型看到结果、继续推理
```

### schema 长什么样、怎么少写

手写 JSON Schema 易错。各 SDK 提供装饰器/类型自动生成：
- Python：`@beta_tool`，从函数签名 + docstring 自动生成 schema。
- Java：`@Tool` 注解。
- TypeScript：`betaZodTool` + Zod schema。

```python
@beta_tool
def get_weather(location: str, unit: str = "celsius") -> str:
    """查某地当前天气。
    Args:
        location: 城市名，如 "北京"。
        unit: 温度单位，celsius 或 fahrenheit。
    """
    return f"{location} 今天晴，22度"
# schema 自动生成，description 来自 docstring
```

**strict 模式**：要强约束格式时给工具加 `strict: true`——模型传的参数必须严格符合 schema（`additionalProperties: false`、所有字段 required），减少参数错。代价是 schema 要写完整（不能有可选字段未声明）。适合参数格式严格、错不起的工具（如订票、转账）。

## 三、使用中会遇到哪些问题

1. **参数错**：模型传错参数（类型错、值非法、缺必填）。
2. **安全**：Agent 能调工具就能执行动作——误操作删数据、越权调敏感工具、凭据泄露。
3. **工具设计差**：模型乱调、选错工具、重复调、该调不调。
4. **prompt 注入**：模型被诱导传恶意参数（SQL 注入字符串等）。

## 四、解决方案

### 参数校验（三层防御）

**1. Schema 层（第一道）**：JSON Schema 约束类型/必填/枚举/格式，`strict: true` 强制。挡掉大部分结构错误。

**2. 运行时校验（第二道，必做）**：函数入口再校验语义——范围、业务合法性。schema 挡不住"类型对但值错"（如 `month=13` 类型是 int 但非法）。
```python
def transfer(amount: int):
    if amount <= 0: raise ValueError("金额必须>0")
    if amount > 10000: raise ValueError("超单笔上限")
```

**3. 失败回填（第三道）**：校验失败/异常**别抛中断循环**，包成 `is_error: true` + **可操作**错误信息回给模型纠错。
```
{"type":"tool_result","tool_use_id":id,"content":"错误：month=13 非法，应为1-12","is_error":true}
```
错误信息要可操作（告诉模型哪里错、怎么改），别只回 "error"。

**4. 防注入**：参数进 SQL/Shell 前必须转义/参数化，别直接拼模型传的字符串——模型可能被 prompt 注入诱导传恶意 payload。

### 权限管控（安全命门）

**1. 分级（按可逆性）**：
- 只读/可逆工具（查库、读文件）：`always_allow`，自动执行。
- 不可逆/高危工具（发邮件、删数据、转账、改线上配置）：`always_ask`，必须人工确认才执行。
- 判断标准是"**这个动作错了能不能回滚**"，难回滚的都该 gating。

**2. 按用户/角色授权**：不同用户能用不同工具集。管理员能用"删除"，普通用户不能。授权表 `(userId, toolName) → allow/deny`，调用前查。

**3. 沙箱隔离**：代码执行/文件操作类工具放 Docker/E2B 沙箱里跑，限制能访问的文件系统/网络，爆炸半径可控。AgentScope、Managed Agents 都内置沙箱——就算模型调了 `rm -rf` 也只删沙箱里的东西。

**4. 凭据绝不进 prompt（重要）**：密钥/token 放 prompt 或 messages 里会被持久化进会话历史、可能泄露。
- **vault 托管**：凭据放 vault（凭据管理服务），Anthropic/框架自动注入、自动刷新 OAuth token，模型和容器永远看不到密钥。
- **自定义工具留己方**：Agent 只发"调用 X"请求，你的代码用凭据执行、把结果回填——容器/模型全程不碰密钥。

**5. 审计日志**：每次工具调用记 `谁/何时/哪个工具/什么参数/结果/是否审批`。出事能追溯；合规必需。

**6. 限流/熔断**：单 Agent 调某工具的频率/总量上限，防失控（模型死循环调删库就完蛋）。超限拒绝。

**7. 危险参数二次确认**：如 `delete where id=*`（全删）、`amount>阈值`，工具内部检测到高危参数强制审批，哪怕工具本身一般免审。

### 工具设计取舍（决定好不好用）

- **描述写"何时调用"**（最被低估）：不只写"做什么"，写"**什么情况该调它**"（如"当用户问当前价格/近期事件时调用"）。模型据此决定调不调，触发条件写进 description 明显提升调用准度——Opus 4.8 等用工具偏保守的模型尤其有效。
- **bash 通用 vs 专用工具**：bash 广但不透明（只有一个不透明命令字符串，难 gating/审计/限流）；需要审批/审计/并发/渲染时提升为专用工具（拿类型化钩子）。先 bash 求广度，需 gating 时再专用化。
- **数量**：几十个直接放；上百个用 **tool search** 按需加载 schema（不全塞 context），模型也不易选错。
- **粒度**：太细的工具（要调好几次才能完成一件事）促使多次调用（费 token/延迟），适当合并成"高阶工具"一次完成一个完整动作；太粗不灵活。按业务动作边界定粒度。
- **幂等**：可重试的工具尽量设计成幂等（重复调用不出错），配合重试机制。

一句话：工具 = 函数 + schema；可靠性靠三层校验（schema/运行时/is_error 回填），安全性靠分级审批 + 沙箱 + 凭据托管 + 审计，好用靠描述写清"何时调用" + 合理粒度。
