# 算法刷题笔记 · 第一类：滑动窗口

> 这是一个系列笔记。每一类算法都按同样的结构组织：
> **直觉类比 → 通用模板 → 两类变体 → 实战示例 → 优化技巧 → 自检清单**。
> 目标不是背代码，而是建立「看到题能套模板、套不上能讲清为什么」的能力。

---

## 0. 一句话本质

滑动窗口 = 用 `left`、`right` 两个指针圈出数组/字符串的一段**连续区间**，
让这个窗口在数组上「**只往右滑，不回头**」，从而把暴力的 O(n²) 枚举降到 **O(n)**。

适用信号（看到这些关键词就该想到它）：
- 「**连续**子数组 / 子串」
- 求「最长 / 最短 / 满足某条件的」区间
- 「至多 / 恰好 / 至少 K 个……」

> ⚠️ 注意「**连续**」是前提。不连续的子序列问题不能用滑窗。

### ⭐ 隐藏前提：单调性（为什么含负数就不能用滑窗）

滑窗能成立，靠的是 `left`、`right` **只进不退**——一旦左指针把某元素移出窗口，它**永远回不来**。这个「永久丢弃」要安全，必须满足：

> **窗口的量（和/计数）随着加入/移除元素单调变化** —— 这样「被丢掉的左边，以后绝不可能再变得有用」。

- **全正数**时安全：加入元素 → 和一定变大；移出 → 和一定变小。所以「和超标就收缩」永远正确，丢掉的左边（正数）以后只会让和更大、更不满足，丢得安心。
- **含负数**时崩溃：加入负数反而让和变小，单调性断了，被丢弃的左边可能后来又变得有用，而滑窗**没有反悔能力**。

**反例**（求「和 ≤ K 的最长连续子数组」，`nums=[5,-3,2]`, K=4，正确答案是整个数组、长度 3）：

```
right=0: 加入5,和=5>4 → 收缩!移出5            ← 把 5 丢了
right=1: 加入-3,和=-3,窗口[-3] 长度1
right=2: 加入2,和=-1,窗口[-3,2] 长度2
结果 2  ❌  (后来的 -3 本可把和拉回 4 以内,但 5 已被永久丢弃)
```

> 所以决策表那条「**含负数 → 滑窗不行**」的真正原因：负数破坏单调性 → 「只进不退」的丢弃不再安全 → 改用**前缀和+哈希**（[05](./05-前缀和.md)）或 **DP**（[02](./02-动态规划.md)，如最大子数组和 Kadane）。
> 对比：对撞双指针要的单调是「**数组本身有序**」；滑窗要的单调是「**窗口的量随增删单调**」——两种不同对象的单调，别混。

## 1. 直觉类比：维护一支排队的队伍

把窗口想象成**一支排队进场的队伍**：

- `left` = 队首的人，`right` = 刚走来的新人
- **进窗口** = 新人排到队尾（`right++`）
- **出窗口** = 队首的人离开（`left++`）
- 整支队伍必须时刻满足某条「**规则**」（不重复 / 总和达标 / 种类不超标…）

所有滑窗题，做的都是同一件事：

> **新人不断从右边加入；一旦队伍违反规则，就从队首开始请人离开，直到规则恢复。**

记住这个画面，后面所有代码都是它的翻译。

## 2. 通用模板（先背这个，万能）

```java
public int slidingWindow(String s) {
    Map<Character, Integer> window = new HashMap<>();  // 窗口内的统计
    int left = 0;
    int res = 0;  // 结果：最长/最短/计数，按题而定

    for (int right = 0; right < s.length(); right++) {
        // ① 进窗口：右指针元素加入
        char c = s.charAt(right);
        window.put(c, window.getOrDefault(c, 0) + 1);

        // ② 判断收缩：窗口需要调整时，从左边请人离开
        while (/* 收缩条件 */) {
            char d = s.charAt(left);
            window.put(d, window.get(d) - 1);  // 出窗口
            left++;
        }

        // ③ 更新答案（位置取决于求最大还是最小）
        res = Math.max(res, right - left + 1);
    }
    return res;
}
```

**三步心法：进窗口 → 判断收缩 → 更新答案。** 所有变化只发生在这三个位置。

## 3. 两大变体：求「最长」 vs 求「最短」

这是滑窗最容易混的地方，记住下面这张表就不会错：

| | **求最长** | **求最短** |
|---|---|---|
| 内层 while 条件 | 窗口**非法**时收缩 | 窗口**合法**时收缩 |
| 更新答案的位置 | while **之后**（此时窗口合法） | while **之内**（合法时尽量缩短） |
| `res` 初值 | `0` | `Integer.MAX_VALUE` |
| 直觉 | 「先撑大，撑不下了再缩」 | 「一够标准就拼命缩」 |

### 变体一：求「最长」窗口

> 例题：无重复字符的最长子串、至多含 K 个不同字符的最长子串

```java
// LeetCode 3. 无重复字符的最长子串
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> window = new HashMap<>();
    int left = 0, res = 0;

    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        window.put(c, window.getOrDefault(c, 0) + 1);

        // 收缩条件：出现了重复字符（窗口非法）
        while (window.get(c) > 1) {
            char d = s.charAt(left);
            window.put(d, window.get(d) - 1);
            left++;
        }

        res = Math.max(res, right - left + 1);  // 窗口合法后更新
    }
    return res;
}
```

### 变体二：求「最短」窗口

> 例题：长度最小的子数组、最小覆盖子串

```java
// LeetCode 209. 长度最小的子数组：和 >= target 的最短连续子数组
public int minSubArrayLen(int target, int[] nums) {
    int left = 0, sum = 0, res = Integer.MAX_VALUE;

    for (int right = 0; right < nums.length; right++) {
        sum += nums[right];  // 进窗口

        // 收缩条件：窗口已合法（满足条件），尝试缩到更短
        while (sum >= target) {
            res = Math.min(res, right - left + 1);  // 收缩中更新
            sum -= nums[left];  // 出窗口
            left++;
        }
    }
    return res == Integer.MAX_VALUE ? 0 : res;
}
```

## 4. 做题时的思考顺序（套模板的 4 个问题）

拿到一道滑窗题，按顺序问自己：

1. **窗口里要维护什么？** → 决定数据结构：`int sum` / `HashMap` 计数 / `HashSet` / 种类计数器
2. **什么时候收缩？** → 写出内层 while 的条件（**这是全题的灵魂**）
3. **求最长还是最短？** → 决定答案更新放在 while 内还是 while 外（见上表）
4. **答案怎么算？** → 通常是窗口长度 `right - left + 1`

## 5. 实战优化：下标跳跃（无重复子串的进阶写法）

模板用「一步步 `left++`」收缩。某些题可以**一步跳到位**，更快：

```java
// LeetCode 3 的下标跳跃版
public int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> map = new HashMap<>();  // 字符 -> 最后出现的下标
    int left = 0, res = 0;

    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        if (map.containsKey(c)) {
            // 直接把 left 跳到「上一个相同字符的后面一位」
            left = Math.max(left, map.get(c) + 1);
        }
        map.put(c, i);
        res = Math.max(res, i - left + 1);
    }
    return res;
}
```

### ⭐ 关键点 1：为什么要 `Math.max(left, ...)`

`map.get(c)` 是字符**上次出现的下标**，但它**可能已经在窗口左边界之外（过期了）**。

以 `"abba"` 为例：

| i | 字符 | 动作 | left |
|---|------|------|------|
| 0 | a | 记录 a→0 | 0 |
| 1 | b | 记录 b→1 | 0 |
| 2 | b | b 重复(在1)，left = max(0, 1+1) = 2 | 2 |
| 3 | a | a 重复(在**0**)，left = max(2, 0+1) = **2** | 2 |

到 `i=3`，字符 `a` 上次出现在下标 0。若**不**用 `max`，`left` 会倒退回 1——
但下标 1 的 `b` 早就被踢出窗口了！`Math.max` 保证 **`left` 只前进、绝不后退**。

### ⭐ 关键点 2：为什么「下标跳跃」只对这类题成立

> **能不能跳跃，取决于：窗口非法时，你能不能 O(1) 算出 left 该跳到哪。**

| 题型 | 「非法」的元凶 | 能否跳跃 |
|------|---------------|----------|
| 无重复子串 | **某个具体字符**，下标已知 ✅ | ✅ 能跳 |
| 至多 K 个不同字符 | 「种类数超了」，非单一元素 | ❌ 逐步缩 |
| 和 ≥ target | 「累加和」，非单一元素 | ❌ 逐步缩 |
| 最小覆盖子串 | 「整体覆盖度」，非单一元素 | ❌ 逐步缩 |

**类比**：无重复子串的麻烦是「队里混进了一个同名的人」——元凶**唯一且位置已知**，
所以能直接喊「从队首到他这段全部出列」。
而「队伍总钱数要 ≥ 100」这种，没有单一元凶，只能**一个个请人走、走一个看一眼**，跳不了。

> 本质：跳跃是用 **HashMap 记住了位置**（记忆换速度）；
> 「和、种类数」这类**聚合量记不住具体位置**，自然跳不了。

### ⚠️ 重要澄清：跳跃 ≠ 复杂度更低

逐步收缩里，每个元素最多进一次、出一次，`left` 总移动量也是 n，**整体仍是 O(n)**。
跳跃只是**常数更小、代码更短**，渐进复杂度和模板版**完全一样**。

> 👉 建议：**先用模板版把题做对（万能、好记），熟练后再用跳跃优化特定题型。**

## 7. 自检清单（写完代码过一遍）

- [ ] `left` 是否只增不减？（绝不能让窗口回头）
- [ ] 收缩条件写对了吗？最长是「非法才缩」，最短是「合法就缩」
- [ ] 答案更新位置对吗？最长在 while **外**，最短在 while **内**
- [ ] 出窗口时，统计结构（map/sum/计数）**对称地**做了减法吗？
- [ ] 边界：空串 / 整个数组都不满足条件 时返回值对吗？
- [ ] `res` 初值：求最长用 `0`，求最短用 `Integer.MAX_VALUE`

## 9. 实战复盘（自己踩过的坑）

> 三道题串起滑窗的三种形态：**变长求最长 → 变长求最短 → 求最短子串（带覆盖判断）**。
> 记录的不是答案，而是**第一次写时栽在哪**——这些才是真正学到的东西。

### 9.1 LC 1004 最大连续 1 的个数 III（求最长）

> 最多翻转 k 个 0，求最长连续 1。转化为：**维护「窗口内 0 的个数 ≤ k」的最长窗口**。

```java
public int longestOnes(int[] nums, int k) {
    int left = 0, zeroCount = 0, res = 0;   // 求最长，res 初值 0
    for (int i = 0; i < nums.length; i++) {
        if (nums[i] == 0) zeroCount++;          // 进窗口
        while (zeroCount > k) {                  // 非法才缩
            if (nums[left] == 0) zeroCount--;    // 出窗口，对称地减
            left++;
        }
        res = Math.max(res, i - left + 1);       // while 外更新
    }
    return res;
}
```

**踩坑**：`res` 初值写成了 `1`。有解时碰巧对，但**空数组应返回 0**，初值 1 会错。
👉 **「求最长」res 初值一律 `0`**。

**不能下标跳跃**：非法由「0 的总数」这个聚合量决定，没有可定位的单一元凶 → 只能逐步缩。

---

### 9.2 LC 209 长度最小的子数组（求最短）

> 正整数数组，求「和 ≥ target」的最短连续子数组，不存在返回 0。

```java
public int minSubArrayLen(int target, int[] nums) {
    int left = 0, sum = 0, res = Integer.MAX_VALUE;   // 求最短，初值 MAX
    for (int i = 0; i < nums.length; i++) {
        sum += nums[i];                          // 进窗口
        while (sum >= target) {                  // 合法就缩
            res = Math.min(res, i - left + 1);   // while 内更新
            sum -= nums[left++];                 // 出窗口
        }
    }
    return res == Integer.MAX_VALUE ? 0 : res;   // 没更新过 → 无解 → 0
}
```

**踩坑**：`res` 初值写成了 `nums.length`。**无解时**（整个数组和都 < target）while 一次不进，
res 停在 `nums.length` 而非 0，输出错误。
👉 **「求最短」res 初值用 `Integer.MAX_VALUE`，结尾三元判「没更新过返回 0」**。

**求最长 vs 求最短 对照**（最容易混，钉死它）：

| | 求最长（1004） | 求最短（209） |
|---|---|---|
| 收缩条件 | 窗口**非法**才缩 | 窗口**合法**就缩 |
| 答案更新 | while **外** | while **内** |
| `res` 初值 | `0` | `Integer.MAX_VALUE` |
| 无解处理 | 无此情况 | 结尾三元 `MAX → 0` |

### 9.3 LC 76 最小覆盖子串（Hard，求最短子串）

> 求 s 中涵盖 t 所有字符（含重复数量）的最短子串。窗口里要判断的是「**是否覆盖 t**」。

**Hard 三连坑**（第一次写全踩到了）：

1. **NPE**：判断覆盖时 `window.get(c)` 在字符没出现时是 `null`，`null < x` 拆箱崩。
   👉 用 `getOrDefault(c, 0)`。
2. **删错字符**：收缩时 `left++` 后再删 `s[left]`，删成了新 left（仍在窗口内）的字符。
   👉 **「先送客，再关门」**：先删旧 left 的字符，`left++` 放最后。
3. **只记 len 没记 start**：求最短**子串**要返回内容，必须在更新 len 时同步记 `start`，
   否则结尾 `substring` 用错位置的 `left` 截取。

**写法 A：`contains` 辅助函数（直观版，O(n × 字符种类)）**

```java
public String minWindow(String s, String t) {
    int left = 0, len = Integer.MAX_VALUE, start = 0;
    Map<Character, Integer> window = new HashMap<>(), need = new HashMap<>();
    for (char c : t.toCharArray()) need.put(c, need.getOrDefault(c, 0) + 1);

    for (int i = 0; i < s.length(); i++) {
        char c = s.charAt(i);
        window.put(c, window.getOrDefault(c, 0) + 1);   // 进窗口
        while (contains(window, need)) {                 // 合法就缩
            if (i - left + 1 < len) { len = i - left + 1; start = left; }  // 更短才记
            char d = s.charAt(left);                      // 先送客
            window.put(d, window.get(d) - 1);
            left++;                                       // 再关门
        }
    }
    return len == Integer.MAX_VALUE ? "" : s.substring(start, start + len);
}

private boolean contains(Map<Character,Integer> window, Map<Character,Integer> need) {
    for (Map.Entry<Character,Integer> e : need.entrySet())
        if (window.getOrDefault(e.getKey(), 0) < e.getValue()) return false;  // 防 NPE
    return true;
}
```

**写法 B：`valid` 计数器（最优版，O(n + m)）**

核心：用 `valid` = 「数量已凑够的字符**种类数**」，把覆盖判断从 O(种类) 降到 O(1)。

```java
public String minWindow(String s, String t) {
    Map<Character, Integer> need = new HashMap<>(), window = new HashMap<>();
    for (char c : t.toCharArray()) need.put(c, need.getOrDefault(c, 0) + 1);

    int left = 0, valid = 0, start = 0, len = Integer.MAX_VALUE;
    for (int right = 0; right < s.length(); right++) {
        char c = s.charAt(right);
        window.put(c, window.getOrDefault(c, 0) + 1);    // ① 进窗口
        if (need.containsKey(c) && window.get(c).equals(need.get(c))) valid++;  // 刚好凑够才 +1

        while (valid == need.size()) {                    // ② 合法（O(1)）就缩
            if (right - left + 1 < len) { start = left; len = right - left + 1; }
            char d = s.charAt(left);                       // 先送客
            left++;
            if (need.containsKey(d) && window.get(d).equals(need.get(d))) valid--;  // 破坏一种才 -1
            window.put(d, window.get(d) - 1);              // 再减计数
        }
    }
    return len == Integer.MAX_VALUE ? "" : s.substring(start, start + len);
}
```

**`valid` 的两个高发 bug 点**：
- 进窗口 `valid++` 用 **`.equals()`**（刚好凑够那一刻 +1），**不能用 `>=`**（多抽会重复加）。
  注意：`Integer` 用 `==` 比较有缓存坑（仅 -128~127），用 `.equals()` 或转 `int`。
- 出窗口是进窗口的**镜像**：**先判断 `valid--`，再减计数**（和进窗口「先加、再判断」对称）。

> A 和 B 思路完全一样，B 只是把「每次重新数覆盖」换成「增量维护 valid」——
> **记忆换速度**，和无重复子串的「下标跳跃」是同一种优化哲学。

## 10. 一句话总结

> 滑动窗口就是**一支只进不退的队伍**：右手进人，左手在违规时请人走。
> 难点永远只有一个——**「什么时候请人走」（收缩条件）**，其余都是模板。
