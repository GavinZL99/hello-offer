# 领域建模科普：概念与方法

> 这篇不走面试速查的路子，目标是把"领域建模到底是什么、为什么有用、怎么动手"讲清楚。多用类比、少堆术语。
> 领域建模是 DDD（Domain-Driven Design，领域驱动设计）的核心实践，下面提到的概念大多来自 DDD。

---

## 一、先说人话：领域建模是什么

"领域"就是你的软件要解决的那块**业务**——电商的买卖、税务系统的开票计税、银行的存贷。"建模"就是**把这块业务里的概念和规则，提炼成一套清晰的结构**。

**领域建模 = 先想清楚业务本身长什么样，再让代码照着这个样子写**，而不是上来就建数据库表、写一堆 if-else。

打个比方：盖房子之前要先画**建筑设计图**——哪里是客厅、哪里是承重墙、水电怎么走。设计图不是房子本身，但它决定了房子结不结实、好不好改。领域模型就是软件的设计图：它不是代码，但它决定了代码清不清晰、能不能长期演进。

很多人写代码是"没有设计图就开始砌砖"：先建几张表，再写 Service 把数据搬来搬去。简单的工具房这么干没事，但盖复杂的大楼（复杂业务），没设计图迟早塌。

---

## 二、为什么需要它：代码是怎么烂掉的

设想一个订单系统，一开始很简单：建个 `order` 表，写个 `OrderService`，增删改查。岁月静好。

然后业务开始长肉：要支持取消、退款、部分发货、优惠券、积分……每来一个需求，就往 `OrderService` 里塞一段逻辑、加几个字段、写几个 if。一年后，这个 Service 几千行，"一个订单能不能取消"的判断散落在七八个地方，谁也说不清完整规则。改一个小需求，得在代码里翻半天、还经常改出 bug。

烂掉的根源有两个：

**第一，代码不像业务。** 代码里全是表名、字段、`updateStatus(3)`，看不出业务概念。产品说"作废这张发票"，开发得在脑子里翻译成"把 status 字段改成 3"。这种翻译每次都有损耗，时间一长，没人说得清"status=3"到底对应什么业务含义。

**第二，规则没人守。** 数据和操作数据的逻辑是分开的——订单对象只是个装数据的袋子（getter/setter），谁都能拿到它随便改。"已支付的订单不能改收货地址"这种规则，全靠写代码的人自觉记得去判断，漏一个地方就出事。

领域建模就是来治这两个病的：**让代码长得像业务**（看代码就懂业务），**让规则锁在它该在的地方**（绕不过去）。

---

## 三、核心理念：给业务建模，而不是给数据库建模

这是领域建模和传统写法最根本的区别。

传统写法是"**面向数据库**"：先想"这数据该怎么存"，设计出表，代码就围着表转。领域建模是"**面向业务**"：先想"这业务是怎么运转的、有哪些概念和规则"，建出模型，**数据库只是模型最后落地存储的一种方式**，不是出发点。

配套有一个很朴素但很重要的理念——**通用语言**（Ubiquitous Language）：业务专家、产品、开发，大家**用同一套词**描述业务，而且**这套词直接出现在代码里**。

业务说"作废发票"，代码里就有 `invoice.void()`；业务说"订单已支付"，代码里就有 `OrderPaid` 这个概念。而不是业务一套话、代码一套黑话，中间靠开发脑补翻译。

好处很实在：代码本身成了业务文档，新人读代码就能懂业务；产品和开发对需求时不会鸡同鸭讲。听起来简单，但"让代码里的名字和业务说法一致"这件事，能消除大量沟通和维护成本。

## 四、最重要的概念：限界上下文

如果整篇只能记一个词，就记**限界上下文**（Bounded Context）。它是领域建模里最关键、也最容易被忽略的一步。下面慢慢讲透。

### 1. 先看一个现象：同一个词，意思会变

复杂业务很大，没法用一个模型装下所有东西，得先切块。切块时会撞到一个有意思的现象：**同一个词，在不同的业务环节，意思根本不一样。**

拿"商品"举例：
- 在**销售**环节，"商品"关心的是价格、促销、图文详情、能不能下单；
- 在**库存**环节，"商品"关心的是 SKU、库存数量、放在哪个仓位；
- 在**物流**环节，"商品"关心的是重量、体积、怎么打包、易不易碎。

"用户"也一样：在**账户**环节是登录名、密码、权限；在**营销**环节是标签、偏好、积分等级；在**客服**环节是工单和历史记录。还是同一个人，但每个环节眼里的"用户"完全不同。

### 2. 为什么不能用"一个大模型"装下所有

很自然的念头是：那就造一个"万能商品类"，把销售、库存、物流关心的字段全塞进去，大家共用。这正是很多系统腐烂的根源，会造出一个所谓的**"上帝类"**：

- **几十上百个字段**，没人能完整理解，新人看了发懵；
- **耦合爆炸**：库存的同事改一下"重量"相关逻辑，可能莫名其妙把销售的"促销"搞崩——因为它们挤在同一个类里；
- **改不动**：任何一个环节想调整，都要担心影响另外两个环节，最后谁都不敢改。

根本矛盾在于：**这三个"商品"本来就是三个不同的东西，硬捏成一个，是在对抗业务的真实结构。**

### 3. 限界上下文：给每个"意思"划一道边界

**限界上下文就是来解决这个的：划出一道边界，在这道边界之内，每个词只有一个确定的含义、对应一个干净的小模型。** 销售上下文有自己的"商品"（只含价格促销）、库存上下文有自己的"商品"（只含 SKU 库存）、物流上下文有自己的"商品"（只含重量体积），**各管各的、各自干净、互不打架**。

类比：同一个词在不同"国家"含义不同。"football"在美国指橄榄球、在英国指足球——你得先说清在哪个国家，这个词才有确定意思。限界上下文就是给业务术语划"国界"：**进了哪个国（上下文），词就按哪个国的意思理解。**

这里还藏着和"通用语言"的关系：**通用语言只在一个限界上下文内部才成立。** 正因为每个上下文有自己的一套语言，同一个"商品"才能在不同上下文里是不同的东西——所以"先划上下文，再在每个上下文里统一语言"，两件事是连着的。

### 4. 边界之间怎么协作

划成几个上下文后，它们当然要打交道——下单时，订单上下文要知道商品的价格、要让库存上下文扣减库存。怎么协作而不把边界又搅浑？两条原则：

- **跨上下文不共享大模型，只传必要的数据。** 订单上下文里需要的"商品"，不是库存那个完整的商品对象，而往往只是"**商品 ID + 名称 + 当时的价格快照**"这么几样。各上下文之间用 **ID 互相引用**，而不是把对方的整个对象搬过来。
- **调外部上下文时加一层"翻译"（防腐层，Anti-Corruption Layer）。** 比如要对接一个老的库存系统，它的数据格式很乱，你不直接拿它的模型，而是加一层适配，把它翻译成自己上下文干净的概念——**不让外部的混乱"污染"自己的模型**。类比涉外翻译：把对方的话翻成你的术语，你内部沟通不受对方语言影响。

### 5. 边界到底怎么找

这是最实际的问题。几个"该划边界了"的信号：

- **语言变了**：同一个词换了含义（"商品"在库存和销售意思不同），或者某个词在另一群人那里根本没人提——这就是跨边界的信号。
- **关注点变了**：销售关心钱、库存关心数量、物流关心体积——关注点截然不同，多半该分开。
- **变化的原因变了**：促销规则和仓储规则是两拨完全不相干的需求来源，把它们放一起就会互相牵连——按"会因为不同原因而改变"来切，是个好准则。
- **团队/部门变了**：通常一个团队负责一个上下文。组织结构和上下文边界往往天然吻合（这背后是"康威定律"：系统结构会长得像组织沟通结构）。

### 6. 几个容易混的关系

- **限界上下文 vs 子域**：子域是"**问题空间**"——业务本身天然分成哪几块（销售、库存、物流）；限界上下文是"**解决方案空间**"——你的软件/模型实际划成哪几块。理想情况一个子域对应一个限界上下文，但现实中可能对不齐（历史包袱导致一个上下文混了两个子域，是常见的待改进信号）。
- **限界上下文 vs 微服务**：这个边界往往就是将来**微服务怎么拆**的依据——按业务边界拆服务，比按技术分层拆健康得多。一个限界上下文常常对应一个微服务：前者是业务上的**逻辑边界**，后者是部署上的**物理边界**。但不强制一对一。
- **多大算合适**：没有死标准，但一个上下文大致对应"一个小团队能完整掌握的一块业务"。大到一个人说不清它在干嘛，就该考虑再拆。

**一句话**：限界上下文之所以最重要，是因为**边界划错了，后面再精细的实体、聚合设计都白搭**——你是在一个错误的范围里精雕细琢。所以领域建模真正的功夫，一大半在"把边界划对"上。

## 五、模型里有哪些基本元素

划好边界后，每个上下文内部要把模型搭起来。有几个最常用、最值得理解的"零件"。

### 实体 vs 值对象：要不要身份证

**实体**（Entity）是**有身份、靠 ID 区分**的东西。一个人改了名字、换了衣服、长大了，凭身份证号还是同一个人。订单、用户都是实体——属性会变，但 ID 不变，它就还是它。

**值对象**（Value Object）是**只看值、没有身份**的东西。一张 100 元的钞票，你不会在意它的编号，换一张等值的没任何区别；金额、地址、坐标都是这种——两个"北京市朝阳区 xx"的地址完全可互换。值对象通常是不可变的，要改就换一个新的。

为什么要分这两者？因为把"金额""地址"建成值对象后，可以把相关的规则收进去（比如金额对象自己保证"币种不同不能直接相加"），比到处用裸的数字安全得多，也更有业务味道。

### 聚合：哪些东西必须当成一个整体

有些对象关系特别紧、必须一起改才能保证正确，就把它们捆成一个**聚合**（Aggregate），并选一个当"代表"，叫**聚合根**（Aggregate Root）。

最经典的例子：订单和订单项。改一个订单项的数量，会牵动订单的总金额——这俩必须一起改、保持一致。所以订单是聚合根，订单项是聚合内部的成员。规则是：**外部只能通过聚合根来操作，不能绕过它直接去改内部成员。** 你不能直接抓住某个订单项改数量，得调 `order.changeItemQuantity(...)`，让订单这个"代表"统一来保证总金额跟着对。

类比：一个家庭对外由户主代表。你想给这家某个孩子办事，得通过户主，不能绕过户主直接处置人家家庭内部——因为家里的事要由户主统筹、保证不乱。

一个实用经验：**聚合要尽量小。** 捆得越大，每次改动要保证一致的范围就越大、越容易互相卡。所以只把"必须强一致、必须同时改"的东西放一个聚合，其他的拆开、用 ID 互相引用就行（比如订单里只存"商品 ID"，而不是把整个商品对象塞进来）。

### 领域事件：把"发生了什么"显式表达出来

**领域事件**（Domain Event）就是"业务上发生了一件值得关心的事"，用过去式命名：订单已创建、发票已作废、支付已成功。

它最大的用处是**解耦**。订单创建后要扣库存、加积分、发通知——如果订单对象自己挨个去调用这些，就和一堆下游绑死了。改成"订单创建后，发布一个'订单已创建'事件"，谁关心谁去订阅处理，订单这边根本不用知道有谁在听。

类比：公司发了个"新员工入职"的通知，IT 去开账号、行政去发工牌、HR 去建档——发通知的人不需要知道有谁在处理、怎么处理。（落地时这种事件常常通过消息队列来传递。）

## 六、怎么动手做：从听业务说话开始

概念讲完了，真正的问题是：**面对一个新业务，这些实体、聚合、上下文，我从哪儿找出来？** 不是拍脑袋设计的，有方法。

**最常用的方法叫"事件风暴"（Event Storming）**：把业务专家和开发拉到一面墙前，用便利贴一起梳理，大致几步：

1. **先找"发生了什么"（事件）**：让业务专家讲整个流程，把发生的事都贴出来——下单了、支付了、发货了、收货了……（这一步最自然，因为业务方最会讲"然后呢"。）
2. **找每件事是谁触发的（命令/动作）**：支付这件事，是"用户点了支付"触发的。
3. **找动作作用在什么东西上（聚合）**：下单作用在"订单"上，支付作用在"支付单"上——这些名词就是候选的聚合/实体。
4. **把相关的事归堆，划出边界（限界上下文）**：和交易有关的归一堆、和履约有关的归一堆，堆与堆之间的缝，就是限界上下文的边界。

你会发现这个过程**全程在用业务语言**——这正好和"通用语言"对上了：墙上便利贴用的词，就该是代码里用的词。

还有个心态很重要：**模型不是一次定死的。** 第一版模型一定不完美，随着对业务理解加深，要不断回头重构它。领域建模是个持续的过程，不是开工前画完图就再也不动了。

## 七、什么时候用、什么时候别用

领域建模不是银弹，用错地方反而添乱。

**值得用的场景**：业务复杂、规则多、要长期演进的**核心系统**。比如交易、计税、风控这种——逻辑绕、改动频繁、出错代价高，花力气建好模型，长期省下的维护成本很可观。

**不该用的场景**：简单的增删改查。一个后台管理页面、一张配置表的维护，业务就是"把数据存进去、查出来"，没什么规则可言——硬套领域建模那套（聚合、值对象、分层、充血对象）纯属过度设计，又慢又啰嗦。这种直接朴素地写就好。

一个常见的坑是**只学了术语、没抓住精神**：纠结"这个该是实体还是值对象"，却没先把限界上下文划对——其实**划边界（想清业务怎么分块）比搬术语重要得多**。

如果只记三句话，就记这三句：
- **给业务建模，别给数据库建模**——先想业务怎么运转，表是最后的事。
- **代码里的词，要和业务说的词一致**（通用语言）——看代码就能懂业务。
- **先划清边界，再搭内部**（限界上下文优先）——边界对了，细节才有意义。

## 八、附：几个术语的快速对照

读 DDD 资料会反复碰到这些词，这里给个速查（中文 + 英文 + 一句话）：

- **领域（Domain）**：软件要解决的业务范围。
- **子域（Subdomain）**：领域切出的小块；分核心域（你的竞争力，自研做精）、支撑域、通用域（能买/开源就别自己造）。
- **通用语言（Ubiquitous Language）**：业务和代码共用的一套词汇。
- **限界上下文（Bounded Context）**：术语含义唯一的边界，常对应一个微服务。
- **实体（Entity）**：有 ID、靠身份区分的对象（人、订单）。
- **值对象（Value Object）**：无 ID、只看值、通常不可变（金额、地址）。
- **聚合 / 聚合根（Aggregate / Aggregate Root）**：必须一起保证一致的一组对象，外部只通过聚合根操作；聚合要小。
- **领域事件（Domain Event）**：已发生的业务事实，用于解耦，常配消息队列。
- **仓储（Repository）**：给聚合根存取的接口，隔离数据库——用起来像操作内存里的对象集合。
- **领域服务（Domain Service）**：放不进单个对象的跨对象逻辑（如转账）。
- **充血模型 vs 贫血模型**：对象自带行为、自我保护规则（充血）vs 对象只有数据、逻辑全在 Service（贫血）；领域建模提倡充血。
- **事件风暴（Event Storming）**：业务和开发一起用便利贴梳理事件、命令、聚合、边界的协作建模方法。
