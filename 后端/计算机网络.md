# 计算机网络（面试深挖版）

> 主线：分层模型为什么这样分 → 一条 URL 请求怎么走完整条链路 → TCP 怎么建连拆连、怎么保证可靠 → HTTP 怎么演进、长连接怎么维持 → HTTPS 为什么安全、TLS 怎么握手 → 认证与跨域怎么落地。

---

## 一、分层模型：为什么要把网络拆成一层一层

### 1. 解决什么问题

网络是个极度复杂的系统——从你手机里的浏览器，到海底光缆、到地球另一端的服务器，中间要经过网卡、交换机、路由器、运营商、DNS、防火墙……如果一坨代码从头管到尾，谁都没法写、没法换、没法调试。

**分层解决的就是"复杂度的分治"**：把一整条链路按职责切成几层，每层**只管自己的事**、对上一层提供统一接口，层与层之间靠"约定好的接口"通信，互不关心对方内部怎么实现。这样某一层的技术换了（比如物理层从铜线换光纤），上面所有层代码不用动。

**类比**：寄国际快递。你（应用层）只管把包裹和收件地址交给快递公司；快递公司（传输层）管打包、保价、丢了赔偿，不关心走哪条路；物流调度（网络层）管规划路线、过几个中转站；运输车/飞机（数据链路+物理层）管实际搬。每层只跟**相邻的上下层**打交道，不越级——你不会自己去开飞机。

### 2. 三套分层模型（OSI / TCP/IP / 实际五层）

| OSI 七层（理论标准） | TCP/IP 四层（工业实际） | 教学五层（折中） | 典型协议 |
|---|---|---|---|
| 应用层 | 应用层 | 应用层 | HTTP、DNS、SMTP、FTP |
| 表示层 | ↑ 合并 | ↑ 合并 | TLS/SSL、JPEG（编解码） |
| 会话层 | ↑ 合并 | ↑ 合并 | （管理会话，少独立协议） |
| 传输层 | 传输层 | 传输层 | TCP、UDP |
| 网络层 | 网际层 | 网络层 | IP、ICMP、路由协议 |
| 数据链路层 | 网络接口层 | 数据链路层 | 以太网、ARP、MAC |
| 物理层 | ↑ 合并 | 物理层 | 光纤、双绞线、电信号 |

**几个关键认知**：
- **OSI 是理论标准**（ISO 定的），但它太理想化（会话/表示层独立成层，现实里基本没单独实现），所以**实际没人用七层协议栈**，都跑 TCP/IP 四层。
- **TCP/IP 才是真正跑在互联网上的**，四层：应用、传输、网际、网络接口。
- 教学常用**五层**是为了既讲清 OSI 的分层思想、又能对应到真实协议。
- **表示层（编解码、加密压缩）和会话层**没死，只是**融进了应用层**：TLS 就干着表示层"加密"的活，但通常算作"应用层之下、传输层之上"的独立一层。

**数据怎么一层层包下去**：发数据时从上往下，每层把上层的整体当**数据载荷**，前面加自己的**头部**（封装，encapsulation）；收数据时从下往上，每层拆掉自己的头、把载荷交给上层（解封装）。像套娃一样：HTTP 报文塞进 TCP 段 → 塞进 IP 包 → 塞进以太网帧 → 变成比特流上线缆。

### 3. 面试一句话

分层 = 复杂度分治 + 接口解耦。OSI 七层是理论标准（会话/表示层基本没独立实现），**TCP/IP 四层才是互联网实际跑的**，教学折中用五层。每层数据靠"封装/解封装"像套娃一样层层包裹传递。

## 二、从输入 URL 到页面渲染：把整条链路串起来

这是面试最经典的"串联题"——它把 DNS、TCP、TLS、HTTP、渲染全揉进一个问题里，答得好能证明你网络链路是通的，不是只会背单点。

**流程**（以 `https://www.example.com/page` 为例）：

```
1. URL 解析
   └─ 拆出协议(https) + 域名(www.example.com) + 路径(/page)

2. DNS 域名解析（域名 → IP）
   └─ 先查本地缓存(浏览器→OSI→hosts)→ 没有问本地 DNS 服务器
   └─ 本地没有 → 递归向上：根 DNS → 顶级域(.com) → 权威 DNS → 拿到 IP
   └─ 缓存逐层写回（DNS 用 UDP，因为查询小、要快）

3. 建立 TCP 连接（三次握手）  ← 传输层
   └─ 客户端 → SYN → 服务端
   └─ 服务端 → SYN+ACK → 客户端
   └─ 客户端 → ACK → 服务端   （连接 ESTABLISHED）

4. TLS 握手（因为是 https）   ← 表示层/安全层
   └─ ClientHello ↔ ServerHello（协商加密套件、交换证书）
   └─ 密钥交换 → 双方算出对称会话密钥
   └─ 后续 HTTP 报文用这个对称密钥加密传输

5. 发送 HTTP 请求
   └─ 构造请求行(GET /page HTTP/1.1)+ 请求头 + body → 通过 TLS 加密 → 走 TCP 发出

6. 服务器处理 + 返回响应
   └─ Web 服务器(Nginx)接收 → 转发到应用(Tomcat/Spring)→ 处理 → 返回响应
   └─ 响应行(200 OK)+ 响应头(Content-Type/Length)+ 响应体(HTML)

7. 浏览器接收 + 渲染
   └─ 解析 HTML → 建 DOM 树；解析 CSS → 建 CSSOM 树
   └─ 两树合成渲染树 → 布局(layout，算每个元素位置)→ 绘制(paint)
   └─ 遇到 JS/CSS/图片等子资源 → 各自再走一遍 2~6（长连接复用同一条 TCP）

8. 连接复用或关闭
   └─ HTTP/1.1 默认 keep-alive：这条 TCP+TLS 连接留着，下个资源接着用
   └─ 空闲超时或 Connection: close → 四次挥手拆 TCP
```

**容易漏/加分点**：
- DNS 有**多级缓存**（浏览器、OS、路由器、本地 DNS），不是每次都走完整递归。
- https 比 http 多一步 **TLS 握手**（在 TCP 建连之后、HTTP 发请求之前），所以 https 首次比 http 慢就慢在这。
- 渲染时子资源是**复用已建好的长连接**发的，不是每个资源都重新握手——这正是长连接的价值。
- 真实链路还会过 **CDN**（静态资源就近取）、**反向代理（Nginx）**、**负载均衡**，应用服务器前面通常不止一跳。

## 三、TCP 三次握手与四次挥手

### 1. 三次握手：为什么是 3 次不是 2 次

**状态和报文**：
```
客户端                        服务端
CLOSED                        LISTEN（一直监听）
  │
  │ ── SYN, seq=x ──────────→   SYN_RCVD
  │                              │（同意，分配资源）
  │ ←── SYN+ACK, seq=y, ack=x+1 ─┤
ESTABLISHED                     │
  │                              │
  │ ── ACK, ack=y+1 ──────────→   ESTABLISHED
  │                              │
  ════════════════ 双方都 ESTABLISHED，可以发数据 ════════════════
```

**为什么必须 3 次不是 2 次**：
- 2 次（只 SYN→SYN+ACK）的问题：**服务端一收到 SYN 就建连、分配资源**，但客户端可能根本没打算连（这个 SYN 是网络里滞留的旧包、或攻击者伪造的）。服务端建一堆半连接白白耗资源。
- 第 3 次（客户端回 ACK）的意义：**让客户端再确认一次"我确实要连"**。服务端只有收到这个 ACK 才真正建连——给客户端一个"反悔/确认"的机会。
- **类比**：打电话。A："喂能听到吗"(SYN)；B："能听到，你呢"(SYN+ACK)；A："我也能"(ACK)。少最后一句，B 不确定 A 到底听没听见，不敢开始说话。

**seq/ack 的作用**：seq 是"我这个包数据从第几字节开始"，ack 是"你发到 seq 的我都收到了，**期望你下次从 seq+1 开始发**"。三次握手除了建连，还**同步了双方的初始序列号（ISN）**——后续数据按序号确认、重传全靠它。

### 2. 四次挥手：为什么是 4 次不是 3 次

```
客户端（主动关闭方）            服务端（被动关闭方）
ESTABLISHED                    ESTABLISHED
  │ ── FIN, seq=u ───────────→   （"我没数据要发了"）
FIN_WAIT_1                     CLOSE_WAIT
  │ ←── ACK, ack=u+1 ──────────┤  （"收到，但我可能还有数据没发完"）
FIN_WAIT_2                     │
  │                              │ 服务端把剩下的数据发完
  │ ←── FIN, seq=v ─────────────┤  （"我也发完了，我也要关"）
  │                              LAST_ACK
TIME_WAIT                       │
  │ ── ACK, ack=v+1 ───────────→   CLOSED
  │（等 2MSL）
CLOSED
```

**为什么挥手要 4 次而握手只要 3 次**：
- 握手时服务端的 SYN 和 ACK 可以**合在一个包里发**（反正都要回）。
- 挥手时，被动方收到 FIN 只能先回 ACK（"知道你要关了"），但**自己可能还有数据没发完**，得等发完再单独发 FIN。所以 **ACK 和 FIN 拆成两次**，加上主动方的 FIN 和被动方的 FIN，共 4 次。
- **如果被动方恰好也没数据要发**，ACK 和 FIN 也能合并（叫"延迟确认"），变成 3 次——但**协议允许拆开**，所以标准答案是 4 次。

### 3. TIME_WAIT：为什么要等 2MSL

主动关闭方发完最后一个 ACK 后进入 TIME_WAIT，**等 2 个 MSL（最大报文生存时间）才真正 CLOSED**。两个原因：

1. **保证最后一个 ACK 能到对端**：这个 ACK 丢了的话，被动方会**重发 FIN**，主动方还在 TIME_WAIT 就能重新回 ACK；如果主动方已经 CLOSED 走人了，被动方重发的 FIN 没人理，被动方会卡在 LAST_ACK 关不掉。等 2MSL 就是给"丢的 ACK + 重发的 FIN"留往返时间。
2. **让本次连接的旧报文在网络中消亡**：防止这次连接滞留的报文，被下一次**复用同端口**的新连接错误接收。2MSL 基本能让旧包 TTL 到期消失。

**TIME_WAIT 的代价**：连接关闭后**端口和少量内存还要占 2MSL**（Linux 默认 60 秒）。短连接高频关闭会堆大量 TIME_WAIT，**耗光端口**——这是为什么服务端用长连接（少挥手）和调小/复用 TIME_WAIT。

### 4. SYN Flood 攻击与防御

攻击者**海量发伪造源 IP 的 SYN**，服务端每个都回 SYN+ACK 并分配资源等第 3 次 ACK（永远不来）→ **半连接队列塞满**，正常用户连不上。

**防御**：**SYN Cookies**——服务端收到 SYN 时**不分配资源**，而是把连接信息编码进一个 cookie 塞进 SYN+ACK 的 seq 里发出去；只有真正收到客户端回的 ACK（带回 cookie）才反算验证、这时才建连。这样半连接不耗资源，伪造的 SYN 也伤不到。代价是丢了一些序列号特性，平时关、被攻击时开。

### 5. 面试一句话

三次握手：双方同步序列号 + 双向确认通信能力，第 3 次是客户端确认给服务端"反悔机会"，防止旧 SYN/伪造 SYN 让服务端空建连。四次挥手：因为被动方收到 FIN 时可能还有数据没发完，ACK 和 FIN 拆两次，所以 4 次。TIME_WAIT 等 2MSL：一保最后 ACK 能补发、二让旧报文消亡防复用端口串包。SYN Flood 靠 SYN Cookies 不预分配资源防御。

## 四、TCP 可靠传输靠什么

TCP 在不可靠的 IP 层之上**造出一条可靠、有序、不丢不重的字节流通道**。它靠的不是某一招，是一套组合拳：

### 1. 序号 + 确认（基础）

每个字节都有序号（seq），接收方按序号**去重、排序、确认**（ack = "我收到这了，期望下一个从这开始"）。乱序到达的包能按 seq 重排，重复的能识别丢弃。这是"可靠+有序"的地基。

### 2. 重传（丢了就重发）

发送方发出去后**启动定时器**，超时没收到 ACK 就重传。问题：超时时间怎么定？网络时延是动态的，定长了要么太早（误重传）、要么太晚（丢了干等）。TCP 用 **RTT（往返时延）动态估算** + 平滑算法自适应。

超时重传太"钝"（要等超时），所以还有更聪明的 **快重传**：接收方每收到一个乱序包就**立刻回重复 ACK**，发送方连收 3 个重复 ACK 就知道"这个包大概率丢了"，**不等超时直接重传**——更快。

### 3. 滑动窗口 + 流量控制（收发速度匹配）

**问题**：发一个等一个 ACK 再发下一个（停等协议）太慢——网络往返 100ms 的话带宽利用率极低。

**滑动窗口**：允许发送方**连续发多个包不等 ACK**（在窗口内），收到 ACK 后窗口往前滑，继续发。窗口大小 = "还能发多少未经确认的数据"。

**流量控制**（flow control）：接收方在自己的 ACK 里带 **窗口大小（rwnd）**，告诉发送方"我接收缓冲区还能装多少"。发送方据此限速，**防发送太快把接收方撑爆**。这是**收发两端之间**的速度协调。

### 4. 拥塞控制（网络不堵）

**和流量控制的区别**（易混点）：
- **流量控制**：管"接收方扛不扛得住"，靠 rwnd。
- **拥塞控制**：管"中间网络扛不扛得住"，靠发送方**自己探测**网络拥塞程度（网络没有谁告诉你"我堵了"）。

四个经典算法：
- **慢开始（slow start）**：刚连上不知道网络多空，先**从 1 个包小步试探**，每收到一个 ACK 窗口**翻倍**增长（指数级，其实不慢，是"试探性快速增长"）。
- **拥塞避免**：窗口涨到阈值（ssthresh）后改成**线性增长**（每 RTT +1），慢慢探到网络极限。
- **快重传**：连收 3 个重复 ACK → 立即重传丢失包（前面讲过）。
- **快恢复**：快重传后**不回到慢开始**，而是把阈值砍半、窗口设为砍半后的值线性增长——因为"还能收到重复 ACK 说明网络还通着"，比超时（网络真堵死）温和。

**判断拥塞的信号**：超时 = 网络严重拥塞（窗口砍半回慢开始）；3 个重复 ACK = 轻度丢包（快恢复，温和砍半）。

### 5. 面试一句话

TCP 可靠 = 序号确认（去重排序）+ 超时/快重传（丢包重发）+ 滑动窗口+流量控制 rwnd（收发匹配）+ 拥塞控制（慢开始/拥塞避免/快重传/快恢复，发送方自探网络）。流量控制管接收方扛不扛得住，拥塞控制管中间网络扛不扛得住——前者靠对端告诉，后者靠自己探测。

### 6. TCP 全双工：同一个端口怎么同时上传下载

**先破误区**：很多人以为"端口像一根水管，水只能单向流或得交替"——**这是错的**。端口（port）就是个 16 位数字（0-65535），是传输层的**逻辑地址**，用来标识"这台机器上哪个 socket 进程"。它没有方向性、不是物理通道。OS 收到报文段看**目的端口**分发到对应 socket 的接收缓冲区；发数据时从发送缓冲区取、**源端口**填这个端口发出去。**同一个端口既能当源端口（发）也能当目的端口（收），完全不冲突**——像你家门牌号，寄件（发）和收件（收）都用同一个门牌号，不矛盾。

**靠两块独立缓冲区实现同时收发**：每个 socket 有**两块独立的内存**——发送缓冲区（send buffer）和接收缓冲区（recv buffer）。
- **发（上传）**：应用 `send()` → 数据进发送缓冲区 → TCP 切段、加序号、发出去，源端口填本地端口。
- **收（下载）**：网卡收到报文段 → TCP 校验序号、去重排序 → 放进接收缓冲区 → 应用 `recv()` 取走。

发送流程和接收流程是**两套独立的流水线**——各自的内存、各自的处理逻辑，互不等待。发不用等收、收不用等发，两套可以**同时跑**（多核并行 / 中断驱动）。**不是端口在分时，是收发根本走两条独立通路**。

**类比**：门牌号 = 端口。你家有**寄件箱**（发送缓冲区）和**收件箱**（接收缓冲区），两个箱子独立。寄件从寄件箱拿、收件放收件箱。快递车来送货（收）和你的寄件被取走（发）可以同时发生，互不干扰——门牌号只是地址，不限制方向。

**一条连接是四元组，不是"一个端口"**：一条 TCP 连接由 `{本地IP, 本地端口, 远端IP, 远端端口}` 唯一标识。同一台机器的同一个本地端口（如服务端 80）能同时和**多个远端**建多条连接——因为四元组不同。OS 靠四元组把收到的报文段**精确路由**到对应 socket，不会串。

**全双工的软件本质——收发两套独立序号和窗口**：TCP 把一条连接当成**两条独立字节流**（A→B 一条、B→A 一条），各自一套：

| | A→B 方向 | B→A 方向 |
|---|---|---|
| 序号空间 | A 发送序号 / B 接收序号 | B 发送序号 / A 接收序号 |
| 滑动窗口 | A 发送窗口、B 报 rwnd 给 A | B 发送窗口、A 报 rwnd 给 B |
| 确认 | B 给 A 回 ACK | A 给 B 回 ACK |

两个方向各自独立的序号、窗口、确认，独立流动、互不影响。**软件层就把收发拆成两套独立机制**，这是全双工的根——也是 WebSocket 能在一条 TCP 上双方随时发帧的基础。

**往下走——物理层也支持全双工（但 TCP 不依赖它）**：
- **双绞线以太网**：网线 4 对线，其中**一对专门发（TX）、一对专门收（RX）**，物理上两根独立线对，当然同时双向。
- **光纤**：两根纤芯（或单纤双向 WDM 用两个波长），收发分离。
- **无线**：半双工（同时只能发或收，靠快速分时 + CSMA/CA），链路层快速切换让上层"看起来全双工"。

**重要**：TCP 全双工是**软件层就保证的**（独立缓冲区 + 独立序号窗口），**不依赖物理层必须全双工**。物理层半双工（无线、老式 Hub）时，TCP 逻辑上仍是全双工——只是物理分时让"同时"变成"快速交替"，对上层透明。

**易混点**：UDP socket 同样有独立收发缓冲区，也是全双工的——只是没有可靠传输/序号/窗口。**全双工是 socket + 缓冲区提供的，不是 TCP 特有**；TCP 强调全双工是因为它有**两个方向的独立序号空间和窗口**，让"可靠的双向字节流"成为可能。

**一句话**：同一端口能同时上传下载，是因为**端口只是地址标签不是单根通道**，每个 socket 有**两块独立的发送/接收缓冲区和两套独立收发流程**（各自序号/窗口/确认），收发走两条独立通路互不等待；连接由**四元组**标识、OS 靠它路由报文段。物理层（独立收发线对）也提供全双工基础，但 TCP 的全双工软件层就保证、不依赖物理层。

## 五、TCP vs UDP

| | TCP | UDP |
|---|---|---|
| 连接 | 面向连接（先握手） | 无连接，直接发 |
| 可靠性 | 可靠（不丢不重有序） | 不可靠，尽力而为 |
| 顺序 | 保证有序 | 不保证 |
| 速度/开销 | 慢、开销大（头部+状态机+重传） | 快、开销小（8 字节头） |
| 传输单位 | 字节流 | 数据报 |
| 拥塞/流量控制 | 有 | 无 |
| 典型应用 | HTTP、SSH、邮件、文件传输 | DNS、视频/直播、游戏、QUIC |

**选型逻辑**：
- **要可靠、有序、不能丢**（网页、文件、交易）→ TCP。代价是握手开销和拥塞控制的"谨慎"。
- **要实时、能容忍少量丢包、不能等重传**（直播、语音、游戏）→ UDP。视频丢一帧观众几乎无感，但等 TCP 重传会卡顿。
- **既要 UDP 的快又要可靠**：别在应用层自己搓重传，**用 QUIC**（HTTP/3 就跑在 QUIC 上，UDP 之上加了可靠传输+多路复用+0-RTT 建连）。

## 六、HTTP 演进与报文

### 1. HTTP 是什么

HTTP（HyperText Transfer Protocol）是**应用层协议**，定义了客户端和服务端怎么交换报文。它本身**不管可靠传输**（那是 TCP 的事），只管"请求长什么样、响应长什么样"。

### 2. 报文结构

```
请求：
  请求行：  GET /page HTTP/1.1       （方法 + 路径 + 版本）
  请求头：  Host: www.example.com    （键值对，元信息）
           Content-Type: application/json
  空行：    （分隔头和体）
  请求体：  { ... }                   （可选，POST 才常有）

响应：
  状态行：  HTTP/1.1 200 OK          （版本 + 状态码 + 原因）
  响应头：  Content-Type: text/html
           Content-Length: 1024
  空行：
  响应体：  <html>...</html>
```

**关键认知**：报文是**纯文本**（HTTP/1 是这样，HTTP/2 才改二进制帧），头是键值对、用空行分隔头和体。这是为什么 HTTP 调试能用 `telnet`/`curl` 肉眼读。

### 3. 状态码（分五类，记语义不记数字）

| 类 | 语义 | 高频 |
|---|---|---|
| 1xx | 信息（继续处理） | 101 Switching Protocols（升级到 WebSocket） |
| 2xx | 成功 | 200 OK、204 No Content、206 Partial Content（断点续传） |
| 3xx | 重定向 | 301 永久、302 临时、304 Not Modified（缓存命中，别传 body 了） |
| 4xx | 客户端错 | 400 参数错、401 未认证、403 无权限、404 没找到、429 限流 |
| 5xx | 服务端错 | 500 内部错、502 网关错、503 不可用、504 网关超时 |

**易混**：401（没登录，要认证）vs 403（登录了但没权限）；502（上游服务挂了/返回无效）vs 504（上游服务太慢超时）。

### 4. 版本演进（这是 HTTP 面试核心）

**HTTP/1.0**：默认短连接（一请求一连接），性能差。靠 `Connection: keep-alive` 头手动开长连接。

**HTTP/1.1**（用最久）：
- 默认长连接（keep-alive），复用 TCP。
- 支持 **pipelining**（管道化，一条连接上可连发多个请求），但响应得按序回——实际几乎没人用，因为**队头阻塞**（前一个慢响应堵住后面）。
- 引入 **Host 头**（一条连接一台机器可虚拟多个域名）、**chunked 分块传输**、缓存控制、断点续传。
- **遗留问题**：文本格式、队头阻塞、为凑并发得开多条 TCP 连接（浏览器一般 6 条）。

**HTTP/2**：
- **二进制分帧**：报文不再是文本，切成带流 ID 的小帧（frame），一条 TCP 连接上**多个流并行交错发送**——**多路复用**，彻底解决 HTTP 层队头阻塞，不用开 6 条连接。
- **头部压缩**（HPACK）：重复的请求头压缩掉。
- **服务端推送**（Server Push，实际用得少、HTTP/3 已弱化）。
- **遗留问题**：还跑在 TCP 上，**TCP 层的队头阻塞没解决**——一个 TCP 包丢了，这条连接上所有流全卡住等重传。

**HTTP/3**：
- **不用 TCP 了**，跑在 **QUIC**（基于 UDP）上。
- QUIC 在 UDP 之上**自己实现可靠传输 + 多路复用 + 加密**，且**一条流丢包不影响其他流**——彻底解决 TCP 队头阻塞。
- **0-RTT 建连**：复用过的连接，下次握手能省一个往返，首屏更快。
- 代价：UDP 在一些网络/防火墙被限流，部署比 TCP 麻烦，但大厂和主流站点都在迁。

### 5. 面试一句话

HTTP/1.0 短连接；1.1 默认长连接+Host 头+chunked，但文本格式+队头阻塞；2.0 二进制分帧+多路复用+头部压缩，解决 HTTP 层队头阻塞但还卡在 TCP 层队头阻塞；3.0 干脆换 QUIC(UDP)，流间不阻塞+0-RTT。状态码记语义：4xx 客户端错(401 认证/403 权限)、5xx 服务端错(502 上游挂/504 上游超时)。

## 七、HTTP 长连接：怎么保持的（重点）

### 1. 解决什么问题：短连接的浪费

**HTTP/1.0 默认短连接**：每个请求建一次 TCP（三次握手），用完就断（四次挥手）。一个网页几十个资源，光握手就花掉大半时间，尤其走 HTTPS 还多一轮 TLS 握手。

**类比**：寄快递每次都重新叫车（建 TCP）、装货、卸货、车开走（断连）。长连接让**车别开走，停门口**，下个请求接着用——一次握手复用多次。

### 2. HTTP 层怎么做

- **HTTP/1.0**：要手动开，请求头加 `Connection: keep-alive`，服务端也愿意才复用。
- **HTTP/1.1**：**默认就是长连接**，不传 Connection 头默认复用；想关就显式发 `Connection: close`。

**底层**：TCP 连接发完一个请求**不主动发 FIN**（不挥断），保持 ESTABLISHED，下个 HTTP 请求直接复用这条 TCP。"HTTP 长连接"本质 = **复用同一条 TCP 连接发多个 HTTP 请求**，HTTP 自己不发心跳维持——依赖 TCP 连接本身还活着。

### 3. 遇到的问题 1：车上的货怎么分批

一条 TCP 上接连发多个请求，服务端怎么知道**这个 body 到哪结束、下一个从哪开始**（framing 分帧问题）？两种方案：

- **Content-Length**（知道总长）：响应头带 `Content-Length: 1024`，读够 1024 字节就知道这个响应完了。**前提是发送前就知道总大小**（静态文件、buffer 好的内容）。
- **分块传输 chunked**（不知道总长）：动态生成的内容发之前不知道多大，用 `Transfer-Encoding: chunked`——把 body 切成一块块，每块前标长度，最后用**长度为 0 的块**表示结束：
  ```
  Transfer-Encoding: chunked

  5\r\n        ← 这块 5 字节
  hello\r\n   ← 内容
  0\r\n       ← 长度0，结束
  \r\n
  ```
  服务端读到"长度0的块"就知道响应完了。**SSE、流式响应就靠这个**。

只有能正确判断请求/响应边界，长连接上多个请求才不会粘成一团。这是长连接能工作的**前置条件**。

### 4. 遇到的问题 2：连接一直闲着怎么办

车停门口一直不用，不能永远占着——端口、内存、文件描述符都是资源。两个层面的"保活/断开"，**千万别混**：

- **HTTP keep-alive timeout（应用层，空闲多久关）**：服务端/客户端配 `Keep-Alive: timeout=5`（或 Nginx 的 `keepalive_timeout`）——连接**空闲超过 5 秒就主动发 FIN 关掉**，释放资源。"车停太久没活干，让车走。"
- **TCP keepalive（传输层，探测连接死没死）**：长连接静静躺着，但**对端可能已崩了/网线断了/断电了**，你这头没收到 FIN 还以为连着。TCP 有个可选保活机制：连接空闲超过 `tcp_keepalive_time`（默认 2 小时）后，**周期性发空 ACK 探测包**，几次没回应就把连接标记为死、关闭。"车停门口，偶尔敲敲门确认公司还在不在。"

**两者的区别（面试重点）**：

| | HTTP keep-alive | TCP keepalive |
|---|---|---|
| 哪一层 | 应用层（HTTP） | 传输层（TCP 内核） |
| 目的 | **复用连接**（别每次重新握手） | **探测死连接**（对端还活着吗） |
| 谁配 | Web 服务器/客户端（Nginx/浏览器） | 操作系统内核参数 |
| 默认 | HTTP/1.1 默认开 | 默认**关**（要手动开），且 2 小时太长很少靠它 |

所以"HTTP 长连接靠什么保持"——靠 **TCP 连接不被主动关闭 + HTTP 层有 framing 能分清请求边界 + 应用层 keep-alive timeout 控制空闲回收**。TCP keepalive 不是为"保持"长连接设计的，是为"探测已死的连接"。

### 5. 遇到的问题 3：队头阻塞

HTTP/1.1 长连接虽能复用，但**一条连接同一时刻只能处理一个请求**——前一个响应没回完，后面请求只能排队等（HTTP 队头阻塞）。浏览器的绕路：**开 6 条并发连接**（域名分片）凑并发。HTTP/2 的多路复用才从根上解决。

### 6. 面试一句话

HTTP 长连接 = 复用同一条 TCP 连接发多个请求（1.1 默认开），靠 **Content-Length/chunked 分清请求边界**、靠**应用层 keep-alive timeout 回收空闲连接**。注意 **HTTP keep-alive(复用) ≠ TCP keepalive(探活)**——后者默认关、2 小时探测，跟"保持"长连接是两回事。队头阻塞靠 HTTP/2 多路复用解决。

### 7. SSE：长连接上的服务端单向推送

**解决什么问题**：HTTP 是请求-响应，客户端不发请求服务端就没法主动推数据。要拿服务端实时更新（行情、通知、AI 逐字生成），有四种方案，对比看：

**类比：等快递进度更新**
- **短轮询**：每分钟打一次电话问"到哪了"，大部分时候答"没动"，白打。
- **长轮询**：打电话过去快递员**不挂**，等到有进展才告诉你；说完立刻再打一个挂着等。
- **SSE**：打通后电话**一直不挂**，快递员有进展就**主动顺着这条线说给你听**，你只能听、不能插话（单向）。
- **WebSocket**：双向**对讲机**，随时互相说。

SSE 就是**建立一条 HTTP 长连接，服务端有数据就顺着推下来**——本质是 **HTTP/1.1 长连接 + chunked 分块传输的规范化用法**（服务端不知道总长、边生成边发，正好就是上一节讲的 chunked）。

**报文格式**：响应头 `Content-Type: text/event-stream`，body 纯文本按固定格式：
```
data: 第一条消息          ← 数据行（多条 data 用 \n 拼成一条）

event: update             ← 自定义事件类型
data: {"price": 100}

id: 42                    ← 事件 ID，断线重连续传用
data: 带ID的消息

retry: 3000               ← 断线后多久重连（毫秒）
: keepalive               ← 注释行，常当心跳包防代理掐断
```
事件之间用**空行（`\n\n`）分隔**，服务端推一条 flush 一次。

**客户端**：浏览器原生 `EventSource`，一行建连、自动监听：
```javascript
const es = new EventSource("/api/updates");
es.onmessage = e => console.log(e.data);              // 默认 message 事件
es.addEventListener("update", e => console.log(e.data)); // 自定义事件
// 断了浏览器自动重连，不用自己写
```

**大杀器——断线自动重连 + Last-Event-ID**：连接断了，浏览器**自动按 retry 间隔重连**，并带上请求头 `Last-Event-ID: 42`，告诉服务端"我上次收到 id=42，从 42 之后接着推"。服务端读这个头跳过已推的、从断点续推——**消息不丢不重**。这是 SSE 相比 WebSocket 最省心的点（WebSocket 断了要自己实现重连+续传协议）。

**为什么 AI 流式输出几乎都用 SSE**：LLM 逐 token 边生成边吐，要实时推给前端，SSE 完美契合——单向够用、标准 HTTP 穿透防火墙/代理/CDN 友好（WebSocket 有时被掐）、浏览器原生 EventSource、自动重连、chunked 正好契合"生成多少推多少"。所以 **Claude API、OpenAI API 的流式响应都是 SSE 格式**（`data: {"delta":{"text":"..."}}` 一条条推，最后 `data: [DONE]` 收尾）；A2A 协议的 `tasks/sendSubscribe` 也用 SSE 推任务状态变化。

**局限**：
- **单向**：只能服务端→客户端，客户端发消息得另发普通 HTTP 请求。
- **只能传 UTF-8 文本**：二进制要 base64，不如 WebSocket 原生二进制高效。
- **HTTP/1.1 下受 6 连接限制**（HTTP/2 多路复用能缓解）。
- **浏览器 EventSource 只支持 GET**：带复杂参数要走 query，或用 `fetch + ReadableStream` 自己解析 SSE 流（SDK 调流式 API 常这么干，能带 body/自定义头）。
- **代理空闲超时**：要用心跳包（`:keepalive`）保活。

**SSE vs WebSocket 选型**：

| | SSE | WebSocket |
|---|---|---|
| 方向 | 单向（服务端→客户端） | 全双工双向 |
| 协议 | HTTP（穿透好） | 独立协议（升级自 HTTP） |
| 数据 | 文本（UTF-8） | 文本 + 二进制 |
| 重连 | 浏览器自动 + Last-Event-ID 续传 | 要自己实现 |
| 适用 | 通知、行情、AI 流式、A2A 任务订阅 | 聊天、游戏、协同编辑 |

**选型逻辑**：只需服务端单向推（实时通知、AI 逐字流式、任务进度订阅）→ SSE，简单可靠；需要双向实时（聊天室、协同、游戏）→ WebSocket。大部分"看起来像实时"的 AI 场景其实单向就够，SSE 比 WebSocket 简单得多——这是大模型流式清一色 SSE 的原因。

**一句话**：SSE = HTTP 长连接上的服务端单向推送，纯文本 `data:` 行 + 空行分隔，靠 chunked 边生成边发；杀手锏是浏览器原生 EventSource + 断线自动重连带 Last-Event-ID 续传。AI 流式输出和 A2A 任务订阅都用它。

### 8. WebSocket：借 HTTP 握手的全双工协议

**解决什么问题**：HTTP 是请求-响应（客户端主动、服务端被动），SSE 是单向服务端推送——都不能"双方随时互发"。聊天、游戏、协同编辑要**双向实时**，就上 WebSocket：一条连接，双方随时主动给对方发消息。

**层次定位**：WebSocket 是**应用层**协议，和 HTTP 同层、都基于 TCP。它不是 HTTP 的一部分——虽然握手复用了 HTTP 报文，但握手一完成，这条 TCP 连接上跑的**就不再是 HTTP，而是 WebSocket 自己的帧协议**。

**握手：借 HTTP 的壳升级（101）**。设计精妙在**复用现有基础设施**（走 80/443 端口、看起来就是普通 HTTP、穿透防火墙/代理/CDN）：
```
客户端 GET 带：Upgrade: websocket / Connection: Upgrade / Sec-WebSocket-Key: <随机> / Sec-WebSocket-Version: 13
服务端回 101：Upgrade: websocket / Sec-WebSocket-Accept: <回执>
```
`Sec-WebSocket-Accept` 怎么算：服务端把客户端的 Key 拼上**固定魔法 GUID**（`258EAFA5-...`）做 **SHA-1 + Base64**。客户端自己算一遍比对——对得上才算握手成功。这个 GUID 防"普通 HTTP 服务器不小心回 101"的误用。握手完成后这条 TCP 连接就**切换成 WebSocket 帧协议**，这就是 `101 Switching Protocols` 的语义。

**握手后：帧（Frame）协议**。双方通过这条 TCP 互发二进制帧，关键字段：
- **FIN**：是否消息最后一帧（一条消息可分多帧发）。
- **opcode**：`0x1` 文本、`0x2` 二进制、`0x0` 续帧、`0x8` 关闭、`0x9`/`0xA` ping/pong。
- **MASK**：客户端发的帧**必须掩码**（用 4 字节 key 异或），服务端发的不掩码——防**中间代理缓存污染攻击**（掩码后数据随机，代理无法误判成 HTTP 缓存恶意内容）。
- **payload length**：7/16/64 位三档，短消息短字段、长消息自动升级——既能发几字节也能发大文件。

**ping/pong 心跳**：一方发 ping、对方必须回 pong，用于**保活**（防代理掐断空闲连接）和**探活**（确认对端活着）。应用层自己控制频率，比 TCP keepalive 的 2 小时灵活。

**ws vs wss**：`ws://` 明文（像 http），`wss://` 是 **WebSocket over TLS**（像 https，先建 TLS 再在里面握手），生产几乎都用 wss。

**全双工怎么实现的（核心）**：关键认知——**全双工不是 WebSocket 发明的新能力，是 TCP 本来就有的；HTTP 的"单向"是应用层语义强加的限制，WebSocket 握手后把这个限制去掉了。**
- **TCP 本身就是全双工的**：一条 TCP 连接有两个独立方向（收/发各一条字节流，各有自己的序号和窗口），双方向可**同时**传输——这是传输层提供的能力。
- **HTTP 为什么看起来单向**：不是 TCP 不行，是 **HTTP 协议语义规定**了"客户端发请求 → 服务端回响应"，服务端**不能在没收到请求时主动发**。这是应用层人为限制。
- **WebSocket 全双工**：握手用掉一次 HTTP 请求-响应（这次还遵循 HTTP 语义）；握手后**不再遵循 HTTP 语义**，双方都能**在任何时刻主动发帧**，不需要对方先发请求。底层 TCP 双向本就通，协议层一放开就是全双工。
- **实现上**：双方各跑一个**独立的读循环和写循环**（两个并发任务/线程），互不阻塞——任何时候有数据就往这条 TCP 写一帧，同时另一个循环一直在读对方发来的帧。读写共享同一条 TCP 连接，但因 TCP 双方向独立、互不冲突。

**类比**：TCP 像**双向车道的高速公路**（两个方向各一条车道，本就能双向同时通车）。HTTP 像规定了"只能从 A 站发车去 B 站、B 站收到后回一辆车"的**调度规则**——不是路不通，是规则限制。WebSocket 握手后 = **取消这个调度规则，两个方向随时都能发车**——路本来就双向，放开规则就全双工了。

**为什么 HTTP/SSE 不算全双工**：
- HTTP 长连接：服务端**不能在没请求时主动推**（每次推送都得"响应"某请求）。
- SSE：服务端能主动推了，但**客户端不能通过这条连接发消息**（只能听，要发得另开 HTTP）。
- WebSocket：双方都能在**同一条连接**上随时主动发——这才是真全双工。

**HTTP / SSE / WebSocket 三者对比**：

| | HTTP | SSE | WebSocket |
|---|---|---|---|
| 网络层 | 应用层（基于TCP） | 应用层（基于HTTP/TCP） | 应用层（基于TCP） |
| 方向 | 请求-响应（客户端主动） | 单向（服务端→客户端） | **全双工双向** |
| 协议 | HTTP | HTTP | 独立协议（握手借HTTP） |
| 数据 | 文本 | 文本（UTF-8） | **文本+二进制** |
| 重连 | — | 浏览器自动+Last-Event-ID | **要自己实现** |
| 穿透性 | 好 | 好（就是HTTP） | 较好（借HTTP握手，但有些代理掐WS帧） |
| 适用 | 常规请求 | 通知/流式输出 | 聊天/游戏/协同 |

**选型**：只需服务端单向推（AI 流式、通知）→ SSE；需要双向实时（聊天、协同、游戏）→ WebSocket；普通请求-响应 → HTTP。WebSocket 的代价：**重连续传要自己实现**、**有些代理对 WS 帧不友好会掐断**（这也是 AI 流式宁用 SSE 不用 WebSocket 的原因之一）。

**一句话**：WebSocket 是**应用层**协议（和 HTTP 同层、基于 TCP），为全双工双向实时设计；**借一次 HTTP 握手完成升级**（`Upgrade: websocket` + `101` + Key 拼 GUID 算 SHA-1 回执验证），握手后切换成二进制帧协议（opcode 区分类型、客户端发帧必须掩码防代理缓存污染、ping/pong 心跳保活）。**全双工的真相是：TCP 本来就全双工，HTTP 的单向是应用层语义限制，WebSocket 握手后去掉了这个限制**——双方各跑独立读写循环，随时主动发帧。

## 八、HTTPS 与 TLS：为什么安全

### 1. 解决什么问题：HTTP 三个不安全

HTTP 明文传输，有三大风险：
- **窃听**：中间人能直接看到内容（密码、token）。
- **篡改**：中间人能改了再转，你发现不了。
- **冒充**：你连的 `bank.com` 可能是攻击者伪造的钓鱼站，你不知道。

**HTTPS = HTTP + TLS**，在 TCP 和 HTTP 之间加一层 TLS，解决这三个：**加密**防窃听、**摘要/MAC**防篡改、**证书**防冒充。

### 2. 为什么用"非对称 + 对称"两种加密

- **对称加密**（一把钥匙加密解密）：快，但**密钥怎么安全传给对方**是个死结——线上传密钥本身就会被窃听。
- **非对称加密**（公钥加密、私钥解密）：公钥可以随便公开，用公钥加密的内容只有私钥能解。解决了密钥传递问题，但**慢得不能用大数据量**。

**TLS 的组合拳**：**握手阶段用非对称加密，安全地协商出一个对称密钥；后续数据全用这个对称密钥加密（快）**。各取所长——非对称解决"密钥怎么安全传"，对称解决"数据怎么快加密"。

### 3. TLS 握手流程（以 TLS 1.2 为例）

```
客户端                                     服务端
  │ ── ClientHello ──────────────→
  │   （支持的加密套件列表、随机数 A、TLS 版本）
  │                              │
  │ ←── ServerHello ─────────────┤
  │   （选定套件、随机数 B）
  │ ←── 证书（服务端公钥，含 CA 签名）──┤
  │ ←── ServerHelloDone ─────────┤
  │                              │
  │  ① 验证证书：用 CA 公钥验签名，确认这公钥真是该域名的 ← 防冒充
  │  ② 生成预主密钥，用服务端公钥加密（非对称）← 防窃听
  │ ── Encrypted PreMasterSecret ─→
  │                              │
  │  双方用 随机数A + 随机数B + 预主密钥 算出对称会话密钥
  │                              │
  │ ── ChangeCipherSpec + Finished（用会话密钥加密）→
  │ ←── ChangeCipherSpec + Finished ─┤
  │                              │
  ═══════ 后续 HTTP 全用会话密钥对称加密 ═══════
```

**为什么握手要带两个随机数（A、B）**：保证每次握手的会话密钥都不同——即使预主密钥相同，随机数不同密钥就不同，防**重放攻击**（攻击者录下旧握手包重发没用）。

### 4. 证书与 PKI：怎么防冒充

非对称加密里，**你怎么确认拿到的公钥真是该域名的**？否则中间人可以给你一个假公钥，你用假公钥加密、中间人用配套私钥解密——这就是**中间人攻击（MITM）**。

**证书（X.509）+ CA（证书颁发机构）解决**：
- 服务端把**自己的公钥 + 域名信息**交给 **CA**，CA 核实后用**CA 的私钥签名**，变成证书。
- 客户端收到证书后，用**操作系统/浏览器内置的 CA 公钥**验签——签名对得上，说明这个公钥是 CA 背书的、可信的。
- **信任链**：根 CA → 中间 CA → 你的服务器证书，层层签名。根 CA 的公钥是操作系统内置的"信任锚"。

**类比**：你要确认一个人是真房东（要转房租给他），不能只信他嘴说的——你要看**房产证（证书）**，房产证上有**房管局（CA）的公章（签名）**，你认房管局的章（系统内置 CA 公钥），所以信这个房产证、信这个人。

### 5. TLS 1.3 的简化

TLS 1.3 把握手从 2-RTT 压到 **1-RTT**（砍掉冗余往返），还支持 **0-RTT**（复用过的连接，下次可零往返恢复）。安全性也更强（砍了一堆被证明不安全的旧算法）。这是 HTTPS 性能持续逼近 HTTP 的关键。

### 6. 面试一句话

HTTPS = HTTP + TLS，解决 HTTP 明文的窃听/篡改/冒充。核心是**握手用非对称加密安全协商出对称密钥、传数据用对称密钥快加密**；防冒充靠**CA 签名的证书 + 系统/浏览器内置 CA 公钥验签的信任链**。握手带随机数防重放。TLS 1.3 把握手压到 1-RTT、支持 0-RTT。

## 九、Cookie / Session / Token：怎么记住"你是谁"

HTTP **无状态**——每个请求独立，服务器不记得你上次来过。但登录后要"记住你登录了"，有三种方案：

### 1. Cookie（存客户端）

服务端在响应头 `Set-Cookie` 下发一小段数据，浏览器**自动存储**并在**后续同域请求自动带上**。它是浏览器机制，不是应用概念。
- **优点**：自动携带，省事。
- **缺点**：有**安全隐患**——同域请求自动带，易被 **CSRF** 利用（见下节）；容量小（4KB）、跨域受限。

### 2. Session（存服务端）

服务端给每个会话生成一个 sessionId，存进**Cookie**下发；后续请求带这个 id，服务端用 id 找到**存在自己内存/Redis 里的会话数据**。
- **优点**：数据在服务端，安全；可随时失效（删掉服务端 session 即可）。
- **缺点**：**服务端有状态**——分布式环境下 session 要共享（存 Redis），否则请求落到不同机器找不到 session；扩容/水平扩展麻烦。

### 3. Token / JWT（无状态，分布式友好）

登录成功后服务端发一个 **token**（JWT 是常见格式），客户端自己存（localStorage/Cookie），后续请求**手动放在 Authorization 头**带上。服务端**不存 token**，靠**验签**确认 token 真实有效。

**JWT 结构**：`Header.Payload.Signature`（三段 base64）——头部（算法）+ 载荷（用户信息、过期时间）+ 签名（用服务端密钥对前两段签名）。服务端只要用密钥验签通过、且没过期，就认。
- **优点**：**无状态**——服务端不存，天然适合分布式/微服务；跨域友好（放 Header 不受 Cookie 同域限制）。
- **缺点**：**发了就难主动失效**（服务端没存，只能等它自然过期，或维护黑名单）；payload 是 base64 **不加密**（别放敏感数据）；续期/吊销要额外设计。

| | Cookie | Session | JWT |
|---|---|---|---|
| 存哪 | 客户端(浏览器) | 服务端(内存/Redis) | 客户端(自存) |
| 服务端有无状态 | — | **有状态** | **无状态** |
| 分布式 | 受限 | 要共享 session | **天然友好** |
| 失效 | 改 Cookie | 删服务端 session 即可 | **难主动失效**(靠过期/黑名单) |
| 携带方式 | 浏览器自动带 | 靠 Cookie 带 id | 手动放 Authorization 头 |

### 4. 面试一句话

HTTP 无状态，要"记住你"有三种：Cookie（存客户端、浏览器自动带、易被 CSRF）、Session（存服务端、安全但有状态、分布式要共享）、JWT（无状态 token、服务端验签不存、分布式友好但难主动失效）。分布式微服务首选 JWT；要强可控(随时踢人下线)用 Session+Redis。

## 十、跨域与 Web 安全

### 1. 同源策略与 CORS

**同源策略**：浏览器安全基石——**JS 脚本默认不能跨源访问资源**（源 = 协议+域名+端口三者全同才算同源）。防的是恶意网站偷读你在别的网站的数据。

**CORS（跨源资源共享）**：跨源要访问，服务端得**显式同意**——通过响应头 `Access-Control-Allow-Origin` 等。**预检请求**：复杂跨源请求（如带自定义头、PUT/DELETE）浏览器先发个 **OPTIONS** 探问"你允许吗"，服务端回了允许头才发真请求。

### 2. CSRF（跨站请求伪造）

攻击者诱导用户（已登录 A 站）访问恶意页面，恶意页面**借浏览器自动带 Cookie 的特性**，向 A 站发请求——A 站收到带着用户有效 Cookie 的请求，以为是用户本人操作，就执行了（转账/改密码）。

**防御**：
- **CSRF Token**：表单里塞一个服务端发的随机 token，提交时校验——攻击者猜不到 token。
- **SameSite Cookie**：限制 Cookie 只在同站请求带，跨站不带。
- **关键操作二次确认**（验证码/密码）。

### 3. XSS（跨站脚本）

攻击者把恶意 JS 注入到页面（评论框存了一段 `<script>`，别人看到时执行），在受害者浏览器里跑，能偷 Cookie/会话、改页面。

**防御**：**输出转义**（把 `<` `>` 等转成实体，让脚本变文本不执行）、**CSP**（内容安全策略，限制能执行的脚本来源）、**HttpOnly Cookie**（JS 读不到 Cookie，偷不了）。

### 4. 面试一句话

同源策略挡跨源读、CORS 靠服务端响应头+OPTIONS 预检放行。CSRF 借浏览器自动带 Cookie 伪造请求，防靠 CSRF Token/SameSite；XSS 注入恶意脚本在受害者浏览器跑，防靠输出转义/CSP/HttpOnly Cookie。

## 十一、面试快问快答

1. **OSI 七层和 TCP/IP 四层区别？** → OSI 理论标准(会话/表示层基本没独立实现)，TCP/IP 四层才是互联网实际跑的，教学折中用五层。
2. **输入 URL 到渲染完整链路？** → DNS 解析→TCP 三次握手→(https)TLS 握手→发 HTTP 请求→服务端处理返回→浏览器解析渲染(建 DOM/CSSOM 树→布局→绘制)，子资源复用长连接。
3. **为什么三次握手不是两次？** → 第 3 次是客户端确认，给"反悔机会"，防旧 SYN/伪造 SYN 让服务端空建连；同时同步双方初始序列号。
4. **为什么四次挥手？** → 被动方收到 FIN 时可能还有数据没发完，ACK 和 FIN 拆两次发，所以 4 次。
5. **TIME_WAIT 等 2MSL 干嘛？** → 保最后 ACK 能补发(对端重发 FIN 时还在) + 让旧报文消亡防复用端口串包。
6. **TCP 怎么保证可靠？** → 序号确认+超时/快重传+滑动窗口流量控制+拥塞控制(慢开始/拥塞避免/快重传/快恢复)。
7. **流量控制 vs 拥塞控制？** → 前者管接收方扛不扛得住(靠 rwnd 对端告诉)，后者管中间网络扛不扛得住(靠自己探测)。
8. **HTTP keep-alive vs TCP keepalive？** → 前者应用层复用连接(HTTP/1.1 默认开)，后者传输层探测死连接(默认关、2 小时)，两码事。
9. **HTTP 各版本区别？** → 1.0 短连接；1.1 默认长连接+Host 头但文本+队头阻塞；2.0 二进制分帧+多路复用解决 HTTP 层队头阻塞但仍卡 TCP 层；3.0 换 QUIC(UDP)流间不阻塞+0-RTT。
10. **HTTPS 为什么用非对称+对称？** → 非对称解决密钥安全传递(慢)、对称解决数据快加密(但密钥难传)；握手用非对称协商出对称密钥、传数据用对称。
11. **怎么防中间人冒充？** → CA 签名的证书 + 系统/浏览器内置 CA 公钥验签的信任链。
12. **Cookie/Session/JWT 选型？** → 分布式微服务选 JWT(无状态)；要随时踢人下线选 Session+Redis；纯客户端小数据用 Cookie。
13. **CSRF 和 XSS 区别？** → CSRF 借浏览器自动带 Cookie 伪造请求(防 CSRF Token/SameSite)；XSS 注入恶意脚本在受害者浏览器跑(防输出转义/CSP/HttpOnly)。
