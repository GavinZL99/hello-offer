## 一、Spring 框架

### 0. 总览：Spring 在解决什么问题？

**要解决的问题**

传统 Java EE 开发存在几个痛点：

1. 对象之间强耦合：业务类自己 `new` 依赖，难以替换实现、难以做单元测试。
2. 横切关注点（事务、日志、权限、监控）散落在每个业务方法里，重复且容易遗漏。
3. 与数据库、消息中间件、缓存等组件整合时配置繁琐、模板代码多。
4. 不同环境下的配置切换、启动参数管理混乱。

**整体解决方案**

Spring 用一个 IoC 容器管理对象的创建和依赖关系，再用 AOP 把横切关注点抽到切面里，配合 Spring Boot 的自动装配把"整合中间件"这件事变成"引一个 starter"。

**核心要点**

- **IoC / DI**：把对象的创建和依赖装配交给容器。
- **AOP**：把横切关注点编织到目标方法前后。
- **声明式事务**：基于 AOP 实现，业务代码只标注 `@Transactional`。
- **Spring Boot 自动装配**：基于 `@Conditional` 系列条件注解 + SPI，按需创建 Bean。
- **Spring MVC**：一个前端控制器（`DispatcherServlet`）+ 多个策略组件（HandlerMapping / HandlerAdapter / ViewResolver）的经典门面。

**衍生机制**

- BeanPostProcessor / BeanFactoryPostProcessor 等扩展点 → 容器开放了 Bean 创建过程的钩子。
- 三级缓存 → 解决单例 Bean 的循环依赖。
- @Conditional 体系 → 实现"按需装配"。
- TransactionManager + 传播行为 → 灵活组合事务边界。
- ApplicationEvent / Listener → 容器内的发布订阅。

### 1. Spring IoC 容器是怎么工作的？

**要点**

- IoC（控制反转）把"谁创建谁、谁依赖谁"的控制权从代码交给容器。DI（依赖注入）是 IoC 的实现方式。
- 容器有两类：`BeanFactory` 是基础接口，懒加载；`ApplicationContext` 在启动时实例化所有非懒加载单例 Bean，并提供事件、国际化、资源访问等企业特性。
- 启动流程：定位资源（XML / 注解 / Java Config）→ 解析成 `BeanDefinition` → 注册到 `BeanDefinitionRegistry` → 实例化 → 属性注入 → 初始化。

**示例**

```java
// 接口
public interface OrderRepository { Order findById(Long id); }

// 实现
@Repository
public class JdbcOrderRepository implements OrderRepository {
    public Order findById(Long id) { /* ... */ }
}

// 业务类，依赖通过构造器注入（推荐：不可变 + 易测）
@Service
public class OrderService {
    private final OrderRepository repo;
    public OrderService(OrderRepository repo) { this.repo = repo; }
}
```

容器在启动时扫描到 `OrderService`，发现它的构造器参数是 `OrderRepository` 类型，到 `BeanDefinitionRegistry` 里找到对应实现 `JdbcOrderRepository`，实例化后注入。

**加分项**

- 区分 `BeanFactory`（容器本身）和 `FactoryBean`（一种特殊 Bean，由它的 `getObject()` 方法生产真正的 Bean，常用于复杂初始化场景，如 MyBatis 的 `MapperFactoryBean`）。
- 通过 `@Lazy` 可以让单例 Bean 也按需创建。

### 2. 描述一个 Bean 的完整生命周期

**要点**

1. **实例化**：通过反射调用构造器（`createBeanInstance`）。
2. **属性填充**：解析 `@Autowired` / `@Value`，注入依赖（`populateBean`）。
3. **Aware 回调**：`BeanNameAware` → `BeanFactoryAware` → `ApplicationContextAware`。
4. **`BeanPostProcessor.postProcessBeforeInitialization`**：典型如 `@PostConstruct` 注解的处理。
5. **初始化**：`InitializingBean.afterPropertiesSet()` → 自定义 `init-method`。
6. **`BeanPostProcessor.postProcessAfterInitialization`**：**AOP 代理就是在这一步生成的**。
7. **使用阶段**。
8. **销毁**：`@PreDestroy` → `DisposableBean.destroy()` → 自定义 `destroy-method`。

**示例**

```java
@Component
public class CacheManager implements InitializingBean, DisposableBean {
    @Value("${cache.size:1000}") private int size;
    private Map<String, Object> cache;

    @PostConstruct
    public void postConstruct() { System.out.println("step 4: @PostConstruct"); }

    @Override
    public void afterPropertiesSet() {
        cache = new ConcurrentHashMap<>(size);
        System.out.println("step 5: afterPropertiesSet");
    }

    @PreDestroy
    public void preDestroy() { System.out.println("step 8a: @PreDestroy"); }

    @Override
    public void destroy() { cache.clear(); }
}
```

**加分项**

- AOP 代理在 `postProcessAfterInitialization` 里生成，所以**初始化方法里调用自身被切面增强的方法是不会走代理的**——构造器、`@PostConstruct`、`afterPropertiesSet` 里的事务注解都会失效。

### 3. Spring 是如何解决循环依赖的？

**要点**

只有**单例 + 字段/Setter 注入**的循环依赖能被解决，**构造器循环依赖**和**多例（prototype）循环依赖**都解决不了。

**三级缓存**

| 缓存 | 类型 | 存放内容 |
|------|------|----------|
| 一级 `singletonObjects` | `Map<String, Object>` | 完整初始化好的成品 Bean |
| 二级 `earlySingletonObjects` | `Map<String, Object>` | 提前暴露的半成品 Bean（可能是代理对象） |
| 三级 `singletonFactories` | `Map<String, ObjectFactory<?>>` | 生产早期引用的工厂 |

**A 和 B 互相依赖时的过程**

1. 创建 A → 实例化后把 `ObjectFactory` 放入三级缓存。
2. 填充 A 的属性时发现需要 B，去创建 B。
3. 创建 B → 填充属性时发现需要 A → 一级、二级都没找到，从三级缓存里调用 `ObjectFactory.getObject()`，拿到 A 的早期引用，**同时把这个引用挪到二级缓存**。
4. B 完成创建并放入一级缓存。
5. A 拿到 B 后完成属性填充和初始化，放入一级缓存。

**为什么需要三级缓存？**

如果只有二级缓存：A 实例化后直接把原始对象暴露出去，B 注入的就是原始 A。但如果 A 需要被 AOP 代理（比如带 `@Transactional`），最终一级缓存里的 A 是代理对象，B 持有的却是原始对象，两者不一致。三级缓存通过 `ObjectFactory` 延迟决定要不要生成代理：调用 `getObject()` 时如果检测到需要代理，就提前生成代理对象返回，保证 B 拿到的也是代理对象。

**为什么解决不了构造器循环依赖？**

构造器执行时 Bean 还没创建完，无法提前暴露引用到三级缓存。

**Spring Boot 2.6+ 默认禁用循环依赖**

需要显式设置 `spring.main.allow-circular-references=true`，或者重构代码（这才是推荐做法）。

### 4. AOP 原理？JDK 动态代理和 CGLIB 的区别？

**要点**

AOP 通过运行时生成代理对象，在目标方法前后织入增强逻辑。

| 维度 | JDK 动态代理 | CGLIB |
|------|------------|-------|
| 实现方式 | 反射 + InvocationHandler | ASM 字节码生成子类 |
| 前提 | 目标必须实现接口 | 不需要接口 |
| 限制 | 只能代理接口方法 | 不能代理 final 类 / final 方法 |
| 性能 | 调用稍慢（反射） | 创建慢，调用接近原生 |
| Spring Boot 2.x+ | 默认 CGLIB | `spring.aop.proxy-target-class=true` |

**示例：自定义切面**

```java
@Aspect
@Component
public class PerformanceAspect {

    @Around("@annotation(monitored)")
    public Object around(ProceedingJoinPoint pjp, Monitored monitored) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            return pjp.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            log.info("method={} cost={}ms", pjp.getSignature().toShortString(), cost);
        }
    }
}
```

**AOP 失效的常见场景**

```java
@Service
public class UserService {

    public void createUser(User u) {
        save(u);          // 同类内部调用 → AOP 失效，事务不会生效
    }

    @Transactional
    public void save(User u) { /* ... */ }
}
```

原因是 `createUser` 通过 `this` 调用 `save`，绕过了代理对象。**解决方式**：

```java
// 方式 1：通过自注入拿到代理
@Autowired private UserService self;
public void createUser(User u) { self.save(u); }

// 方式 2：AopContext.currentProxy()，需要 @EnableAspectJAutoProxy(exposeProxy = true)
public void createUser(User u) { ((UserService) AopContext.currentProxy()).save(u); }

// 方式 3：拆分到不同的 Bean
```

### 5. `@Transactional` 失效的常见场景？

**完整失效场景清单**

| 场景 | 示例 | 原因 |
|------|------|------|
| 方法非 public | `private void save()` | 默认代理只切 public 方法 |
| 同类内部调用 | `this.save()` | 绕过代理 |
| 异常被吞 | `try { ... } catch (Exception e) {}` | 没抛出，不触发回滚 |
| 抛受检异常 | `throw new IOException()` | 默认只回滚 RuntimeException |
| 数据库不支持 | MyISAM | 引擎本身不支持事务 |
| 类没被 Spring 管理 | `new UserService()` | 没经过容器，不会被代理 |
| 错误的传播行为 | `NOT_SUPPORTED` | 显式不参与事务 |
| 多线程 | `executor.submit(() -> save())` | 事务绑定 ThreadLocal，新线程拿不到 |

**示例：异常被吞**

```java
@Transactional
public void transfer(Long from, Long to, BigDecimal amt) {
    accountDao.deduct(from, amt);
    try {
        accountDao.add(to, amt);   // 抛异常
    } catch (Exception e) {
        log.error("ignore", e);    // ❌ 异常被吞，事务不回滚！
    }
}
```

**示例：受检异常**

```java
@Transactional  // ❌ 不会回滚 IOException
public void importFile(String path) throws IOException {
    accountDao.deduct(1L, BigDecimal.TEN);
    Files.readAllBytes(Paths.get(path));  // IOException
}

// ✅ 显式声明 rollbackFor
@Transactional(rollbackFor = Exception.class)
public void importFile(String path) throws IOException { /* ... */ }
```

### 6. 事务传播行为有哪些？

**七种传播行为**

| 传播行为 | 含义 | 典型场景 |
|----------|------|----------|
| `REQUIRED`（默认） | 有则加入，无则新建 | 99% 业务方法 |
| `REQUIRES_NEW` | 挂起当前事务，新建独立事务 | 日志、审计、需要独立提交 |
| `NESTED` | 嵌套事务，基于 savepoint | 部分回滚（外滚内滚，内滚外不滚） |
| `SUPPORTS` | 有则加入，无则非事务 | 查询方法 |
| `NOT_SUPPORTED` | 挂起当前事务，非事务执行 | 长耗时操作 |
| `MANDATORY` | 必须有事务，否则抛异常 | 强约束 |
| `NEVER` | 必须无事务，否则抛异常 | 强约束 |

**示例：REQUIRES_NEW 用于审计日志**

```java
@Service
public class OrderService {
    @Autowired private AuditLogService auditLogService;

    @Transactional
    public void createOrder(Order o) {
        orderDao.insert(o);
        auditLogService.log("CREATE", o.getId());  // 审计日志独立事务
        if (o.getAmount().compareTo(LIMIT) > 0) {
            throw new BizException("amount too large");
        }
    }
}

@Service
public class AuditLogService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String action, Long bizId) {
        auditDao.insert(new AuditLog(action, bizId));
        // 即使外层 createOrder 回滚，这条日志也能保留
    }
}
```

### 7. Spring Boot 自动装配的原理？

**要点**

- 入口注解 `@SpringBootApplication` = `@SpringBootConfiguration` + `@EnableAutoConfiguration` + `@ComponentScan`。
- `@EnableAutoConfiguration` → `@Import(AutoConfigurationImportSelector.class)`。
- 候选配置类来源：
  - **Spring Boot 2.7 之前**：`META-INF/spring.factories`
  - **2.7+**：`META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`
- 通过 `@Conditional` 系列注解决定是否真正装配。

**示例：自定义 starter**

```java
// 1. 配置类
@Configuration
@ConditionalOnClass(RedisTemplate.class)              // 类路径有 RedisTemplate
@ConditionalOnProperty(name = "myapp.cache.enabled",  // 配置开关
                       havingValue = "true",
                       matchIfMissing = true)
@EnableConfigurationProperties(MyCacheProperties.class)
public class MyCacheAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean   // 用户没自定义时才创建
    public MyCache myCache(MyCacheProperties props, RedisTemplate<String, Object> redis) {
        return new MyCache(redis, props.getTtl());
    }
}

// 2. 配置属性
@ConfigurationProperties(prefix = "myapp.cache")
public class MyCacheProperties {
    private Duration ttl = Duration.ofMinutes(10);
    // getter / setter
}

// 3. resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
// com.example.mycache.MyCacheAutoConfiguration
```

**常用条件注解**

- `@ConditionalOnClass / @ConditionalOnMissingClass`：类路径
- `@ConditionalOnBean / @ConditionalOnMissingBean`：容器中是否存在
- `@ConditionalOnProperty`：配置开关
- `@ConditionalOnWebApplication`：Web 环境
- `@ConditionalOnExpression`：SpEL 表达式

### 8. Spring MVC 完整请求处理流程？

```
HTTP 请求
   ↓
DispatcherServlet（前端控制器）
   ↓
HandlerMapping       → 根据 URL 找到 HandlerMethod（Controller 方法）
   ↓
HandlerAdapter       → 适配不同类型 Handler（@Controller / 函数式 / WebFlux）
   ↓
HandlerInterceptor.preHandle
   ↓
参数解析（HandlerMethodArgumentResolver）
   ↓
Controller 方法执行
   ↓
返回值处理（HandlerMethodReturnValueHandler）
   ↓
HandlerInterceptor.postHandle
   ↓
ViewResolver（如果是 ModelAndView）→ View.render
   或 HttpMessageConverter（如果是 @ResponseBody）→ 直接序列化
   ↓
HandlerInterceptor.afterCompletion
   ↓
HTTP 响应
```

**Filter / Interceptor / AOP 的差别**

| 维度 | Filter | Interceptor | AOP |
|------|--------|-------------|-----|
| 所属规范 | Servlet | Spring MVC | Spring |
| 触发时机 | 进入 Servlet 前 | 进入 Controller 前 | Bean 方法调用 |
| 能拿到的上下文 | request/response | request + handler | 方法参数 / 注解 |
| 典型用途 | 编码、CORS、Token 解析 | 权限、日志、流量统计 | 事务、监控、缓存 |

## 二、Redis

### 0. 总览：Redis 在解决什么问题？

**要解决的问题**

1. 数据库读压力大，磁盘 IO 是瓶颈，需要一个**低延迟**（毫秒甚至亚毫秒）的缓存层。
2. 分布式系统需要**共享存储**：分布式锁、计数器、排行榜、会话等。
3. 部分场景需要**高吞吐写**和复杂数据结构（计数 / Set 运算 / 有序集合 / 位图）。

**整体解决方案**

一个**单线程内存数据库** + **多种数据结构** + **多种持久化方式** + **多种集群部署形态**。

**核心要点**

- **内存 + 单线程命令执行**：避免锁竞争和上下文切换；IO 多路复用承接高并发连接。
- **丰富数据结构**：每种结构对应不同业务场景。
- **持久化**：RDB 快照 + AOF 日志，二者结合（混合持久化）。
- **高可用**：主从、哨兵、Cluster 三种部署形态。
- **缓存策略**：过期 + 淘汰，应对内存有限。

**衍生机制**

- 哨兵的故障检测和自动主从切换。
- Cluster 的 16384 个 slot 分片 + Gossip 协议。
- Pipeline / Lua 脚本：减少 RTT、保证原子性。
- 发布订阅 / Stream：消息能力。
- ACL / 多 DB / Module 体系。

### 1. Redis 为什么这么快？

**要点**

1. **基于内存**：内存访问 ns 级别，远快于磁盘。
2. **单线程命令执行**：避免锁竞争和上下文切换。**注意**：6.0+ 引入多线程**只用于网络 IO**，命令执行依然是单线程。
3. **IO 多路复用**：基于 epoll，单线程可以处理大量连接。
4. **高效数据结构**：SDS、跳表、压缩列表 / listpack、quicklist、ziplist。
5. **避免不必要的开销**：自定义内存分配器（jemalloc）、避免不必要的拷贝。

**常被问的延伸**

- 为什么单线程还能这么高 QPS？瓶颈在网络而非 CPU，IO 多路复用 + 命令简单 → 单核能跑到 10w QPS。
- 6.0 多线程的意义？高并发场景网络 IO 占比大，多线程拆分 read / parse / write 提升吞吐。

### 2. 核心数据结构与典型应用场景

| 类型 | 底层结构（7.x） | 典型场景 |
|------|------|----------|
| String | SDS（动态字符串） | 计数器、分布式锁、缓存对象（JSON） |
| List | quicklist（listpack 节点构成的双向链表） | 简单消息队列、最近 N 条记录 |
| Hash | listpack / hashtable | 对象存储、购物车、配置项 |
| Set | intset / hashtable | 去重、共同好友、标签集合 |
| ZSet | listpack / (skiplist + hashtable) | 排行榜、延时队列、按 score 范围查询 |
| Bitmap | String | 用户签到、活跃用户统计 |
| HyperLogLog | 稀疏 / 密集编码 | UV 统计（误差 ~0.81%，常驻 12KB） |
| Geo | ZSet（GeoHash 编码 score） | 附近的人 |
| Stream | radix tree + listpack | 消息流、消费者组（5.0+） |

**示例：用 ZSet 做排行榜**

```bash
# 给用户加积分
ZINCRBY rank:202605 100 user:1001
ZINCRBY rank:202605 50  user:1002

# Top 10
ZREVRANGE rank:202605 0 9 WITHSCORES

# 我的排名（从 0 开始）
ZREVRANK rank:202605 user:1001
```

**示例：用 ZSet 做延时队列**

```bash
# 入队，score 是执行时间戳
ZADD delay_queue 1714915200 "task:1001"

# worker 轮询
ZRANGEBYSCORE delay_queue 0 <now> LIMIT 0 10
# 拿到后用 ZREM 删除（注意原子性，可用 Lua 脚本）
```

**示例：用 Bitmap 做签到**

```bash
# 用户 1001 在 5 月 6 日签到
SETBIT sign:1001:202605 5 1

# 5 月已签到天数
BITCOUNT sign:1001:202605

# 是否在 5 月 6 日签到
GETBIT sign:1001:202605 5
```

### 3. 缓存穿透 / 击穿 / 雪崩 的区别和解决方案

| 问题 | 描述 | 现象 |
|------|------|------|
| **穿透** | 查不存在的数据，缓存和 DB 都没有 | 大量请求直接打到 DB |
| **击穿** | 单个热点 key 失效瞬间，大量请求涌入 | DB 瞬间被打挂 |
| **雪崩** | 大量 key 同时失效 / Redis 宕机 | DB 雪崩、整个系统不可用 |

**穿透的解决：布隆过滤器 + 空值缓存**

```java
public User getUser(Long id) {
    // 1. 布隆过滤器先判断
    if (!bloomFilter.mightContain(id)) {
        return null;  // 一定不存在，直接返回
    }
    // 2. 查缓存
    User u = (User) redis.get("user:" + id);
    if (u != null) return u == NULL_PLACEHOLDER ? null : u;
    // 3. 查 DB
    u = userDao.findById(id);
    if (u == null) {
        // 短 TTL 缓存空值，避免攻击者用不存在的 key 反复打 DB
        redis.set("user:" + id, NULL_PLACEHOLDER, 60, SECONDS);
    } else {
        redis.set("user:" + id, u, 30, MINUTES);
    }
    return u;
}
```

**击穿的解决：互斥锁重建缓存**

```java
public Product getHotProduct(Long id) {
    String key = "product:" + id;
    Product p = (Product) redis.get(key);
    if (p != null) return p;

    String lockKey = "lock:" + key;
    // 只有一个线程能拿到锁去重建
    if (redis.setnx(lockKey, "1", 10, SECONDS)) {
        try {
            p = productDao.findById(id);
            redis.set(key, p, 30, MINUTES);
            return p;
        } finally {
            redis.del(lockKey);
        }
    } else {
        // 其他线程稍等再读缓存
        Thread.sleep(50);
        return getHotProduct(id);
    }
}
```

**雪崩的解决：过期时间打散 + 多级缓存**

```java
// 过期时间加随机抖动
int ttl = 30 * 60 + ThreadLocalRandom.current().nextInt(5 * 60);
redis.set(key, value, ttl, SECONDS);

// + 本地缓存（Caffeine）作为二级
// + Redis 集群高可用
// + 服务端限流降级
```

### 4. Redis 持久化：RDB 和 AOF

| 维度 | RDB | AOF |
|------|-----|-----|
| 形式 | 二进制快照 | 命令追加日志 |
| 触发 | `save`（阻塞）/ `bgsave`（fork 子进程） | 每条写命令追加 |
| 刷盘策略 | 周期性 | `always` / `everysec`（默认） / `no` |
| 文件大小 | 小，紧凑 | 大（重写后变小） |
| 恢复速度 | 快 | 慢（重放命令） |
| 数据安全 | 可能丢最后一次快照后的数据 | 最多丢 1 秒 |

**混合持久化（4.0+，默认 7.x 开启）**

### 5. 主从 / 哨兵 / Cluster 三种部署形态

| 模式 | 解决的问题 | 不能解决 |
|------|-----------|---------|
| 主从 | 读扩展、数据备份 | 自动故障转移、写扩展 |
| 哨兵 | 高可用、自动故障转移 | 写扩展、数据分片 |
| Cluster | 数据分片 + 高可用 | （写扩展和高可用都解决了） |

**主从复制流程**

1. 从节点 `REPLICAOF host port` 后向主请求同步。
2. 主执行 `bgsave` 生成 RDB 发给从。
3. 主把生成 RDB 期间的写命令缓存到**复制积压缓冲区**。
4. 从加载 RDB 后，主把缓冲区命令发过去。
5. 之后增量同步（**PSYNC**），通过 `runid + offset` 实现断线续传。

**哨兵故障转移**

1. 多个哨兵监控主节点。
2. 主观下线（`sdown`）：单个哨兵 ping 不通。
3. 客观下线（`odown`）：超过 `quorum` 个哨兵都判定下线。
4. 哨兵之间通过 Raft 选出 leader 执行故障转移。
5. leader 选一个从节点提升为主，其他从复制新主。
6. 客户端通过哨兵拿到新主地址。

**Cluster 关键点**

- 16384 个 slot，`HASH_SLOT = CRC16(key) mod 16384`。
- 节点之间通过 **Gossip 协议**通信。
- 客户端可缓存 slot 路由表，命中则直连，未命中收到 `MOVED` 重定向。
- **多 key 操作需要在同一 slot**：用 hash tag `{xxx}` 强制路由，例如 `{user:1001}.profile` 和 `{user:1001}.cart` 一定在同一节点。

### 6. Redis 分布式锁的完整方案

**最简单实现**

```bash
SET lock_key request_id NX PX 30000
```

释放时用 Lua 防止误删：

```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

**潜在问题与对应方案**

| 问题 | 后果 | 解决 |
|------|------|------|
| 业务执行超过锁超时 | 其他线程拿到锁，并发问题 | **看门狗续期**（Redisson 默认每 10s 续期到 30s） |
| 锁误删 | 释放别人的锁 | **锁 value 用唯一 ID**，删除前比对 |
| 主从异步复制丢锁 | 主宕机后从升为主，锁丢失 | **RedLock**：在 N 个独立实例上多数派加锁（有争议） |
| 锁不可重入 | 同一线程二次加锁失败 | **Redisson 用 Hash + 计数器** |

**Redisson 示例**

```java
RLock lock = redisson.getLock("order:1001");
try {
    // waitTime: 等待时间；leaseTime: 锁持有时间，-1 表示启用看门狗
    if (lock.tryLock(3, -1, TimeUnit.SECONDS)) {
        // 业务逻辑
    }
} finally {
    if (lock.isHeldByCurrentThread()) lock.unlock();
}
```

**Redisson 看门狗原理**

- `leaseTime = -1` 时启动看门狗，默认每 10 秒检查一次。
- 如果锁还被当前线程持有，自动续期到 30 秒。
- 客户端宕机 → 看门狗停止 → 30 秒后锁自动释放。

### 7. 内存淘汰策略 + 过期 key 删除策略

**8 种淘汰策略**

| 策略 | 范围 | 算法 |
|------|------|------|
| `noeviction`（默认） | - | 不淘汰，写报错 |
| `allkeys-lru` | 所有 key | LRU |
| `allkeys-lfu` | 所有 key | LFU（推荐用于热点） |
| `allkeys-random` | 所有 key | 随机 |
| `volatile-lru` | 设置了过期时间的 key | LRU |
| `volatile-lfu` | 设置了过期时间的 key | LFU |
| `volatile-random` | 设置了过期时间的 key | 随机 |
| `volatile-ttl` | 设置了过期时间的 key | 剩余 TTL 最小 |

**过期 key 的删除**

- **惰性删除**：访问时检查，过期则删并返回不存在。优点：节省 CPU；缺点：内存不释放。
- **定期删除**：每 100ms 随机抽样一批检查，过期比例高继续抽样。
- **内存满时**触发淘汰策略。

### 8. 大 key 和热 key 怎么处理

**大 key**

- **危害**：阻塞单线程（删除大 Hash 耗时 ms 级）、网络阻塞、阻塞集群迁移。
- **排查**：`redis-cli --bigkeys`、`MEMORY USAGE key`、SCAN + 业务分析。
- **删除**：4.0+ 用 `UNLINK`（异步删除）替代 `DEL`。
- **改造**：拆分（按字段 / 按时间分桶）、压缩（protobuf）、用 Hash 替代大 String。

**热 key**

- **危害**：单分片 CPU 打满、流量倾斜。
- **排查**：`redis-cli --hotkeys`（需 LFU 策略）、抓包、监控 QPS。
- **解决**：
  - **本地缓存**（Caffeine + Redis 多级缓存）。
  - **热 key 拆分**：`hot_key_0` ~ `hot_key_N`，写时全量更新，读时随机一个分片。
  - **读写分离**。

```java
// 热 key 分片读
String shardKey = hotKey + ":" + ThreadLocalRandom.current().nextInt(SHARDS);
Object value = redis.get(shardKey);
```

### 10. 缓存与数据库的一致性

**几种典型策略**

| 策略 | 流程 | 一致性问题 |
|------|------|-----------|
| Cache Aside（最常用） | 读：先缓存后 DB；写：更新 DB 后删除缓存 | 极端并发下旧值可能被回填，可用延时双删缓解 |
| Read/Write Through | 读写都走缓存层，缓存层负责同步 DB | 缓存层是单点 |
| Write Behind | 写入缓存立即返回，异步刷 DB | 数据可能丢 |

**为什么是"删除缓存"而不是"更新缓存"？**

1. 计算成本：写多读少时，每次写都更新缓存浪费。
2. 并发问题：A 写 db=1，B 写 db=2，但缓存更新顺序可能反过来变成 cache=1，旧值。

**延时双删**

```java
redis.del(key);
db.update(...);
// 等数据库主从复制完成
Thread.sleep(500);
redis.del(key);  // 兜底删除
```

更稳妥的方案：**订阅 binlog（Canal）异步删除缓存**。

## 三、消息队列

### 0. 总览：消息队列在解决什么问题？

**要解决的问题**

1. **系统耦合**：订单服务直接调用库存、积分、物流等服务，任何一个依赖挂了主链路就卡住。
2. **流量削峰**：秒杀场景瞬间百万 QPS，下游 DB 扛不住。
3. **异步化**：用户下单后不需要等积分、通知等流程。
4. **最终一致性**：跨服务的数据同步、业务状态同步。
5. **数据流处理**：日志、用户行为等数据需要广播给多个消费者。

**整体解决方案**

引入一个独立的中间件，生产者把消息扔进去，消费者按自己节奏消费，把同步调用变成异步消息驱动。

**核心要点**

- **三个端的可靠性**：生产端、Broker、消费端都要保证不丢。
- **三大语义**：at-most-once、at-least-once（最常见）、exactly-once（业务幂等保障）。
- **顺序性**：全局有序代价大，常用按业务 key 局部有序。
- **堆积处理**：临时扩容 + 长期优化消费速率。

**衍生机制**

- 事务消息（保证本地事务和发送消息的一致性）。
- 延时消息（替代传统定时任务）。
- 死信队列（处理多次失败的消息）。
- 消费者组（共享消费 / 广播消费）。

---

### 1. Kafka / RocketMQ / RabbitMQ 选型对比

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| 吞吐 | 极高（百万级/s） | 高（十万级/s） | 中（万级/s） |
| 延迟 | 几 ms | 几 ms | 微秒~ms |
| 消息可靠性 | 高 | 金融级 | 高 |
| 顺序消息 | 分区有序 | 全局/分区有序 | 单队列有序 |
| 事务消息 | 生产者事务 | 半消息 + 回查 | 不原生支持 |
| 延时消息 | 不原生支持 | 18 个等级 / 5.0 任意时间 | 插件 |
| 消费模式 | 拉模式 | 拉模式（伪推） | 推模式 |
| 社区 / 语言 | Scala/Java，活跃 | Java，国内活跃 | Erlang，老牌 |
| 典型场景 | 日志、流处理、大数据 | 业务解耦、金融交易 | 复杂路由、企业集成 |

**选型建议**

- 大数据 / 日志 → Kafka。
- 业务系统 + 国内云原生 → RocketMQ。
- 复杂路由（topic exchange / direct exchange / fanout）→ RabbitMQ。

### 2. 如何保证消息不丢失？

**三个环节都要管**

#### 生产端

```java
// Kafka 同步发送 + ack=all
Properties props = new Properties();
props.put("acks", "all");                    // 所有 ISR 副本都确认
props.put("retries", 3);                     // 重试
props.put("enable.idempotence", true);       // 幂等生产者，避免重试重复

ProducerRecord<String, String> record = new ProducerRecord<>("topic", key, value);
try {
    producer.send(record).get();             // 同步等待
} catch (Exception e) {
    // 失败处理：写本地表 + 定时补偿
}
```

```java
// RocketMQ 同步发送
SendResult result = producer.send(msg);
if (result.getSendStatus() != SEND_OK) {
    // 走兜底逻辑
}
```

#### Broker

- **持久化**：刷盘策略选**同步刷盘**（牺牲性能换安全），或异步刷盘 + 多副本。
- **多副本**：Kafka 的 ISR、RocketMQ 的主从同步复制（`SYNC_MASTER`）。
- **副本数**：至少 3，`min.insync.replicas=2`。

#### 消费端

```java
// Kafka 关闭自动提交，业务成功后手动提交
props.put("enable.auto.commit", false);
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));
    for (ConsumerRecord<String, String> r : records) {
        try {
            handle(r);
            consumer.commitSync(/* 单条提交 */);   // 业务成功才提交
        } catch (Exception e) {
            // 重试或入死信，不要直接 commit
        }
    }
}
```

### 3. 如何保证幂等？

**核心思想**：消息中间件最多保证 at-least-once，**幂等必须由消费方业务保证**。

**常见方案**

| 方案 | 适用场景 | 示例 |
|------|---------|------|
| 数据库唯一索引 | 插入操作 | `INSERT ... ON DUPLICATE KEY` 或 unique constraint |
| 状态机 | 状态流转 | 订单只能从 PAID 到 SHIPPED |
| Redis SETNX | 通用 | 用消息 ID 标记已处理 |
| 版本号 / CAS | 更新操作 | `UPDATE ... WHERE version = ?` |
| Token 表 | 重要业务 | 业务 ID 维护一张去重表 |

**示例：消息 ID 去重**

```java
@RocketMQMessageListener(topic = "order_paid", consumerGroup = "g_order")
public class OrderPaidConsumer implements RocketMQListener<MessageExt> {
    @Override
    public void onMessage(MessageExt msg) {
        String msgId = msg.getKeys();
        // SETNX 标记已处理，过期时间根据业务设
        Boolean first = redis.setIfAbsent("msg:" + msgId, "1", Duration.ofHours(24));
        if (!Boolean.TRUE.equals(first)) {
            log.info("duplicate message {}, skip", msgId);
            return;
        }
        try {
            handle(msg);
        } catch (Exception e) {
            redis.delete("msg:" + msgId);  // 失败回滚标记，允许重试
            throw e;
        }
    }
}
```

### 4. 如何保证消息顺序消费？

**全局有序（吞吐受限，慎用）**

- Kafka：单分区。
- RocketMQ：单 Queue。
- 消费者也只能单线程。

**局部有序（最常用）**

按业务 key 路由到同一分区 / 队列。

**Kafka 示例**

```java
// 按 orderId 路由到同一分区
ProducerRecord<String, String> record =
    new ProducerRecord<>("order_event", String.valueOf(orderId), payload);
// Kafka 默认按 key hash 选分区
```

**RocketMQ 示例**

```java
producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long orderId = (Long) arg;
        return mqs.get((int)(orderId % mqs.size()));
    }
}, orderId);

// 消费端用 MessageListenerOrderly 串行消费
consumer.registerMessageListener(new MessageListenerOrderly() {
    public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs, ConsumeOrderlyContext context) {
        // ...
        return ConsumeOrderlyStatus.SUCCESS;
    }
});
```

**消费端要注意：单分区内不能用线程池并发处理**，否则乱序。

### 5. 消息堆积怎么处理？

**先看堆积原因**

- 生产突增（活动 / 攻击）。
- 消费速度慢（DB 慢、外部依赖慢、消费逻辑重）。
- 消费者宕机。

**临时方案**

- **扩容消费者实例**：但 Kafka 同一消费者组内，一个分区只能被一个消费者消费，**消费者数 > 分区数没有意义**，所以可能要先增加分区数。
- **简化消费逻辑**：异步化、批量化、关闭非核心逻辑。
- **临时新建消费组**，把消息搬运到新 topic 再用大量消费者并行处理。
- **跳过非核心消息**：写到死信队列，活动结束再补偿。

**长期方案**

- 优化消费性能（DB 索引 / 缓存 / 批量写）。
- 合理设置分区数（一般是消费者数的 2-3 倍）。
- 限流 + 降级。
- 监控 Lag，提前告警。

### 6. RocketMQ 事务消息原理

**解决的问题**：本地事务和发消息这两件事如何保证一致——成功就都成功，失败都失败。

**完整流程**

```
1. Producer 发送 half message → Broker
2. Broker 收到后回 ack，但 half message 对消费者不可见
3. Producer 执行本地事务
4. Producer 根据本地事务结果发 commit / rollback
   - commit: half message 转为正常消息，对消费者可见
   - rollback: half message 删除
5. 如果 Broker 长时间没收到二次确认（默认 6 秒）
   → 主动回查 Producer 的事务状态
   → Producer 实现 TransactionListener.checkLocalTransaction
```

**示例**

```java
TransactionMQProducer producer = new TransactionMQProducer("g_order");
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            orderDao.insert((Order) arg);  // 本地事务
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 回查：根据消息里的业务 ID 查 DB
        Order o = orderDao.findById(Long.valueOf(msg.getKeys()));
        return o != null
            ? LocalTransactionState.COMMIT_MESSAGE
            : LocalTransactionState.ROLLBACK_MESSAGE;
    }
});

producer.sendMessageInTransaction(msg, order);
```

**关键点**：`checkLocalTransaction` 必须**幂等查询**且**有最终态**——否则 Broker 反复回查会累。

### 7. Kafka 高吞吐的原因

1. **顺序写磁盘 + PageCache**：顺序写性能接近内存。
2. **零拷贝（sendfile）**：消息从 PageCache 直接走网卡，省去用户态拷贝。
3. **批量发送 + 压缩**：`linger.ms` + `batch.size` + gzip / snappy / lz4 / zstd。
4. **分区并行**：每个分区是独立的提交日志，多分区可并行读写。
5. **异步刷盘 + 副本异步同步**：牺牲一点强一致换吞吐。

```java
// 生产端调优
props.put("linger.ms", 20);          // 等 20ms 凑批
props.put("batch.size", 32 * 1024);  // 32KB 批
props.put("compression.type", "lz4");
props.put("buffer.memory", 64 * 1024 * 1024);
```

### 8. 死信队列（DLQ）的处理

**触发条件**

- RocketMQ：消费失败重试 16 次后进入 `%DLQ%consumerGroup` 主题。
- Kafka：原生没有 DLQ，需业务方自己实现（消费失败的消息发往 dead-letter topic）。

**处理流程**

1. 监控 DLQ 堆积告警。
2. 拉取 DLQ 消息进行分析（业务异常 / 脏数据 / 临时故障）。
3. 修复后**重新投递回原 topic**。
4. 严重的脏数据归档到离线表，人工处理。

```java
// 自实现 DLQ：消费失败 N 次后转发到 dead topic
@KafkaListener(topics = "order")
public void consume(ConsumerRecord<String, String> record) {
    Integer retry = parseRetry(record.headers());
    try {
        handle(record);
    } catch (Exception e) {
        if (retry >= MAX_RETRY) {
            kafkaTemplate.send("order_dlq", record.value());
        } else {
            // 重新投递回原 topic，header 中 retry+1
            kafkaTemplate.send("order", incrementRetry(record));
        }
    }
}
```

## 四、Agent 开发

### 0. 总览：Agent 在解决什么问题？

**要解决的问题**

LLM 本身只能做"输入文本 → 输出文本"，但真实业务场景需要：

1. **拿到实时信息**：LLM 训练数据有截止时间，不知道今天的天气、最新的订单状态。
2. **执行动作**：发邮件、查数据库、调用业务 API、读写文件。
3. **处理超长上下文**：单次对话窗口有限（几十 k ~ 1M token），无法塞下整个公司知识库。
4. **拆解复杂任务**：一个需求往往需要多步骤推理 + 多次决策。
5. **降低幻觉**：纯参数化知识容易胡说，需要外部事实约束。

**整体解决方案**

把 LLM 当作"决策大脑"，外挂**工具（Tools）**让它能行动，外挂**记忆（Memory / RAG）**让它能查资料，加上一个**循环调度器**让它能反思和迭代。

```
┌──────────────────────────────────────────┐
│  User Input                              │
│       ↓                                  │
│  ┌──────────────┐  ┌──────────────────┐ │
│  │     LLM      │←→│  Memory / RAG    │ │
│  │  (决策大脑)  │  │  (短期+长期记忆) │ │
│  └──────┬───────┘  └──────────────────┘ │
│         ↓ tool_call                      │
│  ┌──────────────┐                        │
│  │   Tools      │ ← 业务 API / 检索 / … │
│  └──────┬───────┘                        │
│         ↓ tool_result                    │
│         └→ 回到 LLM 继续推理（循环）     │
└──────────────────────────────────────────┘
```

**核心要点**

- **工具调用（Function Calling）**：让模型能行动。
- **ReAct / Plan-and-Execute**：让模型能多步推理。
- **Memory**：让模型能记住和回忆。
- **RAG**：让模型能利用外部知识。
- **Prompt 工程**：让模型按预期方式工作。

**衍生机制**

- Multi-Agent 协作（Supervisor / Pipeline / Debate）。
- Agent 评测（任务成功率 / 工具准确率 / Trajectory 评估）。
- 可观测性（Trace、Token 监控、LangSmith / Langfuse）。
- 安全（Prompt 注入防御、工具权限隔离）。

---

### 1. Agent / Chat / RAG 的区别

| 维度 | Chat | RAG | Agent |
|------|------|-----|-------|
| 模型行为 | 单纯生成 | 检索后生成 | 决策 + 调用工具 + 反思循环 |
| 复杂度 | 低 | 中 | 高 |
| 典型场景 | 对话助手 | 知识问答 | 任务自动化、数字员工 |
| 可控性 | 高 | 高 | 较低（自主决策） |
| 成本 | 低 | 中 | 高（多轮调用） |

**简单理解**

- Chat：你问它答。
- RAG：你问之前先帮你查资料再答。
- Agent：你给它一个目标，它自己拆解、查资料、调用工具、来回迭代直到完成。

---

### 2. Agent 的核心组件

| 组件 | 职责 | 实现示例 |
|------|------|---------|
| **LLM** | 决策大脑 | Claude、GPT、通义千问 |
| **System Prompt** | 角色 / 目标 / 工具规范 / 输出格式 | 多层结构化 Prompt |
| **Tools** | 模型可调用的能力 | API 调用 / DB 查询 / 代码执行 |
| **Memory** | 短期 + 长期记忆 | 上下文窗口 + 向量库 |
| **Planner** | 任务拆解 | ReAct 隐式 / Plan-and-Execute 显式 |
| **Executor** | 调度循环 | 解析 tool_call、执行、回灌结果 |
| **Observability** | Trace / 日志 | LangSmith / Langfuse |

### 3. Function Calling（工具调用）是怎么工作的？

**完整流程**

```
1. 应用声明工具列表（JSON Schema）→ 随请求传给模型
2. 模型分析用户输入，决定调用哪个工具，输出结构化 tool_call
3. 应用解析 tool_call，执行真正的函数 / API
4. 把 tool_result 回传给模型
5. 模型基于结果继续生成 / 调用下一个工具，直到任务完成
```

**示例：天气查询 Agent**

```python
# 1. 工具定义
tools = [{
    "name": "get_weather",
    "description": "查询某城市当天天气，包括温度、天气状况、降水概率",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "城市名（中文或拼音）"},
            "date": {"type": "string", "description": "YYYY-MM-DD 格式日期，默认今天"}
        },
        "required": ["city"]
    }
}]

# 2. 第一次调用
response = client.messages.create(
    model="claude-opus-4-7",
    tools=tools,
    messages=[{"role": "user", "content": "杭州今天适合出门吗？"}]
)

# 3. 模型决定调用 get_weather
# response.content 包含 tool_use block: {name: "get_weather", input: {city: "杭州"}}
tool_call = response.content[1]
weather = get_weather(**tool_call.input)  # 应用执行

# 4. 把结果回传
response2 = client.messages.create(
    model="claude-opus-4-7",
    tools=tools,
    messages=[
        {"role": "user", "content": "杭州今天适合出门吗？"},
        {"role": "assistant", "content": response.content},
        {"role": "user", "content": [{
            "type": "tool_result",
            "tool_use_id": tool_call.id,
            "content": str(weather)
        }]}
    ]
)

# 5. 模型综合天气信息生成最终答案
print(response2.content[0].text)
# 输出："杭州今天 22°C，多云转晴，降水概率 10%，比较适合出门……"
```

**关键工程实践**

- **工具描述要清晰**：name、description、参数语义、典型场景，模型完全靠描述决策。
- **参数 Schema 要严格**：用 JSON Schema 限制类型、枚举、必填，减少模型乱传。
- **错误返回结构化**：`{"error": "city_not_found", "msg": "..."}`，方便模型自己纠错。
- **限制最大循环步数**：避免死循环。
- **工具结果不要太长**：超长结果先做摘要 / 截断再回灌。

### 4. ReAct 范式

**思想**：交替输出 `Thought → Action → Observation`，把推理和行动结合。

**示例：用 ReAct 完成"查公司今年新员工的人均培训时长"**

```
Question: 公司今年新员工人均培训时长是多少？

Thought: 我需要先查今年的新员工列表，再查每个人的培训记录。
Action: query_employees(filter={"hire_year": 2026})
Observation: 找到 156 名员工，员工 ID 列表 [1001, 1002, ..., 1156]

Thought: 现在查这些员工的培训记录，汇总总时长。
Action: query_training(employee_ids=[1001,...,1156], year=2026)
Observation: 总培训时长 4836 小时，覆盖 142 名员工

Thought: 计算人均（注意分母用所有新员工数还是有培训记录的数）。
Action: 用户题目说"人均"，按所有新员工算。
Final Answer: 人均培训时长 = 4836 / 156 ≈ 31 小时
```

**ReAct vs CoT**

- CoT 只有 Thought（推理链），没有 Action。
- ReAct = CoT + 工具调用，是 Agent 最经典的范式。
- 现代 Function Calling 就是 ReAct 的工程化实现。

### 5. Memory（记忆）怎么设计？

**分层记忆**

| 层 | 时效 | 实现 |
|----|------|------|
| **工作记忆**（短期） | 当前对话 | 直接放上下文 |
| **会话记忆**（中期） | 当前会话 | 摘要 + 最近 N 轮 |
| **长期记忆** | 跨会话 | 向量库 + 结构化存储 |

**短期记忆压缩策略**

- 滑动窗口：只保留最近 N 轮。
- 摘要压缩：每超过 K 轮就让 LLM 摘要前面的。
- 重要信息提取：用一个小模型抽取关键事实存到长期记忆。

**长期记忆的写入**

```python
# 每轮对话结束抽取重要信息
def extract_memory(user_msg, assistant_msg):
    prompt = f"""请从下面的对话中抽取需要长期记忆的事实，按 JSON 数组返回：
对话：
User: {user_msg}
Assistant: {assistant_msg}
返回格式：[{{"fact": "...", "category": "preference|profile|event"}}]"""
    return llm(prompt)

# 入向量库
for fact in extract_memory(...):
    embedding = embed(fact["fact"])
    vector_db.upsert(id=uuid(), vector=embedding, metadata=fact)
```

**长期记忆的召回**

```python
# 对话开始时召回相关记忆
def recall(query, top_k=5):
    embedding = embed(query)
    results = vector_db.query(vector=embedding, top_k=top_k)
    return [r.metadata["fact"] for r in results if r.score > 0.7]

# 拼到 system prompt
memories = recall(user_input)
system_prompt = f"""你是 XX 助理。
关于用户的已知信息：
{chr(10).join(f'- {m}' for m in memories)}
"""
```

### 6. RAG 完整流程 + 优化点

**基础流程**

```
[索引阶段]
文档 → 加载 → 切分 (chunking) → Embedding → 入向量库

[检索阶段]
用户 query → Embedding → 向量库查询 (top-k) → Rerank → 拼 Prompt → LLM 生成
```

**关键优化点**

#### Chunking 策略

- **固定大小**：简单但可能切断语义。
- **语义切分**：按段落、句号切。
- **递归切分**（LangChain RecursiveCharacterTextSplitter）：按层次分隔符递归。
- **父子分片**：小块用于检索（精准），大块用于上下文（信息完整）。
- **结构化切分**：Markdown 按标题层级切。

#### Embedding 选型

中文场景常用：
- `bge-large-zh-v1.5`（智源）
- `m3e-large`（Moka）
- `text-embedding-v3`（通义）
- `text-embedding-3-large`（OpenAI）

#### 检索优化

```python
# 1. 混合检索：向量 + BM25
vector_results = vector_db.search(query_embedding, top_k=20)
bm25_results = bm25.search(query, top_k=20)
merged = rrf_merge(vector_results, bm25_results)  # Reciprocal Rank Fusion

# 2. Rerank：用 cross-encoder 精排
reranker = CrossEncoder("BAAI/bge-reranker-large")
scores = reranker.predict([(query, doc) for doc in merged])
top_5 = sorted(zip(merged, scores), key=lambda x: -x[1])[:5]
```

#### Query 改写

- **HyDE**（Hypothetical Document Embedding）：让 LLM 生成假设答案，用答案 embedding 去检索。
- **Multi-query**：让 LLM 把一个 query 扩展成 N 个相关 query 分别检索后合并。
- **Step-back**：先抽象出一个上层问题，再检索。

#### 评测指标

- **Recall@K**：召回率。
- **MRR**（Mean Reciprocal Rank）：平均倒数排名。
- **NDCG**：归一化折损累计增益。
- **端到端**：忠实度（Faithfulness）、相关性（Relevance）→ 用 RAGAS 框架。

### 7. Multi-Agent 系统的常见模式

| 模式 | 结构 | 典型场景 |
|------|------|---------|
| **Supervisor / Router** | 主 Agent 调度多个子 Agent | LangGraph、CrewAI |
| **Pipeline** | 多 Agent 串行 | 规划 → 编码 → 测试 → 评审 |
| **Debate / Reflection** | 多 Agent 互评 | 提升输出质量 |
| **Group Chat** | 多 Agent 共享对话历史 | AutoGen 经典模式 |

**示例：Supervisor 模式**

```
                ┌──────────────┐
                │  Supervisor  │
                └──────┬───────┘
        ┌──────────────┼──────────────┐
        ↓              ↓              ↓
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ Researcher│  │  Coder   │   │ Reviewer │
  └──────────┘   └──────────┘   └──────────┘
```

```python
# 用 LangGraph 实现
from langgraph.graph import StateGraph

graph = StateGraph(AgentState)
graph.add_node("supervisor", supervisor_node)
graph.add_node("researcher", researcher_agent)
graph.add_node("coder", coder_agent)
graph.add_node("reviewer", reviewer_agent)

# Supervisor 决定下一个走哪个节点
graph.add_conditional_edges(
    "supervisor",
    lambda state: state["next_agent"],
    {"research": "researcher", "code": "coder", "review": "reviewer", "end": END}
)

# 每个子 Agent 完成后回到 Supervisor
graph.add_edge("researcher", "supervisor")
graph.add_edge("coder", "supervisor")
graph.add_edge("reviewer", "supervisor")
```

**关键挑战**

- **角色边界**：避免职责重叠和"踢皮球"。
- **状态共享**：用全局 State 还是消息传递？
- **终止条件**：避免无限对话（设最大轮次 + 显式终止信号）。
- **Token 成本**：多 Agent 容易爆 Token。

### 8. Prompt 工程在 Agent 里的实践

**结构化 System Prompt**

```
# 角色
你是公司差旅风险审核助手，帮助内控团队识别违规用车。

# 目标
分析输入的用车数据，判断是否存在以下违规：
- 跨城用车
- 夜间用车（22:00-06:00）
- 节假日用车
- 高频用车（单日 > 5 次）

# 工具使用规范
- 涉及历史数据时调用 query_history(employee_id, date_range)
- 涉及节假日判断时调用 is_holiday(date, country)
- 不要凭记忆判断节假日

# 输出格式（严格 JSON）
{
  "is_violation": true | false,
  "violation_types": ["cross_city", "night", ...],
  "confidence": 0.0-1.0,
  "reason": "简短说明"
}

# 边界
- 如果数据缺失关键字段，返回 {"error": "missing_field", "field": "..."}
- 不要做主观评判，只基于规则
```

**Few-shot 示例**

```
# 示例
[正例]
输入：员工 1001，2026-05-06 02:30，北京 → 上海
输出：{"is_violation": true, "violation_types": ["night", "cross_city"], "confidence": 0.95, "reason": "凌晨 2:30 跨城用车"}

[反例]
输入：员工 1001，2026-05-06 14:30，北京公司 → 北京机场
输出：{"is_violation": false, "violation_types": [], "confidence": 0.9, "reason": "工作日白天市内出行"}
```

**输出不规范的二次修复**

```python
def parse_output(raw_text, schema):
    try:
        return json.loads(raw_text)
    except json.JSONDecodeError as e:
        # 自动构造修复 Prompt
        repair_prompt = f"""你刚才的输出不符合 JSON Schema：
你的输出：{raw_text}
错误：{e}
期望 Schema：{json.dumps(schema)}
请输出严格符合 Schema 的 JSON，不要任何解释。"""
        fixed = llm(repair_prompt)
        return json.loads(fixed)
```

### 9. 怎么评测一个 Agent 的效果？

**离线评测**

| 指标 | 含义 |
|------|------|
| 任务成功率 | 端到端是否完成目标 |
| 工具调用准确率 | 是否调用了正确的工具 + 正确的参数 |
| 步骤数 | 平均完成所需轮次（越少越好） |
| Token 成本 | 平均 Token 消耗 |
| Trajectory 评估 | 拆 trajectory 评估每一步合理性 |

**在线评测**

- 用户满意度（点踩点赞）。
- 业务转化率。
- A/B 测试。
- LLM-as-Judge：用强模型对结果打分（注意 bias 校准）。

**典型评测集**

- **AgentBench**：通用 Agent 评测。
- **ToolBench**：工具调用评测。
- **SWE-bench**：代码 Agent。
- **GAIA**：通用助手评测。
- **τ-bench**：客服 Agent。

### 10. Agent 落地常见的工程问题

| 问题 | 解决方案 |
|------|---------|
| **幻觉** | RAG、工具约束、结构化输出 + Schema 校验、二次修复 Prompt |
| **延时** | 流式输出、并行工具调用、Prompt Cache、模型分流（小问题用小模型） |
| **成本** | Prompt Cache、Embedding Cache、上下文压缩、小模型分流 |
| **可观测性** | 完整 trace（输入/输出/工具/Token），LangSmith / Langfuse / Phoenix |
| **稳定性** | 重试 + 超时 + fallback 模型 + 最大步数限制 |
| **安全** | Prompt 注入防御（输入过滤、role 隔离）、工具权限分级、敏感操作二次确认 |
| **效果回归** | 评测集 + CI 自动跑分，每次 Prompt 改动都过一遍 |
