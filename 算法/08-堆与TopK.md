# 算法刷题笔记 · 第八类：堆 / Top K

> 结构同前：**直觉类比 → 堆是什么 → 核心套路(维护K大小的堆) → 实战 → 易错点 → 自检清单**。
> 「第 K 大」「前 K 高频」「合并 K 路」这类题，堆（优先队列）是标准解，几乎没有替代。

---

## 0. 一句话本质

> **堆 = 一个能 O(log n) 拿到「当前最大值或最小值」的数据结构。** Top K 类问题靠它把 O(n log n) 排序降到 O(n log k)。

适用信号：
- 「第 K 大 / 第 K 小」「前 K 个高频 / 最接近」
- 「合并 K 个有序链表 / 数组」
- 「数据流中位数 / 实时第 K 大」（边来边查）
- 「每次取最值、取完还要继续取下一个最值」

---

## 1. 直觉类比：一个永远把「最值」浮在顶上的桶

把堆想成一个**特制的桶**：你往里随便扔数，它内部会自动调整，**保证桶口永远是最小（或最大）的那个**。

- **小顶堆**：桶口是**最小值**。`peek()` 看最小、`poll()` 取走最小。
- **大顶堆**：桶口是**最大值**。

你不需要关心桶内部怎么排的，只享受一个能力：**O(1) 看最值、O(log n) 取走最值或放入新值**。

> Java 里就是 `PriorityQueue`，默认**小顶堆**（堆顶最小）。

```java
PriorityQueue<Integer> minHeap = new PriorityQueue<>();                    // 小顶堆(默认)
PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a);     // 大顶堆(自定义比较)
minHeap.offer(x);      // 放入,O(log n)
minHeap.peek();        // 看堆顶(最小),O(1)
minHeap.poll();        // 取走堆顶,O(log n)
```

### 常用操作:peek / poll / offer

| 操作 | 干什么 | 复杂度 | 关键区别 |
|------|--------|--------|---------|
| `peek()` | **看**堆顶,不取出 | O(1) | 只读,堆不变 |
| `poll()` | **取走**堆顶(删除) | O(log n) | 取走后要重新调整堆维持性质 |
| `offer(x)` | 放入 x | O(log n) | 放入后要调整 |

> `peek` vs `poll`:peek 是「**看一眼**」(堆顶还在),poll 是「**拿走**」(堆顶没了,剩下元素重新调整)。
> 第 K 大题最后用 `peek()` —— 堆顶就是答案,看一眼即可,不用拿走。

### 怎么区分小顶堆 / 大顶堆

看比较器:`(a, b) -> a - b`(默认)→ **小顶堆**;`(a, b) -> b - a` → **大顶堆**。
> ⚠️ `b - a` 在值域大时会**整数溢出**(两个相差大的 int 相减溢出),稳妥写 `(a, b) -> Integer.compare(b, a)`。

### ⭐ 重要:堆内部并不「整体有序」

新手常见误解:以为堆是一个排好序的结构。**不是。**

堆**只保证一件事:堆顶是最值**。其余元素是半乱序的。底层是棵二叉树(用数组存),靠**堆性质**维持:

> **小顶堆:每个节点都 ≤ 它的两个孩子**(大顶堆则 ≥)。于是根(堆顶)≤ 所有节点 → 堆顶最小。

```
小顶堆(存 3,1,2,5,4):
        1            ← 堆顶最小
       / \
      3   2          ← 父≤子,但3和2谁大没规定
     / \
    5   4            ← 同层之间无序
```

同层之间(3和2、5和4)**没有顺序**,整体也不是排序的——但「父≤子」就足够保证**堆顶恒为最值**。

**offer / poll 怎么维持这个性质**(每次只 O(log n)):
- **offer(加)**:加到末尾 → **上浮**:和父比,比父小就交换,一路浮上去。
- **poll(取堆顶)**:把末尾元素搬到堆顶 → **下沉**:和较小的孩子交换,一路沉下去。

每次只动「根到叶的一条路径」,不是整体排序。所以**堆 ≠ 有序数组,堆 = 能 O(1) 给最值、O(log n) 增删的桶**。

## 2. 核心套路：求第 K 大 → 维护一个「大小为 K 的小顶堆」

这是 Top K 最反直觉、也最重要的一招。

> **求第 K 大，用「大小为 K 的小顶堆」（不是大顶堆！）。**

**为什么是小顶堆、且只留 K 个？**

想象你要保留「最大的 K 个数」。维护一个**只装 K 个数的小顶堆**：
- 堆里始终是「目前见过的最大的 K 个」。
- 堆顶是这 K 个里**最小的**——也就是「**第 K 大**」的候选。
- 新来一个数，比堆顶大 → 它够资格进 Top K，把堆顶（最弱的）挤掉；比堆顶小 → 它连第 K 大都不如，直接丢。

遍历完，**堆顶就是第 K 大**。

### LC 215 数组中的第 K 个最大元素

```java
public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>();   // 小顶堆
    for (int num : nums) {
        heap.offer(num);
        if (heap.size() > k) heap.poll();   // 超过 K 个,弹掉最小的(堆顶)
    }
    return heap.peek();                      // 堆里剩最大的 K 个,堆顶=第K大
}
```

**复杂度** `O(n log k)`——比「全排序 O(n log n)」更优（k 通常远小于 n）。这就是堆解 Top K 的价值。

> 🧠 记反了会怎样？用「大小为 K 的**大顶堆**」求第 K 大是**错的**——大顶堆顶是最大值，你没法只留最大的 K 个。
> 口诀：**求第 K 大用小顶堆，求第 K 小用大顶堆**（堆顶放的是「最容易被淘汰的那个」）。

## 3. 实战

### 3.1 LC 347 前 K 个高频元素

> 返回出现频率前 K 高的元素。

**两步**：① 哈希表统计频率；② 用「大小为 K 的小顶堆」按**频率**留下 Top K。

```java
public int[] topKFrequent(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int n : nums) freq.put(n, freq.getOrDefault(n, 0) + 1);

    // 小顶堆,按频率比较(频率小的在堆顶,先被淘汰)
    PriorityQueue<Integer> heap = new PriorityQueue<>((a, b) -> freq.get(a) - freq.get(b));
    for (int key : freq.keySet()) {
        heap.offer(key);
        if (heap.size() > k) heap.poll();   // 留下频率最高的 K 个
    }

    int[] res = new int[k];
    for (int i = 0; i < k; i++) res[i] = heap.poll();
    return res;
}
```

> 同样是「大小为 K 的小顶堆」套路，只是比较器从「比数值」换成「比频率」。

### 3.2 LC 23 合并 K 个升序链表

> 把 K 个有序链表合并成一个有序链表。

**思路**：把 K 个链表的**当前头节点**全放进小顶堆，每次取出最小的接到结果后面，再把它的 `next` 放进堆。堆始终维护「K 路里各自最靠前的候选」。

```java
public ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> heap = new PriorityQueue<>((a, b) -> a.val - b.val);
    for (ListNode node : lists) if (node != null) heap.offer(node);

    ListNode dummy = new ListNode(0), tail = dummy;
    while (!heap.isEmpty()) {
        ListNode min = heap.poll();        // 取 K 路里最小的
        tail.next = min;  tail = min;
        if (min.next != null) heap.offer(min.next);   // 补充它的下一个
    }
    return dummy.next;
}
```

> 复杂度 `O(N log k)`，N 是总节点数。比「两两合并」更优雅。`dummy` 哑结点是链表题省边界判断的常用技巧。

### 3.3 LC 295 数据流的中位数（对顶堆，进阶）

> 边接收数字边随时求中位数。**两个堆**：大顶堆存较小的一半，小顶堆存较大的一半，让两堆大小差 ≤ 1。中位数 = 堆顶（或两个堆顶平均）。

> 这是「对顶堆」技巧，思想是**用两个堆把数据从中间劈开**，各自顶住中位数附近。先掌握 3.1/3.2，这个有余力再看。

## 4. 易错点

- [ ] **求第 K 大用小顶堆、第 K 小用大顶堆**——最容易记反。堆顶放「最该被淘汰的」。
- [ ] **堆大小要 `> k` 才 poll**，不是 `>= k`（留够 K 个）。
- [ ] **大顶堆比较器**：`(a, b) -> b - a`；但比较器做减法**可能溢出**，值域大时用 `Integer.compare(b, a)` 或 `Long`。
- [ ] `peek()` / `poll()` 前先判 `isEmpty()`，空堆 poll 返回 null。
- [ ] 别用堆做「全局排序」——那不如直接 `sort`；堆的优势是**只关心 Top K / 流式取最值**。

---

## 5. 堆 vs 排序 vs 快速选择

| 方法 | 求第 K 大复杂度 | 适用 |
|------|---------------|------|
| 全排序后取 | `O(n log n)` | 简单但浪费，K 小时不划算 |
| **堆（大小 K）** | `O(n log k)` | **通用首选**，尤其数据流/在线 |
| 快速选择 (quickselect) | 平均 `O(n)`，最坏 `O(n²)` | 一次性求第 K 大的最快解，但不适合流式 |

> 面试说「第 K 大」，**堆是最稳的答案**（好写、不易错）；想秀复杂度可以提快速选择（基于快排 partition 的思想）。

---

## 6. 一句话总结

> 堆 = **永远把最值浮在顶上的桶**，O(log n) 取最值。
> Top K 的灵魂套路：**求第 K 大维护「大小为 K 的小顶堆」**，堆顶即答案，复杂度 O(n log k)。
> 变体都套这个壳——前 K 高频换比较器、合并 K 路放各路头节点、中位数用对顶堆。
